# Fully Typed API Client

## Introduction & Motivation

Most frontend codebases suffer from poorly typed API layers. You make a `fetch()` call, manually cast the response to an `Interface`, and hope the backend doesn't change. When using raw `fetch` or even basic `axios`, there is no guarantee that your request parameters match the endpoint URL, or that your response actually matches your local TypeScript types.

This project implements a **TypeScript-first API Client Library**. By defining your endpoints centrally using **Zod**, the client guarantees absolute type-safety from end to end:
1. It parses path parameters (`/users/:id` requires `{ id: number }`).
2. It strictly typed query parameters and body payloads via Zod schemas.
3. It validates responses at runtime to catch backend contract breakages early.

## Core Concepts & Architecture

### 1. Endpoint Registry (`registry.ts`)
Instead of magic strings, you define every endpoint in a central object using the `defineEndpoint` factory. This configures the path, HTTP method, caching behavior, retries, and Zod schemas for validation.

### 2. Inferential Client (`client.ts`)
The `ApiClient` wrapper takes an endpoint definition from the registry and deeply infers what `params`, `query`, and `body` you must pass. If the endpoint does not require them, passing them throws a TS error.

### 3. The `Result` Union Pattern
Exceptions (`try/catch`) are difficult to type in TypeScript. Our client catches all network and validation errors and returns a strictly typed discriminated union: `{ ok: true, data: T } | { ok: false, error: Error }`.

### 4. Advanced Networking Features
- **Rate Limiting**: Configured per endpoint using a Token Bucket algorithm to prevent spamming the backend.
- **Caching**: Deterministic caching keyed by Method + URL + Params.
- **Retries**: Configurable exponential backoff for flaky endpoints.

## Usage Examples

Below are the 4 core use cases showing exactly how to implement them, complete with live interactive previews using MDC components.

### 1. Basic GET Request & Path Parameters

If you define a path as `/users/:id`, the client will enforce that `{ params: { id: ... } }` is passed.

```ts
import { ApiRegistry } from '~/utils/api-client/registry';

// Registry definition: path: '/users/:id'

const result = await client.request(ApiRegistry.getUser, {
  params: { id: 1 } 
});

if (result.ok) {
  console.log(result.data.name); // Perfectly typed!
}
```

### 2. POST Request & Body Validation

Zod schemas dictate the exact shape of your request body. If you miss a field, TypeScript will flag it immediately.

```ts
// Registry definition: body requires name (min 2) and email

const result = await client.request(ApiRegistry.createUser, {
  params: undefined,
  body: {
    name: 'Jane Doe',
    email: 'jane@example.com'
  }
});
```

::api-post-demo
::

### 3. Authentication & Interceptors

The `ApiClient` accepts global interceptors. You can mutate headers, append Bearer tokens, or transform responses globally.

```ts
const client = new ApiClient({
  baseUrl: 'https://api.example.com',
  onRequest: (req) => {
    req.headers = { ...req.headers, Authorization: `Bearer ${useToken()}` };
    return req;
  }
});

// Registry definition: query requires 'q'
const result = await client.request(ApiRegistry.searchPosts, {
  params: undefined,
  query: { q: 'vue3' }
});
```

::api-auth-demo
::

### 4. Retries & Error Handling

Endpoints can be configured with exponential backoff retries. If an endpoint is configured with `{ attempts: 3, delayMs: 100 }`, the client will automatically retry on failure, doubling the delay each time before finally yielding an `{ ok: false }` Result object.

```ts
// Registry definition: retry: { attempts: 3, delayMs: 100 }

const result = await client.request(ApiRegistry.flakyEndpoint, {
  params: undefined
});

if (!result.ok) {
  // Exhausted all 3 retries
  console.error(result.error.message); 
}
```

## OpenAPI Integration (Optional Expansion)

Because our central `ApiRegistry` relies on purely typed structures and Zod schemas, it is trivial to integrate with **OpenAPI/Swagger**. 

By using libraries like `@asteasolutions/zod-to-openapi`, you can parse the `ApiRegistry` and export it directly to an OpenAPI `swagger.json` file. Alternatively, you can use CLI tools to reverse-generate the `registry.ts` files directly from a backend's Swagger documentation!
