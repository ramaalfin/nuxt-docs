# Type-Safe Form Library

## Introduction & Motivation

Managing form state in Vue 3 can quickly become chaotic when dealing with deeply nested objects, dynamic arrays, and complex validation rules. While libraries like `vee-validate` are powerful, they often introduce heavy overhead and their TypeScript inference can sometimes feel bolted-on.

We built `useSafeForm` to solve this. It is a lightweight, strictly typed, Zod-first composable. By leaning entirely into **Zod** as the single source of truth, `useSafeForm` guarantees that your form state, error paths, and submit handlers are 100% type-safe. If your schema compiles, your form works.

## Installation Guide

Since this is an internal library pattern, you don't install it via npm. Instead, you integrate the core files into your Nuxt/Vue project.

**Dependencies needed:**
```bash
pnpm add zod
```

**Files required:**
1. Copy `app/utils/formTypes.ts` to your project (contains the type mapping logic).
2. Copy `app/composables/useSafeForm.ts` to your project (the core Vue reactivity).
3. Copy `app/components/SafeField.vue` to your project (the custom input wrapper).

## Core Concepts

### 1. Zod as the Source of Truth
Instead of defining an `interface FormState` and a separate validation schema, you only define the Zod schema. `useSafeForm` automatically infers the deep reactive state from it.

### 2. Flat Error Mapping
When a user submits a form with a nested error (e.g., their nested city string is too short), Zod outputs an error path `["address", "city"]`. Our library flattens this to `"address.city"` and binds it to a strictly typed `FormErrors<T>` object.

### 3. Safe Submission
The `handleSubmit` function intercepts the native form submission, prevents default behavior, and validates against Zod. Your success callback is only ever triggered with **perfectly typed, parsed, and transformed data**.

## API Reference

### `useSafeForm(schema, initialValues)`
The core composable to initialize a form.

- **`schema`**: A Zod schema (`z.ZodTypeAny`).
- **`initialValues`**: An optional partial object matching the schema.

**Returns:**
- `values`: The reactive form state.
- `errors`: A reactive object mapping dot-notation paths to error strings.
- `isSubmitting`: A boolean ref indicating submission status.
- `validate()`: An async function that forces validation and updates `errors`.
- `clearError(path)`: Clears a specific error.
- `setError(path, message)`: Manually sets an error.
- `handleSubmit(onSuccess, onError)`: Returns a form submit event handler.

### `<SafeField>`
A renderless component pattern used to safely extract deeply nested errors and pass them to your UI inputs via scoped slots.

**Props:**
- `errors`: The `errors` object returned from `useSafeForm`.
- `name`: A strictly typed string path (e.g., `'address.city'`). Will throw a TS error if the path doesn't exist in the schema.

## Usage Examples

Here are 4 core use cases showing exactly how to implement them, complete with live interactive previews.

### 1. Basic Form
A standard login form with flat validation.

```vue
<script setup lang="ts">
import { z } from 'zod';
import { useSafeForm } from '~/composables/useSafeForm';

const schema = z.object({
  email: z.string().email(),
  password: z.string().min(8)
});

const { values, errors, handleSubmit } = useSafeForm(schema, {
  email: '',
  password: ''
});

const onSubmit = handleSubmit((data) => {
  // `data` is typed as { email: string, password: string }
  console.log(data);
});
</script>

<template>
  <form @submit="onSubmit">
    <SafeField :errors="errors" name="email" v-slot="{ error }">
      <input v-model="values.email" />
      <span>{{ error }}</span>
    </SafeField>
    
    <button type="submit">Login</button>
  </form>
</template>
```

### 2. Nested Forms
Handling deep objects without losing type safety. Note how the `name` prop on `SafeField` accepts dot-notation.

```vue
<script setup lang="ts">
const schema = z.object({
  name: z.string(),
  address: z.object({
    street: z.string(),
    city: z.string()
  })
});

const { values, errors } = useSafeForm(schema, { address: { city: '' } });
</script>

<template>
  <SafeField :errors="errors" name="address.city" v-slot="{ error }">
    <input v-model="values.address.city" />
  </SafeField>
</template>
```

### 3. Array Fields
Dynamic arrays where the user can add or remove items. Notice how we use template literals to bind specific indices.

```vue
<script setup lang="ts">
const schema = z.object({
  tags: z.array(z.string().min(3)).min(1)
});

const { values, errors } = useSafeForm(schema, { tags: ['Vue'] });
</script>

<template>
  <div v-for="(tag, i) in values.tags" :key="i">
    <!-- Cast to any to bypass Vue template literal dynamic binding limitation -->
    <SafeField :errors="errors" :name="`tags.${i}` as any" v-slot="{ error }">
      <input v-model="values.tags[i]" />
    </SafeField>
    <button @click="values.tags.splice(i, 1)">Remove</button>
  </div>
  <button @click="values.tags.push('')">Add Tag</button>
</template>
```

### 4. Custom Input Component Pattern
When integrating with complex UI libraries (like Nuxt UI or Headless UI), `<SafeField>` acts as the glue. Here is how you can use it to build a highly reusable, masked, or custom input component:

```vue
<!-- CustomPhoneInput.vue -->
<template>
  <SafeField :errors="errors" :name="name" v-slot="{ error }">
    <div class="custom-phone-wrapper">
      <label>{{ label }}</label>
      <input 
        v-model="model" 
        v-mask="'+1 (###) ###-####'"
        :class="{'border-red-500': error}"
      />
      <p class="error-msg" v-if="error">{{ error }}</p>
    </div>
  </SafeField>
</template>

<script setup lang="ts">
const model = defineModel<string>();
defineProps<{
  label: string;
  name: string; // The path key
  errors: Record<string, any>;
}>();
</script>
```

## TypeScript Integration Notes

The magic behind this library lives in `app/utils/formTypes.ts`. 

To provide autocomplete for deeply nested paths, we use a recursive conditional type:
```ts
export type Paths<T, D extends number = 10> = ...
```
This type deeply traverses the generic object `T` provided by `z.infer<Schema>` and constructs string literals. `address: { city: string }` becomes `'address.city'`. 

Arrays are heavily supported. If `tags` is a `string[]`, the type generator yields `'tags.0' | 'tags.1' | 'tags.2' ...`. This ensures that `form.errors['tags.0']` is valid TypeScript, while `form.errors['tags.invalid_key']` will throw a compiler error during `vue-tsc --noEmit`.
