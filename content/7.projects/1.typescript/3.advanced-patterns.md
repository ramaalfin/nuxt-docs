# Advanced Type Patterns: The Principal Engineer's Guide

TypeScript is a Turing-complete type system. While day-to-day work often relies on simple interfaces and type unions, building massive, scalable libraries (like our Form Library or API Client) requires a deep understanding of how TypeScript evaluates constraints, distributes unions, and infers types. 

This is your master reference guide for writing safe, performant, and advanced type patterns.

---

## 1. Generics Deep Dive

Generics are functions that operate on types. Understanding bounded polymorphism and variance is critical.

### Bounded Polymorphism (`extends`)
By default, a generic `<T>` can be anything. Using `extends` constrains it.

```ts
// Bad: T could be a string, number, etc., meaning `.id` isn't guaranteed.
function extractId<T>(obj: T) { ... } 

// Good: Bounded Polymorphism
function extractId<T extends { id: string }>(obj: T): string {
  return obj.id;
}
```

### Covariance vs Contravariance (The `strictFunctionTypes` behavior)
In TypeScript, objects and arrays are *covariant* (you can assign a narrower type to a wider type), but function arguments are *contravariant* (you can assign a wider argument type to a narrower one).

> [!CAUTION]
> If you disable `strictFunctionTypes` in `tsconfig.json`, TypeScript makes function arguments bivariant, which introduces massive soundness holes. Always keep strict mode on.

### Emulating Higher-Kinded Types (HKT)
TypeScript does not natively support HKTs (e.g., `F<T>`), but you can emulate them using interface merging. This is heavily used in libraries like `fp-ts`.

```ts
export interface URItoKind<A> {}
export type URIS = keyof URItoKind<any>;
export type Kind<URI extends URIS, A> = URItoKind<A>[URI];

// Usage:
export interface OptionURI extends URItoKind<any> {
  Option: Option<any>; // Placeholder
}
```

## 2. Conditional Types & Inference (`infer`)

Conditional types take the form `T extends U ? X : Y`. They evaluate lazily and are heavily used to unwrap nested types.

### Distributive Conditional Types
If `T` is a generic and a union type, conditional types automatically distribute over it.

```ts
type ToArray<T> = T extends any ? T[] : never;

// Evaluates to string[] | number[], NOT (string | number)[]
type Distributed = ToArray<string | number>; 
```

**How to prevent distribution:** Wrap the generic in tuples `[]`.

```ts
type ToArrayNonDistributive<T> = [T] extends [any] ? T[] : never;

// Evaluates to (string | number)[]
type NonDistributed = ToArrayNonDistributive<string | number>;
```

### Unpacking with `infer`
The `infer` keyword allows you to declare a local type variable inside the `extends` clause of a conditional type.

```ts
// Unpacking a Promise
type Unpromisify<T> = T extends Promise<infer U> ? U : T;

// Unpacking an Array
type ElementOf<T> = T extends Array<infer E> ? E : T;

// Unpacking Function Arguments
type FirstArgument<T> = T extends (arg1: infer A, ...args: any[]) => any ? A : never;
```

### Real World Use Case: Deep Paths
Remember our Form Library? Here is how we built the `Paths<T>` deep type recursively.

```ts
type IsAny<T> = 0 extends 1 & T ? true : false;

export type Paths<T, D extends number = 10> = [D] extends [never] ? never :
  IsAny<T> extends true ? string :
  T extends object ? {
    [K in keyof T]-?: K extends string | number ?
      `${K}` | (Paths<T[K], Prev[D]> extends infer P ? P extends string ? `${K}.${P}` : never : never)
    : never
  }[keyof T] : never;
```

## 3. Type Inference Tricks

### Contextual Typing
TypeScript infers types in both directions. When the context is known (e.g., passing an inline callback to an array method), TS infers the parameters automatically.

```ts
// 'e' is inferred as MouseEvent because of onClick's definition.
<button onClick={(e) => console.log(e.clientX)} />
```

### Type Predicates (`x is Type`)
When validating data at runtime, return a type predicate to narrow the type locally.

```ts
function isString(val: unknown): val is string {
  return typeof val === 'string';
}

function process(val: string | number) {
  if (isString(val)) {
    val.toUpperCase(); // Safe!
  }
}
```

### Assertion Functions (`asserts condition`)
Similar to predicates, but they throw an error if the condition is false.

```ts
function assertIsUser(user: any): asserts user is User {
  if (!user || !user.id) throw new Error("Not a user");
}

function handle(payload: unknown) {
  assertIsUser(payload);
  console.log(payload.id); // payload is now `User`
}
```

### Key Remapping (`as`)
When mapping over types, you can rename keys dynamically using template literals.

```ts
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};

interface Person { name: string; age: number; }
// Yields: { getName: () => string; getAge: () => number; }
type PersonGetters = Getters<Person>; 
```

## 4. When to Use `any` (Spoiler: Never)

The `any` type completely disables the TypeScript compiler for that variable. It cascades, ruining inference downstream.

### The Superior Alternative: `unknown`
`unknown` is a type-safe `any`. You can assign anything to `unknown`, but you **cannot** interact with it until you narrow it.

```ts
let data: unknown = fetchSomething();

// Error: Object is of type 'unknown'.
console.log(data.id); 

// Safe: You must narrow it first.
if (typeof data === 'object' && data !== null && 'id' in data) {
  console.log(data.id);
}
```

### The `satisfies` Operator
Sometimes you want to validate an object against a type, but keep its *exact* literal type inference intact.

```ts
type RouteConfig = Record<string, string | Record<string, string>>;

// BAD: We lose the exact keys. `routes.home` is just `string | Record...`
const routes: RouteConfig = {
  home: "/",
  about: "/about"
};

// GOOD: `routes` is verified against `RouteConfig`, but `routes.home` remains EXACTLY `"/"`.
const exactRoutes = {
  home: "/",
  about: "/about"
} satisfies RouteConfig;
```

## 5. Performance Implications

As a Principal Engineer, you must know that complex TypeScript types can severely slow down your IDE and build times. The TS compiler has to evaluate these types on every keystroke.

### How to Crash the TS Compiler
Recursive mapped types evaluating massive unions (e.g., standardizing `deepPaths` on a generic DOM event object) will trigger the `Type instantiation is excessively deep and possibly infinite.` error.

### Optimization Strategy 1: Interface vs Type Alias
> [!TIP]
> Always use `interface` over `type` when defining objects.

Interfaces are cached by their name. Type aliases are evaluated structurally and lazily computed. When extending or merging, interfaces resolve much faster.

```ts
// SLOW
type Animal = { name: string };
type Dog = Animal & { bark: () => void };

// FAST
interface Animal { name: string; }
interface Dog extends Animal { bark: () => void; }
```

### Optimization Strategy 2: Cut Down on Deep Mapping
If a mapped type isn't necessary, don't use it. For deeply nested structures (like our Form Library), intentionally cap the recursion depth using a numeric tuple array, preventing the compiler from evaluating past 10 levels deep.

### Optimization Strategy 3: Avoid Heavy Union Distribution
A union of 50 strings passed into a distributive conditional type will evaluate 50 separate times. Wrapping the generic `[T] extends [X]` stops this immediately.

## 6. TypeScript Playground Examples

To master these, copy and paste the following into the [Official TypeScript Playground](https://www.typescriptlang.org/play):

```ts
// 1. Distributive vs Non-Distributive Check
type ToArray<T> = T extends any ? T[] : never;
type ToArrayNonDist<T> = [T] extends [any] ? T[] : never;
type Test1 = ToArray<string | number>; // string[] | number[]
type Test2 = ToArrayNonDist<string | number>; // (string | number)[]

// 2. Inferring Function Returns
type ReturnOf<T> = T extends (...args: any[]) => infer R ? R : never;
type Test3 = ReturnOf<() => { ok: boolean }>; // { ok: boolean }

// 3. Satisfies Operator Magic
type Config = Record<string, string[]>;
const myConfig = {
    servers: ['192.168.1.1'],
    ports: ['8080']
} satisfies Config;
// Hover over `myConfig.servers`! It knows it's exactly a string[]!
```
