# Advanced TypeScript Patterns

Welcome to the comprehensive guide on **Advanced TypeScript Patterns**. As an engineer looking to operate at the Principal level, mastering TypeScript isn't just about adding types to JavaScript—it's about leveraging the compiler to model domain logic, prevent entire classes of runtime errors, and create exceptional developer experiences for your team.

This guide dives deep into Generics, Conditional Types, Type Inference tricks, Compiler Performance, and Real-world Architecture Patterns.

---

## 1. Generics Deep Dive

Generics allow you to write reusable, type-safe code that works over a variety of types rather than a single one.

### Functions and Constraints
You can constrain a generic type using the `extends` keyword. This guarantees the generic type possesses certain properties.

```typescript
// T must be an object with an `id` property of type string or number
function extractId<T extends { id: string | number }>(item: T): T['id'] {
  return item.id;
}

extractId({ id: 1, name: 'Alice' }); // Returns number
extractId({ name: 'Bob' }); // ❌ Error: Property 'id' is missing
```

### Default Types
You can provide fallback types for generics, which is incredibly useful for configuration objects or APIs.

```typescript
type ApiResponse<TData = unknown, TError = Error> = 
  | { ok: true; data: TData }
  | { ok: false; error: TError };

const success: ApiResponse<{ user: string }> = { ok: true, data: { user: 'Alice' } };
const failure: ApiResponse = { ok: false, error: new Error('Network fail') }; // TData defaults to unknown
```

### Higher-Order Generics (Currying)
Sometimes you need to infer types in stages. TypeScript doesn't natively support partial type argument inference, but you can use currying as a workaround:

```typescript
const makeState = <TState>() => <TKey extends keyof TState>(key: TKey, value: TState[TKey]) => {
  return { [key]: value } as Pick<TState, TKey>;
};

interface AppState { theme: 'light' | 'dark'; sidebarOpen: boolean; }

// First call binds AppState, second call infers the keys perfectly!
const setSidebar = makeState<AppState>()('sidebarOpen', true);
// const badTheme = makeState<AppState>()('theme', 'neon'); // ❌ Error: 'neon' is not assignable to 'light' | 'dark'
```

## 2. Conditional Types

Conditional types look like ternary operators (`T extends U ? X : Y`) but operate at the type level.

### Type Filtering (Extract / Exclude)
The built-in utility types `Extract` and `Exclude` are implemented using conditional types and distributive behavior.

```typescript
type Roles = 'admin' | 'user' | 'guest';

// Keep only literal strings that extend 'admin' or 'user'
type AuthenticatedRoles = Extract<Roles, 'admin' | 'user'>; // 'admin' | 'user'

// Remove 'guest'
type NonGuestRoles = Exclude<Roles, 'guest'>; // 'admin' | 'user'
```

### Function Overload Replacement
Instead of writing multiple function signatures, you can often use conditional types combined with generic parameters.

```typescript
// If the input is a string, return a string array. If input is a number, return a number array.
type ProcessResult<T> = T extends string ? string[] : T extends number ? number[] : never;

function processInput<T extends string | number>(input: T): ProcessResult<T> {
  if (typeof input === 'string') return input.split('') as any;
  if (typeof input === 'number') return [input * 2] as any;
  throw new Error('Invalid');
}

const strArr = processInput('hello'); // Inferred as string[]
const numArr = processInput(42);      // Inferred as number[]
```

### Inference with `infer`
The `infer` keyword allows you to extract types from deeply nested structures or function signatures.

```typescript
// Extract the return type of any Promise
type UnpackPromise<T> = T extends Promise<infer U> ? U : T;

type Data = UnpackPromise<Promise<{ id: string }>>; // { id: string }
type Raw = UnpackPromise<number>; // number

// Extract arguments from a function
type GetFirstArg<T> = T extends (first: infer A, ...args: any[]) => any ? A : never;
type Arg = GetFirstArg<(name: string, age: number) => void>; // string
```

## 3. Type Inference Tricks

Mastering the following operators will allow you to construct types out of values, keeping your codebase DRY.

### `typeof` and `keyof`

```typescript
const CONFIG = {
  endpoint: '/api/v1',
  retries: 3,
  timeout: 5000,
} as const;

type ConfigKeys = keyof typeof CONFIG; // 'endpoint' | 'retries' | 'timeout'
type ConfigValues = typeof CONFIG[ConfigKeys]; // '/api/v1' | 3 | 5000
```

### Mapped Types & Template Literals
You can map over unions and manipulate string literals to generate massive types automatically.

```typescript
type Events = 'click' | 'hover' | 'scroll';

// Generates: { onClick: () => void; onHover: () => void; onScroll: () => void; }
type EventHandlers = {
  [K in Events as `on${Capitalize<K>}`]: () => void;
};
```

### The `satisfies` Operator (TS 4.9+)
`satisfies` allows you to validate that an object conforms to a type *without* widening or losing its exact literal values.

```typescript
type ThemeColors = Record<string, string | [number, number, number]>;

const theme = {
  primary: 'blue',
  secondary: [255, 0, 0],
} satisfies ThemeColors;

// Because we used satisfies instead of `: ThemeColors`, TypeScript still knows:
theme.primary.toUpperCase(); // OK! Knows it's a string.
theme.secondary[0]; // OK! Knows it's a tuple.
```

## 4. Common Pitfalls & Solutions

### `any` vs `unknown`
Never use `any`. `any` turns off the TypeScript compiler completely. `unknown` represents "I don't know what this is yet," forcing you to narrow the type via Type Guards before you can use it.

| Feature | `any` | `unknown` |
|---------|-------|-----------|
| Access properties (`val.foo`) | ✅ Allowed (Unsafe) | ❌ Error |
| Call as function (`val()`) | ✅ Allowed (Unsafe) | ❌ Error |
| Requires type narrowing | ❌ No | ✅ Yes |

### Excess Property Checks
TypeScript only checks for excess properties when an object literal is assigned *directly* to a variable or passed *directly* as an argument.

```typescript
interface User { id: number; name: string; }

// ❌ Error: Object literal may only specify known properties
const u1: User = { id: 1, name: 'Alice', age: 30 }; 

// ✅ Compiles (Pitfall!)
const data = { id: 1, name: 'Alice', age: 30 };
const u2: User = data; // Works because of structural typing
```

### `declare` vs `export`
- `declare` tells the compiler "this exists at runtime, trust me." It emits no JavaScript.
- `export` is an ES module feature that emits JavaScript to share code across files.

## 5. Performance Implications

Complex TypeScript projects can suffer from slow compilation and IDE lagging. Understanding how the compiler works can drastically improve speed.

### `interface` vs `type`
**Always prefer `interface` over `type` for object shapes.** 
Interfaces are cached by their name. Type aliases are structurally evaluated immediately. If you have deeply nested type aliases, `tsc` will recursively evaluate them over and over.

### Union Combinatorics
Avoid unions of massive objects if possible. The compiler has to check every single combination.

```typescript
// ⚠️ Slow (Compiler checks every possible variation of large A or large B)
type Data = LargeTypeA | LargeTypeB; 

// ✅ Faster (Tagging allows compiler to narrow instantly)
interface DataA extends LargeTypeA { type: 'A' }
interface DataB extends LargeTypeB { type: 'B' }
type Data = DataA | DataB;
```

## 6. Real-World Type-Safe Patterns

### 1. Discriminated Unions (Tagged Unions)
The absolute best way to model state machines or polymorphic data. 

```typescript
type Shape = 
  | { kind: 'circle'; radius: number }
  | { kind: 'square'; size: number }
  | { kind: 'rectangle'; width: number; height: number };

function getArea(shape: Shape) {
  // TypeScript instantly knows the exact shape inside each block
  switch (shape.kind) {
    case 'circle': return Math.PI * shape.radius ** 2;
    case 'square': return shape.size ** 2;
    case 'rectangle': return shape.width * shape.height;
    default:
      // Exhaustiveness check: TS will error here if you add a new shape to the union but forget to handle it!
      const _exhaustiveCheck: never = shape;
      return _exhaustiveCheck;
  }
}
```

### 2. Builder Pattern
For constructing complex objects incrementally.

```typescript
class QueryBuilder<T extends Record<string, any> = {}> {
  private query: Record<string, any> = {};

  select<K extends string>(key: K): QueryBuilder<T & Record<K, any>> {
    this.query[key] = true;
    return this as any; // Type assertion needed for builder implementation
  }

  build(): T {
    return this.query as T;
  }
}

const query = new QueryBuilder()
  .select('id')
  .select('name')
  .build(); // Inferred as: { id: any; name: any; }
```

### 3. Type-Safe Event Bus / Emitter
Connecting events to strictly defined callback arguments.

```typescript
type EventMap = {
  userLogin: { userId: string; timestamp: number };
  logout: void;
  error: { message: string; code: number };
};

class TypedEventEmitter<T extends Record<string, any>> {
  private listeners: Record<string, Function[]> = {};

  on<K extends keyof T>(event: K, callback: (payload: T[K]) => void) {
    if (!this.listeners[event as string]) this.listeners[event as string] = [];
    this.listeners[event as string].push(callback);
  }

  emit<K extends keyof T>(event: K, payload: T[K]) {
    this.listeners[event as string]?.forEach(cb => cb(payload));
  }
}

const bus = new TypedEventEmitter<EventMap>();
// ✅ Strongly typed payload!
bus.on('userLogin', (data) => console.log(data.userId)); 
// ❌ TS Error: Missing code property
// bus.emit('error', { message: 'Failed' }); 
```

### 4. Typed Redux-like Actions
Using template literals and discriminated unions to enforce action shapes.

```typescript
type ActionFactory<Type extends string, Payload> = { type: Type; payload: Payload };

type Actions = 
  | ActionFactory<'ADD_TODO', { text: string }>
  | ActionFactory<'REMOVE_TODO', { id: number }>
  | ActionFactory<'TOGGLE_TODO', { id: number }>;

function reducer(state: any, action: Actions) {
  if (action.type === 'ADD_TODO') {
    console.log(action.payload.text); // Strongly typed!
  }
}
```

## 7. The "Pitfalls Checklist"

Before you commit your TypeScript code, check against this list:

- [ ] **Did I use `any`?** Replace it with `unknown` and use type narrowing (`typeof x === 'string'`, `instanceof`, or Zod parsing).
- [ ] **Am I casting blindly? (`as AnyType`)** If you find yourself using `as` everywhere, your domain model or generic inference is flawed. Refactor the types so TS infers them naturally.
- [ ] **Did I use `interface` for object shapes?** Avoid `type Alias = { ... }` for simple objects to improve compiler performance.
- [ ] **Did I check for exhaustiveness?** When using `switch` on a discriminated union, always include a `default` block that assigns the variable to `never`.
- [ ] **Is my generic too loose?** Use `extends` to constrain your generics so they don't accept absolute garbage values.
