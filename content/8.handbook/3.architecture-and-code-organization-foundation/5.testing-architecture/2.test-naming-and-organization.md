# Test Naming & Organization in Vue/Nuxt

A clean, predictable structure for test files and naming conventions is vital for navigating a large codebase. When tests are well-organized, developers can easily discover missing coverage, understand the intent of existing tests, and maintain a high standard of quality.

## 1. Test File Organization (Colocation vs Centralization)

For large Vue and Nuxt applications, **Colocation** is the strongly recommended architectural pattern for Unit and Integration tests. 

### Colocation (Recommended)
Place the test file directly adjacent to the file it is testing.

```text
components/
└── UserProfile/
    ├── UserProfile.vue
    ├── UserProfile.spec.ts   <-- Test right next to component
    └── UserProfile.stories.ts

composables/
├── useAuth.ts
└── useAuth.spec.ts           <-- Test right next to composable
```
**Benefits:**
- **Discoverability:** It's immediately obvious if a component lacks tests.
- **Maintainability:** Moving, renaming, or deleting a feature keeps its tests bound to it.
- **Clean Imports:** Relative imports in the test file remain short (e.g., `import UserProfile from './UserProfile.vue'`).

### Centralization (E2E Tests Only)
End-to-End tests (Playwright/Cypress) should **not** be colocated. Because they test user journeys across multiple pages and components, they belong in a dedicated root directory.

```text
/my-nuxt-app
├── e2e/
│   ├── checkout.spec.ts
│   ├── auth.spec.ts
│   └── fixtures/
```

## 2. File Naming Conventions

*   **Unit & Integration Tests:** Use the `.spec.ts` or `.test.ts` extension. Pick one and enforce it via ESLint. (`.spec.ts` is the community standard for Vue/Vitest).
*   **E2E Tests:** Keep them distinct, for example `auth.e2e.spec.ts` or place them in an isolated `e2e/` directory to prevent Vitest from accidentally running them.

## 3. Describe & It Block Conventions

A test file should read like a specification document. Use the AAA pattern (Arrange, Act, Assert) within your blocks, and name them explicitly.

### The `describe` Block
The top-level `describe` block should strictly name the Unit (Component, Composable, or Class) being tested. Nested `describe` blocks should define the specific state, context, or method being tested.

```typescript
// Good
describe('UserProfile.vue', () => { ... })
describe('useAuth() composable', () => { ... })

// Contextual nesting
describe('UserProfile.vue', () => {
  describe('when the user is unauthenticated', () => { ... })
  describe('when the user is an admin', () => { ... })
})
```

### The `it` or `test` Block
The `it` block should describe the expected outcome. It should naturally follow the word "it". 
- **Pattern:** `it('should [expected behavior] when [condition]')`

```typescript
describe('useCart()', () => {
  describe('addItem()', () => {
    
    // GOOD
    it('should increase the total price when a valid item is added', () => { ... })
    it('should throw an error when an item with zero inventory is added', () => { ... })

    // BAD (Too vague, doesn't describe the outcome)
    it('adds item', () => { ... })
    it('tests the error', () => { ... })
  })
})
```

## 4. Nuxt-Specific Organization

When testing Nuxt-specific features like plugins or middleware:
- Tests for `middleware/auth.ts` should live at `middleware/auth.spec.ts`.
- Tests for server APIs (`server/api/users.ts`) should live at `server/api/users.spec.ts`.

By keeping tests colocated and strictly enforcing semantic naming (`describe` = Context, `it` = Expected Outcome), your test suite doubles as living, executable documentation for your Vue/Nuxt architecture.
