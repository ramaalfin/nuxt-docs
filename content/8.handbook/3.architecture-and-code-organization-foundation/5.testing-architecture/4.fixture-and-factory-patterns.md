# Fixture & Factory Patterns for Test Data

In large Vue and Nuxt applications, writing tests often involves passing complex data objects (e.g., deeply nested User profiles, Product catalogs) as props to components, as mocked API responses, or as initial store states.

Manually writing out these objects in every test file leads to massive code duplication, brittle tests, and maintenance nightmares when the underlying data schema changes. The solution is using **Fixtures** and **Factory Patterns**.

## 1. The Problem with Hardcoded Test Data

**Bad Practice:**
```typescript
// UserProfile.spec.ts
it('renders user details', () => {
  // Hardcoded object. If the backend adds a required 'status' field, 
  // you have to update 50 different test files manually.
  const user = {
    id: 1,
    name: 'John Doe',
    email: 'john@example.com',
    role: 'ADMIN',
    preferences: { theme: 'dark', notifications: true }
  }
  
  const wrapper = mount(UserProfile, { props: { user } })
})
```

## 2. Fixtures (Static Test Data)

Fixtures are static, reusable blocks of data placed in a centralized file.

```typescript
// tests/fixtures/users.ts
export const mockAdminUser = {
  id: 1,
  name: 'Admin User',
  role: 'ADMIN',
  // ...
}

export const mockStandardUser = {
  id: 2,
  name: 'Standard User',
  role: 'USER',
  // ...
}
```

**Pros:** Centralizes data. Changes to the schema only happen in one place.
**Cons:** Static. If a specific test needs an admin user named "Alice" instead of "Admin User", you end up creating endless variations (`mockAdminUserNamedAlice`).

## 3. The Factory Pattern (Dynamic Test Data)

The Factory pattern solves the limitations of static fixtures by providing a function that generates a base object, which can optionally be overridden for specific test scenarios. 

This is highly recommended for large applications.

### Creating a Simple Factory
```typescript
// tests/factories/userFactory.ts
import type { User } from '@/types/user'

// Takes optional overrides and merges them with a default base object
export function buildUser(overrides: Partial<User> = {}): User {
  return {
    id: Math.random().toString(),
    name: 'Default User',
    email: 'default@example.com',
    role: 'USER',
    ...overrides
  }
}
```

**Usage in tests:**
```typescript
// UserProfile.spec.ts
import { buildUser } from '~/tests/factories/userFactory'

it('renders standard user', () => {
  // Uses default factory data
  const user = buildUser() 
  mount(UserProfile, { props: { user } })
})

it('renders admin badge for admins', () => {
  // Overrides only the specific field required for this exact test
  const adminUser = buildUser({ role: 'ADMIN' }) 
  const wrapper = mount(UserProfile, { props: { user: adminUser } })
  expect(wrapper.text()).toContain('Admin Badge')
})
```

## 4. Advanced Factories with Tools (Faker & Fishery)

For enterprise-level apps, hand-rolling factories can be tedious. Libraries like **Fishery** (by thoughtbot) combined with **Faker.js** provide robust, typed factories.

*   **Faker.js:** Generates realistic fake data (names, emails, addresses).
*   **Fishery:** Provides a clean API for defining factories with strong TypeScript support.

### Example using Fishery + Faker
```typescript
// tests/factories/postFactory.ts
import { Factory } from 'fishery'
import { faker } from '@faker-js/faker'
import type { Post } from '@/types/post'
import { buildUser } from './userFactory'

export const postFactory = Factory.define<Post>(({ sequence }) => ({
  id: sequence.toString(),
  title: faker.lorem.sentence(),
  content: faker.lorem.paragraphs(2),
  publishedAt: faker.date.recent().toISOString(),
  // Compose factories to generate related data
  author: buildUser() 
}))
```

**Test Usage:**
```typescript
// Generates an array of 5 fully mocked posts with realistic data
const posts = postFactory.buildList(5) 

// Builds a post but forces it to be unpublished
const unpublishedPost = postFactory.build({ publishedAt: null }) 
```

## Summary
- **Never** hardcode large data objects directly inside `it()` blocks.
- **Use Fixtures** for simple, static data (e.g., a list of supported countries).
- **Use Factories** (e.g., `fishery`) for complex entities (Users, Products). This allows your tests to clearly communicate *what* specific data point matters for a given test (via the overrides) while abstracting away the rest.
