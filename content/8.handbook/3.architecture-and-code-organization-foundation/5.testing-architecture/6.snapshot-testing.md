# Snapshot Testing in Vue/Nuxt

Snapshot testing is a technique where a test runner (like Vitest) renders a UI component, takes a "snapshot" of its output (usually the HTML string), and compares it against a reference snapshot stored alongside the test. If the two snapshots do not match, the test fails.

While incredibly fast to write, snapshot tests are highly debated in the testing community due to their fragility. Understanding *when* to use them is key to maintaining a healthy test suite.

## The Danger of Snapshot Fatigue

**Do not use snapshots as a substitute for explicit assertions.** 

When developers overuse snapshot testing for everything, the test suite becomes brittle. A minor, intentional change to a CSS class or a `<div>` wrapper will cause 50 snapshot tests to fail. Developers quickly succumb to "Snapshot Fatigue"—the habit of blindly pressing `u` (update snapshots) whenever tests fail, without actually reviewing if the visual regression was a bug. When this happens, the tests provide zero value.

## When Snapshot Testing is Appropriate

Snapshots are highly effective in very specific scenarios:

### 1. Highly Reusable, "Dumb" UI Components
If you are building a foundational UI library (e.g., a complex `<BaseTable>` or `<MarkdownRenderer>`), and the HTML output is massive but purely deterministic based on props, a snapshot ensures you don't accidentally break the structure.

### 2. Error Messages and Logs
Instead of asserting that an error string exactly equals a 300-character string, snapshotting the output of an error formatter or console logger is very efficient.

### 3. API Response Transformers
If you have a function that transforms a massive, ugly GraphQL response into a clean frontend object, a snapshot test is an easy way to verify the transformation structure remains intact.

## Implementing Snapshots in Vitest

### Standard Snapshots
Standard snapshots write the output to a `.snap` file in a `__snapshots__` directory.

```typescript
// BaseButton.spec.ts
import { mount } from '@vue/test-utils'
import BaseButton from './BaseButton.vue'

it('renders correctly with primary variant', () => {
  const wrapper = mount(BaseButton, { props: { variant: 'primary' } })
  
  // This creates BaseButton.spec.ts.snap
  expect(wrapper.html()).toMatchSnapshot() 
})
```

### Inline Snapshots (Highly Recommended)
Inline snapshots write the snapshot directly into the test file itself. This is **strongly recommended** over standard snapshots because developers can immediately see the expected output without opening a separate file during code review.

```typescript
it('formats currency correctly', () => {
  const result = formatCurrency(1234.56, 'USD')
  
  // Vitest will automatically insert the string "$1,234.56" into this function call
  // upon the first test run.
  expect(result).toMatchInlineSnapshot(`"$1,234.56"`)
})
```

## Best Practices for Vue Component Snapshots

If you must snapshot a Vue component, avoid snapshotting highly dynamic data.

1.  **Mock Timestamps and IDs:** If your component renders a randomly generated ID or `new Date()`, the snapshot will fail every time it runs. Always use deterministic data (via Fixtures/Factories) or mock the system time before taking the snapshot.
2.  **Clean the Output:** Strip out extraneous attributes or comments that don't matter to the component's functionality to keep the snapshot readable.

```typescript
it('renders the alert box', () => {
  vi.setSystemTime(new Date(2023, 1, 1)) // Freeze time for deterministic rendering
  
  const wrapper = mount(AlertBox, { props: { message: 'Warning!' } })
  expect(wrapper.html()).toMatchInlineSnapshot(`
    "<div class="alert alert-warning">
      <span>Warning!</span>
      <time>2/1/2023</time>
    </div>"
  `)
})
```

## Summary
- Use **Inline Snapshots** whenever possible for better code reviews.
- **Never** use a snapshot to avoid writing explicit, meaningful assertions (e.g., `expect(wrapper.find('button').isDisabled()).toBe(true)` is always better than `expect(wrapper.html()).toMatchSnapshot()`).
- Reserve snapshots for pure UI primitives, string formatters, and complex object transformations.
