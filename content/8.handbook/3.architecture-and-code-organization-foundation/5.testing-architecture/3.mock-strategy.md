# Mock Strategy in Vue and Nuxt Testing

In a large application, units of code rarely exist in isolation. Components rely on composables, composables rely on stores, and stores rely on network requests. To test these units reliably and quickly, you must establish a robust **Mocking Strategy**.

Over-mocking makes tests brittle and meaningless (testing the mocks instead of the code). Under-mocking makes tests slow and flaky. Finding the balance is critical.

## 1. Network Mocks (The MSW Standard)

Historically, developers mocked `axios` or `fetch` directly using tools like `vi.mock('axios')`. **This is an anti-pattern in modern architecture.** Mocking the HTTP client tightly couples your tests to implementation details.

**The Standard:** Use **MSW (Mock Service Worker)**.
MSW intercepts actual network requests at the network level (using Service Workers in browser tests, or Node `http` interceptors in Vitest). Your application code executes a real `useFetch` or `axios` call, and MSW intercepts it and returns a mock response.

### Benefits of MSW
- **Refactor Resilient:** If you switch from Axios to native `fetch`, your tests do not break.
- **High Confidence:** You test the actual serialization, error handling, and request formatting of your HTTP client.

### Example MSW Setup for Vitest
```typescript
// mocks/handlers.ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  http.get('/api/user', () => {
    return HttpResponse.json({ id: 1, name: 'John Doe' })
  }),
  http.post('/api/checkout', () => {
    return new HttpResponse(null, { status: 200 })
  })
]
```

## 2. Module Mocks (Vitest `vi.mock`)

You should mock internal modules when the actual implementation is slow, creates unpredictable side effects, or is highly complex to set up.

**When to Mock Modules:**
- `vue-router`: Always mock routing in component tests, as setting up a real router is heavy.
- *Third-party libraries*: E.g., mocking a complex Chart.js library to just render a dummy `<div>` instead of executing heavy canvas math during a unit test.
- *Date/Time*: Always mock timers (`vi.useFakeTimers()`) and System Time (`vi.setSystemTime()`) to prevent flaky tests dependent on the current date.

### Mocking Nuxt Auto-Imports
Nuxt 3 relies heavily on auto-imports (e.g., `useRoute`, `useState`). Testing components that use these without the full Nuxt environment requires module mocking.

```typescript
// Component.spec.ts
import { vi } from 'vitest'

// Mocking a Nuxt auto-imported composable
vi.mock('#imports', () => {
  return {
    useRoute: () => ({ params: { id: '123' } }),
    useRouter: () => ({ push: vi.fn() })
  }
})
```
*Note: If you are using `@nuxt/test-utils`, it handles mounting the Nuxt context for you, reducing the need for manual `#imports` mocking.*

## 3. Mocking Pinia Stores

When testing a component that interacts with a Pinia store, you have two choices:

1.  **Use the Real Store (Integration Testing):** 
    Preferred for most scenarios. Mount the component with a fresh Pinia instance and let it mutate the real store state. It provides higher confidence.
    ```typescript
    import { createTestingPinia } from '@pinia/testing'
    
    wrapper = mount(MyComponent, {
      global: { plugins: [createTestingPinia()] }
    })
    ```

2.  **Mock the Store (Strict Unit Testing):**
    Use this if the store logic is insanely complex and you *only* want to test the component's reaction to specific store states. `@pinia/testing` automatically mocks all actions (`store.myAction` becomes a `vi.fn()`), allowing you to assert if a component called a specific action.

## Summary Strategy
- **Network:** Never mock `fetch`/`axios` directly. Use **MSW**.
- **Time:** Always freeze time (`vi.setSystemTime()`) if logic depends on `Date.now()`.
- **Modules:** Mock third-party heavy dependencies and routing integrations.
- **State:** Prefer using real (but isolated/fresh) Pinia stores using `@pinia/testing`, rather than manually mocking store objects.
