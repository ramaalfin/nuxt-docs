# Test Data Management in Vue & Nuxt

Managing test data effectively is critical for maintaining a reliable test suite. Poor data management leads to test pollution (where one test inadvertently affects the outcome of another) and flaky tests. 

While Fixtures and Factories (discussed in a previous chapter) handle the *creation* of data objects, **Test Data Management** refers to how that data is seeded, utilized, and cleaned up across the entire testing lifecycle.

## 1. Unit & Integration Test Data (Vitest)

In Unit and Component Integration tests, data management focuses on state isolation. Since Vitest runs tests extremely fast, often concurrently or sequentially within the same environment, global state (like Pinia stores) can easily become polluted.

### Principle: Fresh State Per Test
Every `it` block should start with a clean slate. Never rely on the state mutated by a previous test.

**Bad Practice:**
```typescript
describe('useUserStore', () => {
  const store = useUserStore() // Initialized once globally!
  
  it('adds a user', () => {
    store.addUser('Alice')
    expect(store.users).toHaveLength(1)
  })

  it('removes a user', () => {
    // Relies on the fact that 'Alice' was added in the previous test.
    // If the tests are run in a different order, this fails!
    store.removeUser('Alice') 
    expect(store.users).toHaveLength(0)
  })
})
```

**Good Practice (Using `beforeEach`):**
```typescript
import { setActivePinia, createPinia } from 'pinia'

describe('useUserStore', () => {
  beforeEach(() => {
    // Creates a fresh, isolated store environment before EVERY test
    setActivePinia(createPinia()) 
  })

  it('adds a user', () => {
    const store = useUserStore()
    store.addUser('Alice')
    expect(store.users).toHaveLength(1)
  })
})
```

## 2. End-to-End (E2E) Test Data (Playwright / Cypress)

E2E testing is where data management becomes highly complex. You are interacting with a real backend and a real database. 

### Strategy A: Database Seeding & Teardown (Preferred)
Before the E2E suite runs, the testing database is completely wiped and re-seeded with known, consistent data.

*   **Global Setup:** Use Playwright's `globalSetup` to connect to the test database, run migrations, and seed initial data (e.g., creating a standard 'admin' user and a 'test' product).
*   **Per-Test Setup:** Use the UI or API within a specific test to create data required *only* for that test.
*   **Isolation:** If testing destructive actions (e.g., "Delete User"), ensure that test operates on a uniquely created user so it doesn't break other tests relying on the seeded users.

### Strategy B: The API-First Setup Pattern
Instead of running heavy database scripts, you use your application's actual API to set up the data required for an E2E test.

```typescript
// Playwright example
test('User can edit their profile', async ({ request, page }) => {
  // 1. Arrange: Create a temporary user specifically for this test via API
  const response = await request.post('/api/users', { data: { name: 'Temp' } })
  const user = await response.json()

  // 2. Act: Login and navigate
  await page.goto(`/login`)
  // ... perform UI actions

  // 3. Cleanup (Optional, but recommended for large suites)
  await request.delete(`/api/users/${user.id}`)
})
```
*Benefits:* This tests the API boundaries alongside the UI, and avoids hard-coupling tests to specific database structures.

### Strategy C: Database Transactions (Advanced)
Some backends support wrapping a test execution in a database transaction. Before the E2E test runs, a transaction is opened. After the test completes, the transaction is rolled back. This provides absolute isolation and zero teardown latency, though it requires specific backend architecture support (often seen in Laravel or Django testing environments).

## 3. Dealing with Third-Party Data
If your Nuxt app integrates with Stripe, Firebase, or external CMS systems, DO NOT run E2E tests that mutate production data or hit rate limits on live third-party services.

1.  **Use Sandbox Environments:** Always use the "test mode" keys provided by the third-party service for E2E runs.
2.  **Network Mocking in E2E:** Playwright and Cypress can intercept network requests made by the browser. If a third-party sandbox is flaky or slow, you can mock the response directly in the E2E test.

```typescript
// Playwright: Mocking Stripe response
await page.route('https://api.stripe.com/v1/charges', async route => {
  await route.fulfill({ json: { status: 'succeeded' } });
});
```

## Summary
- **Unit/Integration:** Always reset Pinia stores and Vue instances inside `beforeEach`.
- **E2E:** Seed a deterministic database before the suite runs. Create test-specific entities via the API rather than relying on shared state to prevent test pollution.
- **Third-Party:** Never mutate real external data; use sandboxes or intercept network routes.
