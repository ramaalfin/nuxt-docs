# Test Coverage Targets

Test coverage is a metric that measures the percentage of your source code executed during automated tests. While an essential tool for identifying untested parts of a codebase, rigid adherence to arbitrary coverage numbers can lead to a toxic testing culture and brittle test suites.

In large Vue/Nuxt applications, setting a strategic approach to coverage targets is more important than striving for 100%.

## The Trap of 100% Code Coverage

Aiming for 100% test coverage is an anti-pattern in frontend development for several reasons:
1.  **Diminishing Returns:** Getting from 80% to 100% requires exponentially more effort for very little actual value. You end up testing framework internals or trivial boilerplate code.
2.  **Vanity Metrics:** Developers will write meaningless tests (e.g., rendering a component and asserting it exists without testing behavior) just to satisfy the coverage tool.
3.  **Brittle Tests:** When every single line is covered, any minor refactor triggers massive test failures, slowing down feature development.

## Recommended Coverage Targets

Instead of a flat global target, adopt a layered, risk-based approach to coverage.

### 1. Critical Business Logic (Target: 90% - 100%)
This is the core of your application. If this code breaks, the business loses money or data is corrupted.
*   **What it includes:** Pinia store actions handling checkout, authentication composables, price calculation utilities, data sanitizers.
*   **Strategy:** Require exceptionally high unit test coverage here. These are typically pure functions or isolated logic chunks that are fast and easy to test.

### 2. UI Components & Pages (Target: 60% - 80%)
Frontend components involve DOM manipulation, styling, and framework lifecycle hooks. 
*   **What it includes:** Vue components in `components/`, Nuxt pages in `pages/`, layouts.
*   **Strategy:** Focus coverage on component **behavior** (e.g., clicking a button emits an event, validation errors appear on bad input) rather than implementation details. Do not stress over covering every computed property that simply passes a string to a template.

### 3. Boilerplate & Configurations (Target: 0%)
*   **What it includes:** `nuxt.config.ts`, ESLint configs, simple TypeScript interfaces, basic route definitions, Tailwind configurations.
*   **Strategy:** Exclude these files from coverage reports entirely. Testing them provides zero value.

## Configuring Coverage in Vitest

Vitest uses `c8` or `v8` natively to generate coverage reports. You can configure global thresholds and explicit exclusions in `vitest.config.ts`.

```typescript
// vitest.config.ts
import { defineConfig } from 'vitest/config'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  test: {
    coverage: {
      provider: 'v8',
      // Global targets
      lines: 80,
      functions: 80,
      branches: 80,
      statements: 80,
      // Exclude files from coverage entirely
      exclude: [
        'nuxt.config.ts',
        '**/*.d.ts',
        'types/**',
        'constants/**',
        'plugins/**' // Nuxt plugins are often hard to unit test meaningfully
      ]
    }
  }
})
```

## How to use Coverage Reports Effectively

1.  **As a Discovery Tool:** Use the coverage report (HTML format) to find "blind spots" in your application. If a massive new feature was merged and the coverage for that directory is 10%, that indicates a risk.
2.  **Enforce PR Checks (Carefully):** You can set up CI/CD pipelines (like GitHub Actions) to block Pull Requests if coverage *drops* significantly from the baseline, rather than enforcing a strict global threshold. Tools like `Codecov` or `SonarQube` provide excellent PR integration for this.
3.  **Review the "Branches" Metric:** Line coverage is often misleading. The "Branches" metric (which tracks `if/else`, switch statements, and ternaries) is a much stronger indicator of whether edge cases and error states are actually being tested. Focus on Branch coverage over Line coverage.
