# The Test Pyramid in Vue & Nuxt Applications

The Test Pyramid is a foundational concept in software engineering that dictates the optimal distribution of different types of tests. In a large Vue/Nuxt application, adhering to the pyramid ensures a test suite that is fast, reliable, and highly confident.

The pyramid consists of three primary layers, moving from the bottom (fastest, most isolated) to the top (slowest, most integrated).

## 1. Unit Tests (The Base)
Unit tests form the massive foundation of your test suite. They test individual, isolated pieces of code to ensure they work correctly on their own.

*   **What to test:** 
    *   Pure utility functions (e.g., date formatters, math calculators).
    *   Vue Composables (`useCounter`, `useFormatPrice`) that do not rely on heavy Nuxt context.
    *   Pinia store logic (testing actions, getters, and state mutations).
    *   "Dumb" Presentational Components (testing that specific props render specific HTML).
*   **Tools:** Vitest (Standard for modern Vue/Vite apps), Vue Test Utils.
*   **Characteristics:** Extremely fast (milliseconds), strictly isolated (mocking external dependencies), highly deterministic.
*   **Volume:** Should make up ~70% of your test suite.

## 2. Integration Tests (The Middle)
Integration tests verify that multiple units (components, stores, composables) work together correctly. In modern frontend development, the line between Unit and Integration tests is often blurred, leaning heavily towards "Component Integration Tests."

*   **What to test:**
    *   "Smart" or Container Components.
    *   Interactions between a Component and a Pinia Store.
    *   Component trees (Parent component rendering and passing data to Child components).
    *   Nuxt Pages testing routing logic or `useFetch` data hydration.
*   **Tools:** Vitest + Vue Test Utils, `@nuxt/test-utils` (for testing within a simulated Nuxt environment), Testing Library Vue (`@testing-library/vue`).
*   **Characteristics:** Moderately fast. Often requires mounting Vue components and potentially mocking network requests (via MSW - Mock Service Worker).
*   **Volume:** Should make up ~20% of your test suite.

## 3. End-to-End (E2E) Tests (The Peak)
E2E tests evaluate the application as a complete, integrated system from the user's perspective. They spin up a real browser, navigate the actual UI, and interact with a real (or staging) backend/database.

*   **What to test:**
    *   Critical User Journeys (CUJs). Examples: 
        *   User Registration / Login flow.
        *   Complete Checkout process (adding to cart -> payment -> confirmation).
        *   Complex, multi-page data entry wizards.
*   **Tools:** Playwright (Recommended for modern Nuxt apps) or Cypress.
*   **Characteristics:** Slow (seconds to minutes), prone to flakiness (due to network or rendering timing issues), incredibly high confidence.
*   **Volume:** Should make up ~10% of your test suite. Only test the "happy paths" and absolute critical failures. Do not use E2E to test every single edge case of a form validation (use Unit tests for that).

## Summary: Applying the Pyramid in Nuxt

When a bug is discovered, always try to reproduce and fix it at the lowest possible level of the pyramid. 

If a checkout button is disabled incorrectly because of a negative cart total:
- **Don't** write a Playwright E2E test to click through the whole app to verify the button is disabled.
- **Do** write a Vitest Unit test for the `cartTotal` Pinia getter to ensure it never returns a negative number.
