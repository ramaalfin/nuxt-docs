# Hook / Composable Naming Conventions in Vue and Nuxt

In Vue 3 and Nuxt 3, "Hooks" are commonly referred to as **Composables**. Establishing strict naming conventions for these composables is essential for codebase readability, auto-import reliability, and team collaboration.

## 1. The `use` Prefix (Mandatory)
All hooks/composables must begin with the `use` prefix. This is not just a stylistic preference; it is a fundamental convention recognized by the Vue ecosystem, Linters, and Nuxt's auto-import engine.

**Bad:**
```javascript
export function fetchUserData() { ... }
export function authentication() { ... }
```

**Good:**
```javascript
export function useUserData() { ... }
export function useAuth() { ... }
```

## 2. Descriptive Action or Entity Names
The name following `use` should clearly describe either the **Action** the hook performs or the **Entity** it manages.

*   **Entity-focused:** `useUser`, `useCart`, `useProductList`
*   **Action-focused:** `useFetchData`, `useEventListener`, `useScrollLock`

## 3. Nuxt Auto-Imports and File Naming
In Nuxt 3, any file inside the `composables/` directory is automatically imported. The name of the file typically dictates the name of the composable.

**Rule:** The filename should exactly match the composable name in camelCase.

```text
composables/
├── useAuth.ts        // exports function useAuth()
├── useWindowSize.ts  // exports function useWindowSize()
└── useCart.ts        // exports function useCart()
```

### Nested Directories and Nuxt Prefixes
If you group composables in nested directories, Nuxt does **not** automatically prefix them by default (unlike components). However, to prevent naming collisions in large codebases, it is highly recommended to prefix the function name with the directory name conceptually, even if Nuxt doesn't strictly require it.

**Example:**
```text
composables/
└── auth/
    ├── useAuthSession.ts
    └── useAuthCredentials.ts
```

## 4. Boolean Hooks
If a hook returns a single boolean reactive property, the naming should reflect a true/false state, often starting with `is`, `has`, or `should`.

**Good:**
```javascript
const isOnline = useIsOnline()
const hasPermission = useHasPermission('admin')
```

## 5. Avoiding Name Collisions
Since Nuxt auto-imports composables globally, ensure your names do not collide with built-in Vue/Nuxt APIs.
*   Do not name your hook `useFetch` (collides with Nuxt). Name it `useCustomFetch` or `useApiFetch`.
*   Do not name your hook `useRouter` or `useRoute` (collides with Vue Router).

## 6. Exporting State vs Actions
If a hook manages complex state and provides functions to mutate it, destructure the return object with clear property names. The hook itself should maintain a singular focus.

```javascript
// composables/useCheckout.ts
export function useCheckout() {
  // Good: Clear distinction between state and actions
  const isProcessing = ref(false)
  const submitOrder = async () => { ... }
  
  return {
    isProcessing,
    submitOrder
  }
}
```

By adhering to these naming conventions, you ensure that every developer on the team immediately understands what a function does simply by looking at its signature, and you guarantee seamless integration with Nuxt's powerful auto-import system.
