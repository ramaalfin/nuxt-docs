# Hook Dependency Management in Vue and Nuxt

When building Custom Hooks (Composables) in Vue 3, managing the data they depend on is crucial for flexibility, reactivity, and testability. Unlike React, where the dependency array in `useEffect` or `useMemo` strictly governs re-evaluation, Vue's reactivity system handles tracking automatically. 

However, *how* you pass dependencies into a Composable defines its architecture. There are three primary patterns for managing hook dependencies in large Nuxt applications.

## 1. The "unref" Pattern (Flexible Arguments)

The most robust way to handle dependencies passed into a Composable is to accept either raw values or reactive Refs, and normalize them internally using `unref()` or `toValue()` (Vue 3.3+).

### Why?
Sometimes the caller wants to pass a static string, and other times they want to pass a reactive reference that might change. The Composable should handle both seamlessly.

### Example
```typescript
import { ref, watchEffect, toValue, MaybeRefOrGetter } from 'vue'

// The argument can be a string, a ref<string>, or a getter function!
export function useUserTitle(userId: MaybeRefOrGetter<string>) {
  const title = ref('')

  // watchEffect automatically tracks dependencies (toValue(userId))
  watchEffect(async () => {
    const id = toValue(userId) // Unwraps ref or executes getter safely
    if (id) {
      const data = await $fetch(`/api/title/${id}`)
      title.value = data.title
    }
  })

  return { title }
}
```

**Usage:**
```vue
<script setup>
// Static dependency
const { title: staticTitle } = useUserTitle('user_123')

// Reactive dependency
const selectedId = ref('user_123')
const { title: reactiveTitle } = useUserTitle(selectedId) 
// If selectedId.value changes, the hook automatically refetches!
</script>
```

## 2. Dependency Injection (Provide / Inject)

When a Composable relies on heavy dependencies (like an Axios instance, a logging service, or a configuration object) that shouldn't be passed manually into every single function call, use Vue's `provide` and `inject`.

### Example
```typescript
// plugins/logger.ts
export default defineNuxtPlugin((nuxtApp) => {
  const logger = {
    info: (msg) => console.log(`[INFO] ${msg}`),
    error: (msg) => console.error(`[ERROR] ${msg}`)
  }
  
  // Provide at the Nuxt app level
  nuxtApp.vueApp.provide('loggerContext', logger)
})
```

```typescript
// composables/usePayment.ts
import { inject } from 'vue'

export function usePayment() {
  // Inject the dependency rather than hardcoding it
  const logger = inject('loggerContext')
  
  const process = async () => {
    try {
      logger.info('Processing payment...')
      // process logic
    } catch (e) {
      logger.error('Payment failed')
    }
  }
  
  return { process }
}
```
**Benefits:** In unit tests, you can easily `provide` a mock logger, completely isolating `usePayment` from the real logging service.

## 3. Implicit vs Explicit Global Dependencies (Pinia / Vue Router)

In Nuxt, hooks often rely on global dependencies like `useRouter()`, `useRoute()`, or Pinia stores (`useUserStore()`).

### The Implicit Approach (Standard Nuxt)
Calling globals directly inside the hook is fast and utilizes Nuxt's auto-imports.
```typescript
export function useRedirector() {
  const router = useRouter() // Implicit global dependency
  
  const goHome = () => router.push('/')
  return { goHome }
}
```
*Pros:* Clean and minimal boilerplate.
*Cons:* Tightly couples the hook to Vue Router. Harder to test in pure isolation without mounting a Vue app with a router context.

### The Explicit Approach (Dependency via Argument)
For highly reusable hooks that might be published as an NPM package or tested strictly, pass the dependency explicitly.
```typescript
export function useRedirector(routerInstance) {
  const goHome = () => routerInstance.push('/')
  return { goHome }
}
```

## 4. Avoiding Memory Leaks with External Dependencies

If your Composable manages a dependency outside of Vue's reactivity system (e.g., a third-party SDK, a WebSocket, a Chart.js instance, or DOM Event Listeners), you **must** handle cleanup.

Vue provides `onScopeDispose` (and `onUnmounted`) to manage this.

```typescript
import { onScopeDispose } from 'vue'

export function useLiveChat(roomId: string) {
  const chatSocket = new WebSocket(`wss://api.example.com/chat/${roomId}`)
  
  chatSocket.onmessage = (msg) => { /* handle message */ }

  // CRITICAL: Cleanup dependency when the component/scope unmounts
  onScopeDispose(() => {
    chatSocket.close()
  })
}
```
Using `onScopeDispose` is preferred over `onUnmounted` because it ensures the cleanup runs even if the Composable is used outside of a standard component (e.g., inside an `effectScope`).

## Summary Checklist
- Use `toValue()` or `unref()` to accept flexible arguments (Refs or static values).
- Let `watchEffect` or `computed` automatically manage reactive tracking.
- Use `provide/inject` for global services (Loggers, Analytics) to aid testability.
- Always clean up external dependencies using `onScopeDispose`.
