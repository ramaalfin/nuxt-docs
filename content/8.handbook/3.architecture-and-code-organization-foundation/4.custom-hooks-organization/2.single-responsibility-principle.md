# Single Responsibility Principle in Custom Hooks (Composables)

The Single Responsibility Principle (SRP) is a core tenet of solid software design. It states that a module, class, or function should have one, and only one, reason to change. 

When applying the Vue 3 Composition API to build Custom Hooks (Composables) in large Nuxt applications, SRP is critical for preventing "God Hooks"—massive files that try to manage fetching, formatting, event listening, and UI state all at once.

## Identifying a Violation of SRP

Consider a custom hook named `useUserProfile`. 

**Bad Example ("God Hook"):**
```javascript
export function useUserProfile(userId) {
  const user = ref(null)
  const isEditing = ref(false)
  const windowWidth = ref(window.innerWidth)

  // Responsibility 1: Fetching data
  const fetchUser = async () => {
    user.value = await $fetch(`/api/users/${userId}`)
  }

  // Responsibility 2: Window resize tracking (UI behavior)
  const onResize = () => { windowWidth.value = window.innerWidth }
  onMounted(() => window.addEventListener('resize', onResize))
  onUnmounted(() => window.removeEventListener('resize', onResize))

  // Responsibility 3: Formatting data
  const userDisplayName = computed(() => {
    return user.value ? `${user.value.firstName} ${user.value.lastName}` : ''
  })

  return { user, isEditing, fetchUser, windowWidth, userDisplayName }
}
```

**Why this is bad:**
1. It handles HTTP requests.
2. It tracks DOM window events.
3. It formats strings.
If any of these requirements change, this hook must be modified. It is hard to test and impossible to reuse the window resizing logic elsewhere.

## Refactoring for SRP

To adhere to SRP, we break the "God Hook" down into smaller, focused hooks that do exactly one thing.

### 1. The DOM Listener Hook
```javascript
// composables/useWindowSize.ts
export function useWindowSize() {
  const width = ref(0)
  
  const onResize = () => { width.value = window.innerWidth }
  
  onMounted(() => {
    width.value = window.innerWidth
    window.addEventListener('resize', onResize)
  })
  
  onUnmounted(() => window.removeEventListener('resize', onResize))
  
  return { width }
}
```

### 2. The Data Fetching Hook
```javascript
// composables/useUserFetch.ts
export function useUserFetch(userId) {
  const user = ref(null)
  
  const fetchUser = async () => {
    user.value = await $fetch(`/api/users/${userId}`)
  }
  
  return { user, fetchUser }
}
```

### 3. The Formatter (Utility or specific Hook)
```javascript
// utils/formatters.ts
export const formatFullName = (firstName, lastName) => `${firstName} ${lastName}`
```

### 4. The Orchestrator (The Final Component)
Now, the Vue component acts as the orchestrator, pulling in only what it needs, or we can create a higher-level hook that purely *composes* these smaller hooks.

```vue
<script setup>
const props = defineProps(['userId'])

// Each hook does ONE thing well.
const { width: windowWidth } = useWindowSize()
const { user, fetchUser } = useUserFetch(props.userId)

const displayName = computed(() => 
  user.value ? formatFullName(user.value.firstName, user.value.lastName) : ''
)
</script>
```

## Benefits of SRP in Composables

1. **Testability:** You can easily unit test `useWindowSize` without having to mock an API request.
2. **Reusability:** `useWindowSize` can now be used in the Navigation component, the Footer, or anywhere else.
3. **Readability:** Developers don't have to scroll through 300 lines of mixed logic. The filename tells them exactly what the hook does.

## Rules of Thumb for SRP
- Does the name of the hook contain "And" (e.g., `useUserAndSettings`)? It probably violates SRP.
- Does the hook interact with the `window` object *and* `localStorage` *and* fetch data? It definitely violates SRP.
- Keep hooks focused on a single domain (e.g., Auth, Cart), a single API (e.g., LocalStorage, IntersectionObserver), or a single UI interaction (e.g., Modal state).
