# Component Library Principles for Vue/Nuxt Architecture

Building a robust, scalable component library (or Design System) is a crucial foundation for large-scale Nuxt and Vue applications. A well-architected library improves development velocity, ensures UI consistency, and reduces technical debt.

When designing components for a large architecture, adhere to the following core principles: Accessibility (A11y), Themeability, Documentation, and API Consistency.

---

## 1. API Consistency & Design

A predictable API is the most critical factor for developer experience (DX). Component APIs should feel like native HTML elements or standard Vue primitives.

*   **Predictable Prop Naming:**
    *   Use consistent prefixes for boolean props (e.g., `isLoading`, `isDisabled`, `hasIcon`).
    *   Standardize event names. Always use the `update:modelValue` pattern for `v-model` integrations instead of custom change events like `@change` or `@update`, maintaining seamless two-way binding.
*   **Slots over Props for Complex Content:**
    *   Avoid passing complex HTML, rendering logic, or large configuration objects through props. Use Vue slots (default and named) to maintain flexibility.
    *   *Example:* Use `<template #icon>` instead of passing `icon="my-icon-name"` if the icon might need custom styling or a different DOM structure in the future.
*   **Component Modularity:**
    *   Follow the Single Responsibility Principle. A `<Card>` component shouldn't manage API data fetching; it should only handle presentation.

> [!WARNING]
> Do not tightly couple business logic to UI components. If a component fetches its own data, it becomes difficult to reuse in different contexts, test in isolation, or mock effectively.

---

## 2. Accessibility (A11y)

A component library must be usable by everyone, including users relying on assistive technologies like screen readers or keyboard navigation.

*   **Semantic HTML Foundation:**
    *   Always use the correct semantic HTML elements (`<button>` for actions, `<a>` for navigation, `<dialog>` for modals). Never use a `<div @click="doSomething">` as a button.
*   **WAI-ARIA Attributes & Focus Management:**
    *   Implement appropriate `aria-*` attributes (e.g., `aria-expanded`, `aria-hidden`, `aria-describedby`).
    *   Ensure dynamic components (like Modals, Dialogs, or Dropdowns) trap focus correctly and return focus to the trigger element when closed.

> [!IMPORTANT]
> **Leverage Headless Libraries:** Instead of building complex accessibility logic from scratch, use unstyled "headless" libraries like **Radix Vue**, **Headless UI**, or **Ark UI**. These libraries provide fully accessible, logic-only primitives that you can wrap and style according to your design system.

---

## 3. Themeability & Styling

Large applications often require multiple themes (e.g., light/dark mode, enterprise white-labeling). The component library should be styling-agnostic to the specific app consuming it.

*   **CSS Custom Properties (Tokens):**
    *   Use CSS variables for design tokens (colors, spacing, typography). This allows Nuxt apps to override tokens globally in a stylesheet without having to override specific CSS classes.
*   **Utility-First Integration (e.g., Tailwind CSS):**
    *   If using Tailwind CSS, utilize a class management tool like `tailwind-merge` combined with `cva` (Class Variance Authority) or `tailwind-variants`. This allows you to define strict variants while gracefully handling class conflicts.

```ts
// Example of cva usage in a Vue component
import { cva } from 'class-variance-authority'

const buttonClass = cva('base-button-styles', {
  variants: {
    intent: { primary: 'bg-blue-500', danger: 'bg-red-500' },
    size: { sm: 'text-sm', md: 'text-base' }
  }
})
```

*   **Dark Mode Native Support:**
    *   Integrate natively with Nuxt's `@nuxtjs/color-mode` by defining semantic color tokens (e.g., `bg-surface-base`, `text-content-muted`) rather than hardcoding specific hex codes inside component templates.

---

## 4. Documentation & Discoverability

If a component isn't documented, it effectively doesn't exist. Excellent documentation is the bridge between the design system and application developers.

*   **Interactive Playgrounds:**
    *   Deploy an interactive component explorer for the team.
> [!TIP]
> **Histoire** is built natively for Vite/Vue ecosystems and often performs faster and integrates better with Nuxt than Storybook. Alternatively, Nuxt Content is excellent for building markdown-driven documentation sites.
*   **In-Code Type Definitions (TypeScript):**
    *   Define strict prop types using TypeScript (`defineProps<{ ... }>()`).
    *   Use JSDoc comments on props and emits. Modern IDEs (VS Code, WebStorm) will display these comments when a developer hovers over the component in their template, drastically reducing context-switching.

---

## 5. Nuxt-Specific Architectural Considerations

When distributing the component library across multiple Nuxt applications:

*   **Nuxt Layers (`extends`):** Ideal for monorepos and internal tools. A Nuxt layer can export components, composables, and Tailwind configurations effortlessly without a complex build step.
*   **Nuxt Modules:** Best for public open-source libraries or strict enterprise distribution where you need fine-grained control over what gets registered and injected into the user's Nuxt runtime.
*   **Auto-imports Prefixing:**
    *   Utilize Nuxt's `components` directory auto-importing. Use standard prefixes (e.g., `<UiButton>`, `<BaseInput>`) to distinguish library components from app-specific components.
