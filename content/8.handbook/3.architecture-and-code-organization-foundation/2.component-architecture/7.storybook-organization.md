# Storybook Organization & Best Practices for Large Vue/Nuxt Applications

Implementing Storybook in a large-scale Vue 3 and Nuxt 3 environment requires a well-structured component architecture, consistent documentation habits, and robust configuration.

## 1. Folder Structure: Colocation vs. Separation

For a large codebase, **colocating** your stories and tests alongside the components is the recommended best practice.

### Recommended: Colocation (Alongside Components)
Group everything related to a component in its own directory:
```text
components/
└── BaseButton/
    ├── BaseButton.vue        # Component code
    ├── BaseButton.stories.ts # Storybook file
    ├── BaseButton.test.ts    # Unit tests
    └── index.ts              # (Optional) Export for simpler imports
```
**Benefits:**
- **Discoverability:** Developers immediately see that a story and tests exist for the component.
- **Maintainability:** When renaming or deleting a component, all related files are handled together.
- **Scalability:** The `components/` directory inherently acts as a self-contained module for each UI element.

### Discouraged for Large Apps: Separate `stories/` Directory
While keeping a dedicated `stories/` or `/.storybook/stories/` folder separates production code from documentation, it creates friction as the app scales. It leads to mirroring folder structures, making file navigation cumbersome.

## 2. Structuring the Storybook Sidebar (Hierarchy)

In a massive Vue/Nuxt application, an unorganized Storybook sidebar becomes unusable. Use the `title` property in the default export of your `.stories.ts` files to create a structured hierarchy.

### Categorization Strategies
**1. Atomic Design** (Recommended for purely UI-driven codebases):
- `Design System/Atoms/Button`
- `Design System/Molecules/SearchBar`
- `Design System/Organisms/Header`

**2. Domain-Driven Design** (Recommended for feature-heavy Nuxt apps):
- `Common/UI/Button`
- `Common/Form/Input`
- `Features/Authentication/LoginForm`
- `Features/Checkout/PaymentSummary`

**Example:**
```typescript
// BaseButton.stories.ts
import type { Meta } from '@storybook/vue3';
import BaseButton from './BaseButton.vue';

const meta: Meta<typeof BaseButton> = {
  title: 'Design System/Atoms/BaseButton',
  component: BaseButton,
};
export default meta;
```

## 3. Documenting Components Effectively

Effective documentation ensures that other developers know how to consume your components without reading the source code.

### 1. Autodocs
Enable automatic documentation page generation. Storybook extracts props, events, and slots to build a dynamic API table.
```typescript
const meta: Meta<typeof BaseButton> = {
  title: 'Design System/Atoms/BaseButton',
  component: BaseButton,
  tags: ['autodocs'], // Generates a Docs page automatically
};
```

### 2. TSDoc/JSDoc Comments in Vue Components
With `@storybook/vue3`, comments placed directly above your props in `defineProps` (or in standard script tags) are parsed into the Storybook ArgsTable.
```html
<script setup lang="ts">
interface Props {
  /**
   * The variant of the button.
   * @default 'primary'
   */
  variant?: 'primary' | 'secondary' | 'danger';
  /** Disables the button interactions */
  disabled?: boolean;
}
withDefaults(defineProps<Props>(), { variant: 'primary' });
</script>
```

### 3. MDX for Rich Documentation
For complex components or design system guidelines (like Typography or Colors), use `.mdx` files. This allows you to mix Markdown with embedded Storybook stories.

## 4. Best Practices for Writing Stories

### Use Component Story Format (CSF) 3.0
CSF 3.0 utilizes an object-based approach rather than functions, making stories more concise.
```typescript
import type { Meta, StoryObj } from '@storybook/vue3';
import BaseButton from './BaseButton.vue';

const meta: Meta<typeof BaseButton> = {
  title: 'Design System/Atoms/BaseButton',
  component: BaseButton,
  tags: ['autodocs'],
};
export default meta;

type Story = StoryObj<typeof BaseButton>;

export const Primary: Story = {
  args: {
    variant: 'primary',
    label: 'Click Me',
  },
};
```

### Use `args` Instead of Hardcoded Templates
Avoid rewriting `<template>` boilerplate in your stories unless absolutely necessary (e.g., demonstrating scoped slots). Define state via `args` so users can interact with them in the Storybook Controls panel.

### Story Decorators
If a component needs specific margins, a Vue I18n provider, or Pinia store initialization, use Decorators at the story or meta level.
```typescript
const meta = {
  component: BaseButton,
  decorators: [
    (story) => ({
      components: { story },
      template: '<div style="margin: 3em;"><story /></div>',
    }),
  ],
};
```

## 5. Integrating with Nuxt 3

Nuxt 3 relies heavily on auto-imports (components, composables, utilities). Storybook runs its own Vite/Webpack server and is traditionally oblivious to Nuxt's magic.

### Best Approach: `@nuxtjs/storybook`
Use the official Nuxt Storybook module (`@nuxtjs/storybook`) to seamlessly bridge Nuxt's environment (auto-imports, plugins) into Storybook.

If configuring manually with `@storybook/vue3` (Vite builder):
1. **Unplugin Auto Import:** Use `unplugin-auto-import/vite` and `unplugin-vue-components/vite` inside your `.storybook/main.ts` Vite configuration to replicate Nuxt's auto-import behavior.
2. **Setup Global Plugins:** If you rely on Vue plugins like Pinia or Vue I18n, initialize them in `.storybook/preview.ts`.
```typescript
// .storybook/preview.ts
import { setup } from '@storybook/vue3';
import { createPinia } from 'pinia';

const pinia = createPinia();

setup((app) => {
  app.use(pinia);
});
```
