# Component Architecture: Smart vs. Dumb Components in Vue & Nuxt

In modern frontend component architecture, especially within large Vue and Nuxt applications, separating concerns is crucial for maintainability, testability, and reusability. A foundational pattern for achieving this is the **Smart vs. Dumb Component** architecture (also known as Container vs. Presentational components).

## 1. Dumb Components (Presentational / Pure)

Dumb components are solely concerned with **how things look**. They are the building blocks of your UI.

### Characteristics:
- **Stateless (mostly):** They rarely have their own local state (except occasionally for UI state like a toggle or hover state).
- **No External Dependencies:** They do not know about Pinia stores, Vue Router, Vuex, or API endpoints.
- **Data via Props:** All data they need to display is passed down to them via `props`.
- **Actions via Emits:** Any user interaction (clicks, form submissions) is bubbled up to the parent using `emits`.
- **Highly Reusable:** Because they are decoupled from business logic, they can be reused anywhere in the app (e.g., UI libraries like Vuetify, Tailwind UI, or custom base components).

### Vue Example (Dumb Component):
```vue
<!-- components/ui/BaseButton.vue -->
<template>
  <button 
    :class="['btn', variantClass]" 
    :disabled="isLoading"
    @click="$emit('click')"
  >
    <span v-if="isLoading">Loading...</span>
    <slot v-else />
  </button>
</template>

<script setup>
import { computed } from 'vue'

const props = defineProps({
  variant: { type: String, default: 'primary' },
  isLoading: { type: Boolean, default: false }
})

defineEmits(['click'])

const variantClass = computed(() => `btn-${props.variant}`)
</script>
```

## 2. Smart Components (Container / Connected / Stateful)

Smart components are concerned with **how things work**. They act as the glue between the application's data layer and the presentation layer (dumb components).

### Characteristics:
- **Stateful:** They hold or manage application state.
- **Connected:** They interact with Pinia stores, Vue Router, or external APIs (e.g., using Nuxt's `useFetch`).
- **Pass Data Down:** They fetch data and compute values to pass down to dumb components as `props`.
- **Handle Logic:** They listen to `emits` from dumb components and perform business logic (e.g., saving to a database, dispatching store actions).
- **Low Reusability:** They are usually tightly coupled to a specific feature or page route.

### Nuxt Example (Smart Component):
```vue
<!-- pages/users/index.vue (Nuxt Page acting as a Smart Component) -->
<template>
  <div class="user-dashboard">
    <h1>User Directory</h1>
    
    <!-- Using a Dumb Component to show errors -->
    <ErrorMessage v-if="error" :message="error.message" />
    
    <!-- Using a Dumb Component for presentation -->
    <UserList 
      v-else
      :users="users" 
      @delete-user="handleDeleteUser" 
    />
  </div>
</template>

<script setup>
// 1. Connection to Nuxt Data Fetching
const { data: users, error, refresh } = await useFetch('/api/users')

// 2. Connection to global store (Pinia)
const userStore = useUserStore()

// 3. Business logic handling events from dumb components
const handleDeleteUser = async (userId) => {
  try {
    await userStore.deleteUser(userId)
    await refresh() // Refresh Nuxt data
  } catch (e) {
    console.error("Failed to delete user", e)
  }
}
</script>
```

## 3. When to Use Which in Vue/Nuxt

- **Use Dumb Components for:** UI kits (buttons, inputs, modals), layout components (cards, grids), and generic item displays (user avatar, list items). Put these in a generic `components/ui/` or `components/common/` directory.
- **Use Smart Components for:** Page-level components (`pages/` in Nuxt), feature-specific wrappers (e.g., `UserProfileContainer.vue`), and complex forms that need direct store access.

### Nuxt Context:
In Nuxt, your **Pages (`/pages`)** are inherently Smart Components. Your **Layouts (`/layouts`)** are also typically smart, acting as wrappers. 
Your standard `components/` directory should ideally contain a mix, but grouping them logically is highly recommended:
- `components/features/` -> Smart components that bundle logic for a specific domain.
- `components/ui/` -> Purely dumb/presentational components.

## 4. The Impact of Vue 3 Composition API

The introduction of the Composition API (and Nuxt 3 auto-imported composables) has evolved this pattern. 

In Vue 2, Smart components were heavily cluttered with Vuex and lifecycle logic. Now, complex business logic can be extracted into **Composables** (`useUser.js`, `useCart.js`). 

**Does this make the Smart/Dumb pattern obsolete?**
No. However, it shifts *where* the "smarts" live. A component might now visually look like a Dumb component, but internally it simply calls a composable `const { users } = useUsers()`. 
* **Best Practice:** Even with composables, it is best to keep core UI components (like a `UserCard`) purely prop-driven (Dumb). This ensures they can be tested in isolation (e.g., using Storybook or Vue Test Utils) without having to mock composables or Nuxt contexts.

## 5. Architectural Best Practices for Large Vue/Nuxt Apps

1. **Maximize Pure UI Components:** The more dumb components you have, the easier it is to maintain, scale, and test your app visually.
2. **Isolate Side Effects:** Keep side effects (API calls, router pushes) restricted to Smart Components or dedicated Composables.
3. **Prop Validation:** Dumb components should strictly define and validate their props to establish a clear API contract.
4. **Avoid Deep Prop Drilling:** If a Smart Component needs to pass data through many layers of Dumb components, consider using Vue's `provide/inject` or a Pinia store to prevent excessive prop drilling, rather than polluting intermediate dumb components.
5. **Colocate Feature Logic:** In large Nuxt apps, consider a modular or feature-based structure where a domain feature has its own smart containers and dumb components collocated together.
