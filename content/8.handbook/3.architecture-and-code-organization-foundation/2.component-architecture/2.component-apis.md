# Component APIs in Vue & Nuxt: Props, Slots, and Compound Components

Designing robust component APIs is critical for maintainability, reusability, and developer experience in large-scale Vue and Nuxt applications. A well-designed component should be easy to use, hard to misuse, and flexible enough to accommodate changing requirements without becoming a monolithic "God component."

This guide explores three key mechanisms for building powerful component APIs: Props, Slots, and the Compound Component Pattern.

---

## 1. Props: Data & Configuration
Props are the primary way to pass data into a component. In large codebases, strict prop definitions act as documentation and a contract between components.

### Best Practices
* **Strict Typing & Validation**: Always use TypeScript (`defineProps<{ ... }>()`) to enforce prop shapes at compile time, reducing runtime errors.
* **Keep it Flat**: Avoid passing large, deeply nested objects if only a few primitive values are needed. Flat props are easier to track and optimize for reactivity.
* **Avoid Prop Bloat**: If a component has 15+ props to handle every single visual variation, it's a code smell. Consider breaking it down, using slots, or applying the Compound Component pattern instead.
* **One-Way Data Flow**: Never mutate props directly. Use `emit` to notify the parent, or `defineModel` in Vue 3.4+ for seamless two-way bindings.

### Example: Robust Prop Definition
```vue
<!-- BaseButton.vue -->
<script setup lang="ts">
import { computed } from 'vue'

type ButtonVariant = 'primary' | 'secondary' | 'danger';

const props = withDefaults(defineProps<{
  variant?: ButtonVariant;
  isDisabled?: boolean;
  isLoading?: boolean;
}>(), {
  variant: 'primary',
  isDisabled: false,
  isLoading: false
});

const classes = computed(() => ['btn', `btn-${props.variant}`]);
</script>

<template>
  <button :class="classes" :disabled="isDisabled || isLoading">
    <span v-if="isLoading" class="spinner" />
    <slot v-else />
  </button>
</template>
```

## 2. Slots: Content & Inversion of Control
While props are great for data, slots are designed for DOM content. Relying on props to pass HTML strings or configuration for rendering child elements often leads to rigid and hard-to-maintain components. Slots provide "Inversion of Control," allowing the parent to dictate the UI while the child component handles the layout.

### Named Slots
Use named slots when a component has multiple distinct content areas (e.g., a modal with `<slot name="header">`, `<slot name="body">`, and `<slot name="footer">`).

### Scoped Slots
Scoped slots allow the child component to pass data *back up* to the parent's slot template. This is incredibly powerful for building highly reusable structural components like Data Tables, Listboxes, or Autocompletes.

### Example: Scoped & Named Slots in a List Component
```vue
<!-- DataList.vue -->
<script setup lang="ts">
// We take in the raw data
defineProps<{ items: any[] }>();
</script>

<template>
  <ul class="data-list">
    <li v-for="item in items" :key="item.id">
      <!-- Expose 'item' data back up to the parent via scoped slot -->
      <slot name="item" :item="item">
        <!-- Fallback content if no slot is provided -->
        {{ item.name }}
      </slot>
    </li>
  </ul>
</template>
```

**Usage in Parent:**
```vue
<template>
  <DataList :items="users">
    <!-- The parent completely controls how each item is rendered! -->
    <template #item="{ item }">
      <div class="user-card">
        <img :src="item.avatar" alt="Avatar" />
        <div>
          <h4>{{ item.name }}</h4>
          <span>Role: {{ item.role }}</span>
        </div>
      </div>
    </template>
  </DataList>
</template>
```

## 3. The Compound Component Pattern
As components grow in complexity (e.g., Tabs, Accordions, Complex Dropdowns), using just props and slots can become unwieldy. The parent has to manage too much state, or the child component requires dozens of configuration props to toggle features on and off.

The **Compound Component Pattern** solves this by breaking a complex component into multiple smaller, semantic components that work together and share state implicitly. In Vue 3, this is elegantly achieved using the `provide` and `inject` APIs.

### Example: Compound Accordion Component
Instead of passing a complex JSON object to an `<Accordion :items="myItems">` component, we compose it semantically, allowing complete UI flexibility.

**Parent Usage:**
```vue
<Accordion>
  <AccordionItem value="item-1">
    <AccordionTrigger>What is Nuxt?</AccordionTrigger>
    <AccordionContent>Nuxt is a meta-framework for Vue...</AccordionContent>
  </AccordionItem>
  <AccordionItem value="item-2">
    <AccordionTrigger>What is Vue?</AccordionTrigger>
    <AccordionContent>Vue is a progressive framework...</AccordionContent>
  </AccordionItem>
</Accordion>
```

#### Step 1: The Root Component (Provider)
```vue
<!-- Accordion.vue -->
<script setup lang="ts">
import { provide, ref } from 'vue'

const activeItem = ref<string | null>(null)

const toggleItem = (value: string) => {
  activeItem.value = activeItem.value === value ? null : value
}

// Share state and methods with all descendants transparently
provide('accordionContext', {
  activeItem,
  toggleItem
})
</script>

<template>
  <div class="accordion">
    <slot />
  </div>
</template>
```

#### Step 2: The Item Component
```vue
<!-- AccordionItem.vue -->
<script setup lang="ts">
import { provide } from 'vue'

const props = defineProps<{ value: string }>()

// Pass the specific item value down to its specific Trigger and Content
provide('accordionItemContext', props.value)
</script>

<template>
  <div class="accordion-item">
    <slot />
  </div>
</template>
```

#### Step 3: The Trigger (Consumer)
```vue
<!-- AccordionTrigger.vue -->
<script setup lang="ts">
import { inject, computed } from 'vue'

const { activeItem, toggleItem } = inject('accordionContext') as any
const itemValue = inject('accordionItemContext') as string

const isOpen = computed(() => activeItem.value === itemValue)
</script>

<template>
  <button 
    class="accordion-trigger" 
    :aria-expanded="isOpen"
    @click="toggleItem(itemValue)"
  >
    <slot />
    <!-- Child components can react to the shared state internally -->
    <span class="icon" :class="{ 'rotate-180': isOpen }">▼</span>
  </button>
</template>
```

#### Step 4: The Content (Consumer)
```vue
<!-- AccordionContent.vue -->
<script setup lang="ts">
import { inject, computed } from 'vue'

const { activeItem } = inject('accordionContext') as any
const itemValue = inject('accordionItemContext') as string

const isOpen = computed(() => activeItem.value === itemValue)
</script>

<template>
  <div v-show="isOpen" class="accordion-content">
    <slot />
  </div>
</template>
```

### Benefits of Compound Components
1. **Highly Flexible UI**: The parent can easily add icons, change structural wrappers, or reorder elements (e.g., placing the Trigger below the Content) without touching the child's internal code.
2. **Clean API**: Avoids prop drilling and massive configuration object arrays.
3. **Separation of Concerns**: The parent handles the layout and content markup, while the compound components seamlessly coordinate the complex state (accessibility, active states, toggling logic) under the hood.

---

## Summary
* Use **Props** for primitive data and configuration. Keep them strictly typed and flat to avoid API bloat.
* Use **Slots** (named and scoped) to empower the parent component to control the UI and layout, offloading HTML-rendering logic from the child.
* Use **Compound Components** (`provide`/`inject`) for complex, multi-part UI elements. This encapsulates state logic while providing a clean, declarative, HTML-like API to the consumer.
