# Monorepo Structure & Workspace Packages for Scalable Nuxt/Vue Apps

## 1. What is a Monorepo?
A **monorepo** (monolithic repository) is a single version control repository containing multiple independent applications, shared packages, and services. Instead of managing dozens of isolated repositories (polyrepos) for different apps, UI libraries, and utility scripts, a monorepo groups them together while maintaining logical separation.

In the JavaScript/TypeScript ecosystem, this is implemented using **workspace packages**. Workspaces (natively supported by pnpm, npm, and Yarn) allow you to link local packages together. If your main `Web App` depends on your custom `UI Package`, the package manager uses a local symlink. This allows you to import local packages as if they were downloaded from the npm registry, without having to actually publish them.

## 2. Benefits for Scalable Apps
For large-scale architectures, monorepos solve several fundamental growing pains:
*   **Seamless Code Sharing:** Share Vue components, composables (`useAuth`, `useFetch`), TypeScript interfaces, and utility functions effortlessly across multiple applications (e.g., a customer-facing app, an internal admin dashboard, and a documentation site).
*   **Atomic Changes:** Update a shared UI component and all the applications that depend on it in a single commit and Pull Request. This eliminates the "version bumping dance" across multiple repositories and prevents version drift.
*   **Unified Tooling & DX:** Maintain a single source of truth for ESLint, Prettier, TypeScript configs, and CI/CD pipelines. This ensures coding standards are enforced universally across all teams.
*   **Optimized Dependency Management:** Tools like `pnpm` will hoist shared dependencies to the root, dramatically reducing `node_modules` size, preventing phantom dependencies, and speeding up installation times.

## 3. Integration with Nuxt and Vue
The modern Vue and Nuxt ecosystems are highly optimized for monorepos:
*   **Nuxt Layers (Extends):** Nuxt 3 introduced the `extends` feature, which is perfect for monorepos. You can create a `packages/core` Nuxt layer containing your base `nuxt.config.ts`, standard Nuxt modules, core Vue components, and shared server routes (`server/api`). Your main apps (`apps/web`, `apps/admin`) can then extend this base layer, keeping their individual codebases extremely lean.
*   **pnpm Workspaces:** Widely considered the best package manager for Vue/Nuxt monorepos. Its strict dependency resolution and native workspace configuration (`pnpm-workspace.yaml`) provide stability in massive architectures.
*   **Turborepo:** A high-performance task orchestrator for JavaScript. It integrates perfectly with Nuxt by caching the output of your build steps (`.output`, `.nuxt`) and test runs. If you change a button in `packages/ui`, Turborepo knows exactly which Nuxt apps depend on it and need to be rebuilt, and which apps can skip the build and be restored from the cache.
*   **Unbuilt Workspaces:** With Nuxt/Vite, you often don't even need to pre-build your shared Vue component packages. You can export raw `.vue` and `.ts` files from your workspace packages, and Nuxt's Vite bundler will transpile them directly during the app's dev/build process.

## 4. Example Scalable Folder Structure
Here is an industry-standard, scalable monorepo structure using **pnpm + Turborepo + Nuxt 3**:

```text
my-nuxt-monorepo/
├── package.json               # Root dependencies (Turborepo, cross-env, etc.)
├── pnpm-workspace.yaml        # Defines workspaces: apps/*, packages/*
├── turbo.json                 # Turborepo task orchestration and caching rules
├── apps/
│   ├── web/                   # Customer-facing Nuxt 3 App
│   │   ├── nuxt.config.ts     # Extends @my-project/core layer
│   │   └── package.json       # Dependencies: @my-project/ui, @my-project/core
│   └── admin/                 # Internal Nuxt 3 Admin Dashboard
│       ├── nuxt.config.ts
│       └── package.json
└── packages/
    ├── core-layer/            # Shared Nuxt Layer (Base Config, Auth, Plugins)
    │   ├── nuxt.config.ts
    │   └── package.json       # Name: @my-project/core
    ├── ui/                    # Shared Vue 3 Component Library
    │   ├── components/
    │   │   └── BaseButton.vue
    │   ├── index.ts           # Exports components
    │   └── package.json       # Name: @my-project/ui
    ├── composables/           # Shared Vue Composables
    │   ├── useApi.ts
    │   └── package.json       # Name: @my-project/composables
    ├── utils/                 # Shared pure TS utilities (formatting, math, etc.)
    │   ├── index.ts
    │   └── package.json       # Name: @my-project/utils
    └── config/                # Shared Tooling Configurations
        ├── eslint-config/     # Name: @my-project/eslint-config
        └── tsconfig/          # Name: @my-project/tsconfig
```

### Key Integration Points:

**1. `pnpm-workspace.yaml`:**
```yaml
packages:
  - 'apps/*'
  - 'packages/*'
```

**2. Consuming Shared Vue UI (`apps/web/package.json`):**
Using the `workspace:*` protocol ensures the app always uses the local package version.
```json
{
  "dependencies": {
    "@my-project/ui": "workspace:*",
    "@my-project/utils": "workspace:*"
  }
}
```

**3. Sharing Nuxt Layers (`apps/web/nuxt.config.ts`):**
```typescript
export default defineNuxtConfig({
  extends: ['@my-project/core'],
  
  // App-specific overrides can be added here
  modules: ['@nuxtjs/tailwindcss'],
})
```

**4. Turborepo Configuration (`turbo.json`):**
```json
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".output/**", ".nuxt/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```
