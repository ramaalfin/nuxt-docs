# Path Aliases Configuration for Scalable Nuxt/Vue Apps

## What is a Path Alias?
In modern frontend development, as a project grows, the folder structure becomes deeply nested. Importing files across this structure often leads to "relative path hell"—long, confusing, and fragile import statements like `import { formatDate } from '../../../utils/date'`. 

A **Path Alias** is a configuration feature that maps a specific prefix (like `@` or `~`) to an absolute path in your project's directory structure. With an alias, the previous import becomes much cleaner: `import { formatDate } from '@/utils/date'`.

## Benefits for Large Nuxt & Vue Apps
When building scalable architectures (like Feature-Driven Development, Domain-Driven Design, or standard modular patterns), path aliases are a foundational organization tool:

1. **Simplified Refactoring**: If you move a component or a composable to a different directory, you do not need to update the relative paths of all its dependencies. Since aliases resolve from the root/src directory, the import statement remains the same regardless of where the calling file is located.
2. **Cleaner, More Readable Code**: `import Header from '@/components/layout/Header.vue'` is instantly readable compared to `import Header from '../../../../components/layout/Header.vue'`.
3. **Consistency**: Enforces a standard way for teams to import resources, eliminating the guesswork of calculating how many `../` hops are needed.
4. **IDE & TypeScript Integration**: Modern IDEs (like VS Code with Volar) and TypeScript inherently understand these aliases once configured, providing full autocomplete, type checking, and "Go to Definition" capabilities.

## Default Path Aliases in Nuxt 3
Nuxt 3 provides excellent built-in support for path aliases out of the box, meaning you often don't need to configure anything for standard use cases.

- `~` or `@`: Points to the `srcDir` (usually the root directory unless explicitly changed).
- `~~` or `@@`: Points to the `rootDir` of the Nuxt project.
- `#app`: Points to the Nuxt app context.
- `#imports`: Points to the auto-imported utilities and composables.
- `#components`: Points to the auto-imported Vue components.

*Note: In most standard Nuxt 3 setups, `srcDir` and `rootDir` are the same, making `@` and `@@` interchangeable.*

## How to Configure Custom Path Aliases in Nuxt 3

If you are structuring a very large application, you might want domain-specific or layer-specific aliases (e.g., `@core`, `@features`, `@design-system`). 

In Nuxt 3, you **do not** need to manually configure `tsconfig.json` or Vite/Webpack for aliases. You configure them directly in `nuxt.config.ts`. Nuxt automatically generates the underlying Vite/Webpack configurations and updates `.nuxt/tsconfig.json` so that TypeScript and your IDE are aware of them.

### Step 1: Define Aliases in `nuxt.config.ts`

```typescript
import { fileURLToPath } from 'node:url'
import { dirname, join } from 'node:path'

const currentDir = dirname(fileURLToPath(import.meta.url))

export default defineNuxtConfig({
  alias: {
    // Custom domain-specific aliases for a scalable architecture
    '@core': join(currentDir, './core'),
    '@features': join(currentDir, './features'),
    '@design': join(currentDir, './design-system'),
    '@types': join(currentDir, './types')
  }
})
```
*Note: Using Node's `path` and `fileURLToPath` ensures cross-platform compatibility, though Nuxt can often resolve plain relative string paths like `'./core'` as well.*

### Step 2: (Optional) `tsconfig.json` Setup
Because Nuxt dynamically generates a TS config at `.nuxt/tsconfig.json` that includes your `alias` settings from `nuxt.config.ts`, your root `tsconfig.json` only needs to extend it. 

Your root `tsconfig.json` should look like this (this is the default Nuxt setup):
```json
{
  "extends": "./.nuxt/tsconfig.json"
}
```
**Do not manually add `paths` to the root `tsconfig.json`** unless absolutely necessary, as they might conflict with Nuxt's auto-generated paths. Let Nuxt handle it via `nuxt.config.ts`.

## Example: Using Custom Aliases in a Scalable Folder Structure

Imagine a scalable Nuxt 3 project organized by features:

```text
my-nuxt-app/
├── core/
│   ├── services/
│   │   └── api.ts
├── features/
│   ├── auth/
│   │   ├── components/
│   │   │   └── LoginForm.vue
│   │   └── composables/
│   │       └── useAuth.ts
├── design-system/
│   └── Button.vue
├── pages/
│   └── login.vue
├── nuxt.config.ts
└── tsconfig.json
```

**Inside `features/auth/components/LoginForm.vue`**:
Instead of writing painful relative imports...
```vue
<script setup>
// Bad: Relative Path
import { useAuth } from '../composables/useAuth'
import { api } from '../../../core/services/api'
import Button from '../../../design-system/Button.vue'
</script>
```

...you can use your custom aliases configured in `nuxt.config.ts`:
```vue
<script setup>
// Good: Path Aliases
import { useAuth } from '@features/auth/composables/useAuth'
import { api } from '@core/services/api'
import Button from '@design/Button.vue'
</script>
```

This guarantees that if `LoginForm.vue` is ever refactored into a deeper subfolder (e.g., `features/auth/components/ui/forms/LoginForm.vue`), **none of the imports need to change**.
