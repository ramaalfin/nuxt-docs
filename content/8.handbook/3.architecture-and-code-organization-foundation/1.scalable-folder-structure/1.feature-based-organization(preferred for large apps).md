# Feature-Based Organization in Vue & Nuxt Applications

## What is Feature-Based Organization?
Feature-based organization (also known as Domain-Driven or Modular architecture) is an architectural pattern where source code is grouped by its **domain or business feature** rather than its underlying technical file type. 

Instead of spreading the parts of a feature (like Authentication, Checkout, or User Profile) across global directories for components, composables, and state, all related files are encapsulated within a single, dedicated feature folder.

## Feature-Based vs. Traditional Type-Based Organization

### Traditional (Type-Based) Organization
Most frameworks, including Vue and Nuxt by default, encourage grouping files by type. 
* **Structure**: `components/`, `composables/`, `store/`, `services/`, `types/`
* **The Problem**: As an application scales, these directories become massive. If a developer needs to modify the "Checkout" feature, they have to jump between `components/CheckoutCard.vue`, `composables/useCheckout.ts`, `store/checkout.ts`, and `services/checkoutApi.ts`. This drastically reduces cohesion, makes refactoring difficult, and adds cognitive load.

### Feature-Based Organization
Files are grouped by their business capability or domain.
* **Structure**: `features/checkout/`, `features/auth/`, `features/products/`
* **The Solution**: All code related to "Checkout" lives in one place. You achieve high cohesion (related code is together) and low coupling (features interact through strict interfaces rather than internal spaghetti).

## Key Benefits for Large Applications
1. **Scalability & Discoverability**: When onboarding new developers or navigating large codebases, finding where a feature lives takes seconds. The folder structure maps directly to the product requirements.
2. **Encapsulation**: Features can have their own private components, utilities, and types that aren't exposed to the rest of the application, effectively preventing global namespace pollution.
3. **Easier Refactoring & Extraction**: If a feature needs to be extracted into a separate micro-frontend, Nuxt Layer, or npm package, you simply move the entire feature folder.
4. **Fewer Merge Conflicts**: In large teams, different squads can own different feature directories without stepping on each other's toes inside shared global folders.

## Example Folder Structure for Vue/Nuxt

Here is a scalable blueprint. The app is divided into `app` (core setup), `shared` (global utilities and generic UI), `features` (business domains), and `pages` (routing).

```text
src/ (or root)
├── app/                      # Application-wide setup, plugins, layouts, and global styles
│   ├── layouts/
│   ├── plugins/
│   └── router/               # (If Vue; Nuxt uses pages/ automatically)
├── shared/                   # Global, reusable, "dumb" UI components and generic utilities
│   ├── components/           # e.g., BaseButton.vue, Modal.vue
│   ├── composables/          # e.g., useClickOutside.ts
│   ├── utils/                # e.g., date-formatters.ts
│   └── types/                # Generic TS types
├── features/                 # The core of the architecture (Business Domains)
│   ├── auth/                 # Authentication Feature
│   │   ├── components/       # Feature-specific UI (LoginForm.vue, AuthProvider.vue)
│   │   ├── composables/      # Feature logic (useAuth.ts)
│   │   ├── store/            # State management (auth.store.ts)
│   │   ├── services/         # API integrations (auth.api.ts)
│   │   ├── types/            # Domain interfaces (User, AuthResponse)
│   │   └── index.ts          # Public API export (what this feature exposes to others)
│   ├── product-catalog/      # Another Feature
│   │   ├── components/
│   │   └── ...
├── pages/                    # Routing (Vue Router / Nuxt auto-routing)
│   ├── login.vue             # Thin file: Imports <LoginForm /> from features/auth
│   ├── index.vue
│   └── products/
│       └── [id].vue
└── nuxt.config.ts            # Application configuration
```

## Integrating with Nuxt 3

Nuxt 3 is heavily reliant on convention over configuration (auto-imports based on standard directories like `components/`). To adopt a feature-based structure in Nuxt, you have three primary approaches:

### 1. The Nuxt Native Way (Nested Types)
You keep Nuxt's root directories but create feature sub-folders inside them. 
* `components/auth/LoginForm.vue`
* `composables/auth/useAuth.ts`
* *Pros*: Works out of the box with zero configuration.
* *Cons*: Code is still physically separated by type. It's a half-measure for true feature-based architecture.

### 2. Custom Directory Configuration (Recommended)
You create a custom `features/` directory and explicitly tell Nuxt to auto-import from it. You can configure this in your `nuxt.config.ts`:

```typescript
export default defineNuxtConfig({
  components: [
    // Standard shared components
    { path: '~/shared/components', pathPrefix: false },
    // Feature-specific components
    { path: '~/features', pattern: '*/components/**', pathPrefix: false }
  ],
  imports: {
    dirs: [
      'shared/composables/**',
      'features/**/composables/**',
      'features/**/store/**'
    ]
  }
})
```
*Note: In very large apps, some teams prefer to disable auto-imports entirely for features and rely on explicit imports from `features/feature-name/index.ts` to strictly control architecture boundaries.*

### 3. Nuxt Layers (For Enterprise Scale)
For massive applications or mono-repos, Nuxt 3 supports [Layers](https://nuxt.com/docs/getting-started/layers). You can treat each feature as its own Nuxt Layer with its own `components/`, `composables/`, and `pages/` directories, and simply extend them in your main application.

## Best Practices
* **Keep Pages Thin**: Files in the `pages/` directory should only handle routing, fetching initial data, and composing Feature components. They should contain almost zero business logic.
* **Strict Boundaries**: A feature should only interact with another feature through explicit exports (via an `index.ts` barrel file). Feature A should not deeply import from `features/FeatureB/components/InnerComponent.vue`.
* **Keep `shared/` Clean**: Only put code in `shared/` if it is truly domain-agnostic (like a generic button component or a generic text-truncation utility).
