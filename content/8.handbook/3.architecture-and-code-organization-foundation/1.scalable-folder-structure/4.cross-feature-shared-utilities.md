# Cross-Feature Shared Utilities in Nuxt/Vue Architecture

## 1. What are Cross-Feature Shared Utilities?
In a large-scale Nuxt or Vue application, architecture often shifts towards modular, feature-based (or domain-based) structures (like Feature-Sliced Design or Domain-Driven Design). Instead of grouping files by type (e.g., all components together, all composables together), code is grouped by **feature** (e.g., `auth`, `cart`, `user-profile`).

**Cross-feature shared utilities** are the foundational, generic functions, composables, and helpers that do not belong to any specific feature but are used across multiple features. They form the "shared" or "core" layer of your application.

Examples include:
- Date and time formatters (e.g., `formatDate`)
- String manipulators (e.g., `capitalize`, `camelCase`)
- Generic API fetch wrappers or HTTP clients
- Validation rules (e.g., `isEmail`, `isRequired`)
- Browser storage wrappers (LocalStorage/SessionStorage)

## 2. When to Use: Shared vs. Feature-Specific Utilities

Understanding where a utility belongs is critical for maintaining a scalable codebase. 

### Feature-Specific Utilities
- **Scope:** Used only within a single feature/domain.
- **Location:** Reside inside the specific feature's folder (e.g., `features/cart/utils/calculateDiscount.ts`).
- **Characteristics:** Contains business logic specific to that domain. Modifying it shouldn't impact the rest of the application.

### Cross-Feature Shared Utilities
- **Scope:** Used by two or more features, or inherently generic.
- **Location:** Reside in a global `shared/` or `core/` folder (e.g., `shared/utils/date.ts`).
- **Characteristics:** Agnostic of business domains. They act like an internal library.

| Criteria | Feature-Specific Utility | Cross-Feature Shared Utility |
|----------|--------------------------|------------------------------|
| **Domain Logic** | High (e.g., calculateCartTotal) | None (e.g., formatCurrency) |
| **Dependencies** | Can depend on shared utilities | **Must not** depend on feature utilities |
| **Reusability** | Low (specific to a use-case) | High (generic, ubiquitous) |

## 3. How to Structure It in Nuxt/Vue

For a scalable architecture, adopting a layered structure prevents tight coupling and circular dependencies. A standard approach is using a `shared/` layer.

### Folder Structure Example
```text
src/ (or root in Nuxt)
 ├── features/              # Feature modules
 │    ├── auth/
 │    │    ├── components/
 │    │    ├── composables/
 │    │    └── utils/       # <-- Feature-specific utilities (e.g., tokenParser.ts)
 │    └── cart/
 │         ├── components/
 │         └── utils/       # <-- Feature-specific utilities (e.g., cartTotal.ts)
 │
 ├── shared/                # Cross-Feature Shared layer
 │    ├── components/       # UI Primitives (Button, Modal)
 │    ├── composables/      # Shared state/logic (useWindowSize, useStorage)
 │    └── utils/            # <-- Cross-Feature Shared Utilities
 │         ├── date.ts
 │         ├── string.ts
 │         ├── validators.ts
 │         └── index.ts
 │
 └── app/                   # App initialization, router, layouts
```

### Nuxt 3 Auto-Imports Configuration
Nuxt 3 automatically imports files from the `utils/` and `composables/` directories. If you adopt a modular `shared/` folder structure, you can configure Nuxt to auto-import your shared utilities by updating `nuxt.config.ts`:

```typescript
export default defineNuxtConfig({
  imports: {
    dirs: [
      // Auto-import shared utilities
      'shared/utils/**',
      'shared/composables/**',
      // If you also want to auto-import feature specific utils selectively:
      'features/*/utils/**'
    ]
  }
})
```

## 4. Examples in Practice

### Example 1: Generic Date Formatter (Shared Utility)
**Location:** `shared/utils/date.ts`
```typescript
import { format } from 'date-fns';

/**
 * Formats a date string into a readable format.
 * Generic and usable across any feature (Auth, Invoices, Profiles, etc.)
 */
export const formatDate = (date: string | Date, pattern = 'MMM dd, yyyy') => {
  return format(new Date(date), pattern);
};
```

### Example 2: API Client Wrapper (Shared Utility)
**Location:** `shared/utils/http.ts`
```typescript
/**
 * A shared wrapper around Nuxt's $fetch to handle global interceptors,
 * error tracking, and auth headers.
 */
export const useApiClient = (url: string, options = {}) => {
  return $fetch(url, {
    ...options,
    onRequest({ request, options }) {
      // Add global auth headers, etc.
    }
  });
};
```

### Example 3: Feature-Specific Utility (Contrasting Example)
**Location:** `features/cart/utils/cartCalculations.ts`
```typescript
import { formatCurrency } from '~/shared/utils/currency'; // Can import from shared!

/**
 * Domain-specific. ONLY belongs in the Cart feature.
 */
export const calculateDiscountedTotal = (items: CartItem[], discountCode: string) => {
  const total = items.reduce((sum, item) => sum + item.price, 0);
  const discount = applyDiscountLogic(total, discountCode);
  return formatCurrency(total - discount); // Uses shared utility
};
```

## 5. Best Practices & Rules

1. **Strict Dependency Rule (Downward Only):** Features can import from the `shared` layer, but the `shared` layer must **never** import from `features`. This prevents circular dependencies and maintains clear architectural boundaries.
2. **Pure Functions:** Shared utilities should ideally be pure functions (same input always equals same output) with no side effects, making them highly testable.
3. **Avoid the "Junk Drawer":** Don't just throw everything into a generic `utils/index.ts`. Organize shared utilities by their responsibility (e.g., `math.ts`, `object.ts`, `validation.ts`).
4. **Vue/Nuxt Agnosticism:** Where possible, standard utility functions should be written in pure TypeScript/JavaScript so they can be easily migrated or tested outside the Vue ecosystem. Leave Vue-specific reactivity (Refs, Computed) to `shared/composables/`.
