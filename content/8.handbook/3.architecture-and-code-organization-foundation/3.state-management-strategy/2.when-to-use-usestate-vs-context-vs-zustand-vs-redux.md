# State Management in Nuxt: Adapting React Concepts (useState, Context, Zustand, Redux)

When transitioning from React to Vue and Nuxt, state management paradigms shift slightly, but the underlying architectural decisions remain similar. Here is a comprehensive guide mapping React state management concepts to their Vue/Nuxt equivalents, along with when to use each.

---

## 1. Local State: `useState` (React) ➔ `ref` / `reactive` (Vue)

### React Concept
In React, `useState` (or `useReducer`) is used for localized component state that doesn't need to be shared widely across the application.

### Nuxt/Vue Equivalent
Vue 3's Composition API provides `ref()` (for primitives) and `reactive()` (for objects) to handle local state.

### Architectural Decision
**When to use:**
- UI states like dropdown visibility, modal open/close.
- Form inputs and validations localized to a single component.
- Component-specific data loading states (e.g., `isFetching`).

**Example in Nuxt:**
```vue
<script setup>
import { ref, reactive } from 'vue'

const isOpen = ref(false) // primitive
const userForm = reactive({ name: '', email: '' }) // object
</script>
```

---

## 2. Avoiding Prop Drilling: Context API (React) ➔ `provide` / `inject` (Vue)

### React Concept
The React Context API allows passing data deeply through the component tree without passing props manually at every level. It is notorious for causing unnecessary re-renders if the context value changes frequently.

### Nuxt/Vue Equivalent
Vue uses `provide()` in the parent component and `inject()` in the child/descendant components.

### Architectural Decision
**When to use:**
- Broad, application-wide settings that change infrequently (e.g., UI themes, locale/language preferences).
- Providing plugin instances or configuration objects to deeply nested components.

**When NOT to use:**
- Complex, rapidly changing state, as tracking reactivity across multiple decoupled components can become difficult to debug.

**Example in Nuxt:**
```vue
<!-- Parent.vue -->
<script setup>
import { provide, ref } from 'vue'
const theme = ref('dark')
provide('themeContext', theme)
</script>

<!-- DeepChild.vue -->
<script setup>
import { inject } from 'vue'
const theme = inject('themeContext')
</script>
```

## 3. Global Store: Zustand / Redux (React) ➔ Pinia (Vue/Nuxt)

### React Concept
- **Zustand:** A lightweight, unopinionated global state manager.
- **Redux:** A robust, strict Flux architecture for massive applications, heavily reliant on actions and reducers.

### Nuxt/Vue Equivalent
**Pinia** is the official state management library for Vue 3. It replaces Vuex and effectively combines the best of Zustand (simplicity, minimal boilerplate) and Redux (robustness, DevTools integration, SSR support). 

### Architectural Decision
**When to use:**
- Complex global state that must be accessed and mutated by disparate components across the app.
- Shared business logic, like Shopping Carts, User Authentication, or multi-step wizards.
- Whenever you need centralized state that integrates perfectly with Vue DevTools for time-travel debugging.

**Example in Nuxt (Pinia):**
```ts
// stores/cart.ts
import { defineStore } from 'pinia'

export const useCartStore = defineStore('cart', {
  state: () => ({ items: [], discount: 0 }),
  getters: {
    total: (state) => state.items.length,
  },
  actions: {
    addItem(item) {
      this.items.push(item)
    }
  }
})
```

## 4. Nuxt Specific: SSR-Friendly Shared State ➔ `useState` (Nuxt)

### Nuxt Concept
Nuxt introduces a composable named `useState`. *Do not confuse this with React's `useState`.* In Nuxt, `useState` provides an SSR-friendly shared state across components.

### Architectural Decision
**When to use:**
- When fetching data on the server that needs to be passed securely to the client to avoid **hydration mismatches** or **double-fetching**.
- Storing temporary state that should survive across page navigations without needing the heavy machinery of Pinia.

**How it works:**
Nuxt `useState` takes a unique string key. The server populates the state during the SSR pass, serializes it, and sends it to the client. The client then hydrates it seamlessly.

**Example in Nuxt:**
```vue
<script setup>
// Shared across all components that use the 'counter' key.
// Survives SSR hydration seamlessly.
const counter = useState('counter', () => Math.round(Math.random() * 1000))
</script>
```

## Summary: Architectural Decision Matrix

| State Type | React Tool | Vue/Nuxt Tool | Use Case |
| :--- | :--- | :--- | :--- |
| **Component Local** | `useState` | `ref()` / `reactive()` | UI toggles, local form state. |
| **Dependency Injection** | Context API | `provide()` / `inject()` | Avoiding prop drilling for infrequent changes (Theming). |
| **Global/Complex** | Zustand / Redux | Pinia | Global user session, cart, complex cross-component data. |
| **SSR Shared State** | Next.js Server Props | Nuxt `useState()` | SSR data that must be passed to the client during hydration. |
