# Error Recovery Patterns for State Management in Large Vue/Nuxt Apps

In large-scale Vue and Nuxt applications, robust state management (typically via Pinia) is critical. When errors occur—whether due to network failures, server errors, or unexpected application states—recovering gracefully without breaking the user experience is paramount. This document outlines best practices and patterns for error recovery in Vue/Nuxt state management.

## 1. Localized Error Handling in Actions

The most fundamental pattern is handling errors directly within your store's actions. This allows the store to manage its own loading and error states.

### Pattern: Status-driven State
Instead of throwing errors unhandled, catch them and update a dedicated error state.

```typescript
// stores/useUserStore.ts
import { defineStore } from 'pinia'

export const useUserStore = defineStore('user', {
  state: () => ({
    data: null,
    status: 'idle', // 'idle' | 'loading' | 'success' | 'error'
    error: null as string | null,
  }),
  actions: {
    async fetchUser(id: string) {
      this.status = 'loading'
      this.error = null
      
      try {
        const response = await api.getUser(id)
        this.data = response.data
        this.status = 'success'
      } catch (err) {
        this.status = 'error'
        this.error = err instanceof Error ? err.message : 'Unknown error occurred'
        // Optionally rethrow if the component needs to know
      }
    },
    clearError() {
      this.error = null
      this.status = 'idle'
    }
  }
})
```

## 2. Global Action Interception via Pinia Plugins

In large apps, repeating try/catch blocks in every action can be tedious. You can use Pinia plugins to globally intercept action errors, report them to a telemetry service (like Sentry), or show global toast notifications.

### Pattern: Pinia Error Handler Plugin

```typescript
// plugins/pinia-error-handler.ts
import { PiniaPluginContext } from 'pinia'

export function piniaErrorHandler({ store, actionName, name }: PiniaPluginContext) {
  store.$onAction(({ name, store, args, after, onError }) => {
    // Shared variables or setup before action executes
    
    after((result) => {
      // Logic after successful execution
    })

    onError((error) => {
      console.error(`[Pinia Error] in store '${store.$id}' action '${name}':`, error)
      
      // Send to error tracking service
      // useTracking().captureException(error, { tags: { store: store.$id, action: name } })
      
      // Trigger global notification store
      const notificationStore = useNotificationStore()
      notificationStore.showError(`Action ${name} failed. Please try again.`)
    })
  })
}
```

## 3. Optimistic Updates with Rollback

For mutations (e.g., liking a post, updating a profile), optimistic updates make the UI feel snappy by updating the store before the API responds. If the API fails, you must rollback to the previous state.

### Pattern: State Snapshot and Revert

```typescript
// stores/useTodoStore.ts
import { defineStore } from 'pinia'

export const useTodoStore = defineStore('todo', {
  state: () => ({
    todos: [] as Todo[]
  }),
  actions: {
    async toggleTodo(id: string) {
      const todo = this.todos.find(t => t.id === id)
      if (!todo) return
      
      // 1. Snapshot previous state
      const previousState = todo.completed
      
      // 2. Optimistic update
      todo.completed = !todo.completed
      
      try {
        // 3. API Call
        await api.updateTodo(id, { completed: todo.completed })
      } catch (error) {
        // 4. Rollback on failure
        todo.completed = previousState
        useNotificationStore().showError('Failed to update todo. Reverted changes.')
      }
    }
  }
})
```

## 4. Resetting State on Irrecoverable Errors

Sometimes the state becomes corrupted or out-of-sync due to a sequence of errors. Pinia provides a built-in `$reset()` method to safely restore a store to its initial state.

### Pattern: The Clean Slate

```typescript
// Inside a component or error boundary
import { useAuthStore } from '@/stores/auth'

function handleCriticalAuthError() {
  const authStore = useAuthStore()
  
  // Wipe corrupted state
  authStore.$reset()
  
  // Redirect to login
  navigateTo('/login')
}
```

## 5. Nuxt-Specific Fetch and Error Recovery

Nuxt 3 provides composables like `useFetch` and `useAsyncData` which integrate tightly with the Vue ecosystem.

### Pattern: Composable Error State and Retry
When hydrating store data from Nuxt's `useFetch`, utilize its built-in `error` and `refresh` capabilities.

```vue
<script setup>
const userStore = useUserStore()

// fetch with useFetch for SSR support
const { data, error, refresh, status } = await useFetch('/api/user', {
  onResponse({ response }) {
    userStore.setUser(response._data)
  }
})

const handleRetry = () => {
  // Clear the Nuxt error state and retry the fetch
  clearError()
  refresh()
}
</script>

<template>
  <div v-if="status === 'pending'">Loading...</div>
  <div v-else-if="error">
    <p>Failed to load user: {{ error.message }}</p>
    <button @click="handleRetry">Try Again</button>
  </div>
  <div v-else>
    <h1>{{ userStore.data.name }}</h1>
  </div>
</template>
```

## 6. Utilizing Nuxt Error Boundaries

For component-level state errors that throw exceptions during rendering or async setup, wrap chunks of your application in `<NuxtErrorBoundary>`. This prevents the entire app from crashing and provides a localized recovery mechanism.

### Pattern: The Localized Sandbox

```vue
<template>
  <NuxtErrorBoundary @error="someErrorLogger">
    <!-- Component that might throw or have corrupt local state -->
    <DashboardWidget />
    
    <template #error="{ error, clearError }">
      <div class="error-state">
        <p>Widget failed to load.</p>
        <button @click="recover(clearError)">Reset Widget</button>
      </div>
    </template>
  </NuxtErrorBoundary>
</template>

<script setup>
const recover = (clearError) => {
  // 1. Reset relevant stores
  useWidgetStore().$reset()
  // 2. Clear boundary error
  clearError()
}
</script>
```

## 7. Automated Retry Mechanisms (Network Layer)

Before an error even reaches the store, the network layer should attempt to recover from transient faults (e.g., 503, 504, or network timeouts).

### Pattern: Axios / Fetch Interceptor with Exponential Backoff
Using a library like `axios-retry` or a custom Nuxt `$fetch` interceptor prevents minor network hiccups from bleeding into your application state.

```typescript
// plugins/fetch.ts
export default defineNuxtPlugin((nuxtApp) => {
  const $fetch = $fetch.create({
    retry: 3,
    retryDelay: (retryCount) => retryCount * 1000, // 1s, 2s, 3s
    retryStatusCodes: [408, 409, 425, 429, 500, 502, 503, 504],
    onResponseError({ response }) {
      if (response.status === 401) {
        // Auto-refresh token logic here before failing
      }
    }
  })
  
  // Provide globally
  return {
    provide: {
      customFetch: $fetch
    }
  }
})
```

## Summary Checklist for Large Apps:
- [ ] Define standardized error properties (`error`, `status`) in all API-dependent stores.
- [ ] Implement a Pinia plugin for global logging/telemetry of unhandled store errors.
- [ ] Implement optimistic updates with explicit rollback logic for high-frequency user actions.
- [ ] Rely on `$reset()` to sanitize corrupted state.
- [ ] Wrap non-critical sections of the UI in `<NuxtErrorBoundary>` to prevent cascade failures.
- [ ] Catch transient errors at the HTTP client level with automatic retries before allowing them to reach the store.
