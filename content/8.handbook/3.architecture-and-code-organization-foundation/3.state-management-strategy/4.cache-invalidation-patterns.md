# Cache Invalidation Patterns for State Management in Large Vue/Nuxt Apps

In large-scale Vue and Nuxt applications, effective caching and state management are critical for performance and user experience. When data is cached, knowing *when* and *how* to invalidate it is just as important as caching it in the first place. Stale data can lead to bugs, inconsistent UI, and poor user trust.

Here are the primary cache invalidation patterns used in Vue/Nuxt state management ecosystems (like Pinia, Vuex, Nuxt's data fetching, and Nitro).

## 1. Time-To-Live (TTL) / Expiration Pattern

The simplest form of cache invalidation is time-based. Data is considered valid for a specific duration, after which it is marked as stale and refetched upon the next request.

**Implementation in Pinia/Vuex:**
You can store a `lastFetched` timestamp alongside your data in the store.
```typescript
// Pinia Example
export const useUserStore = defineStore('user', {
  state: () => ({
    profile: null,
    lastFetched: null,
    cacheDuration: 5 * 60 * 1000 // 5 minutes
  }),
  actions: {
    async fetchProfile(force = false) {
      const now = Date.now();
      if (!force && this.profile && this.lastFetched && (now - this.lastFetched < this.cacheDuration)) {
        return; // Use cached data
      }
      this.profile = await api.getUserProfile();
      this.lastFetched = now;
    }
  }
})
```

## 2. Event/Mutation-Based Invalidation (Write-Through)

In a large app, data is often mutated (created, updated, or deleted) by the user. When a mutation occurs, the local cache must be invalidated or updated to reflect the new state.

**Pattern A: Optimistic UI Updates (Update Cache)**
Instead of just clearing the cache, you proactively update it to avoid another network request.
```typescript
async updateUser(userData) {
  // Update remote
  await api.updateUser(userData);
  // Update local cache directly without refetching
  this.profile = { ...this.profile, ...userData };
}
```

**Pattern B: Pessimistic Invalidation (Clear Cache)**
If the mutation affects a list or complex computed state, it's safer to clear the cache and let the application refetch it automatically.
```typescript
async deleteItem(itemId) {
  await api.deleteItem(itemId);
  // Invalidate list cache
  this.items = [];
  this.lastFetched = null;
  // Trigger a refetch
  await this.fetchItems();
}
```

## 3. Key-Based Invalidation (Nuxt 3 `useFetch` / `useAsyncData`)

Nuxt 3 introduces powerful built-in caching via `useFetch` and `useAsyncData`. It caches data locally based on a unique key (either auto-generated or explicitly provided).

**Refreshing Specific Data:**
You can invalidate and refresh a specific data call using the `refresh` method returned by `useFetch`.
```typescript
const { data, refresh } = await useFetch('/api/users', { key: 'users-list' })

// Somewhere else, after a user is added:
await refresh() // Refetches the data and updates the state
```

**Global Invalidation with `clearNuxtData`:**
For sweeping cache clearing (e.g., on user logout or major state transitions), Nuxt provides `clearNuxtData`.
```typescript
// Clear all cached Nuxt data
clearNuxtData()

// Clear specific keys (can take a string, array, or custom function)
clearNuxtData('users-list')
clearNuxtData((key) => key.startsWith('user-'))
```

## 4. Stale-While-Revalidate (SWR) Pattern

The SWR pattern serves stale data immediately from the cache while asynchronously fetching fresh data in the background. Once the fresh data is retrieved, the UI updates reactively. This provides a fast perceived performance while ensuring eventual consistency.

**Nuxt Nitro Server Cache (SWR):**
At the server/API layer, Nuxt's Nitro engine supports SWR out of the box using route rules.
```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  routeRules: {
    // Cache the API response for 60 seconds, but serve stale immediately if expired while refetching
    '/api/products': { swr: 60 }
  }
})
```
On the client side, Vue libraries like `@tanstack/vue-query` (Vue Query) natively implement the SWR pattern for state management.

## 5. Tag-Based / Surrogate Key Invalidation (Advanced SSR)

In large e-commerce or content-heavy Nuxt SSR apps (e.g., using Redis or Varnish), you might cache full HTML pages or API responses. When data changes in a headless CMS or database, you need to invalidate specific caches without purging the entire system.

**Nitro Cache Tags:**
Nuxt/Nitro allows you to assign tags to cached responses.
```typescript
// In your server/api/article.ts
export default defineEventHandler((event) => {
  // Tag this response
  event.context.cache = { tags: ['article-content'] }
  return fetchArticle()
})
```
When an article is updated in the CMS, it triggers a webhook to your Nuxt app, which can then purge the specific tag programmatically via Nitro's storage API.

## Summary of Best Practices for Vue/Nuxt:
1. **Use Pinia for Global Client State:** Implement TTL or mutation-based invalidation for explicit control over business logic data.
2. **Leverage Vue Query for Complex Async State:** If your app's state is heavily reliant on server data, tools like `@tanstack/vue-query` abstract away TTL, SWR, and invalidation mechanisms.
3. **Use Nuxt's Built-in Keys:** Explicitly key your `useFetch` calls to make `clearNuxtData` predictable and easy to manage across components.
4. **Nitro Route Rules for SSR:** Offload heavy caching and SWR to the Nitro server layer to reduce client payload size and API load.
