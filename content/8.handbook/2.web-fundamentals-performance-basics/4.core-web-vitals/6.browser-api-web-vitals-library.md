# Browser API & The Web Vitals Library

To capture Field Data (Real User Monitoring), developers need a way to measure performance metrics directly inside the user's browser. 

Historically, this was incredibly difficult. Today, modern browsers expose these metrics natively via the **PerformanceObserver API**. Furthermore, Google maintains a tiny JavaScript library called `web-vitals` that wraps this API, making it trivial to collect and report Core Web Vitals to your analytics dashboard.

---

## 1. The PerformanceObserver API (Under the Hood)

The browser natively tracks performance events as they happen (e.g., when a large image paints, or when the layout shifts). You can subscribe to these events using the `PerformanceObserver` API.

### Example: Manually observing LCP
```javascript
// Create an observer that listens for 'largest-contentful-paint' entries
const observer = new PerformanceObserver((list) => {
  const entries = list.getEntries();
  const lastEntry = entries[entries.length - 1];
  
  console.log('LCP observed:', lastEntry.startTime);
  console.log('LCP Element:', lastEntry.element);
});

// Start observing
observer.observe({ type: 'largest-contentful-paint', buffered: true });
```

### Why manual observation is painful
While the code above looks simple, accurately calculating the *final* Core Web Vitals score is incredibly complex. 
- For LCP, the browser might report several "largest" paints as the page loads before finally settling on the true LCP.
- For CLS, you have to manually filter out expected layout shifts that occur immediately after a user clicks a button.
- For INP, you have to track every single interaction for the entire lifespan of the page and calculate the 98th percentile.

Because of this complexity, you should **never** write raw `PerformanceObserver` code yourself in production.

---

## 2. The `web-vitals` Library

To solve the complexity of the raw browser APIs, Google created the `web-vitals` JavaScript library. 
It is extremely tiny (~1.5KB gzipped), strictly follows the exact same measurement algorithms used by Chrome and Google Search Console, and perfectly handles all edge cases (like background tabs, bfcache, and iframe tracking).

### Installation
You can install it via npm, or load it from a CDN:
```bash
npm install web-vitals
```

### Basic Usage
The library provides dedicated functions for each metric. You simply pass a callback function that will execute whenever the metric is ready to be reported.

```javascript
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

// The callback receives a 'metric' object
function reportMetric(metric) {
  console.log(metric.name);  // e.g., 'LCP'
  console.log(metric.value); // e.g., 1450 (milliseconds)
  console.log(metric.rating); // 'good', 'needs-improvement', or 'poor'
  
  // Here is where you would send the data to your analytics backend (Google Analytics, Datadog, etc.)
}

onLCP(reportMetric);
onINP(reportMetric);
onCLS(reportMetric);
```

## 3. Advanced Features: Attribution

Knowing *that* your page has a bad CLS score is helpful. Knowing *exactly which DOM element* caused the shift is invaluable.

The `web-vitals` library provides an "Attribution" build that includes deep diagnostic data about *why* the metric fired.

### Using the Attribution Build

```javascript
// Note the '/attribution' path
import { onCLS, onLCP } from 'web-vitals/attribution';

onCLS((metric) => {
  console.log('Total CLS Score:', metric.value);

  // The attribution object tells you exactly what went wrong
  const largestShift = metric.attribution.largestShiftTarget;
  console.log('The element that shifted the most was:', largestShift);
});

onLCP((metric) => {
  console.log('LCP Time:', metric.value);
  
  // The attribution object breaks down exactly where the time was spent
  console.log('Time to First Byte:', metric.attribution.ttfb);
  console.log('Resource Load Time:', metric.attribution.resourceLoadTime);
  console.log('LCP Element:', metric.attribution.lcpEntry.element);
});
```

---

## 4. When are metrics reported?

Because metrics like INP and CLS are tracked for the *entire lifespan* of the page, the `web-vitals` library cannot report the final score immediately when the page loads. 

Instead, it intelligently waits for the user to leave the page or background the tab (using the `visibilitychange` event). At that exact moment, it calculates the final score and fires your callback, ensuring you capture the most accurate data possible before the page unloads.

---

## Summary

1. **PerformanceObserver:** The raw browser API that emits performance events. It is too complex to manage manually.
2. **`web-vitals` Library:** A tiny, Google-maintained wrapper that accurately calculates metrics and handles complex edge cases.
3. **Attribution:** Use the `web-vitals/attribution` build to get exact DOM elements and timeline breakdowns to help you debug *why* a metric is failing.
4. **Lifecycle:** Metrics are usually reported at the very end of the page lifecycle (when the user backgrounds the tab or navigates away) to ensure all interactions and layout shifts were captured.
