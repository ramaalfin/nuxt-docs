# Field Data vs. Lab Data

When optimizing for Core Web Vitals, developers rely on two entirely different categories of performance data: **Field Data** and **Lab Data**. 

A common frustration among developers is seeing a perfect 100/100 Lighthouse score (Lab Data), while their Google Search Console reports failing Core Web Vitals (Field Data). Understanding why these two data sources disagree is essential for effective performance engineering.

---

## 1. Lab Data (Synthetic Testing)

Lab data is collected in a highly controlled, simulated environment. 

### How it is collected
Tools like **Google Lighthouse**, **Chrome DevTools**, and **WebPageTest** load your webpage using predefined hardware and network conditions.
For example, a standard Lighthouse mobile test simulates a mid-tier device (like a Moto G4) on a throttled 4G network connection.

### Pros of Lab Data
- **Reproducible:** Because the environment is strictly controlled, running a test 5 times will give you roughly the same result.
- **Immediate Feedback:** You get results instantly, making it perfect for active debugging.
- **Deep Insights:** Lab tools can trace the exact CPU execution timeline, highlight render-blocking scripts, and point to specific lines of unoptimized code.
- **CI/CD Integration:** You can run Lighthouse against a staging server to block PRs that degrade performance *before* they reach production.

### Cons of Lab Data
- **Not Real Users:** It does not reflect the vast diversity of real-world devices, network speeds, or geographical latency.
- **Cannot Measure INP accurately:** Lab tools simulate a page load, but they don't simulate a human clicking around the page. Therefore, metrics like Interaction to Next Paint (INP) cannot be accurately measured in the Lab.

---

## 2. Field Data (Real User Monitoring / RUM)

Field data is collected anonymously from actual human beings interacting with your live website.

### How it is collected
1. **CrUX (Chrome User Experience Report):** Google collects anonymous performance data from opted-in Chrome users. This data powers Google Search Console and PageSpeed Insights.
2. **Custom RUM:** You can install the `web-vitals` JavaScript library on your site to collect this data yourself and send it to analytics tools like Vercel Analytics, Sentry, or Datadog.

### Pros of Field Data
- **The Source of Truth:** This is what your users actually experience. If you have an international audience, Field Data will reveal the latency they experience, which Lab Data misses.
- **SEO Impact:** Google uses Field Data (CrUX) exclusively to determine your search ranking boost. Google does *not* care about your Lighthouse score; it cares about CrUX.
- **Measures Interactions:** It is the only way to accurately measure INP, because it records actual human clicks and taps.

### Cons of Field Data
- **Noisy and Unpredictable:** A user with a 10-year-old Android phone on a 2G connection in a tunnel will generate terrible metrics, dragging down your average.
- **Delayed (CrUX):** The official Google CrUX dataset operates on a rolling 28-day average. If you deploy a massive performance fix today, you won't see the "Pass" in Search Console for nearly a month.
- **Hard to Debug:** Field data tells you *that* a page is slow, but it rarely tells you *why* or which exact line of code caused it.

---

## 3. Why Do They Disagree?

It is perfectly normal for Lab Data and Field Data to show completely different results.

**Scenario A: Great Lab Data, Terrible Field Data**
- *Reason:* Your Lab test was run on a fast MacBook, but 80% of your real users are on slow 3G mobile connections. Or, your site has heavy JavaScript that causes a terrible INP score when users actually try to click things (which Lighthouse didn't test).

**Scenario B: Terrible Lab Data, Great Field Data**
- *Reason:* Lighthouse throttled the CPU and Network to simulate a mid-tier mobile device, but your actual user base consists of corporate executives using high-end laptops on fiber-optic office Wi-Fi. 

---

## 4. The Developer Workflow

To build fast websites, you must use both. They are complementary.

1. **Discover (Field Data):** Check your RUM dashboard or Google Search Console to see *which* pages are failing in the real world, and *which* metric (LCP, CLS, INP) is the problem.
2. **Reproduce (Lab Data):** Open Chrome DevTools on that specific page. Throttle your network and CPU until you can reproduce the failure seen in the Field Data.
3. **Fix & Verify (Lab Data):** Fix the code and run Lighthouse to verify the improvement locally.
4. **Monitor (Field Data):** Deploy the fix to production and monitor the RUM dashboard over the next week to ensure the fix actually improved the real-world experience.

---

## Summary

| Feature | Lab Data (Lighthouse) | Field Data (CrUX / RUM) |
| :--- | :--- | :--- |
| **Source** | Simulated environment | Real human users |
| **Best For** | Debugging, CI/CD, Development | SEO, Business metrics, Monitoring |
| **Reproducibility**| High (Consistent) | Low (Noisy, relies on user hardware) |
| **Metrics** | LCP, CLS, TBT (Total Blocking Time) | LCP, CLS, INP (Interaction to Next Paint) |
| **Feedback Loop**| Immediate | Instant (Custom RUM) or 28 Days (CrUX) |
