# Measuring & Monitoring Core Web Vitals

Optimizing Core Web Vitals (LCP, INP, CLS) is not a one-time task; it is an ongoing process. Code changes, new third-party scripts, and shifting user behaviors can all degrade performance over time.

To ensure your web application remains fast, you must measure and monitor performance using two distinct types of data: **Lab Data** and **Field Data**.

---

## 1. Lab Data vs. Field Data

Understanding the difference between these two data sources is critical for accurate performance monitoring.

### Lab Data (Synthetic)
Lab data is collected in a controlled environment with predefined device and network settings. 
- **Tools:** Google Lighthouse, WebPageTest, Chrome DevTools.
- **Pros:** Excellent for debugging. The results are reproducible and immediate. You can run Lab tests in your CI/CD pipeline before code ever reaches production.
- **Cons:** It does not represent real users. A Lighthouse test runs on a simulated Moto G4 on a 4G connection. If your actual users all have iPhone 15s on 5G, the Lab Data will not reflect their reality. Furthermore, Lab Data cannot accurately measure INP because there is no human interacting with the page.

### Field Data (RUM - Real User Monitoring)
Field data is collected anonymously from actual users navigating your live website in the real world.
- **Tools:** Chrome User Experience Report (CrUX), Vercel Analytics, Datadog RUM, Sentry.
- **Pros:** It is the ultimate source of truth. Google uses Field Data (specifically CrUX data) to determine your search rankings (SEO). It captures the true variety of devices, network conditions, and real-world interactions.
- **Cons:** It is inherently noisy and difficult to debug. If a user in a rural area with terrible cell service visits your site, their slow load time will drag down your metrics, even if your code is perfectly optimized.

## 2. Tools for Measuring Web Vitals

### Google Lighthouse (Lab Data)
Lighthouse is built directly into Chrome DevTools. It runs a synthetic test against your page and generates a comprehensive report on Performance, Accessibility, Best Practices, and SEO.

*Best Practice:* Always run Lighthouse in **Incognito Mode** so your browser extensions (like ad blockers or password managers) don't skew the performance results.

### Chrome User Experience Report - CrUX (Field Data)
CrUX is the official dataset Google uses for SEO rankings. It is an aggregation of real-world performance data from opted-in Chrome users.
You can view your CrUX data in the **Google Search Console** under the "Core Web Vitals" tab, or use PageSpeed Insights to see a quick snapshot.

*Limitation:* CrUX data operates on a 28-day rolling average. If you deploy a massive performance fix today, you won't see the full impact on your CrUX score for a month.

### The `web-vitals` JS Library
If you want real-time Field Data without waiting 28 days, Google provides a tiny (1KB) JavaScript library called `web-vitals` that you can install in your app.

It accurately measures LCP, INP, and CLS on the user's device and allows you to send that data to your own analytics server (like Google Analytics, Datadog, or a custom database).

```javascript
import { onLCP, onINP, onCLS } from 'web-vitals';

function sendToAnalytics(metric) {
  // Send the metric data to your backend
  fetch('/analytics/vitals', {
    method: 'POST',
    body: JSON.stringify(metric),
  });
}

// Measure and report the vitals
onCLS(sendToAnalytics);
onINP(sendToAnalytics);
onLCP(sendToAnalytics);
```

## 3. The Performance Monitoring Workflow

To maintain a fast web application, you need a workflow that combines both Lab and Field data.

1. **Develop (Lab):** Use Chrome DevTools (Network tab, Performance tab) to debug bottlenecks locally.
2. **CI/CD (Lab):** Use Lighthouse CI in your GitHub Actions. Configure it to fail the build if a PR drops the Lighthouse score below 90. This prevents regressions from reaching production.
3. **Monitor (Field):** Implement RUM (using the `web-vitals` library or a service like Vercel Analytics) to track how real users experience the live application.
4. **Investigate (Field + Lab):** When your RUM dashboard alerts you that INP is suddenly failing for mobile users, use that data to write a local test case, reproduce it in the Lab, and fix it.

---

## Summary

1. **Lab Data:** Controlled, synthetic testing (Lighthouse). Great for immediate debugging and CI/CD, but doesn't reflect real-world usage.
2. **Field Data (RUM):** Real data from real users (CrUX, `web-vitals`). This is the source of truth for SEO, but it is noisy and operates on a delay.
3. **The `web-vitals` Library:** The standard way to capture real-time field data and send it to your own analytics provider.
4. **The Workflow:** Use Lab Data to prevent regressions, and Field Data to monitor real-world health.
