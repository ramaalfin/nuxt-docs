# Cumulative Layout Shift (CLS)

**Cumulative Layout Shift (CLS)** is the third Core Web Vitals metric. It measures **visual stability**.

Have you ever been reading an article on your phone, and suddenly an image or an ad loads at the top of the page, pushing the text down and causing you to lose your place? Or worse, you go to click a "Cancel" button, but the page shifts, and you accidentally click "Buy Now"? 

These unexpected movements are called Layout Shifts. CLS measures the total sum of all unexpected layout shifts that occur during the entire lifespan of the page.

To provide a good user experience, pages should maintain a CLS of **less than 0.1**.

---

## 1. How is CLS Calculated?

The browser calculates the CLS score by multiplying two factors together for every unexpected shift:

**`Layout Shift Score = Impact Fraction × Distance Fraction`**

1. **Impact Fraction:** Measures how much of the viewport area was affected by the shift. If a block of text takes up 50% of the screen and moves down, the impact fraction is `0.50`.
2. **Distance Fraction:** Measures the greatest distance the unstable element has moved, relative to the viewport. If the text block moved down by 10% of the screen height, the distance fraction is `0.10`.

`0.50 × 0.10 = 0.05` (This shift contributes 0.05 to your total CLS).

*Note: Layout shifts that occur within 500 milliseconds of a user interaction (like clicking a button to open a menu) are considered "expected" and are excluded from the CLS calculation.*

---

## 2. Common Causes of Poor CLS

There are a few classic mistakes that cause high CLS scores.

### Cause 1: Images without Dimensions
If you don't specify the `width` and `height` attributes on an `<img>` tag, the browser doesn't know how much space to reserve for the image until the image is fully downloaded. 

When the image finally arrives, the browser suddenly forces it into the layout, pushing everything below it downward.

### Cause 2: Ads, Embeds, and Iframes without Dimensions
Similar to images, third-party ads or widgets (like embedded YouTube videos or Twitter posts) are often injected dynamically via JavaScript. If their parent container doesn't have a fixed minimum height, the container collapses to `0px` initially, and then violently expands when the ad loads.

### Cause 3: Dynamically Injected Content
When JavaScript fetches data from an API and then injects a new DOM element (like a banner or an alert) at the *top* of the screen, it pushes all existing content down.

### Cause 4: Web Fonts (FOUT / FOIT)
When custom WebFonts load, they often have slightly different character widths than the fallback font. 
If the browser swaps from the fallback font (Arial) to the custom font (Roboto), the text might suddenly take up fewer lines, causing the paragraph height to shrink and pulling the content below it upwards. This is known as **FOUT** (Flash of Unstyled Text).

---

## 3. How to Optimize CLS (< 0.1)

### 1. Always include `width` and `height` attributes
Always set the aspect ratio of your images and videos in the HTML. Modern browsers are smart enough to use these attributes to calculate the aspect ratio and reserve the exact bounding box space for the image before it even begins downloading.

```html
<!-- ✅ GOOD: The browser reserves space instantly -->
<img src="hero.jpg" width="800" height="400" alt="Hero" />
```

### 2. Reserve space for dynamic content
If you are loading an ad, a YouTube iframe, or dynamic API content, pre-allocate space for it using CSS `min-height` or the CSS `aspect-ratio` property.

```css
.ad-slot {
  /* Reserves space so the page doesn't shift when the ad loads */
  min-height: 250px; 
}
```

### 3. Never inject content *above* existing content
If you must inject dynamic content (like a "Cookie Consent" banner or "New Tweets available" button), try to fix it to the viewport (`position: fixed` or `sticky`) or place it at the *bottom* of the screen so it doesn't push the user's main reading content out of the way.

### 4. Optimize WebFonts
Use `font-display: optional` (which tells the browser to just use the fallback font if the custom font takes too long, preventing a late shift). 
Alternatively, use the modern CSS `size-adjust` property to make your fallback font perfectly match the exact physical dimensions of your custom WebFont.

---

## Summary

1. **The Metric:** CLS measures visual stability and unexpected screen jumps.
2. **The Goal:** A total score of **< 0.1**.
3. **The Causes:** Images/Ads without dimensions, dynamically injected DOM nodes, and WebFont swapping.
4. **The Fix:** Always provide `width` and `height` attributes for media, reserve space for async widgets using CSS `min-height`, and never push existing content down the page programmatically.
