# Interaction to Next Paint (INP)

**Interaction to Next Paint (INP)** is the newest Core Web Vitals metric (replacing First Input Delay, or FID, in March 2024). It measures **responsiveness**.

Specifically, INP measures the latency of all interactions (clicks, taps, and keyboard inputs) throughout the entire lifespan of a user's visit to a page. The final INP value is the longest interaction observed (ignoring a few outliers on pages with heavy interaction).

To provide a good user experience, a page's INP should be **below 200 milliseconds**.

---

## 1. How is INP Calculated?

An "Interaction" consists of three distinct phases. INP measures the total time across all three phases:

1. **Input Delay:** The time from when the user physically clicks/taps until the browser actually begins firing the event handlers. (Usually caused by the Main Thread being busy doing something else).
2. **Processing Time:** The time it takes for your JavaScript event handlers (`onclick`, `keydown`) to finish running.
3. **Presentation Delay:** The time it takes the browser to calculate the new Layout, Paint the pixels, and Composite the new frame to the screen.

```text
[ Click ] ---> [ Input Delay ] ---> [ Processing Time ] ---> [ Presentation Delay ] ---> [ Next Paint ]
```

---

## 2. INP vs. FID (Why did Google change it?)

**First Input Delay (FID)** only measured the *first* interaction on the page, and it only measured the *Input Delay* portion (ignoring processing and presentation time). 
Developers could easily "game" FID by having fast initial loads, even if the page became an unresponsive mess later on. 

INP is a much stricter, comprehensive metric because it tracks *every* interaction and measures the *full* time until the screen visually updates.

---

## 3. How to Optimize INP (< 200ms)

Because INP covers three different phases, you must optimize each phase differently.

### Phase 1: Reducing Input Delay
If a user clicks a button while the browser is busy parsing a massive 2MB JavaScript bundle, the browser cannot respond to the click until it finishes parsing.
- **Solution:** Break up Long Tasks. A "Long Task" is any JS execution that takes longer than 50ms. Use code-splitting, web workers, or `setTimeout` / `scheduler.yield()` to yield control back to the Main Thread so it can respond to user inputs.

### Phase 2: Reducing Processing Time
If your `onclick` handler does massive amounts of math or manipulates a huge virtual DOM, it will delay the next paint.
- **Solution:** Do less work synchronously. If a button click triggers an API call and a complex data transformation, update the UI *first* (e.g., show a spinner) before doing the heavy lifting.
  ```javascript
  button.addEventListener('click', () => {
    // 1. Instantly update UI (Fast processing time)
    spinner.style.display = 'block'; 
    
    // 2. Defer heavy work to a separate task
    setTimeout(() => {
      heavyDataProcessing();
    }, 0);
  });
  ```

### Phase 3: Reducing Presentation Delay
If your JavaScript finishes fast, but it forces the browser to recalculate the layout of 10,000 DOM nodes (Layout Thrashing), the presentation will be delayed.
- **Solution:** 
  - Avoid Forced Synchronous Layouts (reading from the DOM immediately after writing to it).
  - Reduce DOM size (Google recommends < 1,500 total DOM nodes).
  - Use CSS properties that only trigger Compositing (`transform`, `opacity`) instead of ones that trigger Layout (`width`, `margin`).

---

## Summary

1. **The Metric:** INP measures the total time from a user's click/keypress until the browser actually paints the visual result on the screen.
2. **The Goal:** < 200 milliseconds.
3. **The Causes:** Heavy JavaScript blocking the main thread (Input Delay), slow event handlers (Processing Time), and complex DOM updates (Presentation Delay).
4. **The Fix:** Yield to the main thread frequently, update the UI immediately before doing heavy background work, and keep DOM sizes small.
