# Content Security Policy (CSP)

Cross-Site Scripting (XSS) is one of the most common and dangerous vulnerabilities on the web. It occurs when a hacker successfully injects malicious JavaScript into a webpage. Because the browser trusts the code it receives from the server, it executes the malicious script, allowing the hacker to steal cookies, session tokens, or even alter the page content.

**Content Security Policy (CSP)** is an added layer of security that helps detect and mitigate certain types of attacks, primarily XSS and data injection attacks.

---

## 1. What is CSP?

CSP is an HTTP response header that allows site administrators to declare approved sources of content that the browser is allowed to load. 

Instead of blindly trusting all scripts, images, and styles that the HTML document requests, the browser checks every request against the CSP. If the source of the asset is not explicitly whitelisted in the policy, the browser blocks it from loading or executing.

### The Header Syntax
```http
Content-Security-Policy: <directive> <source>; <directive> <source>
```

---

## 2. Common CSP Directives

A CSP is made up of "directives" that control specific types of resources.

- **`default-src`:** The fallback policy for all directives that are not explicitly defined. If you set this, you don't need to specify every single resource type.
- **`script-src`:** Specifies valid sources for JavaScript execution.
- **`style-src`:** Specifies valid sources for CSS stylesheets.
- **`img-src`:** Specifies valid sources for images.
- **`connect-src`:** Restricts the URLs which can be loaded using script interfaces (e.g., `fetch`, XHR, WebSockets).
- **`font-src`:** Specifies valid sources for WebFonts.

---

## 3. Writing a Policy (Examples)

Let's look at how to build a CSP, starting from the most restrictive.

### Example 1: The Ultimate Lockdown
```http
Content-Security-Policy: default-src 'none';
```
This blocks *everything*. No images, no CSS, no JavaScript, nothing. The page will render as plain unstyled HTML text. 

### Example 2: Same-Origin Only
```http
Content-Security-Policy: default-src 'self';
```
This is a very common starting point. `'self'` tells the browser: *"Only allow resources to load if they come from the exact same domain that served this HTML page."*
If the HTML is on `example.com`, an image from `example.com/logo.png` will load, but a script from `cdn.jquery.com` will be blocked.

### Example 3: Whitelisting CDNs and APIs
```http
Content-Security-Policy: default-src 'self'; img-src *; connect-src https://api.mywebsite.com;
```
1. **`default-src 'self'`**: Everything defaults to the same origin.
2. **`img-src *`**: Images can be loaded from anywhere on the internet (the wildcard `*`).
3. **`connect-src https://api.mywebsite.com`**: `fetch` and XHR requests are ONLY allowed to communicate with this specific API domain.

---

## 4. The Danger of Inline Scripts

By default, the strongest feature of CSP is that it completely disables **inline scripts** and **inline styles**.

If a hacker finds an XSS vulnerability in a comment section and injects `<script>alert('Hacked')</script>`, the browser will refuse to run it because the code is inline, rather than loaded from a whitelisted external `.js` file.

### How to allow specific inline scripts
Sometimes, you legitimately need an inline script (e.g., for initializing an analytics tool). You can allow specific inline scripts using a **Nonce** (Number Used Once).

1. The server generates a random cryptographic string (Nonce) for every page load.
2. The server adds the nonce to the CSP header:
   ```http
   Content-Security-Policy: script-src 'self' 'nonce-EDNnf03nceIOfn39fn3e9h3sdfa'
   ```
3. The server adds the exact same nonce to the legitimate inline `<script>` tag:
   ```html
   <script nonce="EDNnf03nceIOfn39fn3e9h3sdfa">
     // Legitimate analytics code
   </script>
   ```

Because the hacker doesn't know the randomly generated nonce for the current page load, their injected `<script>` tag will lack the nonce, and the browser will block it!

---

## 5. Report-Only Mode

Implementing CSP on an older, massive application is terrifying because if you make a mistake, you will break the website for your users (CSS won't load, JS will crash).

To solve this, you can use the **Report-Only** header:
```http
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-violation-report-endpoint/
```

In this mode, the browser does **not** block any resources. It loads everything normally, but it silently sends a JSON report to the `report-uri` server endpoint every time it detects an asset that *would* have been blocked. 

Developers can monitor these logs for a few weeks, fine-tune the policy to ensure no legitimate assets are being flagged, and then finally switch to the enforced `Content-Security-Policy` header.

---

## Summary

1. **Defense in Depth:** CSP is a secondary line of defense against XSS attacks. It does not replace input validation and output encoding.
2. **Whitelisting:** It allows administrators to explicitly whitelist domains where scripts, images, and styles are allowed to load from.
3. **Inline Prevention:** It blocks all inline `<script>` and `<style>` tags by default, crippling most injected XSS attacks. Nonces can be used for legitimate inline scripts.
4. **Report-Only:** Allows safe rollouts by logging violations without actually breaking the webpage.
