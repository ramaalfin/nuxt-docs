# Compression (Gzip, Brotli)

Text-based assets like HTML, CSS, and JavaScript are incredibly verbose. They contain repetitive whitespace, long variable names, and recurring keywords. 

To prevent wasting bandwidth and slowing down page loads, web servers must compress these files before sending them over the network. The browser then decompresses them on the fly. 

The two dominant compression algorithms on the web today are **Gzip** and **Brotli**.

---

## 1. How Web Compression Works

Compression is a negotiation between the browser and the server.

1. **The Request:** When the browser requests an asset, it sends an `Accept-Encoding` header listing the compression algorithms it supports.
   ```http
   Accept-Encoding: gzip, deflate, br
   ```
2. **The Response:** The server reads the header, compresses the file using the best available algorithm, and sends it back with a `Content-Encoding` header so the browser knows how to decompress it.
   ```http
   Content-Encoding: br
   ```

*Note: You should never compress images (JPEG, PNG, WebP) or video files using Gzip or Brotli. Media files are already heavily compressed by their own codecs. Re-compressing them wastes CPU and can actually increase the file size.*

---

## 2. Gzip (The Industry Standard)

**Gzip** was released in 1992 and has been the undisputed king of web compression for decades. 
- It uses the DEFLATE algorithm (a combination of LZ77 and Huffman coding).
- It is universally supported by 100% of browsers, servers, and CDNs.
- It provides excellent compression ratios (often reducing text file sizes by 70% to 80%) with very low CPU overhead.

If you don't configure anything on a modern web server or CDN, it is almost certainly using Gzip by default.

---

## 3. Brotli (The Modern Successor)

**Brotli** (`br`) was open-sourced by Google in 2015 specifically to optimize web assets. It is heavily used by major tech companies and modern CDNs (like Cloudflare and Vercel).

### Why Brotli is Better than Gzip
1. **A Built-in Dictionary:** Brotli was designed specifically for the web. It contains a static dictionary of 13,000 common web words and phrases (like `<html>`, `function`, `Content-Type`, `width`). Instead of sending the word `function` over the network, it just sends a tiny reference ID to the dictionary.
2. **Better Compression Ratios:** On average, Brotli compresses JavaScript 14% smaller than Gzip, HTML 21% smaller, and CSS 17% smaller.
3. **Decompression Speed:** Despite achieving much higher compression ratios, Brotli decompresses on the user's device just as fast as Gzip.

### The Catch: CPU Cost
While Brotli is fast to *decompress*, it is highly CPU-intensive to *compress* files at its maximum setting (Level 11).
If a server tries to compress a dynamic HTML page at Brotli Level 11 on the fly (dynamically), the CPU time required will completely negate the network time saved.

**Best Practice for Brotli:**
- **Dynamic Content (HTML, API responses):** Compress on-the-fly using a low Brotli level (e.g., Level 4). It is still smaller than Gzip and fast enough for real-time.
- **Static Assets (JS, CSS):** Pre-compress these files at maximum Brotli Level 11 during your build step (e.g., using Webpack or Vite), and serve the pre-compressed `.br` files directly.

---

## 4. Minification vs. Compression

It is crucial to understand that Compression does not replace **Minification**. You must do both.

- **Minification** (e.g., Terser, esbuild) alters the *source code*. It removes whitespace, strips comments, and shortens variable names (`const userDatabaseRecord` becomes `const u`).
- **Compression** (Gzip/Brotli) alters the *binary representation* of the file for network transport.

**The Pipeline:**
1. Developer writes readable code.
2. Build tool *Minifies* the code.
3. Server *Compresses* the minified code.
4. Browser downloads, decompresses, and executes the minified code.

---

## Summary

1. **Gzip:** The fast, universally supported, industry-standard compression algorithm.
2. **Brotli:** The modern successor built by Google. It uses a built-in dictionary to compress web assets 15-20% smaller than Gzip.
3. **The Trade-off:** Brotli at max settings is very slow to compress. Pre-compress static assets at max settings, but use low settings for dynamic on-the-fly compression.
4. **Minification:** Always minify your CSS and JS before compressing it. They solve two different problems.
