# Caching Headers (Cache-Control, ETag, Vary)

The fastest network request is the one that never has to be made. HTTP Caching allows the browser to store copies of responses (like images, CSS, and API data) locally on the user's device. 

When the user requests the same asset again, the browser can load it instantly from the local disk instead of fetching it across the network. The server controls this behavior using specific HTTP Headers.

---

## 1. Cache-Control

`Cache-Control` is the most important header for web performance. It dictates exactly *who* can cache the response and for *how long*.

### Key Directives:

- **`max-age=<seconds>`:** Specifies the maximum amount of time (in seconds) the browser is allowed to use the cached file without asking the server if it has changed.
  - *Example:* `Cache-Control: max-age=31536000` (Cache for 1 year. Perfect for static assets with hashed filenames like `app.v1.js`).
- **`no-cache`:** This is horribly named. It does *not* mean "do not cache". It means "you can cache this, but you **MUST** revalidate with the server before using it every single time."
- **`no-store`:** This actually means "do not cache this at all". The browser must download a fresh copy from the server on every request. Used for highly sensitive data (like bank balances).
- **`public` vs `private`:** 
  - `public` means the response can be cached by *anyone* (the browser, a CDN, a proxy).
  - `private` means the response is personalized for a specific user and can *only* be cached by the user's browser, never by an intermediary CDN.

---

## 2. ETags (Revalidation)

If you use `Cache-Control: no-cache` (or if a `max-age` expires), the browser must check with the server to see if the file has changed before using its cached copy. This is called **Revalidation**.

An **ETag** (Entity Tag) is a unique fingerprint (usually a hash) of the file's contents generated by the server.

### The ETag Workflow:
1. **First Request:** The server sends a file and includes an ETag header: `ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"`.
2. **Browser Caches It:** The browser saves the file and the ETag.
3. **Second Request (Revalidation):** The browser asks the server for the file again, but includes a special header: `If-None-Match: "33a64df..."`.
4. **The Server Checks:** The server calculates the hash of the file currently on its disk. 
   - If the hash matches the browser's ETag, the file hasn't changed! The server sends back a tiny, empty **`304 Not Modified`** response. This saves massive amounts of bandwidth.
   - If the hash is different, the server sends the full new file with a **`200 OK`** and a new ETag.

---

## 3. Vary Header

The `Vary` header tells intermediate caches (like CDNs) that the server's response might look completely different depending on the *request* headers sent by the client.

### The Problem it Solves
Imagine a user requests `index.html` from their mobile phone. The server detects the `User-Agent: Mobile` header and sends back a specific mobile-optimized HTML page. 
A CDN sits in the middle and caches that response.
Ten seconds later, a desktop user requests `index.html`. If the CDN isn't careful, it will serve the cached mobile page to the desktop user!

### The Solution
If the server sends `Vary: User-Agent` with its response, it is telling the CDN: *"Do not serve this cached HTML to anyone unless their User-Agent matches exactly."*

Another common use case is Compression:
```http
Vary: Accept-Encoding
```
This tells the CDN to keep separate caches for the Gzip version and the Brotli version of the same file.

---

## Summary

1. **`Cache-Control`:** Determines how long a file can be cached (`max-age`) and whether it needs to be re-checked before use (`no-cache`).
2. **`ETag`:** A unique hash of the file. Used by the browser to ask the server, "Has this file changed?" If not, the server replies with a tiny `304 Not Modified`.
3. **`Vary`:** Warns CDNs that the response content changes based on the user's request headers (like `User-Agent` or `Accept-Encoding`), preventing the CDN from serving the wrong cached file to the wrong user.
