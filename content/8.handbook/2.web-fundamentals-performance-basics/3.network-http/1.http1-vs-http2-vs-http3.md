# HTTP/1.1 vs HTTP/2 vs HTTP/3

The Hypertext Transfer Protocol (HTTP) is the foundation of data communication on the World Wide Web. Over the past decades, it has evolved significantly to handle the increasing complexity and size of modern web applications. 

Understanding the differences between HTTP versions is essential for optimizing network performance and delivering fast user experiences.

---

## 1. HTTP/1.1 (The Workhorse)

Standardized in 1997, HTTP/1.1 has been the backbone of the web for over 20 years. It operates over **TCP (Transmission Control Protocol)**.

### How it works:
When a browser wants to download an asset (like an image or a CSS file), it opens a TCP connection to the server. 
In HTTP/1.0, the connection closed immediately after one file was downloaded. HTTP/1.1 introduced **Keep-Alive**, allowing the browser to reuse the same TCP connection to download multiple files sequentially.

### The Problem: Head-of-Line Blocking
HTTP/1.1 suffers from a severe limitation: it can only download **one file at a time per TCP connection**. 
If the browser requests a massive JavaScript file, it must wait for that file to finish downloading entirely before it can ask for the next image on the same connection. This is called **Head-of-Line (HoL) Blocking**.

*Hack/Workaround:* To bypass this, browsers traditionally opened multiple TCP connections to the same server simultaneously (usually capped at 6 connections). Developers also invented workarounds like CSS Sprites (combining images) and JS concatenations to reduce the number of requests.

---

## 2. HTTP/2 (The Multiplexer)

Released in 2015, HTTP/2 completely overhauled how data is transmitted between the client and server, while keeping the semantics (headers, methods, status codes) exactly the same. It still operates over **TCP**.

### The Solution: Multiplexing
HTTP/2 solves the Head-of-Line blocking problem at the HTTP layer by introducing **Multiplexing**. 
Instead of sending files one by one, HTTP/2 breaks files down into tiny "frames" and sends them all concurrently over a **single TCP connection**. The browser receives these mixed frames and reassembles them on the fly.

### Key Features of HTTP/2:
- **Multiplexing:** Multiple requests and responses fire simultaneously over one connection. (CSS Sprites are no longer needed).
- **Binary Framing:** HTTP/1.1 was plain text. HTTP/2 is binary, making it much faster for machines to parse.
- **Header Compression (HPACK):** HTTP/1.1 sent redundant headers (like cookies and user agents) repeatedly on every request. HTTP/2 compresses and deduplicates headers, saving massive amounts of bandwidth.
- **Server Push (Deprecated):** Allowed the server to push assets to the client before the client asked for them. *(Note: This feature proved too complex and inefficient in practice and has been largely removed from modern browsers).*

### The New Problem: TCP Head-of-Line Blocking
While HTTP/2 fixed HoL blocking at the HTTP layer, it exposed a flaw in the underlying **TCP layer**. 
Because everything travels over one TCP connection, if a single packet of data is lost over a spotty mobile network, TCP halts the *entire* connection until that one packet is retransmitted. This means a single dropped packet delays *all* multiplexed files.

---

## 3. HTTP/3 (The QUIC Revolution)

Standardized in 2022, HTTP/3 is a radical departure from its predecessors. Instead of using TCP, HTTP/3 is built on a entirely new transport protocol called **QUIC**, which runs over **UDP (User Datagram Protocol)**.

### The Solution: QUIC and UDP
UDP is historically known as a fast, "fire-and-forget" protocol used for video streaming and gaming because it doesn't care if packets arrive in order or get lost. 
QUIC takes the speed of UDP and adds its own layer of reliability and congestion control on top.

### Key Features of HTTP/3:
- **No TCP Head-of-Line Blocking:** Because QUIC manages multiplexing natively, if a packet for an image is lost, *only that image* is delayed. The CSS and JS files continue downloading unimpeded.
- **Zero Round Trip Time (0-RTT) Resumption:** Setting up a secure TCP/TLS connection in HTTP/2 requires multiple back-and-forth round trips before any data is sent. QUIC combines the cryptographic handshake with the connection handshake. If the client has talked to the server before, it can start sending data immediately (0-RTT), drastically speeding up load times on mobile devices.
- **Connection Migration:** If your phone switches from Wi-Fi to a 4G cellular network, your IP address changes. In HTTP/2 (TCP), the connection dies and must be completely re-established. QUIC uses unique connection IDs, allowing the download to continue seamlessly as you walk out of your house.

---

## Summary Comparison

| Feature | HTTP/1.1 | HTTP/2 | HTTP/3 |
| :--- | :--- | :--- | :--- |
| **Transport Protocol** | TCP | TCP | UDP (via QUIC) |
| **Data Format** | Plain Text | Binary | Binary |
| **Requests per Connection** | 1 (Sequential) | Unlimited (Multiplexed) | Unlimited (Multiplexed) |
| **HoL Blocking (HTTP)** | Yes | No | No |
| **HoL Blocking (TCP)** | Yes (Per Connection) | Yes (Whole Connection halts) | No (Independent streams) |
| **Connection Migration** | No (Fails on network switch) | No | Yes (Seamless Wi-Fi to 4G) |
| **Handshake Speed** | Slow (TCP + TLS separately) | Slow | Fast (Combined, 0-RTT) |
