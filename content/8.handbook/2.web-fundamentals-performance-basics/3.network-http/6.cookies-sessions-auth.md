# Cookies, Sessions, & Authentication

HTTP is a **stateless** protocol. This means every single request is completely independent. When you click "Login" and the server verifies your password, the server immediately forgets who you are the second the request ends. 

To keep a user logged in across multiple page loads, we need a way to persist "state."

---

## 1. Cookies

A **Cookie** is a tiny piece of text data (max 4KB) that a server asks the browser to store on the user's computer. 

### How it works:
1. The server responds to a login request with a special header: 
   `Set-Cookie: sessionId=abc123`
2. The browser saves this cookie.
3. On **every subsequent request** to that same domain, the browser automatically attaches the cookie to the request headers:
   `Cookie: sessionId=abc123`
4. The server reads the header and remembers who you are.

### Critical Security Attributes
Cookies are vulnerable to theft. You must always secure them using these attributes:
- **`Secure`:** The cookie will only be sent over encrypted HTTPS connections, preventing man-in-the-middle attacks on public Wi-Fi.
- **`HttpOnly`:** The cookie cannot be accessed via JavaScript (`document.cookie`). This entirely prevents XSS (Cross-Site Scripting) attacks from stealing the session.
- **`SameSite=Lax` or `Strict`:** Prevents the browser from sending the cookie if the request originated from a different website. This prevents CSRF (Cross-Site Request Forgery) attacks.

---

## 2. Server-Side Sessions (Stateful)

The traditional way to handle authentication is using **Sessions**.

1. **Login:** The user sends a username/password.
2. **Session Creation:** The server verifies the credentials, creates a unique "Session ID", and saves that ID in its own database (or memory store like Redis) alongside the user's details.
3. **Set-Cookie:** The server sends the Session ID to the browser inside an `HttpOnly` cookie.
4. **Subsequent Requests:** The browser sends the Session ID cookie. The server looks up the ID in its database, finds the user, and authorizes the request.

**Pros:** Highly secure. You can instantly revoke a session (log someone out) just by deleting the ID from the server's database.
**Cons:** Requires database lookups on every single request. Can be difficult to scale if you have multiple servers (they all need access to the shared Redis cache).

---

## 3. JWTs & Bearer Tokens (Stateless)

Modern APIs and SPAs (Single Page Applications) often use **JSON Web Tokens (JWTs)** instead of sessions.

1. **Login:** The user sends a username/password.
2. **Token Creation:** The server verifies the credentials and creates a JWT. The JWT contains the user's ID and an expiration date. The server *cryptographically signs* the JWT using a secret key, but it **does not save it in a database**.
3. **Send Token:** The server sends the JWT to the browser.
4. **Subsequent Requests:** The browser attaches the JWT to the `Authorization: Bearer <token>` header (or stores it in a cookie). 
5. **Verification:** The server receives the JWT, verifies the cryptographic signature to ensure it wasn't tampered with, reads the user ID inside, and authorizes the request—*without ever talking to a database*.

**Pros:** Extremely fast and scalable. The server doesn't need to store anything in memory or Redis.
**Cons:** Because the server doesn't store the token, **you cannot revoke a JWT** before its expiration date. If a hacker steals a JWT that lasts for 30 days, they have 30 days of free access. 

### The JWT Security Compromise
Because you can't revoke JWTs, the industry standard is to use short-lived Access Tokens paired with long-lived Refresh Tokens.
- **Access Token (JWT):** Lasts 15 minutes. Used for API calls. Stored in memory.
- **Refresh Token (Opaque String):** Lasts 30 days. Used to get a new Access Token. Stored in an `HttpOnly` cookie and saved in the database so it can be revoked if stolen.

---

## Summary

1. **Stateless HTTP:** HTTP forgets who you are immediately. We use cookies or headers to remind the server on every request.
2. **Cookies:** Automatically sent by the browser. Must be secured with `HttpOnly`, `Secure`, and `SameSite` to prevent XSS and CSRF attacks.
3. **Sessions:** Stateful. The server stores a Session ID in a database and gives the browser a cookie. Very secure, easy to revoke, but requires database lookups.
4. **JWTs:** Stateless. A signed token containing user data. Extremely fast and scalable, but impossible to revoke before expiration without complex workarounds.
