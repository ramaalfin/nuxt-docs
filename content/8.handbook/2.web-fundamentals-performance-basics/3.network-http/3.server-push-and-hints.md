# Server Push & Resource Hints (Prefetch, Preload)

Modern browsers are incredibly smart, but they can't predict the future. They parse HTML top-to-bottom and only request assets (like CSS, JS, or WebFonts) when they explicitly discover them in the document.

To optimize the Critical Rendering Path, developers can use **Resource Hints** to give the browser a "heads-up" about what assets will be needed soon.

---

## 1. Resource Hints (The Client-Side Solution)

Resource hints are simple HTML `<link>` tags placed in the `<head>` of the document. They tell the browser to start fetching or preparing assets before the browser officially realizes it needs them.

### `preload` (High Priority, Current Page)
Use `preload` for critical assets that are absolutely necessary for rendering the *current* page, but might be discovered late in the parsing process (e.g., a WebFont referenced deep inside a CSS file, or the hero image of a page).

```html
<!-- The browser starts downloading this font immediately, without waiting for the CSS to parse -->
<link rel="preload" href="/fonts/inter-bold.woff2" as="font" type="font/woff2" crossorigin>
```
*Warning:* If you preload an asset but the browser doesn't end up using it within a few seconds, the browser will log a warning in the console. Do not preload everything; only preload critical assets.

### `prefetch` (Low Priority, Future Navigation)
Use `prefetch` for assets that are *not* needed for the current page, but are highly likely to be needed on the *next* page the user navigates to.

```html
<!-- The browser downloads this in the background when it is idle -->
<link rel="prefetch" href="/js/checkout-flow.js" as="script">
```
When the user clicks the "Checkout" button, the JS file is already sitting in the browser's disk cache, resulting in an instant page load.

### `preconnect` (Network Handshake)
If you know your app will fetch data from a third-party API (e.g., `api.stripe.com`), but you don't know the exact URL of the asset yet, you can use `preconnect`. This tells the browser to perform the DNS lookup, TCP handshake, and TLS negotiation ahead of time.

```html
<link rel="preconnect" href="https://api.stripe.com">
```
When the API call is finally made, the connection is already warm, saving hundreds of milliseconds.

## 2. HTTP/2 Server Push (The Server-Side Solution)

Introduced in HTTP/2, **Server Push** was designed to be the ultimate performance optimization. 

### How it Worked
When a browser requested `index.html`, the server could say: *"I see you are asking for the HTML. I know that as soon as you parse this HTML, you will ask me for `styles.css` and `app.js`. So, I'm going to push those to you right now, before you even ask."*

### Why it Failed (and is being deprecated)
In theory, Server Push sounded amazing. In practice, it was a disaster.
1. **Cache Ignorance:** The server had no way of knowing if the browser *already had* `styles.css` in its local cache. Servers would blindly push megabytes of CSS and JS to clients that didn't need them, wasting massive amounts of bandwidth and actually slowing down page loads.
2. **Complexity:** Configuring Server Push on CDNs and web servers was incredibly complex.
3. **Head-of-Line Blocking:** Pushed assets often competed for bandwidth with the actual HTML document the user was waiting for.

As a result, Chrome removed support for HTTP/2 Server Push in 2022, and other browsers have followed suit. 

### The Modern Alternative: `103 Early Hints`
To replace Server Push, the web adopted a new HTTP status code: **103 Early Hints**.

When a server receives a request for an HTML page, it might take the database 200ms to generate the HTML. During that 200ms of "think time", the server can instantly send a `103 Early Hints` response containing `Link: <styles.css>; rel=preload` headers.

The browser receives the hints and starts downloading the CSS *while* the server is still generating the HTML. It solves the performance goal of Server Push without the cache-ignorance flaws.

---

## Summary

1. **`preload`:** High-priority download for assets needed on the *current* page.
2. **`prefetch`:** Low-priority background download for assets needed on the *next* page.
3. **`preconnect`:** Warms up the DNS/TCP/TLS connection to a third-party domain.
4. **Server Push:** A deprecated HTTP/2 feature that pushed assets unprompted. It failed because it ignored browser caches.
5. **103 Early Hints:** The modern, cache-friendly replacement for Server Push.
