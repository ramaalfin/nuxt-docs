# CORS (Cross-Origin Resource Sharing)

One of the most common and frustrating errors developers encounter when building web applications is the dreaded **CORS Error**: 

> *Access to fetch at 'https://api.example.com' from origin 'https://mywebsite.com' has been blocked by CORS policy.*

Understanding what CORS is—and more importantly, why it exists—will save you hours of debugging.

---

## 1. The Same-Origin Policy (SOP)

Before understanding CORS, you must understand the **Same-Origin Policy**. This is a fundamental security mechanism built into all web browsers.

The SOP states that a web browser permits scripts contained in a first web page to access data in a second web page, but **only if both web pages have the same origin**.

An **Origin** consists of three parts:
1. **Protocol** (`http://` vs `https://`)
2. **Domain** (`example.com` vs `api.example.com`)
3. **Port** (`:80` vs `:3000`)

If *any* of those three things are different, it is considered a **Cross-Origin** request.

### Why does the SOP exist?
Imagine you are logged into your bank (`bank.com`), so your browser has an active session cookie. You then visit a malicious website (`evil.com`). 
Without the SOP, a JavaScript file on `evil.com` could secretly run `fetch('https://bank.com/transfer')` in the background. Because your browser automatically attaches cookies to requests, the bank would think *you* made the request and transfer the money.
The SOP prevents `evil.com` from reading the response of `bank.com`.

---

## 2. Enter CORS

The web evolved, and developers needed legitimate reasons to make cross-origin requests. For example, a frontend hosted on `https://mywebsite.com` legitimately needs to fetch data from a backend API hosted on `https://api.mywebsite.com` (different subdomains = different origin).

**CORS (Cross-Origin Resource Sharing)** is an HTTP-header based mechanism that allows a server to tell the browser: *"Hey, it's okay. I explicitly allow `mywebsite.com` to read my data."*

### How CORS works

When the browser realizes your JavaScript is attempting a Cross-Origin `fetch`, it intervenes. 

1. **The Request:** The browser automatically adds an `Origin` header to your request.
   ```http
   Origin: https://mywebsite.com
   ```
2. **The Response:** The server receives the request. If the server wants to allow it, it must respond with a specific header:
   ```http
   Access-Control-Allow-Origin: https://mywebsite.com
   ```
   *(The server can also reply with `*` to allow absolutely anyone to access the API, which is common for public APIs like weather services).*
3. **The Browser Check:** The browser receives the response. It checks if the `Access-Control-Allow-Origin` matches the site the user is on. If it does, the browser hands the JSON data to your JavaScript. If it doesn't match (or the header is missing entirely), the browser throws a CORS Error and refuses to give your JavaScript the data.

---

## 3. Preflight Requests (The `OPTIONS` Ping)

Sometimes, the browser decides a request is too "dangerous" to send blindly. 

If your request does something that could permanently alter data on the server (like a `PUT` or `DELETE` request), or if you attach custom headers (like `Authorization: Bearer <token>`), the browser performs a **Preflight Request**.

Before sending your actual request, the browser pauses and silently sends an HTTP `OPTIONS` request to the server. 
It essentially asks the server: *"I have a client on `mywebsite.com` who wants to send a `DELETE` request with an `Authorization` header. Is that allowed?"*

The server must respond with headers confirming it is okay:
```http
Access-Control-Allow-Origin: https://mywebsite.com
Access-Control-Allow-Methods: GET, POST, DELETE
Access-Control-Allow-Headers: Authorization
```

If the server agrees, the browser proceeds to send the actual `DELETE` request. If the server denies it, the browser throws a CORS error and the `DELETE` request is never sent.

---

## Summary

1. **Same-Origin Policy (SOP):** Browsers block scripts from reading data from different domains to prevent malicious sites from hacking your active sessions on other sites.
2. **CORS:** A set of HTTP headers that allows a server to bypass the SOP and explicitly whitelist specific domains to read its data.
3. **The Enforcer:** CORS is enforced entirely by the **Browser**, not the server. The server still receives the request and might even process it, but the browser refuses to let your JavaScript see the response.
4. **Preflight (`OPTIONS`):** For complex or destructive requests, the browser pings the server first to ask for permission before sending the real request.
