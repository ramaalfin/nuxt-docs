# RequestAnimationFrame Timing

When creating JavaScript-driven animations or manipulating the DOM heavily, developers historically relied on `setTimeout` or `setInterval`. Today, using anything other than `requestAnimationFrame` (rAF) for visual updates is considered a critical anti-pattern.

Understanding the timing of `requestAnimationFrame` reveals exactly how the browser's rendering pipeline syncs with your monitor.

---

## 1. The Problem with `setTimeout`

Standard monitors refresh their screens 60 times per second (60Hz), meaning the browser has exactly **16.6 milliseconds** to construct the DOM, calculate Layout, Paint, and Composite every single frame.

If you use `setInterval(animate, 16)`, you are blindly firing a function every 16ms. 

**The Issues:**
1. **No Sync (Screen Tearing):** The JavaScript timer doesn't know when the monitor is actually drawing the next frame. Your script might execute exactly in the middle of a screen refresh, leading to missed frames (jank) or screen tearing.
2. **Hidden Execution:** If the user switches to a different browser tab, `setInterval` keeps firing relentlessly in the background, draining the user's CPU and laptop battery to animate something they can't even see.

---

## 2. Enter `requestAnimationFrame`

`window.requestAnimationFrame(callback)` was introduced to solve these exact problems. It tells the browser: *"I want to perform an animation. Please call this function right before you repaint the next frame."*

### Perfect Synchronization (V-Sync)
The browser schedules the `rAF` callback to execute at the most optimal time—perfectly aligned with the display's refresh rate (V-Sync). 
- If the monitor is 60Hz, it fires 60 times a second. 
- If the user has a modern 144Hz gaming monitor, the browser will automatically adapt and fire 144 times a second for incredibly smooth motion.

### Battery Optimization
If the user switches tabs or minimizes the browser, the browser pauses all `requestAnimationFrame` calls. The animation simply stops, saving battery life and CPU cycles.

---

## 3. Where rAF Fits in the Rendering Pipeline

To understand rAF, you must understand exactly *when* it executes in the frame lifecycle.

1. **Input Events:** The browser handles user input (clicks, scrolls).
2. **JavaScript / rAF:** The browser executes any pending `requestAnimationFrame` callbacks.
3. **Style Calculation:** The browser calculates CSSOM and applies classes.
4. **Layout:** The browser calculates the geometry of elements.
5. **Paint:** The browser rasterizes pixels.
6. **Composite:** The GPU draws the frame to the monitor.

Because `requestAnimationFrame` fires **before** Style, Layout, and Paint, it is the absolute best place to read from or write to the DOM. If you change a `style` property inside a rAF callback, the browser seamlessly incorporates that change into the imminent Layout and Paint phases without any wasted effort.

---

## 4. The `requestAnimationFrame` Loop

Unlike `setInterval`, `requestAnimationFrame` only fires **once**. To create a continuous animation, you must recursively call `requestAnimationFrame` from inside your callback.

```javascript
const box = document.getElementById('box');
let position = 0;

function animate(timestamp) {
  position += 2; // Move 2 pixels per frame
  box.style.transform = `translateX(${position}px)`;

  // Stop condition (e.g., stop at 500px)
  if (position < 500) {
    // Schedule the next frame
    requestAnimationFrame(animate); 
  }
}

// Kick off the animation
requestAnimationFrame(animate);
```

### The `timestamp` Argument
Notice the `timestamp` argument automatically passed to the callback. This is a high-resolution timestamp (`performance.now()`) indicating exactly when the rAF fired. 

Because frame rates can fluctuate (e.g., dropping from 60fps to 30fps under heavy load), you should always use this `timestamp` to calculate **delta time** (the time passed since the last frame) to ensure your animations move at a consistent physical speed, regardless of the frame rate.

---

## Summary

1. **Never use `setTimeout` for animations:** It is unaware of the monitor's refresh rate and runs even when tabs are hidden.
2. **Perfect Sync:** `requestAnimationFrame` is perfectly synchronized with the monitor's hardware refresh rate (V-Sync).
3. **Pipeline Positioning:** It executes right before the browser calculates Style, Layout, and Paint, making it the most efficient place to mutate the DOM.
4. **Battery Friendly:** Animations automatically pause when the tab is out of focus.
