# Paint (Rasterization, Layers)

After the Layout phase has calculated the exact geometry (size and position) of every element on the screen, the browser still hasn't drawn a single pixel. It only has a mathematical blueprint. 

The next step is to convert that blueprint into actual pixels on the screen. This process happens in two distinct stages: **Paint** and **Compositing**.

---

## 1. The Paint Phase (Rasterization)

**Painting** (or Rasterization) is the process of filling in the pixels based on the calculated layout and the CSS properties (colors, borders, shadows, text, images).

Instead of drawing the entire page at once, the browser breaks the process down into a sequence of drawing commands. It asks: *What draws first?*

### Stacking Contexts and Paint Order
CSS is a 3D environment. Elements can overlap. The browser paints elements in a specific order governed by the **Stacking Context**. 

Even without `z-index`, elements are painted in this order (from bottom to top):
1. Background color of the root element
2. Background color and images of block-level elements
3. Borders
4. Child elements (in document flow order)
5. Text and inline elements
6. Out-of-flow elements (positioned elements)

When you introduce `z-index`, you create new Stacking Contexts, forcing the browser to group and order the painting of those elements explicitly.

---

## 2. Layers and Compositing

If the browser painted everything onto a single, giant canvas, scrolling would be incredibly slow. Every time you scrolled down 1 pixel, the browser would have to repaint the entire screen!

To solve this, modern browsers use **Layers** and a **Compositor Thread**.

### How Layers Work
Instead of one canvas, the browser paints elements onto different independent "Layers" (like transparent sheets of acetate in a projector). 
- The background might be Layer A.
- A scrolling text box might be Layer B.
- A floating header might be Layer C.

### The Compositor Phase
Once the layers are painted (rasterized), they are handed off to the **Compositor Thread** (which usually runs on the GPU). The Compositor's job is simply to stack these layers together, apply any final transformations (like moving or fading them), and display them on the monitor.

When you scroll, the browser *doesn't need to repaint*. The Compositor just takes Layer B and shifts its Y-coordinate upward on the GPU. This is why hardware-accelerated scrolling is so buttery smooth.

---

## 3. Forcing GPU Acceleration

Because compositing is handled by the GPU, it is incredibly fast. We can tell the browser to promote a specific element to its own independent Layer, bypassing the Layout and Paint phases during animations.

This is called **Hardware Acceleration**.

### Properties that trigger Layer creation:
1. **`transform: translate3d(0, 0, 0)`** or **`will-change: transform`**
2. `<video>` and `<canvas>` elements
3. Elements animating `opacity` or `transform`

### The Golden Rule of Web Animation
If you want 60fps (frames-per-second) animations, **you should only animate `transform` and `opacity`.**

- **Animating `width` or `margin` (Bad):**
  Changes Layout -> Triggers Repaint -> Triggers Compositing (Uses CPU, slow, janky).
  
- **Animating `transform: translateX(100px)` (Good):**
  Skips Layout -> Skips Paint -> Only triggers Compositing (Uses GPU, 60fps, buttery smooth).

### A Warning on `will-change`
While it is tempting to put `will-change: transform` on every element to force them onto the GPU, don't. Every layer consumes VRAM (Video RAM). If you create too many layers, the browser will run out of memory, crash, or slow to a crawl. Only promote elements that are actively animating or frequently changing.

---

## Summary

1. **Paint (Rasterization):** Converts the geometric layout into a sequence of drawing commands to fill in pixels.
2. **Stacking Contexts:** Dictates the Z-axis order in which elements are painted.
3. **Layers:** Browsers paint elements onto separate layers to avoid repainting everything during simple interactions (like scrolling).
4. **Compositing:** The GPU stacks the layers together and applies final transforms/opacity.
5. **Animation Performance:** Only animate `transform` and `opacity` to keep work exclusively on the GPU Compositor thread, ensuring 60fps performance.
