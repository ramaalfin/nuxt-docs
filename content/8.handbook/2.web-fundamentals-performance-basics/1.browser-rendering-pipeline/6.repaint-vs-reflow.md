# Repaint vs. Reflow (Layout Thrashing)

Understanding the difference between **Repaint** and **Reflow** (also known as Layout) is the absolute foundation of frontend performance optimization. 

When JavaScript or CSS changes the visual appearance of an element, the browser must update the screen. However, not all visual changes are created equal. Some updates are incredibly expensive (Reflows), while others are relatively cheap (Repaints).

---

## 1. What is a Reflow (Layout)?

A **Reflow** occurs when the browser has to recalculate the positions and geometries (width, height, top, left) of elements in the document.

Because the CSS Box Model is interconnected, changing the geometry of one element often affects its siblings, its children, and its parents. 
For example, if you increase the width of a `<div>`, the browser might have to push the `<div>` next to it further to the right, which might cause a paragraph of text to wrap to a new line, which increases the height of the parent container, and so on.

### Triggers for Reflow:
- Changing geometric properties: `width`, `height`, `padding`, `margin`, `border-width`.
- Changing position: `top`, `bottom`, `left`, `right`.
- Changing the DOM: Adding or removing elements.
- Resizing the browser window or changing font sizes.

**Cost:** 🔴 **Extremely High**. Reflows block the Main Thread and are mathematically expensive.

---

## 2. What is a Repaint?

A **Repaint** occurs when changes are made to an element's appearance that do *not* affect its geometry or layout. The element stays the exact same size and in the exact same place, but its "skin" changes.

When a Repaint happens, the browser skips the Layout phase entirely and jumps straight to drawing new pixels onto the existing layer.

### Triggers for Repaint:
- `color`
- `background-color` or `background-image`
- `box-shadow`
- `border-style` or `border-color` (but *not* `border-width`)
- `visibility: hidden` (The element is invisible, but still takes up its original geometric space).

**Cost:** 🟡 **Moderate**. It skips the Layout phase, but still requires the CPU to recalculate and draw pixels.

---

## 3. The Holy Grail: Compositor-Only Updates

As discussed in previous chapters, if you animate properties that only affect the **Compositor Thread**, you skip *both* Reflow and Repaint. The GPU simply shifts the existing pixel layers around.

### Triggers for Compositor Updates:
- `transform` (Translate, Scale, Rotate, Skew)
- `opacity`

**Cost:** 🟢 **Almost Zero**. Runs at 60fps effortlessly on the GPU.

---

## 4. Layout Thrashing (Forced Synchronous Layout)

The most catastrophic performance issue in modern web apps is **Layout Thrashing**.

Browsers are smart. If your JavaScript changes `element.style.width` three times in a row, the browser doesn't calculate the layout three times. It batches those changes and calculates the layout once at the end of the frame.

However, if you **read** a layout property immediately after **writing** to one, you break the batching mechanism. The browser is forced to pause your JavaScript, recalculate the entire layout immediately (synchronously) just to give you the exact number, and then resume your script.

### Example of Layout Thrashing (BAD)

```javascript
const boxes = document.querySelectorAll('.box');

for (let i = 0; i < boxes.length; i++) {
  // 1. READ: The browser must calculate the layout to find offsetWidth
  const width = document.body.offsetWidth; 
  
  // 2. WRITE: We invalidate the layout we just calculated
  boxes[i].style.width = width + 'px'; 
}
```
In a loop of 100 boxes, this causes 100 Synchronous Layouts. The page will freeze completely.

### The Fix: Read First, Write Last (GOOD)

```javascript
const boxes = document.querySelectorAll('.box');

// 1. READ ONCE
const width = document.body.offsetWidth; 

for (let i = 0; i < boxes.length; i++) {
  // 2. WRITE MANY
  // The browser batches these changes and calculates layout once at the end.
  boxes[i].style.width = width + 'px'; 
}
```

Libraries like `FastDOM` are built specifically to automatically batch DOM reads and writes to prevent Layout Thrashing.

---

## Summary

| Phase Skipped | Property Animated | Performance Cost | Main Thread Blocked? |
| :--- | :--- | :--- | :--- |
| **None (Reflow)** | `width`, `height`, `top`, `left`, `margin` | 🔴 Very High | Yes |
| **Layout (Repaint)** | `color`, `background`, `box-shadow` | 🟡 Moderate | Yes |
| **Layout & Paint (Composite)**| `transform`, `opacity` | 🟢 Extremely Low | No (Runs on GPU) |
