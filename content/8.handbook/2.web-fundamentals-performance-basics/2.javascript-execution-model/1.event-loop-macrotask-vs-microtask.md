# Event Loop (Macrotask vs Microtask)

JavaScript is a **single-threaded**, non-blocking, asynchronous, concurrent language. 
Because it has only one Call Stack, it can only execute one piece of code at a time. So, how does it handle asynchronous operations like fetching data from an API or waiting for a timer to finish without freezing the entire page?

The answer is the **Event Loop**.

---

## 1. The Call Stack and Web APIs

### The Call Stack
The Call Stack is a LIFO (Last In, First Out) data structure that records where we are in the program. If we step into a function, we push it onto the stack. If we return from a function, we pop it off the stack. 
**Crucial Rule:** The browser cannot update the UI, respond to clicks, or paint the screen as long as the Call Stack is not empty.

### Web APIs (The Browser's Background Threads)
When you call `setTimeout` or `fetch`, those functions don't execute on the JavaScript Call Stack. They are handed off to the browser's Web APIs (written in C++). The browser handles the timer or the network request in the background.

When the background task finishes, the Web API doesn't just jam the callback function straight back onto the Call Stack (that would interrupt whatever JavaScript is currently doing). Instead, it pushes the callback into a **Task Queue**.

---

## 2. The Event Loop

The **Event Loop** has one incredibly simple job: it constantly looks at the Call Stack and the Task Queues.

**The Golden Rule of the Event Loop:**
*If the Call Stack is empty, take the first task from the queue and push it onto the Call Stack to execute.*

However, there isn't just one queue. There are two primary queues that dictate execution order: the **Macrotask Queue** and the **Microtask Queue**.

---

## 3. Macrotasks (The Standard Queue)

Macrotasks (often just called "Tasks") represent discrete, independent blocks of work. 

**Common Macrotasks:**
- `setTimeout()` and `setInterval()` callbacks
- UI events (clicks, keypresses)
- I/O (network responses, disk reads)
- `<script>` tag execution

When a Macrotask is pulled from the queue, it runs to completion. When it finishes, the browser *may* choose to update the UI (Paint) before grabbing the next Macrotask.

---

## 4. Microtasks (The Priority Queue)

Microtasks are smaller, highly prioritized pieces of work that usually originate from Promise resolutions or specific API calls.

**Common Microtasks:**
- `Promise.then()`, `.catch()`, `.finally()`
- `queueMicrotask()`
- `MutationObserver` callbacks

### The Microtask VIP Pass
Microtasks have absolute priority over Macrotasks. 
When the Call Stack empties, the Event Loop will check the **Microtask Queue first**. It will pull a Microtask, push it to the stack, and execute it. 

**Here is the dangerous part:** The Event Loop will *not* move on to the next Macrotask, nor will it allow the browser to Paint, until the Microtask Queue is **completely empty**. 
If a Microtask schedules another Microtask, that new Microtask gets appended to the end of the queue and will execute immediately. You can easily create an infinite Microtask loop that freezes the browser forever.

---

## 5. Visualizing the Execution Order

Let's look at the classic interview question to understand exactly how the Event Loop prioritizes queues.

```javascript
console.log('1. Script start'); // Synchronous (Call Stack)

setTimeout(() => {
  console.log('2. setTimeout'); // Macrotask
}, 0);

Promise.resolve().then(() => {
  console.log('3. Promise 1'); // Microtask
}).then(() => {
  console.log('4. Promise 2'); // Microtask
});

console.log('5. Script end'); // Synchronous (Call Stack)
```

### Step-by-Step Execution:
1. `console.log('1')` runs synchronously.
2. `setTimeout` is called. The Web API starts a timer and immediately pushes the callback to the **Macrotask Queue**.
3. `Promise.resolve()` resolves instantly. Its `.then()` callback is pushed to the **Microtask Queue**.
4. `console.log('5')` runs synchronously.
5. The script finishes. **The Call Stack is now empty.**
6. The Event Loop checks the **Microtask Queue**. It finds the Promise 1 callback. It executes it, logging `'3'`.
7. That callback returns, which resolves the next Promise, pushing Promise 2's callback into the **Microtask Queue**.
8. The Event Loop checks the Microtask Queue again (it must be empty before moving on). It finds Promise 2, executes it, logging `'4'`.
9. The Microtask Queue is finally empty. The browser may choose to Paint the screen now.
10. The Event Loop moves to the **Macrotask Queue**. It finds the `setTimeout` callback. It executes it, logging `'2'`.

**Final Output:** `1, 5, 3, 4, 2`.

---

## Summary

1. **Call Stack:** Single-threaded execution. Blocking it freezes the UI.
2. **Macrotasks:** Standard background tasks (`setTimeout`, UI Events). One executes, then the browser can Paint.
3. **Microtasks:** High-priority tasks (Promises, `queueMicrotask`). The Event Loop must empty the *entire* Microtask queue before rendering or taking a Macrotask.
4. **Performance:** Heavy microtask usage can silently block the UI rendering pipeline because the browser is forbidden from painting while the Microtask queue is populated.
