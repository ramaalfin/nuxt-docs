# Memory Management & Garbage Collection

Unlike low-level languages like C or C++ where developers must manually allocate and free memory (using functions like `malloc()` and `free()`), JavaScript is a **garbage-collected** language. 

The JavaScript engine automatically allocates memory when you create variables, objects, or functions, and it frees that memory when it determines they are no longer needed. However, understanding *how* this works is crucial for avoiding Memory Leaks in complex applications.

---

## 1. The Memory Lifecycle

Regardless of the programming language, the memory lifecycle is always the same:

1. **Allocate Memory:** The OS reserves a block of memory for your program. (In JS, this happens automatically when you declare variables).
2. **Use Memory:** The program reads and writes to the allocated memory.
3. **Release Memory:** The memory is freed so the OS can give it to other programs. (In JS, this happens automatically via the Garbage Collector).

### The Stack vs. The Heap
JavaScript engines allocate memory in two primary places:
- **The Stack:** Used for static data. Primitive values (numbers, strings, booleans) and references (pointers) to objects are stored here. The Stack is fast, ordered (LIFO), and has a fixed size limit.
- **The Heap:** Used for dynamic data. Objects, Arrays, and Functions are stored here. The Heap is a large, unstructured region of memory.

```javascript
const name = "Alice"; // Allocated on the Stack
const user = { age: 30 }; // The object is on the Heap. The 'user' pointer is on the Stack.
```

---

## 2. Garbage Collection Algorithms

How does the JavaScript engine know when to free memory from the Heap? It uses an algorithm to determine if an object is "reachable".

### The Old Way: Reference Counting (Obsolete)
Historically, early browsers (like IE6) used a simplistic algorithm called Reference Counting. If an object had zero references pointing to it, it was garbage collected.

*The fatal flaw:* **Circular References**. If two objects reference each other, their reference count is never zero, even if they are completely detached from the rest of the application. This caused massive memory leaks.

### The Modern Way: Mark-and-Sweep
All modern JavaScript engines (V8, SpiderMonkey, JavaScriptCore) use the **Mark-and-Sweep** algorithm. 

Instead of counting references, the engine starts from a set of "Roots" (global objects like `window`, and the current Call Stack). 
1. **Mark Phase:** The engine crawls the object tree, starting from the roots, and "marks" every object it can reach.
2. **Sweep Phase:** It then scans the entire Heap. Any object that is *not* marked is considered unreachable garbage, and its memory is freed.

Because a detached island of circularly referenced objects cannot be reached from the global Root, Mark-and-Sweep effortlessly cleans them up.

---

## 3. Common Causes of Memory Leaks

Even with a highly optimized Mark-and-Sweep Garbage Collector, developers can still create Memory Leaks by accidentally keeping objects "reachable" from the Root when they are no longer needed.

### 1. Accidental Global Variables
If you assign a value to an undeclared variable (in non-strict mode), it becomes attached to the `window` object. Since the `window` object is a Root, it will never be garbage collected.

```javascript
function loadData() {
  // ❌ Missing 'const' or 'let'. 'data' is attached to window!
  data = new Array(1000000).fill('Heavy Data'); 
}
```
*Fix: Always use `'use strict';` or rely on ES6 modules (which are strict by default).*

### 2. Forgotten Timers and Callbacks
If you start a `setInterval` that references objects in a component, and that component is destroyed, the timer will keep the component's scope alive forever.

```javascript
function startPolling() {
  const massiveData = { /* ... */ };
  
  // ❌ If this timer isn't cleared, massiveData is never freed!
  setInterval(() => {
    console.log(massiveData.id);
  }, 1000);
}
```
*Fix: Always store the timer ID and call `clearInterval()` when the component unmounts.*

### 3. Lingering DOM References
If you remove an element from the DOM tree, but you still have a JavaScript variable pointing to it, the browser cannot free that DOM node's memory.

```javascript
let detachedNodes = [];

function removeButton() {
  const btn = document.getElementById('my-btn');
  document.body.removeChild(btn); // Removed from screen
  
  // ❌ The JS array still holds a reference. The node cannot be garbage collected!
  detachedNodes.push(btn); 
}
```

### 4. Closures
As discussed in the Closures chapter, a closure keeps its parent's lexical scope alive. If a closure accidentally captures a massive variable that it doesn't need, and the closure itself lives a long time, it creates a leak.

---

## 4. `WeakMap` and `WeakSet`

To help manage complex memory relationships, ES6 introduced `WeakMap` and `WeakSet`. 

Unlike standard `Map` and `Set`, the references held by a `WeakMap` are "weak". If the only remaining reference to an object is inside a `WeakMap`, the Garbage Collector will ignore the `WeakMap` and delete the object.

```javascript
let cache = new WeakMap();
let user = { name: "Alice" };

cache.set(user, "Some heavy cached data");

user = null; 
// Because it's a WeakMap, the GC sees that 'user' is null elsewhere.
// It automatically deletes the object and the associated cache data!
```

---

## Summary

1. **Stack vs Heap:** Primitives and pointers live on the fast Stack; objects and functions live on the large Heap.
2. **Mark-and-Sweep:** The modern algorithm that finds unreachable objects by tracing paths from Global Roots. It solves the circular reference problem.
3. **Memory Leaks:** Caused by unintentionally keeping variables "reachable." Watch out for globals, uncleared intervals, lingering DOM references, and heavy closures.
4. **Weak References:** Use `WeakMap` and `WeakSet` to associate data with objects without preventing them from being garbage collected.
