# Closures & Scope Chain

Two of the most powerful and frequently misunderstood concepts in JavaScript are **Scope** and **Closures**. Understanding them is essential for writing secure, modular, and memory-efficient code.

---

## 1. Scope and the Scope Chain

**Scope** determines the accessibility (visibility) of variables and functions at any given part of your code. 

JavaScript has three main types of scope:
1. **Global Scope:** Variables declared outside any function or block. Accessible everywhere.
2. **Function Scope:** Variables declared inside a function using `var`, `let`, or `const`. Accessible only within that function.
3. **Block Scope:** Variables declared inside a block (`{ ... }`) using `let` or `const` (but *not* `var`). Accessible only within that block.

### The Scope Chain
When JavaScript needs to find the value of a variable, it looks in the current scope. If it doesn't find it, it moves *up* to the parent scope, and then the grandparent scope, all the way up to the Global Scope. This hierarchical path is called the **Scope Chain**.

```javascript
const globalVar = 'I am global';

function parentFunc() {
  const parentVar = 'I am in parentFunc';

  function childFunc() {
    const childVar = 'I am in childFunc';
    
    // The engine looks for parentVar in childFunc's scope -> Not found
    // It moves UP the scope chain to parentFunc's scope -> Found!
    console.log(parentVar); 
    
    // It moves up to the Global scope to find globalVar -> Found!
    console.log(globalVar); 
  }
  
  childFunc();
}
```

*Crucial Rule:* The scope chain only goes UP (from inner to outer). An outer scope can *never* access variables defined in an inner scope.

---

## 2. What is a Closure?

A **Closure** is created when a function "remembers" and accesses variables from its lexical scope (its parent's scope) *even after the parent function has finished executing and returned*.

This happens because functions in JavaScript are first-class citizens; they can be returned from other functions.

### The Classic Closure Example

```javascript
function createCounter() {
  let count = 0; // This variable is in the parent's scope

  return function increment() {
    count++; // The inner function accesses the parent's variable
    return count;
  };
}

const myCounter = createCounter();

// createCounter() has finished executing. Its execution context is gone.
// However, myCounter STILL has access to 'count'!

console.log(myCounter()); // 1
console.log(myCounter()); // 2
console.log(myCounter()); // 3
```

### How does this work?
Normally, when a function finishes executing, its variables are garbage-collected and destroyed. However, because `increment()` relies on the `count` variable, the JavaScript engine creates a **Closure**. It preserves `count` in a special heap memory area attached to the `myCounter` function instance.

## 3. Practical Uses for Closures

Closures are not just a theoretical quirk; they are heavily used in modern JavaScript development.

### 1. Data Privacy (Encapsulation)
Before classes and private `#fields` were introduced, closures were the only way to create truly private variables in JavaScript.

```javascript
function createBankAccount(initialBalance) {
  let _balance = initialBalance; // Private variable

  return {
    deposit: function(amount) {
      _balance += amount;
      return _balance;
    },
    getBalance: function() {
      return _balance;
    }
  };
}

const account = createBankAccount(100);
account.deposit(50);
console.log(account.getBalance()); // 150
console.log(account._balance); // undefined (It is completely hidden!)
```

### 2. Function Currying & Partial Application
Closures allow you to create pre-configured functions.

```javascript
function multiplyBy(factor) {
  return function(number) {
    return number * factor;
  };
}

const double = multiplyBy(2);
const triple = multiplyBy(3);

console.log(double(5)); // 10
console.log(triple(5)); // 15
```

### 3. Event Listeners and Callbacks
Whenever you attach an event listener that uses variables from its parent component (e.g., in React or Vue components), you are using closures.

---

## 4. The Dark Side: Memory Leaks

Because closures preserve variables in memory long after their parent functions have died, they can cause **Memory Leaks** if used carelessly.

If a closure holds onto a massive object or a detached DOM element, the Garbage Collector cannot clean it up as long as the closure function itself is still referenced anywhere in the application.

```javascript
function attachHandler() {
  const massiveData = new Array(1000000).fill('Data');
  const btn = document.getElementById('my-btn');

  btn.addEventListener('click', () => {
    // This closure forces the browser to keep massiveData in memory forever,
    // even if we never click the button!
    console.log(massiveData[0]); 
  });
}
```

---

## Summary

1. **Scope Chain:** The mechanism where an inner function looks outward to parent scopes to resolve variable names. It only goes one way: UP.
2. **Closure:** A function that remembers the variables in its parent's scope, even after the parent has finished executing.
3. **Use Cases:** Data privacy, function currying, and callbacks heavily rely on closures.
4. **Memory:** Closures prevent Garbage Collection. Be careful not to accidentally hold onto massive objects or DOM nodes in long-living closures.
