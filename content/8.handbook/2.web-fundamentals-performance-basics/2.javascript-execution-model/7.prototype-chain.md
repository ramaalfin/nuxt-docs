# The Prototype Chain

Unlike languages like Java or C++ which use Classical Inheritance (where classes inherit from other classes), JavaScript uses **Prototypal Inheritance**. 

In JavaScript, objects inherit directly from other objects. This mechanism of objects linking to other objects creates the **Prototype Chain**.

---

## 1. What is a Prototype?

Every object in JavaScript (with very few exceptions) has an internal, hidden property called `[[Prototype]]`. 

This `[[Prototype]]` is simply a reference (a pointer) to another object. When you attempt to access a property or method on an object, the JavaScript engine looks for it on the object itself. If it cannot find it, it follows the `[[Prototype]]` link and looks at the parent object.

*Note: In modern JavaScript, you can access an object's prototype using `Object.getPrototypeOf(obj)`, or the legacy (and discouraged) `__proto__` property.*

### The Lookup Process

```javascript
const grandparent = { lastName: "Smith", wealth: "High" };

// Create a parent object that inherits from grandparent
const parent = Object.create(grandparent);
parent.firstName = "John";
parent.wealth = "Medium"; // This shadows the grandparent's property

// Create a child object that inherits from parent
const child = Object.create(parent);
child.firstName = "Timmy";

// 1. Engine looks at child: 'firstName' exists. Returns "Timmy".
console.log(child.firstName); 

// 2. Engine looks at child: 'lastName' missing. 
// Moves to parent: 'lastName' missing. 
// Moves to grandparent: 'lastName' exists. Returns "Smith".
console.log(child.lastName); 

// 3. Engine looks at child: 'wealth' missing. 
// Moves to parent: 'wealth' exists. Returns "Medium" (Shadowing).
console.log(child.wealth); 
```

---

## 2. The End of the Chain: `Object.prototype`

If the engine keeps following the Prototype Chain, where does it stop?

Almost all objects ultimately inherit from a master object called `Object.prototype`. 
This object contains the fundamental methods you use every day, such as `.toString()`, `.hasOwnProperty()`, and `.valueOf()`.

If you ask for a property that doesn't exist anywhere in the chain, the engine will eventually reach the prototype of `Object.prototype`, which is `null`. At that point, it stops searching and returns `undefined`.

```javascript
const obj = {}; // Inherits directly from Object.prototype

console.log(obj.toString()); // "[object Object]" (Found on Object.prototype)
console.log(obj.fakeProperty); // undefined (Hit null at the end of the chain)
```

---

## 3. Constructor Functions and `.prototype`

Before ES6 `class` syntax, developers used **Constructor Functions** to create multiple objects with the same prototype.

Every function in JavaScript automatically gets a property called `.prototype`. This is **NOT** the function's own prototype. Instead, it is the object that will be assigned as the `[[Prototype]]` to any instance created when that function is called with the `new` keyword.

```javascript
function Animal(name) {
  this.name = name;
}

// We attach a method to the constructor's prototype object
Animal.prototype.speak = function() {
  console.log(`${this.name} makes a noise.`);
};

const dog = new Animal("Rex");

dog.speak(); // "Rex makes a noise."
```
Because `dog` was created with `new Animal()`, the engine automatically set `dog`'s hidden `[[Prototype]]` to point to `Animal.prototype`. This allows 10,000 `Animal` instances to all share the exact same `speak` function in memory, rather than creating 10,000 copies of the function!

## 4. ES6 Classes (Syntactic Sugar)

In ES6, JavaScript introduced the `class` keyword. It looks exactly like Classical Inheritance from Java/C#, but under the hood, **it is still just the Prototype Chain**.

```javascript
class Animal {
  constructor(name) {
    this.name = name;
  }

  // This is automatically placed on Animal.prototype!
  speak() {
    console.log(`${this.name} makes a noise.`);
  }
}

class Dog extends Animal {
  // 'extends' simply sets up the prototype chain: 
  // Dog.prototype inherits from Animal.prototype
  bark() {
    console.log("Woof!");
  }
}

const myDog = new Dog("Rex");
myDog.bark(); // Found on Dog.prototype
myDog.speak(); // Found on Animal.prototype
```

The `class` syntax is incredibly useful because it hides the messy boilerplate of manual prototype wiring, but understanding that it is just "syntactic sugar" over prototypes is vital for advanced debugging.

---

## 5. Performance Considerations

Traversing the prototype chain is generally very fast, but looking up properties that are deep in the chain (or don't exist at all) forces the engine to traverse the *entire* chain every time. 

Furthermore, you should **never** modify the prototypes of built-in objects (e.g., `Array.prototype.myCustomMethod = ...`). This practice, called "Monkey Patching," can break native browser optimizations, cause naming collisions with future ECMAScript updates, and make your code unpredictable to other developers.

---

## Summary

1. **Prototypal Inheritance:** Objects inherit directly from other objects via a hidden `[[Prototype]]` link.
2. **The Lookup:** If an object doesn't have a property, the engine walks up the Prototype Chain to find it.
3. **`Object.prototype`:** The top of the chain. Its prototype is `null`, which ends the search.
4. **Memory Efficiency:** Attaching methods to a constructor's `.prototype` ensures all instances share one copy of the function in memory.
5. **ES6 Classes:** Just syntactic sugar over the exact same prototypal system.
