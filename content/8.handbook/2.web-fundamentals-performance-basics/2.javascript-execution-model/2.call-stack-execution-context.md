# Call Stack & Execution Context

Before diving into closures, scope, or the event loop, you must understand exactly how the JavaScript engine reads and executes your code. It does this by creating **Execution Contexts** and managing them via the **Call Stack**.

---

## 1. What is an Execution Context?

An **Execution Context** is an abstract concept describing the environment in which JavaScript code is evaluated and executed. Whenever any code is run in JavaScript, it runs inside an execution context.

There are three types of Execution Contexts:

1. **Global Execution Context (GEC):** The default context. Code that is not inside any function is in the global execution context. It creates a global object (e.g., `window` in the browser) and sets the value of `this` to point to that global object. There is only *one* Global Execution Context per page.
2. **Function Execution Context (FEC):** Every time a function is invoked, a brand new execution context is created for that function.
3. **Eval Function Execution Context:** Code executed inside an `eval` function gets its own context (rarely used, usually bad practice).

---

## 2. The Two Phases of an Execution Context

When an Execution Context is created, it goes through two distinct phases:

### Phase 1: Creation Phase (Memory Allocation)
Before the code actually executes line-by-line, the JavaScript engine scans the code and sets up memory space for variables and functions.
- **Variables:** Variables declared with `var` are allocated in memory and initialized to `undefined`. Variables declared with `let` and `const` are allocated but remain *uninitialized* (in the Temporal Dead Zone).
- **Functions:** Function *declarations* (not function expressions) are placed entirely into memory, meaning they can be called before they appear in the code (Hoisting).

### Phase 2: Execution Phase (Code Execution)
The engine runs the code line-by-line. Variables are assigned their actual values, and functions are executed.

---

## 3. The Call Stack

Because JavaScript is single-threaded, it can only execute one Execution Context at a time. To manage multiple contexts (like a function calling another function), it uses the **Call Stack**.

The Call Stack is a data structure that operates on the **LIFO (Last In, First Out)** principle.

### How it works step-by-step:

Consider this code:
```javascript
function greet() {
  console.log('Hello');
}

function start() {
  greet();
  console.log('Done');
}

start();
```

1. **Script loads:** The engine creates the **Global Execution Context** and pushes it to the bottom of the Call Stack.
2. **`start()` is invoked:** The engine creates a new **Function Execution Context** for `start()` and pushes it *on top* of the Global Context. The engine begins executing the `start` function.
3. **`greet()` is invoked:** Inside `start`, `greet()` is called. A new **Function Execution Context** for `greet()` is created and pushed *on top* of the `start` context.
4. **`console.log('Hello')` is invoked:** Pushed to the stack, executed, and immediately popped off.
5. **`greet()` finishes:** Its execution context is **popped off** the stack. The engine resumes executing `start()`.
6. **`console.log('Done')` is invoked:** Pushed, executed, popped.
7. **`start()` finishes:** Its execution context is **popped off**.
8. **Script finishes:** The Global Execution Context remains until the browser tab is closed.

### Visualizing the Stack (LIFO):

| Stack State | Action |
| :--- | :--- |
| `[Global]` | Engine starts |
| `[Global, start]` | `start()` called |
| `[Global, start, greet]` | `greet()` called |
| `[Global, start, greet, log]` | `console.log()` called inside `greet` |
| `[Global, start, greet]` | `log` finishes and is popped |
| `[Global, start]` | `greet()` finishes and is popped |
| `[Global]` | `start()` finishes and is popped |

---

## 4. Stack Overflow

The Call Stack has a maximum size (usually around 10,000 function calls, depending on the browser). If you push too many execution contexts onto the stack without popping them off, the browser will throw an error: `Uncaught RangeError: Maximum call stack size exceeded`.

This almost always happens due to infinite recursion without a base case:

```javascript
function inception() {
  inception(); // Calls itself forever
}

inception(); // Crash!
```

---

## Summary

1. **Execution Context:** The environment where JavaScript code runs. It has a Creation Phase (hoisting, memory allocation) and an Execution Phase (running the code).
2. **Global Context:** The default environment, creating the global object and `this`.
3. **Function Context:** Created instantly whenever a function is *invoked*.
4. **Call Stack:** A LIFO queue that keeps track of which Execution Context is currently running. If the stack overflows, the browser crashes.
