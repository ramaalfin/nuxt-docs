# Type-Safe API Clients

When building frontend applications, communicating with backend APIs is a major source of potential bugs. If the backend changes a response shape or requires a new query parameter, the frontend often won't know until it crashes at runtime.

**Type-Safe API Clients** solve this problem by ensuring that your frontend knows the exact types of your backend requests and responses at compile time.

---

## 1. The Traditional (Untyped) Approach

Historically, developers have used tools like `fetch` or `axios` directly.

```typescript
// ❌ The old way: No type guarantees
async function getUser(id: string) {
  const response = await fetch(`/api/users/${id}`);
  const data = await response.json(); 
  // 'data' is typed as 'any'!
  
  return data;
}

const user = await getUser("123");
console.log(user.first_name); // Is it first_name? firstName? Who knows!
```

You can manually cast the result using `as`, but this is just a **lie to the compiler**. If the backend changes `first_name` to `firstName`, your TypeScript code will compile perfectly fine, but fail at runtime.

```typescript
type User = { first_name: string };
// ❌ Still dangerous: A blind cast
const data = (await response.json()) as User; 
```

---

## 2. End-to-End Type Safety

The modern standard is **End-to-End (E2E) Type Safety**. This means the frontend and backend share a single source of truth for their types. If the backend schema changes, the frontend build instantly fails if it hasn't been updated.

There are three primary ways to achieve this depending on your tech stack.

### Approach A: tRPC (The Standard for Monorepos)

If you have a full-stack TypeScript repository (e.g., Next.js, Nuxt, or a separate Node backend in a monorepo), [tRPC](https://trpc.io/) is the gold standard.

tRPC allows you to define routers on your server and call them on your client *as if they were local functions*. There is no build step or code generation required.

**Server Setup:**
```typescript
import { initTRPC } from '@trpc/server';
import { z } from 'zod';

const t = initTRPC.create();

export const appRouter = t.router({
  getUser: t.procedure
    // 1. Define the input schema
    .input(z.string())
    // 2. Define the resolver
    .query(async (opts) => {
      // opts.input is strictly typed as a string
      const user = await db.user.findById(opts.input);
      // tRPC automatically infers the return type from this function!
      return { id: user.id, name: user.name }; 
    }),
});

export type AppRouter = typeof appRouter; // Export the type!
```

**Client Usage:**
```typescript
import { createTRPCProxyClient, httpBatchLink } from '@trpc/client';
import type { AppRouter } from '../server/router'; // Import ONLY the type

const trpc = createTRPCProxyClient<AppRouter>({
  links: [httpBatchLink({ url: '/api/trpc' })],
});

// ✅ Type-safe! 'user' is inferred as { id: string, name: string }
// If you pass a number to `getUser`, TypeScript throws an error.
const user = await trpc.getUser.query("123"); 
```

### Approach B: OpenAPI / Swagger Generation (For cross-language teams)

If your backend is written in Python, Go, Java, or Rust, you can't share TypeScript types directly. Instead, you use an **OpenAPI (Swagger) Specification**.

You can use tools like [`openapi-fetch`](https://openapi-ts.dev/openapi-fetch/) or [`orval`](https://orval.dev/) to generate a fully typed client from your backend's `openapi.json` file.

**Usage with openapi-fetch:**
```typescript
import createClient from "openapi-fetch";
// The generated types from your backend
import type { paths } from "./my-openapi-types"; 

const client = createClient<paths>({ baseUrl: "https://api.mybackend.com" });

// ✅ TypeScript knows exactly what endpoints exist and what they return
const { data, error } = await client.GET("/users/{user_id}", {
  params: {
    path: { user_id: "123" }, // Strict typing on URL params!
    query: { include_posts: true } // Strict typing on Query strings!
  }
});

if (data) {
  // data is strictly typed based on the backend schema
  console.log(data.firstName); 
}
```

### Approach C: Framework-Specific Tools (Nuxt/Next)

Meta-frameworks are starting to build this in by default. For example, Nuxt 3 provides `$fetch` via `ofetch` and integrates it with its server routes.

If you hit a local API route in Nuxt, the types are automatically inferred!

```typescript
// --- server/api/users/[id].ts ---
export default defineEventHandler((event) => {
  return { id: event.context.params.id, name: "Alice" };
})

// --- pages/users.vue ---
<script setup lang="ts">
// ✅ Nuxt automatically knows the response type of this local endpoint!
const { data: user } = await useFetch('/api/users/123');

console.log(user.value?.name); 
</script>
```

## 3. The "Manual Wrapper" Fallback

If you can't use tRPC or OpenAPI, you should at least build a manual wrapper around `fetch` that uses Zod for runtime validation. This prevents invalid data from entering your app.

```typescript
import { z } from "zod";

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
});

async function fetchUserSafe(id: string): Promise<z.infer<typeof UserSchema>> {
  const response = await fetch(`/api/users/${id}`);
  const data = await response.json();
  
  // Actually validate the data at runtime!
  const parsed = UserSchema.parse(data); 
  
  return parsed;
}
```

While this requires manually writing schemas, it is infinitely safer than a blind `as User` cast, because it will throw a loud error immediately if the backend sends unexpected data.

---

## Summary

- Never use `as Type` casts when fetching data; they are a lie to the compiler and hide real bugs.
- **tRPC** is the best option for full-stack TypeScript monorepos, requiring zero build steps to share types.
- **OpenAPI Code Generation** is the best option when your backend is written in another language (Go, Python, etc.).
- When tools aren't available, use **Zod** to validate raw API responses at runtime to enforce type safety.
