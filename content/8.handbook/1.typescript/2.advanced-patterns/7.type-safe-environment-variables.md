# Type-Safe Environment Variables

While we previously discussed validating configuration objects at runtime (using tools like Zod), there's another approach to handling environment variables specifically: **Augmenting global types** to ensure autocomplete works natively everywhere `process.env` or `import.meta.env` is used, and using dedicated library ecosystems like **T3 Env**.

---

## 1. Augmenting `NodeJS.ProcessEnv`

If you are using Node.js, `process.env` is typed as an object where every key returns `string | undefined`.

You can override this behavior globally by augmenting the `NodeJS` namespace in a `.d.ts` (declaration) file.

### Step 1: Create a Declaration File

Create an `env.d.ts` file in your project root (or inside your `src` or `types` folder).

```typescript
// env.d.ts

declare namespace NodeJS {
  interface ProcessEnv {
    // We declare the variables we expect to exist
    NODE_ENV: 'development' | 'production' | 'test';
    PORT?: string;
    DATABASE_URL: string;
    NEXT_PUBLIC_API_URL?: string;
  }
}
```

### Step 2: Use it in your code

Now, when you type `process.env.`, TypeScript will provide autocomplete for `DATABASE_URL` and `NODE_ENV`. 

```typescript
// ✅ Autocompletes `DATABASE_URL` and knows it is a `string` (not undefined)
const db = connect(process.env.DATABASE_URL);

// ✅ Knows `NODE_ENV` is specifically one of the three literal strings
if (process.env.NODE_ENV === 'production') {
  // ...
}
```

**The Catch:** This is a *compile-time* guarantee only. It assumes your DevOps team or CI/CD pipeline correctly injects `DATABASE_URL`. If the variable is missing at runtime, your app will still crash. This is why runtime validation (like Zod) is still highly recommended alongside it.

---

## 2. Augmenting `import.meta.env` (Vite / Vue)

If you are using Vite (common in Vue 3 and Nuxt applications), you access environment variables via `import.meta.env` rather than `process.env`.

Vite provides a specific interface you can augment: `ImportMetaEnv`.

```typescript
// env.d.ts
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_APP_TITLE: string;
  readonly VITE_API_ENDPOINT: string;
  // more env variables...
}

interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

Usage in your Vue components:

```typescript
// ✅ Strictly typed!
const title = import.meta.env.VITE_APP_TITLE;
```

## 3. The T3 Env Pattern (@t3-oss/env-core)

While global type augmentation is nice for autocomplete, it doesn't provide runtime safety. Validating `process.env` manually with Zod is great, but it can be boilerplate-heavy.

The **T3 Stack** community created a widely adopted standard for environment variables that combines the best of both worlds: [T3 Env](https://env.t3.gg/).

T3 Env enforces strict validation at runtime while automatically providing full TypeScript types. It also explicitly separates **Server** variables from **Client** variables, preventing you from accidentally leaking API keys to the frontend.

### Setting up `@t3-oss/env-core`

```typescript
import { createEnv } from "@t3-oss/env-core";
import { z } from "zod";

export const env = createEnv({
  // 1. Server-only variables
  server: {
    DATABASE_URL: z.string().url(),
    STRIPE_SECRET_KEY: z.string().min(1),
  },

  // 2. Client (Public) variables
  clientPrefix: "PUBLIC_",
  client: {
    PUBLIC_CLERK_PUBLISHABLE_KEY: z.string().min(1),
  },

  // 3. Manual destructuring (Required for frameworks like Next.js/Nuxt)
  runtimeEnv: process.env,
  
  // 4. Treat empty strings as undefined
  emptyStringAsUndefined: true,
});
```

### Why is this better?

1. **Guaranteed Runtime Safety:** If `DATABASE_URL` is missing, the app crashes with a beautiful, descriptive error message on startup.
2. **Guaranteed Type Safety:** The exported `env` object is perfectly typed.
3. **Leak Prevention:** If you try to access `env.STRIPE_SECRET_KEY` in a client-side component, the compiler will yell at you (or it will crash), ensuring you never leak secrets to the browser.

---

## Summary

- **Global Augmentation:** You can use `declare namespace NodeJS` or `interface ImportMetaEnv` to add autocomplete to raw `process.env` or `import.meta.env` calls.
- **Compile-Time vs Runtime:** Global augmentation only helps you write code; it doesn't protect you from missing variables when deployed.
- **T3 Env:** The modern standard for handling environment variables. It forces you to define Zod schemas, strictly separates client and server variables, and ensures your app fails fast if the environment is misconfigured.
