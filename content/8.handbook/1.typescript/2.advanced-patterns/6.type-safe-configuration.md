# Type-Safe Configuration

Configuration files, environment variables, and settings objects are notoriously difficult to type correctly. They usually originate from strings (like `.env` files) or untyped JSON files, making them a frequent source of runtime crashes if a key is misspelled or a required value is missing in production.

**Type-Safe Configuration** ensures that your app will not even *start* unless the configuration is fully present, valid, and strictly typed.

---

## 1. The Danger of `process.env`

The most common way developers access environment variables in Node.js or modern frontend frameworks is via `process.env` or `import.meta.env`.

```typescript
// ❌ Dangerous and untyped
const dbUrl = process.env.DATABASE_URL; // Type is string | undefined
const port = parseInt(process.env.PORT); // What if PORT is undefined? parseInt(undefined) -> NaN!

// You end up writing manual runtime checks everywhere
if (!dbUrl) {
  throw new Error("DATABASE_URL is missing");
}
```

This scatters validation logic across your entire codebase and provides zero autocomplete.

---

## 2. Centralized Type-Safe Config with Zod

The best pattern for handling environment variables and configuration is to parse and validate them immediately at application startup using a schema library like [Zod](https://zod.dev/).

If the configuration is invalid, the app throws an error immediately before any other code runs.

### Step 1: Define the Schema

Create a central file (e.g., `config.ts` or `env.ts`).

```typescript
import { z } from 'zod';

// Define the schema for your environment variables
const EnvSchema = z.object({
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  PORT: z.coerce.number().default(3000), // Automatically converts "3000" to 3000
  DATABASE_URL: z.string().url("Must be a valid URL"),
  API_KEY: z.string().min(10, "API Key must be at least 10 characters")
});

// Infer the TypeScript type
export type EnvConfig = z.infer<typeof EnvSchema>;
```

### Step 2: Validate and Export

In the same file, parse `process.env` (or your framework's equivalent) through the schema.

```typescript
// Parse the actual environment variables
const parsed = EnvSchema.safeParse(process.env);

if (!parsed.success) {
  console.error("❌ Invalid environment variables:", parsed.error.flatten().fieldErrors);
  // Crash the app immediately if config is invalid
  process.exit(1); 
}

// Export the strictly typed configuration object
export const config = parsed.data;
```

### Step 3: Use the Config Safely

Now, everywhere else in your app, import `config` instead of using `process.env`.

```typescript
import { config } from './config';

// ✅ Type-safe! TypeScript knows config.PORT is a guaranteed `number`.
console.log(`Starting server on port ${config.PORT}`); 

// ✅ Type-safe! TypeScript knows config.DATABASE_URL is a guaranteed `string`.
db.connect(config.DATABASE_URL);
```

## 3. Type-Safe Configuration with "const assertions"

Sometimes you have static configuration objects in your code (like feature flags, theme colors, or API endpoints).

By default, TypeScript infers object properties as mutable.

```typescript
// ❌ Mutable inference
const theme = {
  primary: "#ff0000",
  secondary: "#00ff00"
};

// Type is { primary: string, secondary: string }
theme.primary = "not-a-color"; // TypeScript allows this!
```

If you want the configuration to be strictly typed to its exact literal values and made immutable, use a **`as const`** assertion.

```typescript
// ✅ as const (Readonly Tuple / Literal Type)
const theme = {
  primary: "#ff0000",
  secondary: "#00ff00"
} as const;

// Type is:
// {
//   readonly primary: "#ff0000";
//   readonly secondary: "#00ff00";
// }

// ❌ ERROR: Cannot assign to 'primary' because it is a read-only property.
theme.primary = "#000000"; 
```

`as const` is incredibly powerful for generating union types from configuration objects using the `typeof` and `keyof` operators.

```typescript
// Extracting types from our const config
type ThemeColors = keyof typeof theme; // "primary" | "secondary"
type HexCodes = typeof theme[ThemeColors]; // "#ff0000" | "#00ff00"
```

## 4. Nuxt/Next.js Runtime Config

If you are using modern meta-frameworks like Nuxt, they often have built-in ways to handle runtime configuration with TypeScript support.

### Nuxt 3 Example

Nuxt allows you to define your config in `nuxt.config.ts`, which automatically generates TypeScript interfaces for you.

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  runtimeConfig: {
    // Private keys are only available on the server
    apiSecret: process.env.API_SECRET,
    
    // Public keys are exposed to the client
    public: {
      apiBase: process.env.NUXT_PUBLIC_API_BASE || '/api'
    }
  }
})
```

When you use `useRuntimeConfig()`, Nuxt provides full autocomplete based on what you defined in your config file.

```vue
<script setup lang="ts">
const config = useRuntimeConfig();

// ✅ Type-safe: Autocompletes `config.public.apiBase`
console.log(config.public.apiBase); 
</script>
```

*(Note: Even with Nuxt's built-in config, it is still highly recommended to use Zod to validate `process.env` before passing it into `nuxt.config.ts` to ensure the values actually exist at startup!)*

---

## Summary

- **Never trust `process.env`:** It is fundamentally untyped (`string | undefined`) and leaves validation up to the consumer.
- **Fail Fast:** Parse and validate your environment variables using **Zod** at the very entry point of your application.
- Use **`as const` assertions** to lock down static configuration objects and infer strict literal types.
- Leverage framework tools (like `useRuntimeConfig` in Nuxt) to safely pass typed configuration from the server to the client.
