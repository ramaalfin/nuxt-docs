# Type-Safe Reducers

Reducers are pure functions that take the current state and an action, and return a new state. They are the foundation of state management libraries like Redux, `useReducer` in React, and complex state management in Vue.

However, writing reducers in JavaScript often involves a `switch` statement over string literals, which can easily lead to typos, unhandled actions, and unsafe payload access. 

By leveraging **Discriminated Unions** and **Exhaustiveness Checking**, we can create 100% type-safe reducers.

---

## 1. The Untyped Reducer Anti-Pattern

Let's look at a traditional, unsafe reducer:

```javascript
// ❌ Untyped Javascript Reducer
function shoppingCartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM':
      return { ...state, items: [...state.items, action.payload] };
    case 'REMOVE_ITEM':
      // Did we pass an ID in the payload? Or the whole item? Who knows!
      return { ...state, items: state.items.filter(i => i.id !== action.payload.id) };
    default:
      return state;
  }
}

// ❌ Nothing stops us from dispatching garbage
dispatch({ type: 'ADD_ITEM', payload: "Just a string?" }); 
dispatch({ type: 'ADD_ITME', paylod: { id: 1 } }); // Typo in type and payload!
```

---

## 2. Defining the Action Union

The secret to a type-safe reducer is defining the `Action` type as a **Discriminated Union**. Every action must have a common property (usually `type`) that acts as the discriminant.

```typescript
// Define the domain models
interface CartItem {
  id: string;
  name: string;
  price: number;
}

interface CartState {
  items: CartItem[];
  total: number;
}

// ✅ Define the Discriminated Union
type CartAction =
  | { type: 'ADD_ITEM'; payload: { item: CartItem } }
  | { type: 'REMOVE_ITEM'; payload: { id: string } }
  | { type: 'CLEAR_CART' }; // Note: No payload here!
```

By explicitly defining `CartAction`, we enforce exactly what shapes of objects can be dispatched to our reducer.

---

## 3. Implementing the Type-Safe Reducer

Now, we use our `CartState` and `CartAction` types in the reducer function.

```typescript
function shoppingCartReducer(state: CartState, action: CartAction): CartState {
  switch (action.type) {
    case 'ADD_ITEM':
      // ✅ TypeScript KNOWS action is strictly the ADD_ITEM object here.
      // It auto-completes action.payload.item
      return { 
        ...state, 
        items: [...state.items, action.payload.item],
        total: state.total + action.payload.item.price
      };

    case 'REMOVE_ITEM':
      // ✅ TypeScript KNOWS action is strictly the REMOVE_ITEM object here.
      // It auto-completes action.payload.id
      const itemToRemove = state.items.find(i => i.id === action.payload.id);
      return {
        ...state,
        items: state.items.filter(i => i.id !== action.payload.id),
        total: state.total - (itemToRemove?.price || 0)
      };

    case 'CLEAR_CART':
      // ❌ ERROR: Property 'payload' does not exist on type '{ type: "CLEAR_CART"; }'
      // console.log(action.payload);
      return { items: [], total: 0 };

    default:
      return state;
  }
}
```

TypeScript's Control Flow Analysis narrows the `action` type in every `case` block based on the `action.type` check!

---

## 4. Exhaustiveness Checking (The `never` Trick)

What happens if another developer comes along and adds an `UPDATE_QUANTITY` action to the `CartAction` union, but forgets to handle it in the `shoppingCartReducer`? 

By default, the reducer will just hit the `default` case and do nothing. This is a silent bug.

We can force a compile-time error by assigning the unhandled action to the `never` type in the default block.

```typescript
// Helper function for exhaustiveness checking
function assertUnreachable(x: never): never {
  throw new Error(`Unhandled action type: ${JSON.stringify(x)}`);
}

function shoppingCartReducer(state: CartState, action: CartAction): CartState {
  switch (action.type) {
    case 'ADD_ITEM':
      // ...
      return state;
    case 'REMOVE_ITEM':
      // ...
      return state;
    case 'CLEAR_CART':
      // ...
      return state;
    // Oops! We forgot 'UPDATE_QUANTITY'
    default:
      // ❌ ERROR: Argument of type '{ type: "UPDATE_QUANTITY"; ... }' 
      // is not assignable to parameter of type 'never'.
      return assertUnreachable(action);
  }
}
```

Because `UPDATE_QUANTITY` falls through to the `default` block, `action` is not `never`. TypeScript correctly flags this as a compilation error, forcing the developer to implement the missing case!

## 5. Type-Safe Dispatch Wrappers (Optional)

If you find writing object literals like `dispatch({ type: 'ADD_ITEM', payload: ... })` tedious, you can use **Action Creators**, which are simple factory functions.

```typescript
// Action Creators provide great autocomplete when dispatching
const actions = {
  addItem: (item: CartItem): CartAction => ({ 
    type: 'ADD_ITEM', 
    payload: { item } 
  }),
  removeItem: (id: string): CartAction => ({ 
    type: 'REMOVE_ITEM', 
    payload: { id } 
  }),
  clearCart: (): CartAction => ({ 
    type: 'CLEAR_CART' 
  })
};

// Usage:
dispatch(actions.addItem({ id: "1", name: "Shirt", price: 20 }));
```

Alternatively, libraries like **Redux Toolkit** (RTK) abstract almost all of this away for you, generating type-safe action creators and reducers automatically without needing `switch` statements or manual unions.

---

## Summary

- Never use `any` or loose string validation for Reducer actions.
- Model your actions as a **Discriminated Union** with a strict `type` property.
- Rely on TypeScript's **Control Flow Analysis** inside `switch` statements to safely access payloads.
- Use the **`never` type trick** in your `default` block to enforce exhaustiveness checking, guaranteeing that all future actions are handled.
