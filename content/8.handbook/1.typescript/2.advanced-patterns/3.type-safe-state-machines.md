# Type-Safe State Machines

In frontend development, we often deal with complex, multi-step processes: fetching data, multi-step wizards, authentication flows, or video player states.

Usually, developers model these states using a collection of boolean flags (`isLoading`, `isError`, `isSuccess`) or a string literal (`status: 'idle' | 'loading' | 'success'`). While string literals are better than booleans, they still don't prevent you from accessing data that shouldn't exist in a certain state.

**Type-Safe State Machines** use TypeScript's discriminated unions to mathematically prove that certain properties only exist when the application is in a specific state.

---

## 1. The "Boolean Explosion" Anti-Pattern

Let's look at a typical data fetching state:

```typescript
interface UserDataState {
  isLoading: boolean;
  isError: boolean;
  isSuccess: boolean;
  error?: Error;
  data?: User;
}
```

This structure is problematic because it allows **impossible states**. What does it mean if `isLoading` is true, but `data` is also present? What if `isError` is true, but there is no `error` object?

You have to manually check these things at runtime, and TypeScript can't help you if you mess up.

---

## 2. Using Discriminated Unions

We can solve this by modeling our state as a **Finite State Machine (FSM)** using a discriminated union. We define exactly what states exist, and exactly what data is associated with each state.

```typescript
type FetchState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: Error };
```

Now, the shape of the object changes depending on the `status` property (the discriminant).

### The Magic of Control Flow Analysis

When we use this type-safe state machine, TypeScript's control flow analysis forces us to check the state before accessing the data.

```typescript
function renderUI(state: FetchState<User>) {
  // ❌ ERROR: Property 'data' does not exist on type 'FetchState<User>'.
  // console.log(state.data); 

  switch (state.status) {
    case 'idle':
      return "Ready to fetch!";
    case 'loading':
      return "Loading spinner...";
    case 'success':
      // ✅ TypeScript KNOWS state is the 'success' variant here.
      // `state.data` is guaranteed to exist and be of type `User`.
      return `Welcome, ${state.data.name}!`;
    case 'error':
      // ✅ TypeScript KNOWS state is the 'error' variant.
      // `state.error` is guaranteed to exist.
      return `Oops: ${state.error.message}`;
  }
}
```

## 3. Modeling Complex Workflows (XState)

For very complex UI components, manual state machines can become tedious. Libraries like [XState](https://stately.ai/docs/xstate) take this concept to the next level.

XState allows you to define not just the states, but also the **transitions** (events) that move you from one state to another.

### A Basic XState Example

```typescript
import { createMachine } from 'xstate';

// 1. Define the Context (the data)
interface TrafficLightContext {
  carsPassed: number;
}

// 2. Define the Events
type TrafficLightEvent = 
  | { type: 'TIMER' }
  | { type: 'EMERGENCY_OVERRIDE' };

// 3. Create the Machine
const trafficLightMachine = createMachine({
  id: 'trafficLight',
  initial: 'green',
  context: { carsPassed: 0 },
  states: {
    green: {
      on: {
        TIMER: 'yellow',
        EMERGENCY_OVERRIDE: 'red'
      }
    },
    yellow: {
      on: {
        TIMER: 'red'
      }
    },
    red: {
      on: {
        TIMER: 'green'
      }
    }
  }
});
```

Modern versions of XState infer types automatically, providing strict autocomplete for state nodes and event types. If you try to send a `TIMER` event to a state that doesn't accept it, or check if the machine is in a state called `'blue'`, TypeScript will throw a compile-time error.

## 4. State Machines in Frameworks (Vue/React)

You can easily wrap a discriminated union in a Vue `ref` or a React `useState` hook.

### Vue 3 Example

```vue
<script setup lang="ts">
import { ref } from 'vue';

type FormState = 
  | { status: 'editing'; formData: { name: string } }
  | { status: 'submitting'; formData: { name: string } }
  | { status: 'success'; receiptId: string }
  | { status: 'error'; errorMessage: string };

// Initialize in the 'editing' state
const state = ref<FormState>({ 
  status: 'editing', 
  formData: { name: '' } 
});

const submit = async () => {
  if (state.value.status !== 'editing') return;

  // Transition to submitting
  state.value = { status: 'submitting', formData: state.value.formData };

  try {
    const receipt = await api.submit(state.value.formData);
    // Transition to success
    state.value = { status: 'success', receiptId: receipt.id };
  } catch (err) {
    // Transition to error
    state.value = { status: 'error', errorMessage: "Failed to submit" };
  }
};
</script>

<template>
  <div v-if="state.status === 'editing' || state.status === 'submitting'">
    <!-- Safe to access formData! -->
    <input v-model="state.formData.name" :disabled="state.status === 'submitting'" />
    <button @click="submit" :disabled="state.status === 'submitting'">Submit</button>
  </div>
  
  <div v-else-if="state.status === 'success'">
    <!-- Safe to access receiptId! -->
    Success! Receipt: {{ state.receiptId }}
  </div>
  
  <div v-else-if="state.status === 'error'">
    <!-- Safe to access errorMessage! -->
    Error: {{ state.errorMessage }}
    <button @click="state = { status: 'editing', formData: { name: '' } }">Retry</button>
  </div>
</template>
```

Vue's template compiler is smart enough to understand discriminated unions inside `v-if` and `v-else-if` blocks, providing full type safety within your HTML template!

---

## Summary

- Avoid boolean explosions (`isLoading`, `isError`, `isSuccess` all living side-by-side). They allow for impossible states.
- Use **Discriminated Unions** to strictly define what data exists at what phase of your component's lifecycle.
- For complex logic with strict transitions and side-effects, consider a dedicated state machine library like **XState**.
- Frameworks like Vue and React natively support discriminated unions for conditional rendering, ensuring template type safety.
