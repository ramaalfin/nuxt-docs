# Type-Safe Form Handlers

Forms are the backbone of most web applications, but they are also a common source of runtime errors. By default, the web platform's APIs (like `FormData` and form submission events) are inherently untyped. They return `string | File | null` or just `any`.

To build robust applications, we need to bridge the gap between untyped HTML forms and our strongly-typed TypeScript domain models. This is where **Type-Safe Form Handlers** come in.

---

## 1. The Problem with Untyped Forms

Consider a standard HTML form submission:

```typescript
function handleSubmit(event: SubmitEvent) {
  event.preventDefault();
  
  const form = event.target as HTMLFormElement;
  const formData = new FormData(form);
  
  // ❌ formData.get() returns `FormDataEntryValue | null` (string | File | null)
  const username = formData.get("username");
  const age = formData.get("age");
  
  // We have no guarantees about the shape or type of our data here!
  api.updateUser({ 
    username: username as string, // Dangerous cast
    age: Number(age)              // Might be NaN
  });
}
```

This approach requires manual casting and validation, leaving a massive surface area for bugs. What if the input name in HTML changes from `username` to `name`? TypeScript won't catch it.

---

## 2. Using Zod for Type-Safe Parsing

The most modern and reliable way to handle form data in TypeScript is to use a schema validation library like [Zod](https://zod.dev/). 

Zod allows you to define a schema that acts as both a **runtime validator** and a **TypeScript type generator**.

### Defining the Schema

```typescript
import { z } from "zod";

const UserFormSchema = z.object({
  username: z.string().min(3, "Username must be at least 3 characters"),
  age: z.coerce.number().min(18, "Must be at least 18"), // Coerce strings to numbers automatically!
  newsletter: z.coerce.boolean().optional() // Checkboxes often send "on" or undefined
});

// Extract the inferred TypeScript type!
// type UserForm = { username: string; age: number; newsletter?: boolean }
export type UserForm = z.infer<typeof UserFormSchema>;
```

### Applying the Schema to FormData

Instead of manually extracting fields, we can convert the `FormData` into an object and pass it to Zod.

```typescript
function handleSubmit(event: SubmitEvent) {
  event.preventDefault();
  
  const form = event.target as HTMLFormElement;
  const formData = new FormData(form);
  
  // Convert FormData to a standard object: { username: "alice", age: "25" }
  const rawData = Object.fromEntries(formData.entries());
  
  // Safely parse the data
  const result = UserFormSchema.safeParse(rawData);
  
  if (!result.success) {
    // ❌ Handle validation errors
    console.error("Validation failed:", result.error.flatten().fieldErrors);
    return;
  }
  
  // ✅ Success! `result.data` is perfectly typed as `UserForm`
  const typedData = result.data;
  
  api.updateUser({
    username: typedData.username, // Auto-completes safely!
    age: typedData.age            // Guaranteed to be a valid number
  });
}
```

## 3. Type-Safe Form State with Generic Hooks (Vue/React)

If you are using a framework like Vue or React, you often want to bind your form inputs to a reactive state object. You can create generic wrappers to ensure the state perfectly matches your Zod schemas.

### Vue Example: A Type-Safe Form Wrapper

```typescript
// useForm.ts
import { ref } from 'vue';
import { z } from 'zod';

export function useForm<T extends z.ZodTypeAny>(schema: T, initialValues: z.infer<T>) {
  // `state` is strictly typed based on the schema!
  const state = ref<z.infer<T>>(initialValues);
  const errors = ref<z.inferFlattenedErrors<T>['fieldErrors']>({});

  const validate = () => {
    const result = schema.safeParse(state.value);
    if (!result.success) {
      errors.value = result.error.flatten().fieldErrors;
      return false;
    }
    
    errors.value = {};
    return true;
  };

  return { state, errors, validate };
}
```

**Usage:**

```vue
<script setup lang="ts">
const { state, errors, validate } = useForm(UserFormSchema, {
  username: "",
  age: 0,
  newsletter: false
});

const submit = () => {
  if (validate()) {
    // state.value is 100% valid and type-safe here
    api.updateUser(state.value);
  }
};
</script>

<template>
  <form @submit.prevent="submit">
    <!-- Vue provides type checking on `state.username` -->
    <input v-model="state.username" />
    <span v-if="errors.username">{{ errors.username[0] }}</span>
    
    <!-- ... -->
  </form>
</template>
```

## 4. End-to-End Type Safety (tRPC / Nuxt Server Routes)

The ultimate goal of type-safe forms is **End-to-End Type Safety**. If you change a database column or an API requirement, your frontend forms should immediately show red squiggly lines if they aren't updated.

With tools like **tRPC** or **Nuxt Server Actions / API Routes** (using `h3` and `zod`), you share the exact same Zod schema between your frontend form validation and your backend API validation.

```typescript
// --- shared/schemas.ts ---
export const UpdateUserSchema = z.object({ /* ... */ });

// --- server/api/user.post.ts (Nuxt) ---
import { readValidatedBody } from 'h3';

export default defineEventHandler(async (event) => {
  // Validates incoming data against the shared schema
  const body = await readValidatedBody(event, UpdateUserSchema.parse);
  
  // Save to DB...
  return { success: true };
});

// --- client/components/Form.vue ---
import { UpdateUserSchema } from '~/shared/schemas';

// The client uses the EXACT SAME schema to validate before sending, 
// guaranteeing the server will accept it.
const { state, validate } = useForm(UpdateUserSchema, { ... });
```

---

## Summary

- **Avoid raw `FormData` parsing:** It lacks types and requires dangerous casting.
- **Use Zod (or similar schema libraries):** Define a single source of truth that acts as both runtime validation and compile-time TypeScript types (`z.infer<typeof Schema>`).
- **Leverage `z.coerce`:** Useful for turning string-based form inputs (like `<input type="number">`) into actual JavaScript numbers/booleans during parsing.
- **Share schemas:** The gold standard is sharing your validation schemas between your frontend forms and your backend APIs for complete end-to-end type safety.
