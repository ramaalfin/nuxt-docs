# Type-Safe Error Handling

Error handling in JavaScript and TypeScript is inherently difficult to type. When you use a `try/catch` block, the `catch` clause receives an `error` object of type `unknown` (or `any` in older TS versions). 

Because JavaScript allows you to `throw` literally anything—a string, a number, null, or a custom class—TypeScript cannot guarantee the shape of the error you catch.

**Type-Safe Error Handling** involves using specialized patterns (like the Result/Either pattern) or careful type narrowing to bring compile-time safety to your failure states.

---

## 1. The Problem with `try/catch`

By default, TypeScript enforces that caught errors are `unknown`.

```typescript
try {
  await fetchUserData();
} catch (error: unknown) {
  // ❌ ERROR: Object is of type 'unknown'.
  console.error(error.message); 
}
```

You are forced to write type guards to safely interact with the error.

```typescript
try {
  await fetchUserData();
} catch (error: unknown) {
  // ✅ Manual Type Guarding
  if (error instanceof Error) {
    console.error(error.message); // Safe!
  } else if (typeof error === "string") {
    console.error(error); // Safe!
  } else {
    console.error("An unknown error occurred");
  }
}
```

While safe, this requires lots of boilerplate scattered throughout your codebase. More importantly, **the function signature `fetchUserData()` does not tell the consumer what specific errors it might throw**.

---

## 2. The Result Pattern (Functional Approach)

Languages like Rust and Go handle errors differently. Instead of throwing exceptions that halt the program, functions return an object representing either a Success or a Failure. This is often called the **Result** or **Either** pattern.

We can recreate this in TypeScript using Discriminated Unions.

### Step 1: Define the Result Type

```typescript
// The Discriminated Union
type Result<T, E = Error> = 
  | { success: true; data: T }
  | { success: false; error: E };

// Helper constructors
export const success = <T>(data: T): Result<T, never> => ({ success: true, data });
export const failure = <E>(error: E): Result<never, E> => ({ success: false, error });
```

### Step 2: Use the Result Type

Now, instead of throwing, our functions return a `Result`. The beautiful part is that **the function signature now strictly documents its errors**.

```typescript
// Specific error types
class NetworkError extends Error {}
class ValidationError extends Error {}

// Look at this incredible function signature!
async function getUser(id: string): Promise<Result<User, NetworkError | ValidationError>> {
  if (!id) return failure(new ValidationError("ID is required"));
  
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) return failure(new NetworkError("Failed to fetch"));
    
    const data = await response.json();
    return success(data);
  } catch (e) {
    return failure(new NetworkError("Network crash"));
  }
}
```

### Step 3: Consume the Result

When the consumer calls `getUser`, TypeScript forces them to check `.success` before accessing `.data` or `.error`.

```typescript
const result = await getUser("123");

// ❌ ERROR: Property 'data' does not exist on type 'Result<User, ...>'
// console.log(result.data); 

if (result.success) {
  // ✅ TypeScript knows this is a User!
  console.log(result.data.name);
} else {
  // ✅ TypeScript knows this is NetworkError | ValidationError!
  if (result.error instanceof ValidationError) {
    console.log("Bad input:", result.error.message);
  } else {
    console.log("Retry connection:", result.error.message);
  }
}
```

This pattern is widely considered the safest way to handle expected domain errors in TypeScript.

## 3. The Tuple Pattern (The Go Approach)

If creating an entire `Result` type feels too heavy, you can use a lighter approach inspired by the Go programming language: returning a tuple of `[Error, null]` or `[null, Data]`.

```typescript
// The Tuple Type
type AsyncTuple<T, E = Error> = Promise<[E, null] | [null, T]>;

async function readFileSafe(path: string): AsyncTuple<string, Error> {
  try {
    const data = await fs.readFile(path, 'utf8');
    return [null, data]; // Success
  } catch (error) {
    return [error instanceof Error ? error : new Error(String(error)), null]; // Failure
  }
}

// Usage
async function run() {
  const [err, data] = await readFileSafe('./config.json');

  if (err) {
    console.error("Failed to read:", err.message);
    return;
  }

  // ✅ Safe to use `data` here because `err` was null!
  console.log(data.toUpperCase()); 
}
```

*(Note: While popular, this tuple pattern doesn't type-narrow as perfectly as the Discriminated Union `Result` object in older versions of TypeScript, but works great in TS 4.6+).*

---

## 4. `never` as a Return Type

When dealing with functions that *always* throw errors (like an assert or panic function), you should type their return value as `never`.

```typescript
function throwFatal(message: string): never {
  throw new Error(`FATAL: ${message}`);
}

function process(data: string | null) {
  // If data is null, the function throws and execution stops
  const validData = data ?? throwFatal("Data cannot be null");
  
  // ✅ TypeScript knows validData is strictly `string` here
  console.log(validData.toLowerCase());
}
```

---

## Summary

- Default `try/catch` blocks erase type safety, typing the caught error as `unknown`.
- To make errors visible in function signatures, use the **Result Pattern** (returning discriminated unions indicating success/failure).
- Alternatively, use the **Tuple Pattern** (`[err, data]`) for a lightweight, Go-style approach.
- Use `never` for helper functions that unconditionally throw exceptions, allowing TypeScript to accurately map control flow.
