# Generics with Constraints & Bounds

Generics are TypeScript's most powerful mechanism for creating **reusable, type-safe abstractions**. While basic generics let you parameterize types, **constraints and bounds** let you control _what kinds of types_ are acceptable — turning your generics from wide-open wildcards into precisely scoped contracts.

This guide covers everything from foundational concepts to advanced real-world patterns that senior engineers encounter daily.

---

## 1. Foundation: What Are Generics?

At their core, generics allow you to write functions, classes, and types that work with **any type** while preserving type information throughout the operation.

### The Problem Without Generics

```typescript
// Without generics — you lose type information
function getFirst(arr: any[]): any {
  return arr[0];
}

const val = getFirst([1, 2, 3]); // val is `any` — useless!
val.toFixed(); // No error at compile time, but could crash at runtime
```

### The Solution With Generics

```typescript
// With generics — full type safety preserved
function getFirst<T>(arr: T[]): T {
  return arr[0];
}

const val = getFirst([1, 2, 3]); // val is `number` ✅
val.toFixed(); // Fully type-safe ✅

const str = getFirst(['a', 'b', 'c']); // str is `string` ✅
str.toUpperCase(); // Fully type-safe ✅
```

The `T` is a **type parameter** — a placeholder that gets replaced with a concrete type when the function is called. TypeScript infers `T` from the argument automatically.

### Multiple Type Parameters

```typescript
function makePair<A, B>(first: A, second: B): [A, B] {
  return [first, second];
}

const pair = makePair('hello', 42); 
// Inferred as [string, number] ✅

// You can also explicitly specify type arguments
const explicit = makePair<string, boolean>('yes', true);
```

## 2. Generic Constraints with `extends`

Unconstrained generics accept **anything** — but often your function needs to assume something about the type. The `extends` keyword introduces a **constraint** (also called an **upper bound**) on what types `T` can be.

### Basic Constraint Syntax

```typescript
// T must be an object that has at least a `length` property of type number
function logLength<T extends { length: number }>(item: T): T {
  console.log(`Length: ${item.length}`);
  return item;
}

logLength('hello');       // ✅ string has `.length`
logLength([1, 2, 3]);     // ✅ array has `.length`
logLength({ length: 10 }); // ✅ object literal with `.length`

logLength(42);            // ❌ Error: number doesn't have `.length`
logLength(true);          // ❌ Error: boolean doesn't have `.length`
```

### How Constraints Work Internally

When you write `T extends Constraint`:
1. **At call site**: TypeScript checks that the actual type satisfies `Constraint`
2. **Inside the function**: You can safely access any property defined in `Constraint`
3. **Return type**: The full original type `T` is preserved (not narrowed to `Constraint`)

```typescript
function cloneWithId<T extends { name: string }>(
  obj: T
): T & { id: string } {
  return {
    ...obj,          // All original properties preserved
    id: crypto.randomUUID(),
  };
}

const user = cloneWithId({ name: 'Alice', age: 30, role: 'admin' });
// Type: { name: string; age: number; role: string; } & { id: string }
// All properties are preserved! ✅
user.age;  // ✅ number
user.role; // ✅ string
user.id;   // ✅ string
```

This is a crucial distinction: the **constraint** is not the type — it's the **minimum requirement**. `T` retains its full shape.

## 3. Constraining with Interfaces and Types

You can constrain generics against any TypeScript type — interfaces, type aliases, unions, or inline shapes.

### Interface Constraints

```typescript
interface Identifiable {
  id: string | number;
}

interface Timestamped {
  createdAt: Date;
  updatedAt: Date;
}

// T must implement both Identifiable AND Timestamped
function updateRecord<T extends Identifiable & Timestamped>(
  record: T,
  updates: Partial<Omit<T, 'id' | 'createdAt'>>
): T {
  return {
    ...record,
    ...updates,
    updatedAt: new Date(),
  };
}

interface User extends Identifiable, Timestamped {
  name: string;
  email: string;
}

const user: User = {
  id: '1',
  name: 'Alice',
  email: 'alice@example.com',
  createdAt: new Date(),
  updatedAt: new Date(),
};

const updated = updateRecord(user, { name: 'Bob' });
// updated is fully typed as User ✅
```

### Union Type Constraints

```typescript
// T is constrained to a specific set of allowed types
function serialize<T extends string | number | boolean>(value: T): string {
  return JSON.stringify(value);
}

serialize('hello'); // ✅
serialize(42);      // ✅
serialize(true);    // ✅
serialize({});      // ❌ Error: {} is not assignable to string | number | boolean
```

## 4. The `keyof` Constraint Pattern

One of the most common constraint patterns is using `keyof` to restrict a generic to the keys of another type. This enables **type-safe property access**.

### Basic `keyof` Constraint

```typescript
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const person = { name: 'Alice', age: 30, email: 'alice@example.com' };

const name = getProperty(person, 'name');  // string ✅
const age = getProperty(person, 'age');    // number ✅
getProperty(person, 'phone');              // ❌ Error: 'phone' is not in keyof person
```

### Multi-Key Patterns with `keyof`

```typescript
// Pick multiple keys type-safely
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  const result = {} as Pick<T, K>;
  keys.forEach(key => {
    result[key] = obj[key];
  });
  return result;
}

const user = { id: 1, name: 'Alice', email: 'a@b.com', role: 'admin' };
const subset = pick(user, ['name', 'email']);
// Type: Pick<typeof user, 'name' | 'email'>
// Value: { name: 'Alice', email: 'a@b.com' }

pick(user, ['phone']); // ❌ Error: 'phone' is not assignable to keyof typeof user
```

### Nested `keyof` — Deep Property Access

```typescript
function getDeep<
  T,
  K1 extends keyof T,
  K2 extends keyof T[K1]
>(obj: T, key1: K1, key2: K2): T[K1][K2] {
  return obj[key1][key2];
}

const config = {
  database: { host: 'localhost', port: 5432 },
  cache: { ttl: 300, maxSize: 1000 },
};

const host = getDeep(config, 'database', 'host');   // string ✅
const ttl = getDeep(config, 'cache', 'ttl');         // number ✅
getDeep(config, 'database', 'ttl');                  // ❌ Error: 'ttl' not in keyof database
```

## 5. Multiple Constraints (Intersection Constraints)

TypeScript allows multiple constraints through **intersection types** (`&`).

### Combining Multiple Requirements

```typescript
interface Serializable {
  serialize(): string;
}

interface Validatable {
  validate(): boolean;
}

interface Loggable {
  toLog(): string;
}

// T must implement ALL three interfaces
function processEntity<T extends Serializable & Validatable & Loggable>(
  entity: T
): string {
  if (!entity.validate()) {
    throw new Error(`Validation failed: ${entity.toLog()}`);
  }
  return entity.serialize();
}

// A class that satisfies all three constraints
class UserDTO implements Serializable, Validatable, Loggable {
  constructor(
    public name: string,
    public email: string
  ) {}
  
  serialize(): string {
    return JSON.stringify({ name: this.name, email: this.email });
  }
  
  validate(): boolean {
    return this.name.length > 0 && this.email.includes('@');
  }
  
  toLog(): string {
    return `UserDTO(${this.name}, ${this.email})`;
  }
}

processEntity(new UserDTO('Alice', 'alice@example.com')); // ✅
processEntity({ serialize: () => '{}' }); // ❌ Missing validate and toLog
```

## 6. Conditional Type Constraints

Conditional types (`T extends U ? X : Y`) act as **type-level if/else** — they compute different result types based on constraints.

### Basic Conditional Types

```typescript
// Determine the container type based on input type
type Wrapper<T> = T extends string
  ? StringWrapper
  : T extends number
    ? NumberWrapper
    : GenericWrapper<T>;

interface StringWrapper {
  kind: 'string';
  value: string;
  uppercase(): string;
}

interface NumberWrapper {
  kind: 'number';
  value: number;
  toFixed(digits: number): string;
}

interface GenericWrapper<T> {
  kind: 'generic';
  value: T;
}
```

### Distributive Conditional Types

When a conditional type acts on a **union**, it **distributes** — applying the condition to each member individually:

```typescript
type ToArray<T> = T extends unknown ? T[] : never;

// Distribution in action:
type Result = ToArray<string | number>;
// Evaluates as:
// = (string extends unknown ? string[] : never) | (number extends unknown ? number[] : never)
// = string[] | number[]  ✅

// To PREVENT distribution, wrap both sides in tuples:
type ToArrayNonDist<T> = [T] extends [unknown] ? T[] : never;
type Result2 = ToArrayNonDist<string | number>;
// = (string | number)[]  — single array type, not distributed
```

### The `infer` Keyword — Extracting Types from Constraints

`infer` lets you **capture** (extract) type information from within a conditional type:

```typescript
// Extract the return type of a function
type ReturnOf<T> = T extends (...args: any[]) => infer R ? R : never;

type A = ReturnOf<() => string>;                     // string
type B = ReturnOf<(x: number) => boolean>;           // boolean
type C = ReturnOf<typeof JSON.parse>;                // any

// Extract the element type of an array
type ElementOf<T> = T extends (infer E)[] ? E : T;

type D = ElementOf<string[]>;    // string
type E = ElementOf<number[]>;    // number
type F = ElementOf<boolean>;     // boolean (not an array, returns T)

// Extract the resolved type of a Promise
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;

type G = Awaited<Promise<Promise<string>>>;  // string (recursively unwraps!)
```

### Constraint Narrowing with `infer`

```typescript
// Extract different parts of a function signature
type FunctionParts<T> = T extends (...args: infer A) => infer R
  ? { args: A; returnType: R }
  : never;

type Parts = FunctionParts<(name: string, age: number) => boolean>;
// { args: [name: string, age: number]; returnType: boolean }

// Extract constructor parameter types
type ConstructorParams<T> = T extends new (...args: infer P) => any ? P : never;

class Service {
  constructor(public name: string, public port: number) {}
}

type ServiceParams = ConstructorParams<typeof Service>; // [string, number]
```

## 7. Generic Default Types

Default types provide **fallback values** for generic parameters when the caller doesn't specify them.

### Basic Defaults

```typescript
// TError defaults to Error if not provided
type Result<TData, TError = Error> =
  | { success: true; data: TData }
  | { success: false; error: TError };

// Using with defaults — TError is Error
const result1: Result<string> = { success: true, data: 'hello' };
const result2: Result<string> = { success: false, error: new Error('fail') };

// Overriding the default
interface ApiError {
  code: number;
  message: string;
  details?: unknown;
}
const result3: Result<string, ApiError> = {
  success: false,
  error: { code: 404, message: 'Not found' },
};
```

### Defaults with Constraints

You can combine defaults with constraints. The default must satisfy the constraint:

```typescript
interface BaseConfig {
  debug: boolean;
}

// TConfig must extend BaseConfig, defaults to BaseConfig itself
function createApp<TConfig extends BaseConfig = BaseConfig>(
  config: TConfig
) {
  return {
    config,
    isDebug: config.debug,
  };
}

// Using default — just pass a BaseConfig
createApp({ debug: true });

// Using a more specific config
createApp({ debug: false, port: 3000, host: 'localhost' });
```

### Ordering Rules

Default type parameters must come **after** non-default type parameters (just like default function arguments):

```typescript
// ✅ Correct — defaults come last
type Query<TTable, TSelect = '*', TLimit extends number = 100> = {
  table: TTable;
  select: TSelect;
  limit: TLimit;
};

// ❌ Error — default before required
// type BadQuery<TSelect = '*', TTable> = { ... };
```

## 8. Bounded Polymorphism & Upper/Lower Bounds

### Upper Bounds (Most Common)

An **upper bound** constrains `T` to be a **subtype of** the bound. In TypeScript, `extends` always creates an upper bound:

```typescript
// Upper bound: T must be a subtype of Animal
interface Animal {
  name: string;
  sound(): string;
}

interface Dog extends Animal {
  breed: string;
  fetch(): void;
}

interface Cat extends Animal {
  indoor: boolean;
  purr(): void;
}

function makeNoise<T extends Animal>(animal: T): string {
  return `${animal.name} says ${animal.sound()}`;
}

const dog: Dog = { name: 'Rex', breed: 'Lab', sound: () => 'Woof', fetch: () => {} };
const cat: Cat = { name: 'Whiskers', indoor: true, sound: () => 'Meow', purr: () => {} };

makeNoise(dog); // ✅ Dog extends Animal
makeNoise(cat); // ✅ Cat extends Animal
makeNoise({ name: 'Rock' }); // ❌ Missing sound() method
```

### Simulating Lower Bounds

TypeScript doesn't natively support lower bounds (`T super U` as in Java/Scala), but you can simulate them using conditional types:

```typescript
// Simulate lower bound: T must be a SUPERTYPE of the Bound
type LowerBound<T, Bound> = Bound extends T ? T : never;

// Usage: ensure the container type is at least as wide as the element type
function addToCollection<T>(
  collection: T[],
  item: LowerBound<T, string>
): void {
  collection.push(item);
}

const strings: string[] = [];
addToCollection(strings, 'hello');        // ✅ string extends string

const mixed: (string | number)[] = [];
addToCollection(mixed, 'hello');          // ✅ string extends string | number
```

### Bounded Return Types — Preserving Specificity

A key benefit of bounded generics is that the **return type preserves the specific subtype**, not just the bound:

```typescript
interface Shape {
  area(): number;
}

interface Circle extends Shape {
  radius: number;
}

interface Rectangle extends Shape {
  width: number;
  height: number;
}

// Without generics — returns Shape, losing specificity
function scaleBasic(shape: Shape, factor: number): Shape {
  return { area: () => shape.area() * factor };
}

// With bounded generics — preserves the exact subtype
function scale<T extends Shape>(shape: T, factor: number): T {
  return {
    ...shape,
    area: () => shape.area() * factor,
  };
}

const circle: Circle = { radius: 5, area: () => Math.PI * 25 };
const scaled = scale(circle, 2);
// scaled is Circle ✅ — not just Shape!
scaled.radius; // ✅ still accessible
```

## 9. Recursive Generic Constraints

Generics can reference themselves to model recursive or self-referential data structures.

### Recursive Data Structures

```typescript
// A tree node where each node contains children of the same type
interface TreeNode<T> {
  value: T;
  children: TreeNode<T>[];
}

const orgChart: TreeNode<string> = {
  value: 'CEO',
  children: [
    {
      value: 'CTO',
      children: [
        { value: 'Lead Engineer', children: [] },
        { value: 'Senior Engineer', children: [] },
      ],
    },
    {
      value: 'CFO',
      children: [
        { value: 'Accountant', children: [] },
      ],
    },
  ],
};

// Generic tree traversal
function traverse<T>(node: TreeNode<T>, visit: (value: T) => void): void {
  visit(node.value);
  node.children.forEach(child => traverse(child, visit));
}

traverse(orgChart, name => console.log(name));
```

### Self-Referencing Constraints (F-Bounded Polymorphism)

A powerful pattern where `T` constrains itself — ensuring that subtypes can reference their own type:

```typescript
// T must extend Comparable<T> — it must be comparable to its OWN type
interface Comparable<T extends Comparable<T>> {
  compareTo(other: T): number;
}

class Money implements Comparable<Money> {
  constructor(
    public amount: number,
    public currency: string
  ) {}

  compareTo(other: Money): number {
    if (this.currency !== other.currency) {
      throw new Error('Cannot compare different currencies');
    }
    return this.amount - other.amount;
  }
}

class Temperature implements Comparable<Temperature> {
  constructor(public celsius: number) {}
  
  compareTo(other: Temperature): number {
    return this.celsius - other.celsius;
  }
}

// Generic sort that works with any Comparable
function sortItems<T extends Comparable<T>>(items: T[]): T[] {
  return [...items].sort((a, b) => a.compareTo(b));
}

sortItems([new Money(100, 'USD'), new Money(50, 'USD')]); // ✅
sortItems([new Temperature(30), new Temperature(20)]);    // ✅
// sortItems([new Money(100, 'USD'), new Temperature(30)]); // ❌ Type error!
```

### Recursive Type Utilities

```typescript
// Deep Readonly — recursively makes every property readonly
type DeepReadonly<T> = T extends (infer E)[]
  ? ReadonlyArray<DeepReadonly<E>>
  : T extends object
    ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
    : T;

interface Config {
  server: {
    host: string;
    port: number;
    cors: {
      origins: string[];
      methods: string[];
    };
  };
  features: boolean[];
}

type FrozenConfig = DeepReadonly<Config>;
// Every level is now readonly:
// {
//   readonly server: {
//     readonly host: string;
//     readonly port: number;
//     readonly cors: {
//       readonly origins: readonly string[];
//       readonly methods: readonly string[];
//     };
//   };
//   readonly features: readonly boolean[];
// }

// Deep Partial — recursively makes every property optional
type DeepPartial<T> = T extends object
  ? { [K in keyof T]?: DeepPartial<T[K]> }
  : T;

type PartialConfig = DeepPartial<Config>;
// Every property at every level is now optional
```

## 10. Generic Constraints in Classes

Classes use generic constraints extensively for type-safe data structures and design patterns.

### Generic Repository Pattern

```typescript
interface Entity {
  id: string;
  createdAt: Date;
  updatedAt: Date;
}

class Repository<T extends Entity> {
  private items: Map<string, T> = new Map();

  save(item: T): T {
    const now = new Date();
    const saved = {
      ...item,
      updatedAt: now,
      createdAt: item.createdAt || now,
    };
    this.items.set(item.id, saved);
    return saved;
  }

  findById(id: string): T | undefined {
    return this.items.get(id);
  }

  findAll(): T[] {
    return Array.from(this.items.values());
  }

  findWhere<K extends keyof T>(key: K, value: T[K]): T[] {
    return this.findAll().filter(item => item[key] === value);
  }

  delete(id: string): boolean {
    return this.items.delete(id);
  }
}

// Concrete entity
interface User extends Entity {
  name: string;
  email: string;
  role: 'admin' | 'user';
}

const userRepo = new Repository<User>();

userRepo.save({
  id: '1',
  name: 'Alice',
  email: 'alice@example.com',
  role: 'admin',
  createdAt: new Date(),
  updatedAt: new Date(),
});

// Type-safe queries!
const admins = userRepo.findWhere('role', 'admin');   // ✅ User[]
userRepo.findWhere('role', 'superadmin');              // ❌ Error: not assignable to 'admin' | 'user'
userRepo.findWhere('phone', '123');                    // ❌ Error: 'phone' not in keyof User
```

### Generic Builder Pattern

```typescript
class FormBuilder<TFields extends Record<string, unknown> = {}> {
  private fields: Record<string, FieldConfig> = {};

  addField<
    TName extends string,
    TType
  >(
    name: TName,
    config: FieldConfig<TType>
  ): FormBuilder<TFields & Record<TName, TType>> {
    this.fields[name] = config;
    return this as any;
  }

  build(): Form<TFields> {
    return new Form<TFields>(this.fields);
  }
}

interface FieldConfig<T = unknown> {
  type: 'text' | 'number' | 'email' | 'select';
  required?: boolean;
  defaultValue?: T;
  validate?: (value: T) => boolean;
}

class Form<T> {
  constructor(private fields: Record<string, FieldConfig>) {}
  
  getValues(): T {
    // Implementation that collects values
    return {} as T;
  }
}

// Usage — the type builds up incrementally with each addField call
const loginForm = new FormBuilder()
  .addField('email', {
    type: 'email',
    required: true,
    defaultValue: '',
    validate: (v: string) => v.includes('@'),
  })
  .addField('password', {
    type: 'text',
    required: true,
    defaultValue: '',
  })
  .addField('rememberMe', {
    type: 'select',
    defaultValue: false,
  })
  .build();

// loginForm is Form<{ email: string; password: string; rememberMe: boolean }>
const values = loginForm.getValues();
// values is { email: string; password: string; rememberMe: boolean } ✅
```

## 11. Variance and Generic Constraints

Understanding **variance** is critical for knowing when one generic type can substitute for another.

### Covariance (Output Position)

A type is **covariant** in `T` when `Dog extends Animal` implies `Container<Dog> extends Container<Animal>`. This applies when `T` appears in **output** (return) positions:

```typescript
interface Producer<out T> {
  produce(): T;
}

// Dog extends Animal, so Producer<Dog> can be assigned to Producer<Animal>
declare const dogProducer: Producer<Dog>;
const animalProducer: Producer<Animal> = dogProducer; // ✅ Covariant
```

### Contravariance (Input Position)

A type is **contravariant** in `T` when `Dog extends Animal` implies `Handler<Animal> extends Handler<Dog>`. This applies when `T` appears in **input** (parameter) positions:

```typescript
interface Consumer<in T> {
  consume(item: T): void;
}

// Animal is wider than Dog, so Consumer<Animal> can handle Dogs
declare const animalConsumer: Consumer<Animal>;
const dogConsumer: Consumer<Dog> = animalConsumer; // ✅ Contravariant
```

### Invariance (Both Positions)

When `T` appears in **both** input and output positions, the type is **invariant** — only exact type matches are allowed:

```typescript
interface Box<in out T> {
  get(): T;       // output position — covariant
  set(val: T): void; // input position — contravariant
  // Combined → invariant: only Box<Dog> is assignable to Box<Dog>
}

declare const dogBox: Box<Dog>;
// const animalBox: Box<Animal> = dogBox; // ❌ Error — invariant
```

### Variance Annotations (TypeScript 4.7+)

TypeScript 4.7 introduced explicit variance annotations to improve performance and clarity:

```typescript
// `out` = covariant (T only appears in output positions)
interface ReadonlyStore<out T> {
  get(id: string): T;
  getAll(): T[];
}

// `in` = contravariant (T only appears in input positions)
interface WriteOnlyStore<in T> {
  save(item: T): void;
  saveAll(items: T[]): void;
}

// `in out` = invariant (T appears in both positions)
interface MutableStore<in out T> {
  get(id: string): T;
  save(item: T): void;
}
```

## 12. Advanced Constraint Patterns

### Constrained Mapped Types

```typescript
// Make only specific keys optional
type PartialBy<T, K extends keyof T> = Omit<T, K> & Partial<Pick<T, K>>;

interface User {
  id: string;
  name: string;
  email: string;
  bio: string;
}

// Only bio is optional, everything else required
type UserInput = PartialBy<User, 'bio'>;
// { id: string; name: string; email: string; bio?: string }

// Make only specific keys required
type RequiredBy<T, K extends keyof T> = Omit<T, K> & Required<Pick<T, K>>;
```

### Type-Safe Event System with Constrained Generics

```typescript
type EventMap = {
  'user:login':    { userId: string; timestamp: number };
  'user:logout':   { userId: string };
  'item:created':  { itemId: string; name: string };
  'item:deleted':  { itemId: string };
  'error':         { code: number; message: string };
};

class TypedEventBus<TEvents extends Record<string, any>> {
  private handlers: Map<string, Set<Function>> = new Map();

  on<K extends keyof TEvents & string>(
    event: K,
    handler: (payload: TEvents[K]) => void
  ): () => void {
    if (!this.handlers.has(event)) {
      this.handlers.set(event, new Set());
    }
    this.handlers.get(event)!.add(handler);
    
    // Return unsubscribe function
    return () => this.handlers.get(event)?.delete(handler);
  }

  emit<K extends keyof TEvents & string>(
    event: K,
    payload: TEvents[K]
  ): void {
    this.handlers.get(event)?.forEach(fn => fn(payload));
  }

  // Only allow listening to events that match a pattern
  onPattern<P extends string>(
    pattern: P,
    handler: (
      event: Extract<keyof TEvents & string, `${P}${string}`>,
      payload: TEvents[Extract<keyof TEvents & string, `${P}${string}`>]
    ) => void
  ): void {
    // Implementation would use pattern matching
  }
}

const bus = new TypedEventBus<EventMap>();

// Fully type-safe!
bus.on('user:login', (payload) => {
  console.log(payload.userId);     // ✅ string
  console.log(payload.timestamp);  // ✅ number
});

bus.emit('user:login', { userId: '123', timestamp: Date.now() }); // ✅
bus.emit('user:login', { userId: '123' }); // ❌ Missing timestamp
bus.emit('unknown', {});                   // ❌ 'unknown' not in EventMap
```

### Constrained Higher-Order Functions

```typescript
// A function that creates a memoized version of any function
function memoize<
  TArgs extends unknown[],
  TReturn
>(
  fn: (...args: TArgs) => TReturn,
  keyGenerator?: (...args: TArgs) => string
): (...args: TArgs) => TReturn {
  const cache = new Map<string, TReturn>();
  
  return (...args: TArgs): TReturn => {
    const key = keyGenerator
      ? keyGenerator(...args)
      : JSON.stringify(args);
      
    if (cache.has(key)) {
      return cache.get(key)!;
    }
    
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

// Usage
const expensiveCalc = (x: number, y: number): number => {
  console.log('Computing...');
  return x * y + Math.sqrt(x);
};

const memoized = memoize(expensiveCalc);
memoized(4, 5); // "Computing..." → 22
memoized(4, 5); // Returns cached 22, no log
```

### Type Guards with Generic Constraints

```typescript
// A generic type guard factory
function isType<T>(
  check: (value: unknown) => boolean
): (value: unknown) => value is T {
  return (value): value is T => check(value);
}

const isString = isType<string>(v => typeof v === 'string');
const isNumber = isType<number>(v => typeof v === 'number');

// More sophisticated: structural type guard
function hasProperty<K extends string>(
  obj: unknown,
  key: K
): obj is Record<K, unknown> {
  return typeof obj === 'object' && obj !== null && key in obj;
}

function processInput(input: unknown) {
  if (hasProperty(input, 'name') && hasProperty(input, 'age')) {
    // TypeScript knows: input is Record<'name', unknown> & Record<'age', unknown>
    console.log(input.name, input.age); // ✅
  }
}
```

### Type-Safe Route Params

```typescript
// Extract route parameter names from a path pattern
type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<Rest>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

type Params = ExtractParams<'/users/:userId/posts/:postId'>;
// 'userId' | 'postId'

// Build a type-safe router
type RouteParams<T extends string> = {
  [K in ExtractParams<T>]: string;
};

function navigate<T extends string>(
  path: T,
  params: RouteParams<T>
): string {
  let result: string = path;
  for (const [key, value] of Object.entries(params)) {
    result = result.replace(`:${key}`, value as string);
  }
  return result;
}

navigate('/users/:userId/posts/:postId', {
  userId: '123',
  postId: '456',
}); // ✅ Both params required

navigate('/users/:userId/posts/:postId', {
  userId: '123',
}); // ❌ Error: missing postId
```

## 14. Practical Design Patterns

### Pattern 1: Type-Safe API Response Handler

```typescript
interface ApiEndpoints {
  '/users': { list: User[]; get: User; create: User };
  '/posts': { list: Post[]; get: Post; create: Post };
  '/comments': { list: Comment[]; get: Comment; create: Comment };
}

interface User { id: string; name: string }
interface Post { id: string; title: string; body: string }
interface Comment { id: string; text: string }

type HttpMethod = 'list' | 'get' | 'create';

async function apiCall<
  TPath extends keyof ApiEndpoints,
  TMethod extends keyof ApiEndpoints[TPath]
>(
  path: TPath,
  method: TMethod
): Promise<ApiEndpoints[TPath][TMethod]> {
  const response = await fetch(path as string);
  return response.json();
}

// Fully typed return values!
const users = await apiCall('/users', 'list');     // User[]
const post = await apiCall('/posts', 'get');        // Post
const comment = await apiCall('/comments', 'create'); // Comment
apiCall('/unknown', 'list');                        // ❌ Error
```

### Pattern 2: Discriminated Union Factory

```typescript
type ActionDef = Record<string, Record<string, unknown> | void>;

type Actions<T extends ActionDef> = {
  [K in keyof T]: T[K] extends void
    ? { type: K }
    : { type: K; payload: T[K] }
}[keyof T];

// Define your action shapes
interface TodoActions extends ActionDef {
  ADD_TODO: { text: string; priority: 'low' | 'medium' | 'high' };
  REMOVE_TODO: { id: string };
  TOGGLE_TODO: { id: string };
  CLEAR_COMPLETED: void;
}

type TodoAction = Actions<TodoActions>;
// Automatically becomes:
// | { type: 'ADD_TODO'; payload: { text: string; priority: 'low' | 'medium' | 'high' } }
// | { type: 'REMOVE_TODO'; payload: { id: string } }
// | { type: 'TOGGLE_TODO'; payload: { id: string } }
// | { type: 'CLEAR_COMPLETED' }

function todoReducer(state: any, action: TodoAction) {
  switch (action.type) {
    case 'ADD_TODO':
      console.log(action.payload.text); // ✅ Narrowed perfectly
      break;
    case 'CLEAR_COMPLETED':
      // action has no payload here ✅
      break;
  }
}
```

### Pattern 3: Fluent Pipe with Constrained Types

```typescript
class Pipe<TInput, TOutput> {
  constructor(private fn: (input: TInput) => TOutput) {}

  then<TNext>(nextFn: (input: TOutput) => TNext): Pipe<TInput, TNext> {
    return new Pipe((input: TInput) => nextFn(this.fn(input)));
  }

  execute(input: TInput): TOutput {
    return this.fn(input);
  }
}

function pipe<TInput, TOutput>(
  fn: (input: TInput) => TOutput
): Pipe<TInput, TOutput> {
  return new Pipe(fn);
}

// Each step's output must match the next step's input
const transform = pipe((s: string) => s.split(','))       // string → string[]
  .then(arr => arr.map(Number))                            // string[] → number[]
  .then(nums => nums.filter(n => n > 0))                   // number[] → number[]
  .then(nums => nums.reduce((sum, n) => sum + n, 0))       // number[] → number
  .then(total => `Total: ${total}`);                       // number → string

const result = transform.execute('1,2,-3,4,5');
// result is string ✅ → "Total: 12"
```

## 15. Common Mistakes & How to Fix Them

### Mistake 1: Over-Constraining

```typescript
// ❌ Too restrictive — only works with arrays of objects with exactly an `id` field
function findById<T extends { id: string }>(items: T[], id: string): T | undefined {
  return items.find(item => item.id === id);
}

// ✅ Better — allow any key type for id
function findByKey<
  T,
  K extends keyof T
>(items: T[], key: K, value: T[K]): T | undefined {
  return items.find(item => item[key] === value);
}
```

### Mistake 2: Forgetting That `T` Is Preserved

```typescript
// ❌ Unnecessarily losing type info by returning the constraint type
function clone<T extends object>(obj: T): object {
  return { ...obj };
}
const user = clone({ name: 'Alice', age: 30 });
user.name; // ❌ Error: property 'name' doesn't exist on type 'object'

// ✅ Return T to preserve the full type
function cloneCorrect<T extends object>(obj: T): T {
  return { ...obj };
}
const user2 = cloneCorrect({ name: 'Alice', age: 30 });
user2.name; // ✅ string
```

### Mistake 3: Using `any` Instead of `unknown` in Generics

```typescript
// ❌ `any` in the constraint defeats the purpose
function process<T extends any>(value: T): T {
  return value; // T is effectively `any`
}

// ✅ Use `unknown` if you truly accept anything
function processCorrect<T>(value: T): T {
  return value; // T preserves the actual type
}
```

### Mistake 4: Not Using `const` Type Parameters (TS 5.0+)

```typescript
// Without `const` — literal types are widened
function createRoute<T extends string>(path: T) {
  return { path };
}
const route1 = createRoute('/users'); // { path: string } — widened! 😞

// ✅ With `const` — literal types are preserved
function createRouteConst<const T extends string>(path: T) {
  return { path };
}
const route2 = createRouteConst('/users'); // { path: '/users' } — preserved! 🎉

// Works with objects and arrays too
function defineConfig<const T extends Record<string, unknown>>(config: T): T {
  return config;
}
const config = defineConfig({
  mode: 'production',
  port: 3000,
});
// { mode: 'production'; port: 3000 } — exact literal types preserved!
```

## 16. Quick Reference Cheat Sheet

| Pattern | Syntax | Purpose |
|---|---|---|
| Basic constraint | `<T extends Type>` | T must be a subtype of Type |
| Multiple constraints | `<T extends A & B>` | T must satisfy both A and B |
| `keyof` constraint | `<K extends keyof T>` | K must be a key of T |
| Default type | `<T = DefaultType>` | Fallback when T is not specified |
| Conditional type | `T extends U ? X : Y` | Type-level branching |
| `infer` keyword | `T extends F<infer U> ? U : never` | Extract types from structures |
| F-bounded | `<T extends Comparable<T>>` | Self-referencing constraint |
| `const` modifier | `<const T extends Type>` | Preserve literal types |
| Covariance | `<out T>` | T in output positions only |
| Contravariance | `<in T>` | T in input positions only |
| Invariance | `<in out T>` | T in both positions |
| Distributive conditional | `T extends U ? X : Y` (with union T) | Applies to each union member |
| Non-distributive | `[T] extends [U] ? X : Y` | Treats union as whole |

## Summary

Mastering generics with constraints and bounds is the difference between writing types that are **merely correct** and types that are **precisely engineered**. The key principles to remember:

1. **Use constraints to encode domain knowledge** — don't leave generics unconstrained unless you truly accept anything
2. **Preserve specificity** — always return `T`, not the constraint type
3. **Leverage `keyof`** — it's the bridge between runtime property names and compile-time type safety
4. **Use conditional types for branching** — think of `extends` in conditional types as "matches" rather than "inherits"
5. **Embrace recursive types** — they model real-world data structures (trees, nested configs, deep paths)
6. **Prefer `const` type parameters** — when you need literal type preservation (TS 5.0+)
7. **Understand variance** — it determines when generic types are substitutable
