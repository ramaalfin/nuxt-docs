# Conditional Types (T extends U ? X : Y)

Conditional types are TypeScript's way of introducing **type-level logic**. They allow you to define types that depend on a condition, acting essentially as `if/else` statements for types. When combined with generics, they enable you to build highly dynamic, adaptive type systems.

The syntax mirrors JavaScript's ternary operator:

```typescript
SomeType extends OtherType ? TrueType : FalseType;
```

If `SomeType` is assignable to `OtherType`, the result is `TrueType`. Otherwise, it's `FalseType`.

---

## 1. The Basics of Conditional Types

In their simplest form, conditional types take an input type and return one of two output types:

```typescript
type IsString<T> = T extends string ? true : false;

type A = IsString<string>;  // true
type B = IsString<number>;  // false
type C = IsString<"hello">; // true (literal string extends string)
```

This becomes much more powerful when used to adapt types dynamically based on input:

```typescript
interface Animal { live(): void; }
interface Dog extends Animal { bark(): void; }

type Example1 = Dog extends Animal ? number : string; 
// Type is number

type Example2 = RegExp extends Animal ? number : string; 
// Type is string
```

### Real-world Example: API Response Wrapper

Imagine an API where the response format changes depending on whether the data is a single item or an array of items:

```typescript
type ApiResponse<T> = T extends any[] 
  ? { data: T, count: number } 
  : { data: T, cacheId: string };

type UsersResponse = ApiResponse<User[]>; 
// { data: User[], count: number }

type UserResponse = ApiResponse<User>;    
// { data: User, cacheId: string }
```

## 2. Distributive Conditional Types

When conditional types act on a **generic type**, and that generic type is a **union**, they become **distributive**.

This means that `Condition<A | B | C>` is evaluated as `Condition<A> | Condition<B> | Condition<C>`.

```typescript
type ToArray<Type> = Type extends any ? Type[] : never;

type StrArrOrNumArr = ToArray<string | number>;
// Evaluates as: (string extends any ? string[] : never) | (number extends any ? number[] : never)
// Result: string[] | number[]
```

Notice that the result is `string[] | number[]`, NOT `(string | number)[]`.

### Preventing Distribution

Sometimes distribution is exactly what you want. But if you want to check if the *entire union* extends something, or you want the result to be a single array containing the union, you must **wrap the types in tuples `[]`** to disable distribution:

```typescript
type ToArrayNonDistributive<Type> = [Type] extends [any] ? Type[] : never;

type StrOrNumArr = ToArrayNonDistributive<string | number>;
// Result: (string | number)[]
```

### Filtering Unions with `never`

A common pattern with distributive conditional types is using `never` to filter out members of a union. Since `Union | never` is just `Union` (because `never` is the empty set), we can remove types we don't want:

```typescript
type ExcludeNullAndUndefined<T> = T extends null | undefined ? never : T;

type CleanType = ExcludeNullAndUndefined<string | number | null | undefined>;
// Result: string | number
```

TypeScript provides this exact utility built-in as `NonNullable<T>`.

## 3. The `infer` Keyword

Conditional types become infinitely more powerful with the `infer` keyword. `infer` allows you to **declare a type variable within the `extends` clause** and then use it in the true branch.

It's essentially pattern matching for types.

### Extracting Function Return Types

```typescript
type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : any;

type Num = MyReturnType<() => number>;   // number
type Str = MyReturnType<(s: string) => string>; // string
```

Here, we are saying: "If `T` is a function, infer its return type and assign it to `R`, then return `R`. Otherwise, return `any`."

### Extracting Array Element Types

```typescript
type UnpackArray<T> = T extends (infer U)[] ? U : T;

type ItemType = UnpackArray<string[]>; // string
type NotAnArray = UnpackArray<number>; // number (condition failed, returned T)
```

### Extracting Promise Resolution Types

```typescript
type UnpackPromise<T> = T extends Promise<infer U> ? U : T;

type ResolvedType = UnpackPromise<Promise<User>>; // User
```

TypeScript provides a built-in `Awaited<T>` that does this recursively to handle nested Promises (`Promise<Promise<T>>`).

## 4. Advanced `infer` Patterns

### Multiple `infer` Declarations

You can use `infer` multiple times in a single condition:

```typescript
type TupleHeadAndTail<T extends any[]> = 
  T extends [infer Head, ...infer Tail] ? { head: Head, tail: Tail } : never;

type Parts = TupleHeadAndTail<[string, number, boolean]>;
// { head: string, tail: [number, boolean] }
```

### String Template Inference

Since TypeScript 4.1, `infer` works with template literal types to parse strings at the type level:

```typescript
type ExtractProtocol<URL extends string> = 
  URL extends `${infer Protocol}://${string}` ? Protocol : "unknown";

type Http = ExtractProtocol<"http://example.com">;   // "http"
type Https = ExtractProtocol<"https://example.com">; // "https"
type Ftp = ExtractProtocol<"ftp://server.local">;    // "ftp"
```

### Conditional Type Constraints in `infer` (TS 4.7+)

In TypeScript 4.7 and later, you can add `extends` constraints directly to `infer` declarations:

```typescript
// Before TS 4.7
type FirstIfString<T> = 
  T extends [infer S, ...unknown[]] 
    ? S extends string 
      ? S 
      : never 
    : never;

// TS 4.7+
type FirstIfStringModern<T> = 
  T extends [infer S extends string, ...unknown[]] ? S : never;

type Yes = FirstIfStringModern<[string, number]>; // string
type No = FirstIfStringModern<[number, string]>;  // never
```

## 5. Built-in Utility Types Using Conditional Types

Many of TypeScript's standard utility types are built entirely using conditional types. Studying them is a great way to learn:

### `Exclude<UnionType, ExcludedMembers>`
Removes types from a union.
```typescript
type Exclude<T, U> = T extends U ? never : T;
```

### `Extract<Type, Union>`
Keeps only types that are assignable to the union.
```typescript
type Extract<T, U> = T extends U ? T : never;
```

### `Parameters<Type>`
Extracts the parameter types of a function as a tuple.
```typescript
type Parameters<T extends (...args: any) => any> = 
  T extends (...args: infer P) => any ? P : never;
```

## 6. Real-World Advanced Pattern: Deep Object Omit

A common challenge is removing a property from an object deeply (at all nested levels). We can solve this with recursive conditional types and mapped types.

```typescript
type DeepOmit<T, K extends string | number | symbol> = 
  // If T is an array, apply DeepOmit to its elements
  T extends Array<infer E> ? Array<DeepOmit<E, K>> :
  // If T is an object, map over its keys
  T extends object ? {
    [Key in keyof T as Key extends K ? never : Key]: DeepOmit<T[Key], K>
  } :
  // If primitive, return as is
  T;

interface ComplexObject {
  id: string;
  name: string;
  metadata: {
    id: number;
    createdAt: Date;
    author: {
      id: string;
      username: string;
    }
  };
  tags: { id: string, name: string }[];
}

// Remove 'id' from all levels!
type WithoutIds = DeepOmit<ComplexObject, 'id'>;
/*
{
  name: string;
  metadata: {
    createdAt: Date;
    author: {
      username: string;
    }
  };
  tags: { name: string }[];
}
*/
```

## Summary

Conditional types (`T extends U ? X : Y`) are the foundation of TypeScript's metaprogramming capabilities:

1. They provide **type-level branching** (`if/else`).
2. They **distribute over unions** automatically (unless explicitly disabled).
3. The `infer` keyword enables **type-level pattern matching** to extract internal types from functions, arrays, promises, and even strings.
4. They can be **recursive** to build deep utilities (like `DeepOmit` or `DeepReadonly`).
