# Discriminated Unions & Exhaustiveness Checking

When working with complex, varying data structures in TypeScript, **Discriminated Unions** (also known as tagged unions or algebraic data types) are one of the most powerful and commonly used patterns. They allow you to represent data that can take one of several different, distinct shapes, and process that data with complete type safety.

Combined with **Exhaustiveness Checking**, discriminated unions guarantee that you've handled every possible variation of your data, eliminating a whole category of runtime bugs.

---

## 1. What is a Discriminated Union?

A discriminated union consists of three key ingredients:

1. **Types that have a common, singleton property** (the discriminant or tag).
2. A **Type Alias** that unions those types together.
3. A **Type Guard** on that common property to narrow the type.

### The Problem it Solves

Imagine you are fetching data from an API that can be in one of three states: Loading, Success, or Error.

```typescript
// ❌ BAD: A monolithic interface where everything is optional
interface ApiState {
  state: "loading" | "success" | "error";
  data?: string[];
  error?: Error;
}

function handleState(api: ApiState) {
  if (api.state === "success") {
    // TypeScript doesn't know if data exists!
    console.log(api.data.length); // ❌ Error: Object is possibly 'undefined'
  }
}
```

### The Discriminated Union Solution

Instead of making properties optional, we define each exact possible state as its own distinct type, and give them a common literal property (`status` in this case).

```typescript
// ✅ GOOD: Distinct types united by a discriminator
interface StateLoading {
  status: "loading";
}

interface StateSuccess {
  status: "success";
  data: string[];
}

interface StateError {
  status: "error";
  error: Error;
}

type ApiState = StateLoading | StateSuccess | StateError;
```

Now, when we check the discriminant (`status`), TypeScript's control flow analysis **narrows** the union to the exact type.

```typescript
function handleState(state: ApiState) {
  if (state.status === "success") {
    // TypeScript KNOWS state is exactly StateSuccess here.
    // Data is guaranteed to exist.
    console.log(state.data.length); // ✅ OK
  } else if (state.status === "error") {
    console.error(state.error.message); // ✅ OK
  } else {
    // TypeScript knows state must be StateLoading here
    console.log("Loading...");
  }
}
```

## 2. Using `switch` for Type Narrowing

While `if/else` works, `switch` statements are often cleaner for discriminated unions, especially when the number of cases grows.

```typescript
type Shape = 
  | { kind: "circle"; radius: number }
  | { kind: "square"; sideLength: number }
  | { kind: "rectangle"; width: number; height: number };

function getArea(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2; // narrowed to Circle
    case "square":
      return shape.sideLength ** 2;       // narrowed to Square
    case "rectangle":
      return shape.width * shape.height;  // narrowed to Rectangle
  }
}
```

## 3. Exhaustiveness Checking

What happens if another developer comes along and adds a new shape to the `Shape` union?

```typescript
type Shape = 
  | { kind: "circle"; radius: number }
  | { kind: "square"; sideLength: number }
  | { kind: "rectangle"; width: number; height: number }
  | { kind: "triangle"; base: number; height: number }; // NEW!
```

If we don't update our `getArea` function, it will silently return `undefined` for triangles. This is a common source of bugs in JavaScript.

**Exhaustiveness Checking** is a TypeScript technique to force a compile-time error if you forget to handle a case in a discriminated union.

### The `never` Assignment Technique (Recommended)

We can use the `never` type to ensure every possible case has been handled. The `never` type represents a state that should not exist. If our `switch` statement handles all cases, the leftover type in the `default` block will be `never`.

```typescript
function getArea(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.sideLength ** 2;
    case "rectangle":
      return shape.width * shape.height;
    default:
      // We assign the remaining value of `shape` to a variable of type `never`.
      // If we handled all cases, `shape` is `never`, and this is valid.
      // If we missed a case (like "triangle"), `shape` will be `{ kind: "triangle" ... }`
      // and TypeScript will throw an error because we can't assign it to `never`.
      const _exhaustiveCheck: never = shape;
      return _exhaustiveCheck;
  }
}
// ❌ ERROR: Type '{ kind: "triangle"; base: number; height: number; }' is not assignable to type 'never'.
```

### The Exhaustive Helper Function

Many teams prefer to abstract this into a reusable helper function that throws a runtime error (just in case) while providing the same compile-time safety:

```typescript
function assertUnreachable(x: never): never {
  throw new Error(`Unhandled case: ${JSON.stringify(x)}`);
}

function getArea(shape: Shape): number {
  switch (shape.kind) {
    case "circle": return Math.PI * shape.radius ** 2;
    case "square": return shape.sideLength ** 2;
    case "rectangle": return shape.width * shape.height;
    default:
      // Throws at runtime, errors at compile time if cases are missing
      return assertUnreachable(shape); 
  }
}
```

### Using Compiler Flags (Alternatives)

You can also rely on the `noImplicitReturns` or `strictNullChecks` compiler flags. If your function is typed to return a `number`, and you miss a `switch` case, TypeScript will complain that the function might implicitly return `undefined`.

However, the `assertUnreachable(never)` technique is preferred because it explicitly documents *intent* and guarantees an error exactly at the point of the unhandled state, regardless of function return types.

## 4. Advanced Patterns

### Redux / Vuex / Pinia Actions
Discriminated unions are the absolute best way to type state management actions/events.

```typescript
type Action =
  | { type: "ADD_TODO"; payload: { text: string } }
  | { type: "TOGGLE_TODO"; payload: { id: number } }
  | { type: "REMOVE_TODO"; payload: { id: number } }
  | { type: "CLEAR_COMPLETED" }; // Note: payload is completely absent here!

function reducer(state: State, action: Action) {
  switch (action.type) {
    case "ADD_TODO":
      // action is strictly the ADD_TODO object
      return [...state, action.payload]; 
    case "CLEAR_COMPLETED":
      // action.payload doesn't exist on this branch!
      // action.payload; // ❌ Error
      return state.filter(t => !t.completed);
    default:
      assertUnreachable(action);
  }
}
```

### Multiple Discriminants
You aren't restricted to a single property like `type` or `kind`. You can use combinations of literal types to narrow down data.

```typescript
type Event = 
  | { source: "mouse"; event: "click"; x: number; y: number }
  | { source: "mouse"; event: "scroll"; deltaY: number }
  | { source: "keyboard"; event: "keydown"; key: string };

function handleEvent(e: Event) {
  if (e.source === "mouse") {
    // e is narrowed to the two mouse events
    if (e.event === "click") {
      console.log(e.x, e.y); // ✅
    }
  }
}
```

### Booleans as Discriminants
While string literals are most common, you can also use boolean literals (`true` / `false`) as discriminants! This is great for representing validation results.

```typescript
type ValidationResult<T> = 
  | { isValid: true; data: T }
  | { isValid: false; error: string };

function processInput(result: ValidationResult<number>) {
  if (result.isValid) {
    console.log(result.data * 2); // ✅ data exists
  } else {
    console.error(result.error);  // ✅ error exists
  }
}
```

---

## Summary

- **Discriminated Unions** are the standard way to model data that can be in mutually exclusive states.
- They require a **common literal property** (like `type`, `kind`, `status`, or a boolean).
- Use `switch` or `if/else` on that common property to **narrow** the type safely.
- Always use **Exhaustiveness Checking** via the `never` type (`assertUnreachable`) to ensure your code won't break silently when new variants are added to the union in the future.
