# DeepPartial & Recursive Types

TypeScript's type system is powerful enough to define types that reference themselves. These are known as **Recursive Types**. They are essential for modeling data structures that can nest infinitely, such as JSON objects, file system trees, or deeply nested configuration objects.

One of the most common and practical uses of recursive types is the `DeepPartial` utility type.

---

## 1. What are Recursive Types?

A recursive type is simply a type or interface that refers to itself in its own definition.

### Example: A File System Tree

Imagine modeling a directory structure where a folder can contain files or other folders.

```typescript
// Using an interface
interface TreeNode {
  name: string;
  isFolder: boolean;
  // recursive reference
  children?: TreeNode[]; 
}

const fileSystem: TreeNode = {
  name: "root",
  isFolder: true,
  children: [
    {
      name: "src",
      isFolder: true,
      children: [
        { name: "index.ts", isFolder: false }
      ]
    },
    { name: "package.json", isFolder: false }
  ]
};
```

### Example: A JSON Object

A JSON object is inherently recursive. It can be a string, number, boolean, null, an array of JSON values, or an object containing JSON values.

```typescript
// Using a type alias
type JSONValue = 
  | string
  | number
  | boolean
  | null
  | JSONValue[]
  | { [key: string]: JSONValue };

const data: JSONValue = {
  user: {
    id: 1,
    name: "Alice",
    tags: ["admin", "staff", { nested: true }]
  }
};
```

## 2. The Problem with built-in `Partial<T>`

TypeScript provides a built-in utility type called `Partial<T>`. It takes a type `T` and makes all of its top-level properties optional.

```typescript
interface UserConfig {
  theme: string;
  notifications: {
    email: boolean;
    sms: boolean;
  };
}

// Partial only affects the top level!
type PartialConfig = Partial<UserConfig>;

/*
Equivalent to:
{
  theme?: string;
  notifications?: { // Note: email and sms inside here are STILL REQUIRED!
    email: boolean;
    sms: boolean;
  };
}
*/
```

If you try to pass an object that only updates a deeply nested property, TypeScript will complain:

```typescript
const updateConfig = (config: Partial<UserConfig>) => { /* ... */ };

// ❌ ERROR: Property 'sms' is missing in type '{ email: true; }' 
// but required in type '{ email: boolean; sms: boolean; }'.
updateConfig({
  notifications: {
    email: true 
  }
});
```

`Partial<T>` is **shallow**. We need a deep version of it.

---

## 3. Creating `DeepPartial<T>`

To make all properties optional, regardless of how deeply nested they are, we must write a recursive utility type.

Let's build `DeepPartial<T>` step by step.

### Step 1: The Basic Recursion

We use a Mapped Type to iterate over the keys of `T`, and for each property, we apply `DeepPartial` to its value type.

```typescript
type DeepPartial<T> = {
  [P in keyof T]?: DeepPartial<T[P]>;
};
```

This works for basic nested objects!

```typescript
// ✅ Works now!
const updateConfig = (config: DeepPartial<UserConfig>) => { /* ... */ };

updateConfig({
  notifications: {
    email: true // No error about missing 'sms'!
  }
});
```

### Step 2: Handling Arrays and Primitives

The simple version above has a flaw: it tries to map over primitives (like `string` or `number`) and arrays in weird ways, turning `Array<string>` into `{ [key: number]: DeepPartial<string> }` which loses array prototype methods.

We need to add conditional types to stop the recursion when it hits something that shouldn't be mapped.

```typescript
type DeepPartial<T> = 
  // If T is a primitive (or a Function/Date/RegExp), just leave it as T.
  T extends Function | Date | RegExp | string | number | boolean | null | undefined
    ? T 
  // If T is an Array, apply DeepPartial to its elements.
  : T extends Array<infer U> 
    ? Array<DeepPartial<U>>
  // If T is a ReadonlyArray, handle that too.
  : T extends ReadonlyArray<infer U>
    ? ReadonlyArray<DeepPartial<U>>
  // Otherwise, it's an object, so recurse!
  : { [P in keyof T]?: DeepPartial<T[P]> };
```

This is a highly robust `DeepPartial` implementation, very similar to what libraries like Redux or TypeORM use internally.

## 4. Other Common Recursive Utility Types

The concept of `DeepPartial` can be applied to other built-in utility types as well.

### `DeepReadonly<T>`

Makes every property deeply immutable. Useful for freezing configuration objects or Redux state.

```typescript
type DeepReadonly<T> = 
  T extends Function | Date | RegExp | string | number | boolean | null | undefined
    ? T
  : T extends Array<infer U>
    ? ReadonlyArray<DeepReadonly<U>>
  : { readonly [P in keyof T]: DeepReadonly<T[P]> };
```

### `DeepRequired<T>`

The opposite of `DeepPartial`. Removes all `?` modifiers deeply.

```typescript
type DeepRequired<T> = 
  T extends Function | Date | RegExp | string | number | boolean | null | undefined
    ? T
  : T extends Array<infer U>
    ? Array<DeepRequired<U>>
  : { [P in keyof T]-?: DeepRequired<T[P]> };
```

---

## 5. Performance Warning

While recursive types are incredibly powerful, they come with a cost. 

If you apply a complex recursive type like `DeepPartial` to a massive, deeply nested object with hundreds of properties, the TypeScript compiler has to perform an enormous amount of work to evaluate the type. 

**Excessive use of deep recursive types can lead to:**
1. Slow IDE performance (autocompletion lag).
2. Slower build times (`tsc`).
3. "Type instantiation is excessively deep and possibly infinite" compilation errors.

**Best Practice:** Use recursive types like `DeepPartial` only when truly necessary, and avoid applying them to excessively large "everything" objects (like the global Window object or complex DOM nodes).

---

## Summary

- **Recursive types** are types that reference themselves, useful for modeling trees, graphs, and JSON data.
- **`Partial<T>` is shallow**, meaning it only makes top-level properties optional.
- **`DeepPartial<T>`** uses conditional types and mapped types recursively to make all nested properties optional.
- We must use conditional checks (`T extends ...`) to stop recursion when hitting primitives or arrays to prevent malformed types.
- Be mindful of compiler performance when using complex recursive types on large data structures.
