# Branded Types (Newtype Pattern)

TypeScript's type system is structural, meaning that types are compatible if their shapes match. While this is incredibly flexible, it can sometimes lead to logic errors when you want to enforce that two values with the exact same structure (like two `string`s) are actually treated as fundamentally different types.

The **Branded Types** pattern (often called the **Newtype pattern** after its equivalent in languages like Haskell or Rust) is a technique used to simulate nominal typing in TypeScript's structural type system.

---

## 1. The Problem with Structural Typing

Consider an application that uses multiple different types of IDs, all represented as strings.

```typescript
type UserId = string;
type PostId = string;

function deleteUser(id: UserId) {
  // Deletes the user
}

const myPostId: PostId = "post_12345";

// ❌ TypeScript allows this, even though it's logically a bug!
deleteUser(myPostId);
```

Because `UserId` and `PostId` are just aliases for `string`, TypeScript sees them as identical. You can accidentally pass a `PostId` into a function expecting a `UserId`, leading to catastrophic bugs at runtime.

---

## 2. What is a Branded Type?

A branded type takes a primitive type (like `string` or `number`) and intersects it with an object containing a unique "brand" property. This brand property doesn't exist at runtime, but it forces TypeScript to treat the type as distinct.

```typescript
// The generic Branded Type helper
type Brand<K, T> = K & { __brand: T };

// Creating our distinct types
type UserId = Brand<string, "UserId">;
type PostId = Brand<string, "PostId">;
```

Now, `UserId` and `PostId` are no longer just strings. They are strings *that also secretly have a `__brand` property*. 

---

## 3. How to Use Branded Types

Because the `__brand` property doesn't exist at runtime, you cannot create a branded type via a normal object literal. You must use a factory function or a type assertion to "mint" the branded type.

### Minting the Type (Type Assertions)

```typescript
// Minting function for UserId
function makeUserId(id: string): UserId {
  // We assert that the string is a UserId. 
  // No runtime overhead is introduced.
  return id as UserId; 
}

function deleteUser(id: UserId) {
  console.log(`Deleting user: ${id}`);
}

const myPostId = "post_123" as PostId;
const myUserId = makeUserId("user_456");

// ✅ Works perfectly
deleteUser(myUserId);

// ❌ ERROR: Argument of type 'PostId' is not assignable to parameter of type 'UserId'.
deleteUser(myPostId);
```

TypeScript will now catch you if you accidentally mix up your IDs!

### Type Safety with Primitives

You can still use branded types anywhere the primitive type is expected, because a `Brand<string, "UserId">` is still fundamentally a `string`.

```typescript
const userId = makeUserId("user_789");

// ✅ Allowed: String methods still work!
console.log(userId.toUpperCase()); 

// ✅ Allowed: Can be assigned to a normal string
const regularString: string = userId; 
```

However, you *cannot* assign a normal string to a branded type without an explicit assertion.

```typescript
// ❌ ERROR: Type 'string' is not assignable to type 'UserId'.
const id: UserId = "just a string"; 
```

---

## 4. Advanced: Flavoring (Weak Branded Types)

Sometimes, strict branding is too restrictive. You might want a type that prevents you from accidentally mixing up IDs, but still allows you to pass a raw string *if* you're feeling confident, without having to use a factory function.

This is called "Flavoring". It uses an optional brand property instead.

```typescript
type Flavor<K, T> = K & { __flavor?: T };

type EmailAddress = Flavor<string, "EmailAddress">;

function sendEmail(email: EmailAddress) {
  // ...
}

const userEmail = "test@example.com" as EmailAddress;
const someString = "hello world";

// ✅ Works with the flavored type
sendEmail(userEmail);

// ✅ Works with a raw string (unlike strict Branding)
sendEmail("raw@string.com"); 

// ❌ However, if you explicitly flavor it wrong, it fails:
type PhysicalAddress = Flavor<string, "PhysicalAddress">;
const address = "123 Main St" as PhysicalAddress;

// ERROR: Types of property '__flavor' are incompatible.
sendEmail(address); 
```

Flavoring provides a balance: it stops you from passing a `PhysicalAddress` to an `EmailAddress` function, but doesn't force you to explicitly cast raw `"test@example.com"` strings everywhere in your codebase.

---

## 5. Best Practices & Considerations

1.  **Use specific naming:** The `__brand` or `__flavor` property names should be unique or obscure to avoid accidental collisions with real runtime properties. Some developers use a unique `symbol`.
    ```typescript
    declare const brand: unique symbol;
    type Brand<K, T> = K & { readonly [brand]: T };
    ```
2.  **Centralize factories:** If you are using strict branding, export factory functions (like `makeUserId()`) or validation functions (`parseUserId()`) rather than casting `as UserId` everywhere.
3.  **Validation:** Branded types pair perfectly with runtime validation schemas like Zod. You can create a Zod schema that parses a string and returns a branded type if it matches the correct format (e.g., an email or UUID schema).
4.  **Avoid over-engineering:** Don't brand every single primitive in your app. Reserve it for critical domain concepts (IDs, Tokens, Currency, validated structures) where confusing them would cause severe bugs.

---

## Summary

- TypeScript's **structural typing** can sometimes let distinct domain concepts mix if their shapes are the same (like two strings).
- **Branded Types (Newtype)** intersect a primitive with an imaginary property to create distinct, incompatible types.
- They incur **zero runtime overhead** and are created via type assertions (`as Type`).
- They are an excellent tool for making your domain model extremely robust and preventing accidental data mix-ups.
