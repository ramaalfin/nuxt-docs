# Generic Variance: Covariance & Contravariance

If you've spent enough time with TypeScript's advanced type system, you might have run into mysterious errors where TypeScript refuses to accept one generic type in place of another, even though they seem compatible. 

This behavior is governed by rules of **Variance**—specifically **Covariance**, **Contravariance**, **Bivariance**, and **Invariance**. Understanding these concepts will level up your mental model of how TypeScript determines if types are compatible.

---

## 1. What is Variance?

Variance describes how the subtyping relationship of complex types (like generics or functions) relates to the subtyping relationship of their component types.

Let's establish a basic subtyping relationship:
```typescript
class Animal { name: string; }
class Dog extends Animal { bark: boolean; }
```
Here, `Dog` is a subtype of `Animal` (`Dog <: Animal`). Anywhere an `Animal` is required, a `Dog` can be provided.

Now, what happens if we put these types inside a generic like `Array<T>` or a function `(arg: T) => void`? How do `Array<Dog>` and `Array<Animal>` relate? 

That relationship is Variance.

---

## 2. Covariance (Going with the flow)

**Covariance** means the complex type preserves the subtyping relationship of the inner type. If `Dog` is a subtype of `Animal`, then `Container<Dog>` is a subtype of `Container<Animal>`.

In TypeScript, **Objects, Arrays, and Promises are covariant.**

### Example: Arrays

```typescript
let animals: Array<Animal> = [];
let dogs: Array<Dog> = [new Dog()];

// ✅ Covariance in action!
// Since Dog <: Animal, Array<Dog> <: Array<Animal>
animals = dogs; 
```

This makes intuitive sense. If you have a function that expects an array of Animals to read from, passing it an array of Dogs is perfectly safe because every Dog is an Animal.

*(Note: In fully sound type systems, mutable arrays are invariant, but TypeScript intentionally makes them covariant for developer convenience, even though it can lead to theoretical runtime errors if you mutate the array).*

---

## 3. Contravariance (Going against the flow)

**Contravariance** means the subtyping relationship is *reversed*. If `Dog` is a subtype of `Animal`, then `Container<Animal>` is a subtype of `Container<Dog>`.

This sounds incredibly counter-intuitive until you look at **Function Arguments**. In TypeScript (with `strictFunctionTypes: true`), function parameters are contravariant.

### Example: Function Arguments

Let's look at a function that takes a callback.

```typescript
type AnimalHandler = (a: Animal) => void;
type DogHandler = (d: Dog) => void;

let handleAnimal: AnimalHandler = (a: Animal) => { console.log(a.name); };
let handleDog: DogHandler = (d: Dog) => { console.log(d.name, d.bark); };

// ❌ ERROR: Type 'AnimalHandler' is not assignable to type 'DogHandler'.
// handleAnimal = handleDog; 
```

Why is `handleAnimal = handleDog` bad? If you call `handleAnimal(new Animal())`, the underlying `handleDog` function would try to access `.bark`, which a regular `Animal` doesn't have. Crash!

However, the reverse works:

```typescript
// ✅ Contravariance in action!
// AnimalHandler <: DogHandler
handleDog = handleAnimal;
```

Why is this safe? `handleDog` guarantees that it will be called with a `Dog`. If it's secretly `handleAnimal` under the hood, `handleAnimal` only cares about the `name` property. Since a `Dog` has a `name` property, it's perfectly safe!

**Rule of thumb:** 
- Return types are **Covariant** (they produce data).
- Argument types are **Contravariant** (they consume data).

---

## 4. Invariance (Strict matching)

**Invariance** means there is no subtyping relationship at all. `Container<A>` and `Container<B>` are completely incompatible unless `A` and `B` are exactly the same type.

In TypeScript, if you create a generic interface where the type parameter `T` is used in *both* covariant (return type/property) and contravariant (function argument) positions, the generic becomes invariant.

```typescript
interface StateManager<T> {
  getValue(): T;          // Covariant position
  setValue(val: T): void; // Contravariant position
}

let animalManager: StateManager<Animal>;
let dogManager: StateManager<Dog>;

// ❌ ERROR: Incompatible
// animalManager = dogManager; 

// ❌ ERROR: Incompatible
// dogManager = animalManager; 
```

This is because:
1. `animalManager = dogManager` fails because `animalManager.setValue(new Animal())` would pass an `Animal` to `dogManager.setValue`, which expects a `Dog` (needs contravariance, but gets covariance).
2. `dogManager = animalManager` fails because `dogManager.getValue()` expects a `Dog`, but `animalManager.getValue()` might return a standard `Animal` (needs covariance, but gets contravariance).

Therefore, they must be strictly the same type.

---

## 5. Bivariance (The wild west)

**Bivariance** means both covariant and contravariant relationships are allowed. It's essentially "loose" typing.

Historically, function arguments in TypeScript were bivariant. This was changed in TypeScript 2.6 with the `--strictFunctionTypes` flag, which made function arguments properly contravariant.

However, method syntax in interfaces and classes is *still* intentionally bivariant in TypeScript today!

```typescript
// Method syntax (Bivariant)
interface EventLogger {
  logEvent(event: Animal): void; 
}

// Property syntax (Contravariant - Strict)
interface StrictLogger {
  logEvent: (event: Animal) => void;
}
```

If you are designing APIs where you want strict type checking on callbacks (like React event handlers), **always use property syntax (`logEvent: (e: T) => void`) instead of method syntax (`logEvent(e: T): void`)**.

---

## 6. TypeScript 4.7+ Variance Annotations

Sometimes, the TypeScript compiler struggles to infer the correct variance for deeply complex recursive generic types, leading to huge performance hits or confusing errors.

TypeScript 4.7 introduced explicit Variance Annotations: `in` and `out`.

- `out T`: Indicates `T` is strictly covariant (it only comes "out" of the type).
- `in T`: Indicates `T` is strictly contravariant (it only goes "in" to the type).
- `in out T`: Indicates `T` is invariant (it goes both in and out).

```typescript
// We explicitly tell TS that T is only ever produced, never consumed.
type Provider<out T> = () => T;

// We explicitly tell TS that T is only ever consumed, never produced.
type Consumer<in T> = (arg: T) => void;
```

This is mostly an advanced optimization tool for library authors (like the developers of React or Vue) to help the TypeScript compiler type-check faster, but it's great to know if you see `in` or `out` in a `.d.ts` file!

---

## Summary

- **Covariance (`out`)**: Type relationships go the *same* direction. (`Dog` <: `Animal`, so `Array<Dog>` <: `Array<Animal>`). Common for read-only properties, arrays, and function return types.
- **Contravariance (`in`)**: Type relationships are *reversed*. (`Dog` <: `Animal`, but `AnimalHandler` <: `DogHandler`). Common for function arguments.
- **Invariance (`in out`)**: Types must match exactly. Occurs when a generic is used as both an input and an output.
- **Bivariance**: Both directions are allowed. (Avoid this; it exists mainly for legacy reasons in TS method signatures).
