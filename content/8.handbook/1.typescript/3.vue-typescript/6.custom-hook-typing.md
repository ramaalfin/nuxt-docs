# Custom Hook (Composable) Typing

In Vue 3, "Hooks" are referred to as **Composables**. Writing type-safe composables requires a deep understanding of Vue's reactivity system (`Ref`, `ComputedRef`, `Reactive`) and how to make your functions flexible enough to accept both raw JavaScript values and reactive Vue variables.

---

## 1. The `MaybeRef` and `MaybeRefOrGetter` Utilities

When a developer uses your composable, they might want to pass a raw string:
`useGreeting("Alice")`

Or they might want to pass a reactive reference that updates over time:
`const name = ref("Alice"); useGreeting(name);`

Or even a getter function (Vue 3.3+):
`useGreeting(() => props.name)`

To type this perfectly, Vue provides `MaybeRef` and `MaybeRefOrGetter`.

```typescript
import { ref, watchEffect, toValue } from 'vue';
import type { MaybeRefOrGetter } from 'vue';

// 1. Accept MaybeRefOrGetter so the consumer can pass anything!
export function useGreeting(name: MaybeRefOrGetter<string>) {
  const greeting = ref('');

  watchEffect(() => {
    // 2. Use `toValue()` (Vue 3.3+) to unwrap refs AND execute getters safely
    const rawName = toValue(name);
    greeting.value = `Hello, ${rawName}!`;
  });

  return { greeting };
}
```

*Note: In versions prior to Vue 3.3, you would use `MaybeRef<T>` and the `unref()` function instead.*

## 2. Typing the Return Object

Composables almost always return an object containing multiple reactive variables and methods.

**Best Practice: Do not use `reactive()` for the return object.**

If you return a `reactive` object, the consumer will destroy reactivity if they destructure it without using `toRefs`. Instead, always return a plain JavaScript object containing `Ref`s.

```typescript
import { ref, readonly, type Ref, type DeepReadonly } from 'vue';

// Define the return type explicitly (Optional, but good for complex composables)
interface MousePosition {
  x: DeepReadonly<Ref<number>>;
  y: DeepReadonly<Ref<number>>;
}

export function useMouse(): MousePosition {
  const x = ref(0);
  const y = ref(0);

  // ... implementation (add event listeners, etc.)

  return {
    // 1. Return plain object with Refs
    // 2. Use `readonly()` so the consumer cannot mutate the composable's internal state
    x: readonly(x),
    y: readonly(y)
  };
}
```

Because we returned an object of Refs, the consumer can destructure safely:
```typescript
// Reactivity is preserved!
const { x, y } = useMouse(); 
```

## 3. Async Composables and `Promise<...>`

If your composable fetches data asynchronously, it shouldn't be an `async function` itself (unless it's strictly used inside `<script setup await>`).

Instead, the composable should return synchronous reactive state (`isLoading`, `data`, `error`) and fire the async work in the background.

```typescript
import { ref, shallowRef, type Ref, type ShallowRef } from 'vue';

export function useFetchUser<T>(userId: MaybeRefOrGetter<string>) {
  // Use ShallowRef for performance on large objects
  const data: ShallowRef<T | null> = shallowRef(null);
  const isLoading: Ref<boolean> = ref(false);
  const error: Ref<Error | null> = ref(null);

  const execute = async () => {
    isLoading.value = true;
    error.value = null;
    
    try {
      const id = toValue(userId);
      const res = await fetch(`/api/users/${id}`);
      data.value = await res.json();
    } catch (err) {
      error.value = err instanceof Error ? err : new Error(String(err));
    } finally {
      isLoading.value = false;
    }
  };

  // Automatically fetch on initialization
  execute();

  return { data, isLoading, error, execute };
}
```

**Usage:**
```typescript
interface User { name: string; age: number; }

// Generics map seamlessly into the returned Ref<T | null>
const { data: user, isLoading } = useFetchUser<User>("123");
```

## 4. State vs Options Arguments

When a composable takes many configuration options, use a typed `Options` interface. If some options can be reactive, type them specifically with `MaybeRefOrGetter`.

```typescript
interface UsePollingOptions {
  intervalMs?: number; // Static configuration
  enabled?: MaybeRefOrGetter<boolean>; // Reactive configuration!
}

export function usePolling(action: () => void, options: UsePollingOptions = {}) {
  const { intervalMs = 1000, enabled = true } = options;

  let timer: number | null = null;

  watchEffect((onCleanup) => {
    // toValue() checks if enabled is a ref/getter and unwraps it dynamically
    if (toValue(enabled)) {
      timer = window.setInterval(action, intervalMs);
    }

    onCleanup(() => {
      if (timer) clearInterval(timer);
    });
  });
}
```

This allows incredible flexibility for the consumer:

```typescript
const isUserActive = ref(true);

// The polling will automatically start/stop when `isUserActive` changes!
usePolling(() => console.log("Ping"), {
  enabled: isUserActive,
  intervalMs: 5000 
});
```

---

## Summary

- Use **`MaybeRefOrGetter<T>`** for arguments so consumers can pass raw values, Refs, or Getter functions.
- Use **`toValue(arg)`** inside your composable to safely unwrap those arguments.
- Always return a plain object containing `Ref` properties so consumers can safely destructure the results.
- Protect internal state by wrapping returned Refs in **`readonly()`**.
- For async composables, return synchronous reactive state (`isLoading`, `data`) rather than making the composable itself `async`.
