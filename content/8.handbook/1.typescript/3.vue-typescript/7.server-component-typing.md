# Server Component Typing

With the advent of Nuxt 3 and modern Vue 3 features, **Server Components** (often denoted by `.server.vue` suffixes or specific experimental flags) allow you to render UI components entirely on the server. The client receives pure, static HTML with zero JavaScript bundle overhead for that component.

However, crossing the boundary from Client to Server introduces strict constraints on what types of data can be passed. Typing Server Components requires understanding these serialization boundaries.

---

## 1. The Serialization Boundary

Standard Vue components run in the same memory space as their parent. If a parent passes a function, a `Date` object, or a massive class instance, the child can access it instantly by reference.

**Server Components run in a completely different environment (Node/Edge).** 

When a Client component renders a Server component, any props passed to it must be serialized (usually into JSON) and sent over the network, then deserialized on the server.

### The Rule: Props MUST be Serializable

When defining props for a Server Component, you must ensure that all types are JSON-serializable.

```vue
<!-- ✅ Valid Server Component Props -->
<script setup lang="ts">
interface ValidProps {
  id: number;
  title: string;
  metadata: {
    tags: string[];
    isActive: boolean;
  };
}

defineProps<ValidProps>();
</script>
```

```vue
<!-- ❌ Invalid Server Component Props -->
<script setup lang="ts">
interface InvalidProps {
  onClick: () => void;      // Functions cannot be serialized!
  createdAt: Date;          // Dates are stringified to ISO strings!
  userClass: UserModel;     // Class instances lose their methods!
  icon: Component;          // Vue component definitions cannot be passed!
}

defineProps<InvalidProps>();
</script>
```

---

## 2. Enforcing Serialization Types

If you want to be extremely strict, you can create a utility type to ensure that a given prop type only contains JSON-safe values, throwing a TypeScript error if a developer tries to pass a function or complex object.

```typescript
// A utility type that forces T to only contain primitives, arrays, and plain objects.
type JSONSafe<T> = T extends string | number | boolean | null 
  ? T 
  : T extends Function 
  ? never 
  : T extends object 
  ? { [K in keyof T]: JSONSafe<T[K]> } 
  : never;

// Now use it in your component
interface Props {
  data: JSONSafe<{ 
    id: number; 
    name: string; 
    // ❌ Error: Type '() => void' is not assignable to type 'never'
    // onClick: () => void; 
  }>;
}
```

While creating explicit `JSONSafe` utility types is an advanced pattern, it perfectly illustrates the constraints of Server Components.

---

## 3. Emits and Events

Because Server Components execute on the server and send static HTML to the client, **they cannot emit events back to the client**. They do not exist in the browser's memory.

Therefore, a `.server.vue` component should never have a `defineEmits` declaration.

```vue
<!-- MyStaticBanner.server.vue -->
<script setup lang="ts">
// ❌ Do not do this in a Server Component!
const emit = defineEmits<{
  (e: 'close'): void;
}>();
</script>

<template>
  <div>
    <!-- The @click handler will do nothing because no JS is sent to the client -->
    <button @click="emit('close')">Close</button>
  </div>
</template>
```

If you need interactivity (like closing a banner), you must use a standard Client Component, or wrap the Server Component inside a Client Component that handles the state.

---

## 4. Server-Only Data Fetching

The primary reason to use Server Components is to perform heavy, secure data fetching directly next to your database without exposing APIs to the client.

When typing the data fetched *inside* a Server Component, you can freely use heavy database ORM types (like Prisma models) without worrying about leaking those types to the client bundle.

```vue
<!-- UserProfile.server.vue -->
<script setup lang="ts">
// We can safely import server-only libraries!
import { prisma } from '~/server/db';
import type { PrismaUser } from '~/server/types';

const props = defineProps<{ userId: string }>();

// This runs ONLY on the server
const user: PrismaUser | null = await prisma.user.findUnique({
  where: { id: props.userId }
});
</script>

<template>
  <div v-if="user">
    <!-- Render static HTML safely using server data -->
    <h1>{{ user.name }}</h1>
    <p>Secret Server Hash: {{ user.passwordHash }}</p> 
  </div>
</template>
```

The client will never see the `PrismaUser` type, the `prisma` import, or the Javascript required to render this template. It only receives the resulting `<h1>` and `<p>` tags.

---

## Summary

- **Props must be JSON-Serializable**: Never type props as Functions, Dates, or Class Instances in a Server Component.
- **No Emits**: Do not type or use `defineEmits()`. Server Components generate static HTML and cannot listen to client-side DOM events.
- **Isolate Server Types**: It is perfectly safe to import and use heavy backend typings (like database models) inside a `.server.vue` component, as they will be stripped entirely from the client bundle.
