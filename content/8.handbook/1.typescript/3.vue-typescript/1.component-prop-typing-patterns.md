# Component Prop Typing Patterns

When building Vue 3 applications with TypeScript, typing component props correctly is essential for building robust, reusable UI components. Vue's `<script setup>` syntax provides several powerful ways to declare and type props, ranging from simple primitive types to complex objects and generic types.

---

## 1. Type-Based Declaration (Recommended)

The most idiomatic way to type props in Vue 3 with TypeScript is using the **Type-Based Declaration**. You define an interface or a type alias and pass it as a generic argument to `defineProps()`.

```vue
<script setup lang="ts">
// 1. Define the Prop interface
interface ButtonProps {
  label: string;
  color?: 'primary' | 'secondary' | 'danger'; // Optional with specific literal types
  disabled?: boolean;
  icon?: string;
}

// 2. Pass it to defineProps
const props = defineProps<ButtonProps>();
</script>

<template>
  <button :class="['btn', `btn-${color}`]" :disabled="disabled">
    <Icon v-if="icon" :name="icon" />
    {{ label }}
  </button>
</template>
```

**Why this is great:**
- Pure TypeScript syntax.
- You can extract `ButtonProps` to a separate file and import it (supported natively in Vue 3.3+).
- Immediate IDE autocomplete for consumers of this component.

---

## 2. Providing Default Values

When using Type-Based Declarations, you lose the ability to provide default values directly in the interface. Vue provides `withDefaults` to solve this.

```vue
<script setup lang="ts">
interface CardProps {
  title: string;
  elevation?: number;
  isHoverable?: boolean;
}

// Wrap defineProps with withDefaults
const props = withDefaults(defineProps<CardProps>(), {
  elevation: 1,         // Default for primitive
  isHoverable: false    // Default for primitive
});
</script>
```

### Defaults for Objects and Arrays
If a prop is an object or an array, you **must** provide a factory function to return the default value, preventing reference sharing across component instances.

```vue
<script setup lang="ts">
interface ListProps {
  items?: string[];
  config?: { sort: boolean };
}

const props = withDefaults(defineProps<ListProps>(), {
  items: () => [], // Factory function!
  config: () => ({ sort: true }) // Factory function!
});
</script>
```

## 3. Reactive Prop Destructuring (Vue 3.4+)

Historically, destructuring `props` in Vue destroyed reactivity. You had to use `toRefs(props)` or always access them via `props.title`.

Starting in Vue 3.4+, **Reactive Prop Destructuring** is fully supported and stable! It also provides a much cleaner way to assign default values without needing `withDefaults`.

```vue
<script setup lang="ts">
interface UserProfileProps {
  name: string;
  role?: string;
  avatarUrl?: string;
}

// ✅ Vue 3.4+ Syntax: Destructure directly and provide defaults!
const { 
  name, 
  role = 'guest', 
  avatarUrl = '/default-avatar.png' 
} = defineProps<UserProfileProps>();

// `role` remains reactive in the template!
</script>

<template>
  <div>
    <img :src="avatarUrl" />
    <h2>{{ name }}</h2>
    <p>Role: {{ role }}</p>
  </div>
</template>
```

*Note: This feature must be enabled in your `vite.config.ts` or `nuxt.config.ts` if you are on an older minor version, but it is standard moving forward.*

## 4. Generic Components

Sometimes you build a component (like a Select dropdown or a Table) that needs to handle *any* type of data, and you want to pass that type through to the slots or emitted events.

Vue 3.3 introduced `<script setup generic="...">`.

```vue
<!-- GenericSelect.vue -->
<script setup lang="ts" generic="T extends Record<string, any>">
interface Props {
  items: T[];
  itemText: keyof T; // Must be a key of the provided generic type
  itemValue: keyof T;
  modelValue?: T;
}

const props = defineProps<Props>();

const emit = defineEmits<{
  (e: 'update:modelValue', value: T): void;
}>();

const selectItem = (item: T) => {
  emit('update:modelValue', item);
};
</script>

<template>
  <ul>
    <li v-for="item in items" :key="String(item[itemValue])" @click="selectItem(item)">
      <!-- TypeScript knows `item` is type `T` here -->
      {{ item[itemText] }}
    </li>
  </ul>
</template>
```

**Usage:**
```vue
<script setup lang="ts">
import GenericSelect from './GenericSelect.vue';

interface User { id: number; name: string; }
const users: User[] = [{ id: 1, name: "Alice" }];
</script>

<template>
  <!-- Vue infers T = User automatically! -->
  <GenericSelect 
    :items="users" 
    item-text="name"  
    item-value="id"   
    @update:model-value="val => console.log(val.name)" 
  />
  
  <!-- ❌ Error: 'title' is not a key of User -->
  <!-- <GenericSelect :items="users" item-text="title" ... /> -->
</template>
```

## 5. Runtime Declaration (The Old Way)

For completeness, you can still use runtime declarations (like standard Vue 2 / Options API). This is generally discouraged in modern Vue+TS codebases because it is more verbose and relies on Vue's proprietary `PropType` utility.

```vue
<script setup lang="ts">
import type { PropType } from 'vue';

interface ComplexConfig { /* ... */ }

const props = defineProps({
  title: {
    type: String,
    required: true
  },
  config: {
    type: Object as PropType<ComplexConfig>, // Manual cast required
    default: () => ({})
  }
});
</script>
```

Always prefer **Type-Based Declarations** unless you are migrating legacy code or explicitly need runtime validation that TypeScript cannot provide.

---

## Summary

- Use **Type-Based Declarations** (`defineProps<MyProps>()`) as your default pattern.
- Prioritize **Reactive Prop Destructuring** for setting default values cleanly (if using Vue 3.4+).
- Fallback to `withDefaults` if using older Vue versions or strict organizational linting rules.
- Utilize `<script setup generic="T">` for highly reusable components like tables, lists, and dropdowns to maintain strict type safety from parent to child.
