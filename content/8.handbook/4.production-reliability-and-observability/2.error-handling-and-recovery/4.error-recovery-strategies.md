# Error Recovery Strategies

Detecting and displaying an error is only half the battle. A production-quality application provides clear, automatic, or user-guided pathways back to a working state. This is **error recovery**.

## 1. Automatic Retry with Exponential Backoff

For transient errors (network hiccups, temporary server overload), automatically retrying the failed operation without any user intervention is the best experience.

**Exponential Backoff** means waiting progressively longer between each retry attempt to avoid flooding a struggling server.

```typescript
// utils/retry.ts
export async function withRetry<T>(
  fn: () => Promise<T>,
  options: { maxAttempts?: number; baseDelay?: number } = {}
): Promise<T> {
  const { maxAttempts = 3, baseDelay = 500 } = options

  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await fn()
    } catch (error) {
      if (attempt === maxAttempts) throw error // Re-throw on final attempt
      
      // Exponential backoff: 500ms, 1000ms, 2000ms...
      const delay = baseDelay * Math.pow(2, attempt - 1)
      await new Promise(resolve => setTimeout(resolve, delay))
    }
  }
  throw new Error('Max retry attempts exceeded') // TypeScript safety
}
```

**Usage in a Pinia store action:**
```typescript
// stores/order.ts
const submitOrder = async (payload: OrderPayload) => {
  try {
    const result = await withRetry(
      () => $fetch('/api/orders', { method: 'POST', body: payload }),
      { maxAttempts: 3, baseDelay: 1000 }
    )
    // Success
  } catch (error) {
    // Only triggers after all retries are exhausted
    errorMessage.value = getErrorMessage(error)
  }
}
```

## 2. User-Triggered Retry

When automatic retries fail, the user must be able to manually trigger a retry with a single, obvious action.

```vue
<script setup lang="ts">
const { data, error, refresh } = await useFetch('/api/products')
</script>

<template>
  <div v-if="error" class="error-state">
    <ErrorIcon />
    <p>Failed to load products.</p>
    <!-- The `refresh` function from useFetch re-executes the fetch -->
    <button @click="refresh()" class="btn-retry">Try Again</button>
  </div>
  <ProductGrid v-else :products="data" />
</template>
```

## 3. State Rollback (Optimistic Update Recovery)

When an optimistic update fails (you updated the UI before the server confirmed), you must roll back to the previous state.

```typescript
// stores/todo.ts
export const useTodoStore = defineStore('todo', () => {
  const todos = ref<Todo[]>([])

  const toggleTodo = async (id: string) => {
    // 1. Cache the current state before mutation
    const previousTodos = [...todos.value]

    // 2. Optimistically update the UI
    const todo = todos.value.find(t => t.id === id)
    if (todo) todo.completed = !todo.completed

    try {
      // 3. Persist to server
      await $fetch(`/api/todos/${id}/toggle`, { method: 'PATCH' })
    } catch (error) {
      // 4. ROLLBACK: Restore the previous state on failure
      todos.value = previousTodos
      showNotification({ type: 'error', message: 'Failed to update. Changes reverted.' })
    }
  }

  return { todos, toggleTodo }
})
```

## 4. Page-Level Recovery: `clearError`

In Nuxt, when a fatal error is thrown (rendering `error.vue`), use `clearError` to dismiss the error and redirect the user to a safe location.

```vue
<!-- error.vue -->
<script setup lang="ts">
const props = defineProps<{ error: NuxtError }>()

const recover = () => clearError({ redirect: '/' })
</script>

<template>
  <div class="error-page">
    <h1>{{ props.error.statusCode }}</h1>
    <p>{{ props.error.statusMessage }}</p>
    <button @click="recover">Return to Homepage</button>
  </div>
</template>
```

## 5. Recovery Strategy Decision Tree

When designing an error flow, ask these questions in order:

```
1. Is this a transient error (network, timeout)? 
   → YES: Auto-retry with exponential backoff first.
   → NO: Go to 2.

2. Can the user fix it themselves (validation, missing input)?
   → YES: Show inline error with clear instructions.
   → NO: Go to 3.

3. Is this a recoverable server/app error?
   → YES: Show section-level error with a "Try Again" button.
   → NO: Go to 4.

4. Is this a fatal/unrecoverable error?
   → YES: Show full-page error.vue with a "Go Home" link.
```

## 6. Dead Letter Queue for Critical Operations

For extremely critical actions (e.g., financial transactions) that fail, do not silently lose them. Queue them in localStorage for re-processing when the application recovers.

```typescript
// composables/useActionQueue.ts
export function useActionQueue() {
  const queue = useLocalStorage<QueuedAction[]>('action_queue', [])

  const enqueue = (action: QueuedAction) => {
    queue.value.push(action)
  }

  const processQueue = async () => {
    const pending = [...queue.value]
    queue.value = [] // Clear before processing to avoid double-processing
    
    for (const action of pending) {
      try {
        await $fetch(action.endpoint, { method: action.method, body: action.payload })
      } catch {
        queue.value.push(action) // Re-enqueue on failure
      }
    }
  }
  
  // Process queue when app comes back online
  useEventListener(window, 'online', processQueue)

  return { enqueue, processQueue }
}
```
