# User-Friendly Error Messages

Technical error messages like `Error: 500 Internal Server Error` or `TypeError: Cannot read properties of undefined (reading 'name')` are meaningless—and alarming—to end users. Great error UX communicates what happened, what it means for the user, and what they can do next.

## 1. The Three Principles of Good Error Messages

### Principle 1: Be Human
Avoid jargon, error codes, and stack traces in the user-facing UI. Translate technical failures into plain language.

| ❌ Technical (Don't Show) | ✅ Human (Show This) |
|---------------------------|----------------------|
| `ERR_NETWORK_CHANGED` | "We couldn't connect. Please check your internet connection." |
| `422 Unprocessable Entity` | "Some of your information looks incorrect. Please review the highlighted fields." |
| `500 Internal Server Error` | "Something went wrong on our end. Our team has been notified." |

### Principle 2: Be Helpful
Every error message should suggest a next action whenever possible.

```
❌ Bad: "Error: Failed to load."

✅ Good: "We couldn't load your orders right now. 
         Try refreshing the page, or contact support if the problem continues."
                                                [Refresh] [Contact Support]
```

### Principle 3: Be Honest (But Reassuring)
Don't hide that an error occurred, but avoid language that creates panic. Never say "Fatal Error" to a user if the issue is a failed API request.

## 2. Error Message Architecture in Nuxt

### Centralizing Error Messages
Define all user-facing error messages in a single dictionary, rather than hardcoding strings throughout the codebase.

```typescript
// utils/errorMessages.ts
export const ERROR_MESSAGES: Record<string, string> = {
  NETWORK_ERROR: 'Unable to connect. Please check your internet connection.',
  UNAUTHORIZED: 'Your session has expired. Please log in again.',
  NOT_FOUND: 'We couldn\'t find what you were looking for.',
  SERVER_ERROR: 'Something went wrong on our end. Please try again later.',
  VALIDATION_ERROR: 'Some information is incorrect. Please review and try again.',
  DEFAULT: 'An unexpected error occurred. Please try again.'
}

export function getErrorMessage(error: unknown): string {
  if (error instanceof Error) {
    // Map HTTP status codes to friendly messages
    if ('statusCode' in error) {
      const code = (error as any).statusCode
      if (code === 401 || code === 403) return ERROR_MESSAGES.UNAUTHORIZED
      if (code === 404) return ERROR_MESSAGES.NOT_FOUND
      if (code >= 500) return ERROR_MESSAGES.SERVER_ERROR
    }
    // Map specific network errors
    if (error.message.includes('NetworkError') || error.message.includes('Failed to fetch')) {
      return ERROR_MESSAGES.NETWORK_ERROR
    }
  }
  return ERROR_MESSAGES.DEFAULT
}
```

### Using it in a Composable
```typescript
// composables/useUserProfile.ts
import { getErrorMessage } from '~/utils/errorMessages'

export function useUserProfile(userId: string) {
  const { data, error } = useFetch(`/api/users/${userId}`)
  
  const friendlyError = computed(() => 
    error.value ? getErrorMessage(error.value) : null
  )

  return { data, friendlyError }
}
```

```vue
<template>
  <div v-if="friendlyError" class="error-alert" role="alert">
    {{ friendlyError }}
  </div>
</template>
```

## 3. Form Validation Error Messages

Form errors need special treatment. They must be:
- **Field-specific:** Placed adjacent to the field that caused the error, not at the top of the page.
- **Actionable:** Tell users *what* to fix, not just *that* something is wrong.

```
❌ Bad: "Invalid input."
✅ Good: "Password must be at least 8 characters and include one number."

❌ Bad: "Email error."
✅ Good: "Please enter a valid email address (e.g., name@example.com)."
```

## 4. Error Toasts vs. Inline Errors vs. Full-Page Errors

Choose the right error presentation pattern based on the severity of the failure:

| Severity | Pattern | When to Use |
|---|---|---|
| Low (non-blocking) | Toast / Snackbar | Background sync failed, optional feature unavailable |
| Medium (action failed) | Inline error near the action | Form submission failed, button action failed |
| High (section failed) | Section-level error state with retry | A card/widget couldn't load its data |
| Critical (page failed) | Full-page error (`error.vue`) | Authentication required, fatal server error |

## 5. Accessibility of Error Messages

Error messages must be accessible to screen reader users:
- Use `role="alert"` on dynamically appearing error containers so screen readers announce them immediately.
- Use `aria-describedby` on form inputs to link them to their field-level error message.
- Ensure sufficient color contrast (never rely on red color alone to convey an error state; pair with an icon or text label).
