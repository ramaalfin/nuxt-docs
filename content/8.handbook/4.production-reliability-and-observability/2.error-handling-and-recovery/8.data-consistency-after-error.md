# Data Consistency After Error

One of the most dangerous failure modes in a web application is when an error leaves the application in an **inconsistent state**—where the UI shows data that no longer matches what is stored in the backend database. This can lead to phantom UI elements, duplicate submissions, or incorrect financial calculations.

Maintaining data consistency after an error is a critical aspect of production reliability.

## 1. The Consistency Problem

Consider this sequence of events:
1. User adds an item to their cart (local state: `qty = 2`).
2. A `PATCH /api/cart` request is sent to the server.
3. The server processes the update (database: `qty = 2`).
4. The server response is lost due to a network error.
5. The client never receives a `200 OK`, interprets this as a failure.
6. Rollback triggers: local state reverts to `qty = 1`.
7. **Result:** UI shows 1 item, database has 2 items. ⚠️ **Inconsistent.**

## 2. Strategies for Maintaining Consistency

### Strategy 1: Re-fetch After Mutation (Source of Truth Refresh)
After any mutation (successful or failed), re-fetch the authoritative state from the server. This is the simplest and most reliable strategy.

```typescript
// stores/cart.ts
const updateCartItem = async (itemId: string, quantity: number) => {
  try {
    await $fetch(`/api/cart/items/${itemId}`, {
      method: 'PATCH',
      body: { quantity }
    })
  } catch (error) {
    // Log the error, notify the user
    showNotification({ type: 'error', message: 'Failed to update cart. Refreshing...' })
  } finally {
    // ALWAYS re-fetch the cart after a mutation attempt, regardless of success or failure
    // This guarantees the local state matches the server state
    await fetchCart()
  }
}
```

The trade-off is an extra network request, but it eliminates the risk of inconsistency.

### Strategy 2: Idempotency Keys for Write Operations

For critical mutations (payments, order submissions), use **idempotency keys**. An idempotency key is a unique identifier sent with the request. If the same key is sent multiple times, the server processes it only once and returns the same response.

This solves the "did my request go through?" problem that causes double submissions.

```typescript
// composables/useIdempotentFetch.ts
import { v4 as uuidv4 } from 'uuid'

export function useIdempotentFetch() {
  const submit = async (url: string, body: object) => {
    const idempotencyKey = uuidv4()
    
    return $fetch(url, {
      method: 'POST',
      body,
      headers: {
        'Idempotency-Key': idempotencyKey
      }
    })
  }
  
  return { submit }
}
```

The backend must be designed to honor these keys (checking if a request with the same key was already processed and returning the cached result).

### Strategy 3: Optimistic Update with Server Reconciliation

When using optimistic updates, always reconcile the local state with the server's response on success.

```typescript
const likePost = async (postId: string) => {
  // 1. Optimistic update
  const post = posts.value.find(p => p.id === postId)
  if (!post) return
  post.likeCount++
  post.isLiked = true

  try {
    // 2. Server mutation
    const serverPost = await $fetch<Post>(`/api/posts/${postId}/like`, { method: 'POST' })
    
    // 3. RECONCILE: Replace our optimistic guess with the server's truth
    // The server may have a different like count (e.g., concurrent likes from other users)
    Object.assign(post, serverPost)
  } catch {
    // 4. Rollback on failure
    post.likeCount--
    post.isLiked = false
  }
}
```

## 3. Detecting and Resolving Stale State

If a user has had the app open for a long time (e.g., left a tab open overnight), the local state may be severely stale. When the user returns to the tab (`visibilitychange` event), trigger a fresh data load.

```typescript
// composables/useStaleDataRefresh.ts
export function useStaleDataRefresh(refreshFn: () => Promise<void>, staleDurationMs = 5 * 60 * 1000) {
  const lastFetchedAt = ref(Date.now())

  useEventListener(document, 'visibilitychange', async () => {
    if (document.visibilityState !== 'visible') return
    
    const ageMs = Date.now() - lastFetchedAt.value
    if (ageMs >= staleDurationMs) {
      await refreshFn()
      lastFetchedAt.value = Date.now()
    }
  })
}
```

## 4. Consistency Across Multiple Browser Tabs

If a user modifies data in Tab A, Tab B still shows the old data. Use the **BroadcastChannel API** to notify all tabs of state changes.

```typescript
// composables/useCrossTabSync.ts
export function useCrossTabSync(channelName: string) {
  const channel = new BroadcastChannel(channelName)
  
  const broadcast = (event: string, payload?: any) => {
    channel.postMessage({ event, payload })
  }

  const onMessage = (handler: (data: { event: string; payload: any }) => void) => {
    channel.onmessage = ({ data }) => handler(data)
  }

  onUnmounted(() => channel.close())
  
  return { broadcast, onMessage }
}
```

```typescript
// stores/cart.ts - Broadcast cart updates to all tabs
const { broadcast, onMessage } = useCrossTabSync('cart-sync')

// After successful cart mutation:
broadcast('cart:updated', newCartState)

// Listen for changes from other tabs:
onMessage(({ event, payload }) => {
  if (event === 'cart:updated') {
    items.value = payload.items
  }
})
```
