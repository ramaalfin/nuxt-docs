# Fallback UI Patterns

A fallback UI is the interface shown to users when the primary content cannot be displayed—either because data is loading, an error occurred, or a feature is unavailable. Designing intentional, polished fallback states is what separates a production-quality application from a prototype.

## 1. The Three States of Every Data-Driven Component

Every component that fetches or renders dynamic data must handle three distinct states:

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Loading    │ →  │   Success    │ →  │    Error     │
│  (Skeleton)  │    │  (Content)   │    │  (Fallback)  │
└──────────────┘    └──────────────┘    └──────────────┘
```

Never ship a component that only handles the "Success" state.

## 2. Skeleton Screens (Preferred Loading Fallback)

Skeleton screens are the modern standard for loading states. They are grey, animated placeholder shapes that mimic the layout of the actual content. They are superior to spinners because they:
- Reduce perceived loading time.
- Prevent layout shift when content arrives.
- Set accurate visual expectations for the user.

```vue
<!-- components/ProductCard.vue -->
<script setup lang="ts">
defineProps<{ loading?: boolean; product?: Product }>()
</script>

<template>
  <!-- LOADING STATE: Skeleton -->
  <div v-if="loading" class="product-card skeleton" aria-hidden="true">
    <div class="skeleton-image" />
    <div class="skeleton-line skeleton-line--title" />
    <div class="skeleton-line skeleton-line--price" />
  </div>

  <!-- SUCCESS STATE: Real content -->
  <div v-else-if="product" class="product-card">
    <img :src="product.imageUrl" :alt="product.name" />
    <h3>{{ product.name }}</h3>
    <p>{{ product.price }}</p>
  </div>
</template>

<style scoped>
.skeleton { animation: pulse 1.5s ease-in-out infinite; }
.skeleton-image { height: 200px; background: var(--color-skeleton); border-radius: 8px; }
.skeleton-line { height: 16px; background: var(--color-skeleton); border-radius: 4px; margin-top: 12px; }
.skeleton-line--title { width: 70%; }
.skeleton-line--price { width: 30%; }

@keyframes pulse {
  0%, 100% { opacity: 1; }
  50% { opacity: 0.4; }
}
</style>
```

## 3. Empty State UI

An empty state is shown when a query returns zero results—distinct from a loading or error state. It must be:
- **Helpful:** Explain why it's empty.
- **Actionable:** Provide a next step to fill it.

```vue
<!-- components/OrderHistory.vue -->
<template>
  <div v-if="orders.length === 0" class="empty-state">
    <ShoppingBagIcon class="empty-state__icon" />
    <h3>No orders yet</h3>
    <p>When you place your first order, it will appear here.</p>
    <NuxtLink to="/shop" class="btn">Start Shopping</NuxtLink>
  </div>
  <OrderList v-else :orders="orders" />
</template>
```

## 4. Error Fallback State

The error state must be clearly distinct from the loading and empty states.

```vue
<template>
  <div v-if="error" class="error-state" role="alert">
    <ExclamationIcon class="error-state__icon" />
    <p>{{ friendlyError }}</p>
    <button @click="refresh()" class="btn btn--secondary">Try Again</button>
  </div>
</template>
```

## 5. Vue's `<Suspense>` for Async Components

Vue 3's `<Suspense>` component provides a declarative way to manage loading fallbacks for async component trees.

```vue
<!-- pages/dashboard.vue -->
<template>
  <Suspense>
    <!-- Renders when all async setup() calls in the tree resolve -->
    <template #default>
      <DashboardWidgets />
    </template>

    <!-- Renders while the default slot is loading -->
    <template #fallback>
      <DashboardSkeleton />
    </template>
  </Suspense>
</template>
```

## 6. Nuxt's `<NuxtLoadingIndicator>`

For page-level navigation, Nuxt provides a built-in loading bar that activates during route transitions.

```vue
<!-- app.vue -->
<template>
  <div>
    <NuxtLoadingIndicator color="#6366f1" :height="3" />
    <NuxtLayout>
      <NuxtPage />
    </NuxtLayout>
  </div>
</template>
```

## Summary: Fallback Checklist

Before shipping any data-driven component, verify:
- [ ] **Loading state** is handled (preferably with a skeleton screen).
- [ ] **Empty state** is handled with a helpful, actionable message.
- [ ] **Error state** is handled with a user-friendly message and retry option.
- [ ] **Partial failure** is handled (e.g., if one of three API calls fails, the other two still render).
