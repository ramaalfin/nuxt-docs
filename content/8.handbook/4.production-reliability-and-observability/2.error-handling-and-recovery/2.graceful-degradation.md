# Graceful Degradation

Graceful degradation is the design principle that a system should continue to function—albeit at a reduced capacity—when a part of it fails. Rather than completely crashing or showing a blank screen, the application should degrade to a simpler but still usable state.

In a Nuxt application, failures can originate from network errors, broken third-party integrations, failed data fetches, or JavaScript runtime errors. The goal is to contain each failure and provide the best possible experience to the user despite it.

## 1. The Spectrum of Degradation

Think of degradation in levels, from least to most severe:

```text
Level 1: Feature Degraded    → Widget shows cached/stale data
Level 2: Feature Unavailable → Widget is hidden, rest of page works
Level 3: Page Degraded       → Page loads with placeholder content
Level 4: Full Fallback       → Static error.vue page is shown
```

The goal is always to stay at the **lowest level of degradation possible**.

## 2. Graceful Degradation Patterns in Nuxt

### Pattern 1: Feature Toggle on Failure
Hide non-critical UI elements when their data fails to load.

```vue
<script setup lang="ts">
const { data: recommendations, error } = await useFetch('/api/recommendations')
</script>

<template>
  <main>
    <HeroSection />
    <ProductList />

    <!-- Degrade gracefully: If recommendations fail, simply don't show them -->
    <!-- The core page (Hero + ProductList) still works perfectly -->
    <RecommendationsWidget v-if="!error && recommendations" :data="recommendations" />
  </main>
</template>
```

### Pattern 2: Stale Cache as Fallback
When a network request fails, serve the last successfully cached response instead of showing an error.

```typescript
// composables/useProductsWithFallback.ts
export function useProductsWithFallback() {
  const cachedProducts = useLocalStorage<Product[]>('products_cache', [])
  const isFromCache = ref(false)

  const { data, error } = useFetch('/api/products', {
    onResponse({ response }) {
      if (response.ok) {
        // Update the cache on a successful response
        cachedProducts.value = response._data
        isFromCache.value = false
      }
    }
  })

  const products = computed(() => {
    if (error.value && cachedProducts.value.length > 0) {
      isFromCache.value = true
      return cachedProducts.value // Degrade to stale cache
    }
    return data.value ?? []
  })

  return { products, isFromCache }
}
```

```vue
<template>
  <div>
    <!-- Warn users they are seeing stale data, but don't block them -->
    <Banner v-if="isFromCache" type="warning" message="Showing offline data. Some info may be outdated." />
    <ProductGrid :products="products" />
  </div>
</template>
```

### Pattern 3: Third-Party Script Degradation
If a third-party script (analytics, chat widget, maps) fails to load, the core application must continue working.

```typescript
// plugins/analytics.client.ts
export default defineNuxtPlugin(() => {
  try {
    // Attempt to initialize third-party analytics
    window.analytics.initialize({ key: 'YOUR_KEY' })
  } catch (error) {
    // Log silently — never let a marketing tool crash the app
    console.warn('[Analytics] Failed to initialize:', error)
    
    // Provide a no-op shim so the rest of the app can still call analytics methods
    window.analytics = { track: () => {}, identify: () => {} }
  }
})
```

## 3. Progressive Enhancement vs Graceful Degradation

These are complementary, not opposing concepts:
- **Progressive Enhancement:** Start with a baseline that works for everyone, then layer advanced features on top (e.g., a form that works without JavaScript, enhanced with `v-model` when JS loads).
- **Graceful Degradation:** Build the full-featured experience first, then ensure it degrades cleanly when parts fail (e.g., an interactive map that falls back to a static image if the Maps API fails).

In Nuxt 3's SSR context, **Progressive Enhancement** is particularly powerful. Your server renders a fully functional HTML page. JavaScript hydrates it client-side for enhanced interactivity. If JavaScript fails to hydrate, the user still gets a working HTML page.

## 4. Designing for Degradation from the Start

Graceful degradation cannot be bolted on as an afterthought. It must be designed into every feature:
- **Identify non-critical vs. critical data:** Which API responses are required for the page to function? Which are "nice to have"?
- **Use `v-if` / `v-show` defensively:** Every dynamic component should have a fallback when its data is absent.
- **Test degraded states explicitly:** Write integration tests that mock API failures and assert that the degraded UI (not a blank page) is shown.
