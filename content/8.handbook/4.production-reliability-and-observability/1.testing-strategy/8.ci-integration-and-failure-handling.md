# CI Integration & Failure Handling

Automated tests only deliver value when they run consistently and automatically. The Continuous Integration (CI) pipeline is the backbone of your testing strategy—it acts as the gatekeeper that ensures no broken code reaches the main branch or production.

## 1. CI Pipeline Architecture

A well-structured CI pipeline for a Nuxt application has multiple, ordered stages. Faster checks run first to give rapid feedback. Slower checks (like E2E) run after earlier checks pass.

### Recommended Pipeline Stages

```text
┌─────────────────────────────────────────────────────────┐
│ Stage 1: Static Analysis (~30s)                         │
│   ├── ESLint / vue-tsc type checking                    │
│   └── Dependency audit (npm audit)                      │
├─────────────────────────────────────────────────────────┤
│ Stage 2: Unit & Integration Tests (~2min)               │
│   ├── vitest run --coverage                             │
│   └── Coverage threshold enforcement                    │
├─────────────────────────────────────────────────────────┤
│ Stage 3: Build (~3min)                                  │
│   └── nuxt build (validates SSR compilation)            │
├─────────────────────────────────────────────────────────┤
│ Stage 4: E2E Tests (~10min) [runs against built app]    │
│   ├── npx playwright test                               │
│   └── Upload artifacts (traces, screenshots)            │
└─────────────────────────────────────────────────────────┘
```

*Each stage gates the next. A lint failure saves 10+ minutes of waiting for E2E results.*

## 2. GitHub Actions Configuration

```yaml
# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [main, develop]

jobs:
  lint:
    name: Lint & Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'pnpm' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck

  unit-tests:
    name: Unit & Integration Tests
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'pnpm' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm test:coverage
      - uses: codecov/codecov-action@v4  # Upload coverage to Codecov

  e2e-tests:
    name: E2E Tests (Playwright)
    needs: unit-tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'pnpm' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec playwright install --with-deps chromium
      - run: pnpm build
      - run: pnpm test:e2e
      
      # Always upload traces on failure for debugging
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 7
```

## 3. Failure Handling Policies

### Policy 1: Fail Fast, Fix Forward
A CI failure on a PR blocks merging. This is non-negotiable. Do not merge "with a note to fix later." The main branch must always be in a deployable state.

### Policy 2: The "Broken Window" Rule  
Never leave a failing test in the suite for more than 24 hours. If a legitimate test is failing due to a real bug, that bug must either be fixed immediately or the failing code must be reverted. Do not disable or skip the test.

```typescript
// DO NOT DO THIS without an immediate plan to fix:
it.skip('checks out correctly', () => { ... }) // Creates a "broken window"
```

### Policy 3: Differentiate Failure Types in Slack/Email Notifications
Configure CI notifications to distinguish between:
- **Real Code Failures:** A test caught a real bug → High priority alert.
- **Suspected Flakiness:** A test failed and then passed on retry → Medium priority, log for the flaky test tracker.
- **Infrastructure Failure:** The CI runner ran out of memory or disk space → Notify DevOps.

## 4. PR Coverage Comments

Integrate Codecov or a custom GitHub Actions script to post a coverage summary comment directly on every Pull Request. This makes coverage changes transparent and reviewable.

```
## Coverage Report 📊

| Coverage Type | Before | After  | Delta  |
|---------------|--------|--------|--------|
| Lines         | 82.3%  | 84.1%  | +1.8%  |
| Branches      | 78.1%  | 79.5%  | +1.4%  |
| Functions     | 85.0%  | 85.0%  | 0.0%   |
```

This creates a culture where every PR author is aware of the testing impact of their changes.
