# Flaky Test Elimination

A **flaky test** is a test that produces inconsistent results—passing sometimes and failing other times—without any change to the application code. Flaky tests are one of the most toxic elements of a CI/CD pipeline. They erode trust in the entire test suite, causing developers to ignore test failures and ultimately weakening your confidence in deployments.

Treating flaky tests as a critical bug is a cultural and technical imperative.

## 1. Root Causes of Flaky Tests

### Time & Date Dependencies
Tests that rely on `Date.now()` or `new Date()` will produce different results every time they run.

**Fix:** Always freeze time using `vi.useFakeTimers()` and `vi.setSystemTime()` in Vitest, or Playwright's clock API.

```typescript
// vitest
beforeEach(() => {
  vi.useFakeTimers()
  vi.setSystemTime(new Date('2024-01-15'))
})

afterEach(() => {
  vi.useRealTimers()
})
```

### Race Conditions (Async Operations Not Awaited)
The most common source of E2E flakiness. A test clicks a button, then immediately asserts on the next page before the navigation completes.

**Fix:** Always use Playwright's web-first assertions and `await` on every action. Never use `page.waitForTimeout()`.

```typescript
// BAD: Will flake depending on network speed
await page.click('#submit-btn')
await page.waitForTimeout(1000) // Hope the page loads in 1 second!
expect(await page.url()).toContain('/success')

// GOOD: Playwright waits until the URL contains '/success' or throws on timeout
await page.getByRole('button', { name: 'Submit' }).click()
await expect(page).toHaveURL(/\/success/)
```

### Test Order Dependencies
A test that passes only when run *after* a specific other test has already mutated global state is fundamentally broken.

**Fix:** Enforce isolation. Never share state between test files. In Vitest, use `--isolate` (on by default). In Playwright, each test file gets its own browser context.

### Non-Deterministic Data
Using `Math.random()`, `faker.internet.email()`, or `Date.now()` as test inputs or user identifiers can cause assertion failures when the random data happens to trigger unexpected behavior.

**Fix:** Use deterministic data via Factories/Fixtures with seeded random values.

### External Service Flakiness
E2E tests that make real calls to third-party APIs (Stripe, Sendgrid) will fail whenever those external services experience downtime.

**Fix:** Use MSW at the E2E level to mock critical third-party integrations, or use dedicated sandbox environments with stable test keys.

## 2. Detecting and Tracking Flaky Tests

### Playwright Retries
Configure Playwright to retry tests a fixed number of times. A test that passes on retry is almost certainly flaky.

```typescript
// playwright.config.ts
export default defineConfig({
  retries: process.env.CI ? 2 : 0, // Retry in CI only, so local runs expose the issue
})
```

### Flakiness Quarantine
When a flaky test is identified, do **not** just delete it or comment it out. Move it into a `quarantine` test suite that runs with retries enabled but does not block the main CI pipeline.

```
e2e/
├── journeys/           # Stable, must-pass tests
│   └── checkout.spec.ts
└── quarantine/         # Known flaky tests being actively fixed
    └── payment-flow.spec.ts
```

Log all quarantined tests in a dedicated tracking issue in your project management tool. Quarantined tests must be fixed or removed within a sprint.

## 3. The "Fix Flaky Tests First" Rule

Adopt this team rule: **Any developer who encounters a flaky test in CI is responsible for opening a bug ticket for it, regardless of whether it was their code that triggered it.** The test is treated with the same severity as a production bug.

When the pipeline goes red, the first question must be "Is this a real failure?" not "Is it just flaky again?".

## 4. Playwright Tracing for Debugging

When a Playwright test fails, it's often hard to know why. Enable tracing to get a full, visual replay of the test.

```typescript
// playwright.config.ts
export default defineConfig({
  use: {
    trace: 'on-first-retry', // Record trace only when a test fails and retries
    screenshot: 'only-on-failure',
    video: 'on-first-retry'
  }
})
```

Upload traces to CI artifacts so developers can open them in the Playwright Trace Viewer to see exactly what the DOM state was when the assertion failed.
