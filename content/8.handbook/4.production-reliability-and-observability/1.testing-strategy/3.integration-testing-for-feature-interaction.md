# Integration Testing for Feature Interaction

In modern frontend architecture, the lines between isolated Unit Tests and End-to-End Tests are bridged by **Integration Tests**. While a unit test verifies that a single composable or component works in isolation, an integration test verifies that *multiple units work together properly* to deliver a feature.

Integration tests are the "sweet spot" of testing: they offer high confidence (like E2E tests) but run much faster (like Unit tests).

## What Constitutes an Integration Test?

An integration test usually evaluates a "Feature Module" or a complex "Page". It tests the interaction between:
1.  **UI Components:** A parent rendering children and passing props/slots.
2.  **State Management:** The components interacting with Pinia stores or Vue `ref`s.
3.  **Data Fetching:** The feature initiating a network request and handling the response.
4.  **Routing:** (Optional) Clicking a link to push a new route to the Vue Router.

## 1. Setting the Boundary

The most important aspect of integration testing is defining the boundary. What is real, and what is mocked?

*   **Real:** The Vue Component tree, Pinia state, `userEvent` interactions.
*   **Mocked:** The actual network (using MSW), Browser APIs (like `IntersectionObserver`), and sometimes the Router.

## 2. Example: Testing a Checkout Feature

Let's test a `CheckoutFlow.vue` component that interacts with a `CartStore` and an external API.

```typescript
import { render, screen, waitFor } from '@testing-library/vue'
import userEvent from '@testing-library/user-event'
import { createTestingPinia } from '@pinia/testing'
import CheckoutFlow from './CheckoutFlow.vue'

// Note: MSW is assumed to be running and intercepting network calls.

it('allows the user to complete checkout and shows success state', async () => {
  // 1. Arrange: Mount the component with a real (but isolated) Pinia store
  render(CheckoutFlow, {
    global: {
      plugins: [
        createTestingPinia({ 
          stubActions: false, // We want the REAL actions to execute!
          initialState: {
            cart: { items: [{ id: 1, name: 'Shoes', price: 100 }] }
          }
        })
      ]
    }
  })

  // 2. Act: User fills out the shipping form
  const user = userEvent.setup()
  await user.type(screen.getByLabelText(/address/i), '123 Vue Street')
  await user.click(screen.getByRole('button', { name: /confirm purchase/i }))

  // 3. Assert: Loading state appears
  expect(screen.getByText(/processing/i)).toBeInTheDocument()

  // 4. Assert: Success message appears after API (mocked via MSW) resolves
  await waitFor(() => {
    expect(screen.getByRole('heading', { name: /order successful/i })).toBeInTheDocument()
  })
})
```

## 3. Why Not Just E2E Test This?

You *could* write this as a Playwright test. However:
1.  This integration test takes ~200ms to run in Vitest. A Playwright test takes ~3 seconds.
2.  If the checkout API goes down in the testing environment, the E2E test fails. The integration test (using MSW) still passes, proving the *frontend logic* is correct.

## 4. Handling Nuxt Context (`useNuxtApp`, `useAsyncData`)

When writing integration tests in Nuxt 3, you often hit errors because Nuxt-specific composables (`useAsyncData`, `useNuxtApp`) require the Nuxt context to be present.

To resolve this, use `@nuxt/test-utils` for integration tests that rely heavily on Nuxt features.

```typescript
import { mountSuspended } from '@nuxt/test-utils/runtime'
import MyNuxtPage from './pages/index.vue'

it('renders the page data fetched via useAsyncData', async () => {
  // mountSuspended wraps the component in the Nuxt context and resolves Suspense boundaries
  const wrapper = await mountSuspended(MyNuxtPage)
  
  expect(wrapper.text()).toContain('Data from API')
})
```

## Summary
- Integration tests verify that your components, stores, and composables talk to each other correctly.
- Use **Testing Library** to drive the UI interactions.
- Let Pinia actions run for real, but mock the network layer with **MSW**.
- Use `@nuxt/test-utils` (`mountSuspended`) when testing features deeply tied to Nuxt 3's hydration and data fetching lifecycle.
