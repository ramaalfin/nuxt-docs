# Test Performance Optimization

A slow test suite is a test suite that developers will avoid running locally, skip in their workflow, and resent. As a project grows, a naively-structured suite that takes 20+ minutes in CI directly impacts developer velocity and feedback cycles.

Optimizing test performance is an ongoing investment that pays dividends daily.

## 1. Vitest Performance Optimization

### Run Tests in Parallel
Vitest runs test *files* in parallel by default across multiple worker threads (powered by Vite). However, there are configuration options to tune this.

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    // Use all available CPU cores (default)
    pool: 'threads',
    poolOptions: {
      threads: {
        // Set a minimum number of workers to prevent cold starts
        minThreads: 2,
        // Cap maximum workers if memory is a constraint
        maxThreads: 8,
      }
    }
  }
})
```

### Only Test Changed Files (Watch Mode)
During active development, use `vitest --watch`. It uses Vite's HMR graph to identify which test files are affected by a specific code change and **only reruns those tests**.

```bash
pnpm vitest --watch
```

### `vitest --changed` in CI
In CI, on feature branches, you can run only the tests affected by changed files since the last commit or merge base.

```bash
# Only run tests related to files changed vs. main branch
pnpm vitest run --changed origin/main
```
*Note: Use this with care. Always run the full suite on the `main` branch itself.*

### Avoid Heavy Setup in `beforeAll`
If every test file runs a `beforeAll` that spawns a server or connects to a database, parallelism is wasted. Prefer lightweight mocking strategies (MSW, Pinia Testing) that have near-zero setup time.

## 2. Playwright E2E Performance Optimization

### Parallelize E2E Tests via Sharding
Playwright supports test **sharding**—splitting the entire E2E suite across multiple CI workers that run simultaneously.

```yaml
# In GitHub Actions (runs 4 shards concurrently)
strategy:
  matrix:
    shardIndex: [1, 2, 3, 4]
    shardTotal: [4]

steps:
  - run: npx playwright test --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
```

This can reduce a 20-minute E2E suite to ~5 minutes.

### Reuse Browser Contexts for Auth
As discussed in the E2E chapter, saving and reusing `storageState` (authentication cookies) prevents expensive login UI flows from running for every test.

### Limit Trace and Video Recording
Recording traces and videos for every test is expensive. Limit them to only failures.
```typescript
use: {
  trace: 'on-first-retry',   // Not 'on'
  video: 'on-first-retry',   // Not 'on'
  screenshot: 'only-on-failure'
}
```

## 3. CI Caching

The single biggest win for CI performance is caching `node_modules` and Playwright browsers between pipeline runs.

```yaml
# GitHub Actions
- name: Cache node_modules
  uses: actions/cache@v4
  with:
    path: ~/.pnpm-store
    key: ${{ runner.os }}-pnpm-${{ hashFiles('**/pnpm-lock.yaml') }}
    restore-keys: |
      ${{ runner.os }}-pnpm-

- name: Cache Playwright Browsers
  uses: actions/cache@v4
  with:
    path: ~/.cache/ms-playwright
    key: ${{ runner.os }}-playwright-${{ hashFiles('**/pnpm-lock.yaml') }}
```
A full Playwright browser installation can take 60-90 seconds. Caching reduces this to near-zero on cache hit.

## 4. Monitoring Suite Performance Over Time

Track your test suite's total execution time as a metric over time. A sudden spike in test duration (e.g., someone added a 5-second `setTimeout` inside a composable and didn't mock timers) should trigger an investigation.

Use CI pipeline analytics (GitHub Actions Insights, Datadog CI Visibility) to identify:
- The **slowest test files** (prime candidates for optimization).
- The **most frequently failing tests** (prime candidates for flakiness investigation).
- Long-term trends in total pipeline duration.

## Performance Budget: Recommended Targets

| Test Suite | Local Dev | CI Pipeline |
|---|---|---|
| Vitest (full) | < 60 seconds | < 3 minutes |
| Vitest (watch, affected) | < 3 seconds | N/A |
| Playwright E2E (with sharding) | < 5 minutes | < 8 minutes |
| Full CI Pipeline (all stages) | N/A | < 15 minutes |

Treat a breach of these budgets as a performance regression that must be investigated and resolved.
