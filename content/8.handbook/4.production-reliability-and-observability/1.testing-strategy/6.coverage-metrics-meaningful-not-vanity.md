# Coverage Metrics: Meaningful, Not Vanity

Code coverage is one of the most misunderstood metrics in software engineering. A 90% coverage score does not mean your application is 90% bug-free. Chasing high coverage numbers for their own sake—a phenomenon known as **"Vanity Coverage"**—actively harms your codebase by incentivizing bad tests.

## The Danger of Vanity Coverage

Imagine a developer who writes this test to push coverage from 85% to 90%:

```typescript
it('exists', () => {
  const wrapper = mount(CheckoutButton)
  expect(wrapper.exists()).toBe(true) // This "covers" every line of the component
})
```

This test provides **zero confidence**. It doesn't verify any behavior. It is purely a line-coverage artifact. Yet it counts towards your 90% goal.

## What Coverage Actually Measures

Coverage tools (like Vitest's built-in `v8` coverage or `istanbul`) measure:

| Metric | What it tracks |
|--------|---------------|
| **Line** | Whether a specific line of code was executed. |
| **Statement** | Whether a specific expression was executed (more granular than lines). |
| **Branch** | Whether **both sides** of a conditional (`if/else`, ternary, `&&`, `??`) were executed. |
| **Function** | Whether a specific function was called at all. |

## Prioritize Branch Coverage

**Branch coverage is the most meaningful metric.** It reveals whether your tests cover the different *paths* through your code, not just whether they touch specific lines.

**Example:**
```typescript
function getDiscount(isMember: boolean, totalPrice: number) {
  if (isMember && totalPrice > 100) { // 4 possible branches
    return totalPrice * 0.2
  }
  return 0
}
```

A test that only calls `getDiscount(true, 200)` achieves 100% **line** coverage but only 50% **branch** coverage. A meaningful test suite also tests `getDiscount(false, 200)`, `getDiscount(true, 50)`, and `getDiscount(false, 50)`.

## Recommended Coverage Targets (Risk-Based)

Rather than a single global coverage target, we use a **risk-based** approach:

| Code Area | Line/Statement Target | Branch Target | Rationale |
|-----------|----------------------|---------------|-----------|
| Business Logic / Utils (`utils/`, store actions) | **90%+** | **85%+** | Core revenue-impacting logic. Must be thoroughly tested. |
| Vue Components (`components/`) | **70%+** | **65%+** | Focus on behavior over rendering. |
| Nuxt Pages (`pages/`) | **60%+** | **55%+** | Integration tests handle most coverage here. |
| Config / Boilerplate | **Excluded** | **Excluded** | Zero value to test. |

## Configuring Meaningful Coverage in Vitest

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html', 'lcov'], // 'html' for local review, 'lcov' for Codecov
      
      // Enforce thresholds - CI will FAIL if these are not met
      thresholds: {
        lines: 80,
        branches: 75, // Prioritize branches!
        functions: 80,
        statements: 80,
      },
      
      // Exclude files with zero testing value
      exclude: [
        'nuxt.config.ts',
        'tailwind.config.ts',
        '**/*.d.ts',
        '**/types/**',
        '**/constants/**',
        '**/mocks/**',
        '**/fixtures/**',
      ]
    }
  }
})
```

## How to Use Coverage Reports Effectively

1.  **Open the HTML Report (`npx vitest --coverage --ui`)**: Navigate through the report to find red (uncovered) branches, not just red lines.
2.  **Track Coverage Trends, Not Absolute Numbers**: Use Codecov, SonarQube, or GitHub Actions to alert when coverage *drops* on a PR, not just when it falls below a threshold.
3.  **The "Coverage Delta" Rule**: Any PR that introduces new untested business logic which reduces branch coverage by more than 2% must include new tests before it can be merged.

## A Meaningful Coverage Checklist
- [ ] Are all `if/else` paths in Pinia store actions tested?
- [ ] Are error paths (`.catch()`, `try/catch`) tested?
- [ ] Are null/undefined guard clauses tested?
- [ ] Are all major props/slots of a reusable component tested?

If the answer to all of the above is "yes," you have meaningful coverage, regardless of whether the number is 75% or 95%.
