# Comprehensive Test Pyramid & Strategy

Our testing strategy balances confidence, execution speed, and maintenance cost. By adhering to a strict Test Pyramid, we ensure that bugs are caught as early in the development lifecycle as possible, without bogging down CI pipelines with slow, brittle tests.

## The Pyramid Distribution

Our Nuxt application relies on a 70/20/10 distribution model:

### 1. Unit Tests (70%) - Vitest
- **Scope:** Pure functions, isolated composables, Pinia getters/actions, Vue components lacking external dependencies.
- **Speed:** < 10ms per test.
- **Goal:** Verify that a specific piece of business logic behaves as expected given a set of inputs.
- **Failure implication:** "My code is logically broken."

### 2. Integration Tests (20%) - Vitest + Testing Library + MSW
- **Scope:** Feature components (Parent/Child interactions), Component + Pinia interaction, UI rendering of Mocked Network responses.
- **Speed:** ~100ms - 500ms per test.
- **Goal:** Verify that the "wiring" between isolated units is connected properly and the UI reacts correctly to user input and state changes.
- **Failure implication:** "My components aren't communicating properly, or my API contract expectation is wrong."

### 3. E2E Tests (10%) - Playwright
- **Scope:** Complete end-to-end user flows navigating through multiple pages against a real, running backend.
- **Speed:** 5 - 30 seconds per test.
- **Goal:** Verify Critical User Journeys (CUJs). Verify the Nuxt SSR hydration pipeline and actual database/API integrity.
- **Failure implication:** "The application is fundamentally broken for the user."

## Shift-Left Testing Philosophy

"Shifting left" means moving the discovery of bugs as early in the pipeline as possible.

1.  **TypeScript & ESLint:** The very first layer of testing. Catches structural and syntax errors before code is even run.
2.  **Local Unit Tests (Vitest):** Run locally on the developer's machine using `vitest --watch` during active development.
3.  **Pre-Commit Hooks:** Husky + lint-staged run ESLint and affected Unit tests before allowing a `git commit`.
4.  **CI Pipeline:** Runs the full Vitest suite and Playwright E2E suite on every Pull Request.

## Anti-Patterns to Avoid

- **The Ice Cream Cone:** Having 10 Unit tests and 500 E2E tests. This results in hour-long CI pipelines and massive flakiness. If you find yourself writing a Playwright test to check if a specific CSS class is applied on a hover state, you are building an Ice Cream Cone. Stop, and write a Component test instead.
- **The Hourglass:** Having lots of Unit tests and E2E tests, but no Integration tests. This leads to high confidence in individual functions, but massive failures when they are wired together in the UI.
- **Testing the Framework:** Do not write tests to verify that Vue's reactivity system works. Assume Nuxt and Vue work. Test *your* code.
