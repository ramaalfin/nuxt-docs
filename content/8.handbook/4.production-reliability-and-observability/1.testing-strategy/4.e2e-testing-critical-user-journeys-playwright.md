# E2E Testing Critical User Journeys (Playwright)

End-to-End (E2E) testing is the top of the test pyramid. It validates the application exactly as a real user would experience it—spinning up an actual browser, navigating the DOM, interacting with real backends, and ensuring the entire system functions as a cohesive unit.

For Nuxt 3 applications, **Playwright** is the industry standard due to its speed, auto-waiting mechanisms, and seamless cross-browser support.

## 1. Focus on Critical User Journeys (CUJs)

Because E2E tests are slow, expensive to run in CI, and prone to flakiness, **do not use them to test edge cases**. Leave form validation edge cases to Unit/Integration tests.

E2E tests should exclusively cover **Critical User Journeys (CUJs)**. These are the flows that, if broken, directly impact revenue or core business function.

**Examples of CUJs:**
- "A user can sign up, confirm email, and log in."
- "A user can add an item to the cart, enter payment details, and complete a checkout."
- "An admin can access the dashboard and delete a post."

## 2. Playwright Best Practices

### Auto-Waiting and Assertions
Playwright automatically waits for elements to be actionable (visible, enabled) before clicking or typing. Do not use manual `page.waitForTimeout(3000)`. Use web-first assertions.

```typescript
// BAD: Prone to flakiness depending on network speed
await page.click('button#submit')
await page.waitForTimeout(2000)
expect(await page.isVisible('.success-msg')).toBeTruthy()

// GOOD: Playwright automatically polls the DOM until the condition is met or times out
await page.getByRole('button', { name: 'Submit' }).click()
await expect(page.getByText('Purchase Successful')).toBeVisible()
```

### Locators
Like Testing Library, Playwright encourages semantic locators over CSS selectors.
Prefer `getByRole`, `getByText`, and `getByLabel`.

## 3. Dealing with Authentication

Authenticating a user via the UI (typing in username/password) is slow. If you have 50 E2E tests, you don't want to log in 50 times.

**Strategy: Save and Reuse Authentication State**
Playwright allows you to authenticate once in a global setup, save the browser's storage state (cookies, local storage), and inject it into subsequent tests.

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  projects: [
    { name: 'setup', testMatch: /.*\.setup\.ts/ },
    {
      name: 'chromium',
      use: { 
        ...devices['Desktop Chrome'],
        // Use the saved state for all tests in this project
        storageState: 'playwright/.auth/user.json', 
      },
      dependencies: ['setup'],
    },
  ],
});
```

## 4. Test Data Strategy for E2E

An E2E test requires a predictable environment. If your test relies on a product called "Test Shirt" existing in the database, it will fail if someone deletes it.

**The Golden Rule of E2E Data:** The test must create the data it needs, and (ideally) clean it up afterwards.

```typescript
test('user can update their profile', async ({ request, page }) => {
  // 1. ARRANGE: Create test-specific data via direct API calls (bypassing UI)
  const user = await createTestUserViaAPI(request)
  
  // 2. ACT: Navigate via UI
  await page.goto(`/settings`)
  await page.getByLabel('Bio').fill('This is a new bio')
  await page.getByRole('button', { name: 'Save' }).click()
  
  // 3. ASSERT
  await expect(page.getByText('Profile updated')).toBeVisible()
})
```

## 5. Visual Regression Testing

Playwright excels at visual regression testing (Snapshot testing for images). It takes a screenshot of a component or page and compares it pixel-by-pixel to a baseline.

```typescript
test('homepage visual regression', async ({ page }) => {
  await page.goto('/')
  // Masks dynamic content like dates or rotating ads to prevent flakiness
  await expect(page).toHaveScreenshot('homepage.png', {
    mask: [page.locator('.dynamic-timestamp')]
  })
})
```
*Note: Use visual regression sparingly (e.g., for core design system components or static marketing pages), as cross-OS rendering differences can introduce flakiness in CI pipelines.*
