# Unit Testing Philosophy with Vitest

In the context of production reliability, unit testing is your first line of defense. Vitest is the testing framework of choice for modern Nuxt/Vue applications because it shares Vite's execution pipeline, making it insanely fast and perfectly aligned with our build configuration.

## 1. The Goal of a Unit Test

A unit test should answer one question: **"Does this specific, isolated piece of code behave correctly under different inputs?"**

It should **not** answer:
- "Does the page load successfully?" (That's E2E).
- "Does the database update?" (That's Integration).

## 2. Test Behavior, Not Implementation

The most critical philosophy in our testing strategy is to test *behavior*, not *implementation details*. 

If you refactor the internal logic of a function without changing its inputs and outputs, **your tests should not break.**

**Bad (Testing Implementation):**
```typescript
it('fetches data correctly', () => {
  const store = useDataStore()
  store.fetch()
  // Fragile! If the variable name changes to 'isFetching', the test breaks.
  expect(store.loading).toBe(true) 
})
```

**Good (Testing Behavior):**
```typescript
it('disables the submit button while fetching', () => {
  const wrapper = mount(SubmitButton)
  // We don't care how the component tracks loading, 
  // we just care that the button is disabled.
  expect(wrapper.find('button').attributes('disabled')).toBeDefined()
})
```

## 3. Pure Functions First

The easiest code to unit test is a pure function. Whenever possible, extract complex logic out of Vue components and Pinia stores into pure TypeScript utility functions (e.g., `utils/math.ts` or `utils/dateFormatter.ts`). 

Testing a pure function in Vitest takes milliseconds and requires zero mocking.

```typescript
// utils/price.spec.ts
import { calculateDiscount } from './price'

it('applies a 10% discount', () => {
  expect(calculateDiscount(100, 10)).toBe(90)
})
```

## 4. Mocking Boundaries

Unit tests must be isolated. If a composable relies on a network request, that request must be mocked. 

**Our Mocking Philosophy:**
1.  **Mock Network:** Use MSW (Mock Service Worker) for API calls. Never mock `fetch` or `axios` directly.
2.  **Mock Time:** Use `vi.useFakeTimers()` for anything involving `setTimeout` or `setInterval`. Use `vi.setSystemTime()` when testing date logic.
3.  **Mock Browser APIs:** Use `vi.spyOn(window, 'localStorage')` when testing browser storage.

## 5. Fast Execution Over Everything

Vitest is chosen for its speed. If a unit test suite takes more than a few seconds to run, developers will stop running it locally.
- Do not mount massive component trees for a unit test.
- Avoid deep `mount` if `shallowMount` is sufficient (though `Testing Library` prefers deep mounting for integration-style tests).
- Clean up after every test using `beforeEach` and `afterEach` to prevent state leakage, which leads to slow, sequential debugging sessions.

By adhering to these philosophies, Vitest becomes an enabler of rapid development rather than a hindrance.
