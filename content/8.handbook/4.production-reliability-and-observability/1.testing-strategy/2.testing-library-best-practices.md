# Vue Testing Library Best Practices

*Note: While many architectures refer to "React Testing Library", the core philosophies apply perfectly to `@testing-library/vue`. The foundational principle remains the same across frameworks.*

## The Guiding Principle
> **"The more your tests resemble the way your software is used, the more confidence they can give you."** — Kent C. Dodds

Testing Library forces you to test your Vue components from the perspective of a user. Users do not care about your component's internal `ref`, `reactive` state, or `v-model` implementations. They care about what they can see, read, and click.

## 1. Querying the DOM (The Right Way)

Testing Library provides various ways to find elements. You must prioritize queries based on how accessible they are to a user.

### 🥇 Priority 1: Accessible Queries (Use these 95% of the time)
These queries mimic how humans or screen readers find elements.
- **`getByRole`**: The absolute best query. Finds buttons, headings, textboxes. (`getByRole('button', { name: /submit/i })`)
- **`getByLabelText`**: The best way to interact with forms.
- **`getByText`**: Finding text on the screen.

### 🥈 Priority 2: Semantic Queries
- **`getByAltText`**: For images.
- **`getByTitle`**: For `title` attributes (SVGs).

### 🥉 Priority 3: Test IDs (Use as a last resort)
- **`getByTestId`**: Use this **only** when semantic queries fail (e.g., finding a highly dynamic SVG element or a generic container that cannot be identified by text or role).

**Bad Example (Testing Implementation):**
```javascript
// Relies on a specific CSS class that a designer might change tomorrow
const button = wrapper.find('.btn-primary')
```

**Good Example (Testing Library):**
```javascript
// Relies on the user-facing text and role. Resilient to styling changes.
const button = screen.getByRole('button', { name: /checkout/i })
```

## 2. Avoid Testing Vue Internals

Do not test component methods, computed properties, or internal data states directly.

**Anti-Pattern:**
```javascript
it('computes the total', () => {
  const wrapper = mount(ShoppingCart)
  wrapper.vm.items = [{ price: 10 }] // Mutating internal state directly!
  expect(wrapper.vm.totalPrice).toBe(10) // Testing a computed property directly!
})
```

**Testing Library Pattern:**
```javascript
import { render, screen } from '@testing-library/vue'
import userEvent from '@testing-library/user-event'

it('displays the total when an item is added', async () => {
  render(ShoppingCart)
  
  // User adds an item via the UI
  const input = screen.getByLabelText(/item price/i)
  await userEvent.type(input, '10')
  await userEvent.click(screen.getByRole('button', { name: /add/i }))
  
  // Assert the result is visible to the user
  expect(screen.getByText(/Total: \$10/i)).toBeInTheDocument()
})
```

## 3. Use `userEvent` over `fireEvent`

Testing Library provides two ways to simulate events: `fireEvent` and `userEvent`.
Always prefer **`userEvent`**.

*   `fireEvent.change(input, { target: { value: 'a' } })` just directly modifies the DOM value.
*   `userEvent.type(input, 'a')` simulates the exact sequence of events a real user triggers: mouse down, click, focus, keydown, keypress, input, keyup. It uncovers hidden bugs in event bubbling and focus management.

## 4. Assertions and waitFor

When dealing with async updates in Vue, use Testing Library's async utilities.

- Use **`findBy`** instead of `getBy` when expecting an element to appear asynchronously.
- Use **`waitFor`** for asserting that an element disappears or a state changes over time.

```javascript
// Wait for loading spinner to disappear
await waitFor(() => {
  expect(screen.queryByRole('progressbar')).not.toBeInTheDocument()
})
```

By strictly adhering to Testing Library principles, your tests will act as accessibility validators and will survive massive refactors, provided the UI remains functionally identical to the user.
