# Automated & Manual Accessibility Testing

Accessibility testing is not a one-time audit—it is an ongoing practice embedded into the development workflow. Automated tools can catch approximately 30-40% of WCAG violations. The remaining 60-70% require manual testing and real-world validation with assistive technologies.

## 1. The Testing Stack

A complete accessibility testing strategy combines multiple complementary layers:

```
┌─────────────────────────────────────────────────────────┐
│ Layer 1: Static Analysis (Lint time)                     │
│   eslint-plugin-vuejs-accessibility                      │
├─────────────────────────────────────────────────────────┤
│ Layer 2: Unit / Component Tests (Vitest)                 │
│   @axe-core/vue + @testing-library/vue                  │
├─────────────────────────────────────────────────────────┤
│ Layer 3: E2E Tests (Playwright)                          │
│   @axe-core/playwright                                   │
├─────────────────────────────────────────────────────────┤
│ Layer 4: Browser DevTools                                │
│   Chrome Lighthouse, Accessibility Tree panel            │
├─────────────────────────────────────────────────────────┤
│ Layer 5: Manual Testing                                  │
│   Keyboard navigation + Screen readers (VoiceOver, NVDA) │
└─────────────────────────────────────────────────────────┘
```

## 2. Layer 1: ESLint for Vue Accessibility

`eslint-plugin-vuejs-accessibility` catches common accessibility errors at the point of authoring.

```bash
pnpm add -D eslint-plugin-vuejs-accessibility
```

```javascript
// eslint.config.js
import vueA11y from 'eslint-plugin-vuejs-accessibility'

export default [
  ...vueA11y.configs['flat/recommended'],
  {
    rules: {
      // Enforce alt text on images
      'vuejs-accessibility/alt-text': 'error',
      // Enforce label associations on form controls
      'vuejs-accessibility/label-has-for': 'error',
      // Prevent using click events only (require keyboard equivalent)
      'vuejs-accessibility/click-events-have-key-events': 'warn',
      // Prevent non-interactive elements from having interactive roles
      'vuejs-accessibility/no-static-element-interactions': 'warn',
    }
  }
]
```

## 3. Layer 2: Automated Component Tests with axe-core

`axe-core` is the industry-standard accessibility rules engine used by the majority of automated testing tools.

```bash
pnpm add -D axe-core @axe-core/vue vitest-axe
```

```typescript
// components/ProductCard.spec.ts
import { render } from '@testing-library/vue'
import { axe, toHaveNoViolations } from 'vitest-axe'
import ProductCard from './ProductCard.vue'

expect.extend(toHaveNoViolations)

it('is accessible', async () => {
  const { container } = render(ProductCard, {
    props: {
      product: { id: '1', name: 'Running Shoes', price: '$99', imageUrl: '/shoes.jpg' }
    }
  })
  
  const results = await axe(container)
  
  // Fails the test if any axe violations are found
  expect(results).toHaveNoViolations()
})
```

### Testing Specific Accessibility Rules

```typescript
it('has no color contrast violations', async () => {
  const { container } = render(Button, { props: { variant: 'primary' } })
  
  const results = await axe(container, {
    // Only run specific rules for focused testing
    runOnly: { type: 'rule', values: ['color-contrast'] }
  })
  
  expect(results).toHaveNoViolations()
})
```

## 4. Layer 3: E2E Accessibility with Playwright

Run axe-core against full pages in a real browser during Playwright E2E tests.

```bash
pnpm add -D @axe-core/playwright
```

```typescript
// e2e/accessibility.spec.ts
import { test, expect } from '@playwright/test'
import AxeBuilder from '@axe-core/playwright'

test.describe('Accessibility Audits', () => {
  
  test('Homepage has no axe violations', async ({ page }) => {
    await page.goto('/')
    
    const results = await new AxeBuilder({ page })
      .withTags(['wcag2a', 'wcag2aa', 'wcag21aa']) // Target WCAG AA compliance
      .analyze()
    
    expect(results.violations).toEqual([])
  })

  test('Checkout page has no axe violations', async ({ page }) => {
    await page.goto('/checkout')
    
    const results = await new AxeBuilder({ page })
      .withTags(['wcag2a', 'wcag2aa'])
      .exclude('.third-party-widget') // Exclude known third-party violations
      .analyze()
    
    expect(results.violations).toEqual([])
  })
})
```

## 5. Layer 4: Browser DevTools

Use browser built-in tools for rapid investigation:
- **Chrome Accessibility Tree:** DevTools → Elements → Accessibility tab. Inspect what the accessibility tree looks like for any element.
- **Chrome Lighthouse:** DevTools → Lighthouse → Run Audit. Provides a scored accessibility report.
- **Firefox Accessibility Panel:** DevTools → Accessibility. Shows the full accessibility tree with contrast ratios.

## 6. Layer 5: Manual Testing Checklist

Automated tools cannot test user experience. Every feature before release should pass this manual checklist:

### Keyboard Navigation
- [ ] Tab through every interactive element on the page.
- [ ] Verify tab order is logical (top-to-bottom, left-to-right).
- [ ] Open and close all modals, dropdowns, and menus via keyboard.
- [ ] Test skip link functionality (press Tab once, a "Skip to content" link should appear).

### Screen Reader (VoiceOver on macOS)
- [ ] Enable VoiceOver (`Cmd+F5`).
- [ ] Navigate by headings (`VO+Cmd+H`) — is the heading structure logical?
- [ ] Navigate by landmarks (`VO+U` → Landmarks) — are all regions announced?
- [ ] Navigate the page in reading order (arrow keys) — is content in a sensible order?
- [ ] Submit a form and verify error messages are announced.
- [ ] Trigger a modal and verify focus moves inside and is trapped.

### Visual Checks
- [ ] Zoom page to 200% — no content loss or horizontal scroll.
- [ ] Test with Windows High Contrast Mode (or Chrome's emulation).
- [ ] Verify all images have meaningful alt text (right-click → Inspect).
- [ ] Run contrast checks on all text using WebAIM Contrast Checker.

## 7. Continuous Accessibility Monitoring

Add axe-core to your CI pipeline to catch regressions before they reach production.

```yaml
# .github/workflows/a11y.yml
- name: Run Accessibility Audit
  run: pnpm playwright test e2e/accessibility.spec.ts
```

Track your Lighthouse accessibility score over time using tools like **Calibre**, **SpeedCurve**, or a custom GitHub Action that posts scores as PR comments.
