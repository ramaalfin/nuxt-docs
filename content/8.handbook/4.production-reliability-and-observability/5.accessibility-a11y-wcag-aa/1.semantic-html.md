# Semantic HTML

Semantic HTML is the practice of using HTML elements that carry inherent meaning—elements that describe their content's purpose, not just its visual appearance. It is the bedrock of web accessibility, SEO, and maintainable code.

When you use `<button>` instead of `<div>`, you get keyboard accessibility, screen reader announcements, and ARIA role semantics for free—with zero JavaScript.

## 1. The Semantic vs. Non-Semantic Problem

```html
<!-- ❌ Non-semantic: A div pretending to be a button -->
<!-- Requires: manual tabindex, manual keyboard events, manual ARIA -->
<div class="btn" onclick="doSomething()">Submit</div>

<!-- ✅ Semantic: A real button -->
<!-- Comes with: focusable, keyboard-triggerable (Enter/Space), role="button" -->
<button type="button" onclick="doSomething()">Submit</button>
```

The `<div>` version requires you to manually implement everything the `<button>` gives you for free. And you'll almost certainly forget something.

## 2. Document Structure Elements

Use landmark elements to give structure to your page. Screen reader users navigate by landmarks—they can jump directly to `<main>` or `<nav>` without reading every element.

```html
<body>
  <header>
    <!-- Site logo, primary navigation -->
    <nav aria-label="Primary">
      <ul>
        <li><a href="/">Home</a></li>
        <li><a href="/about">About</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <!-- The page's primary content -->
    <article>
      <h1>Page Title (Only one h1 per page!)</h1>
      <section aria-labelledby="section-1-title">
        <h2 id="section-1-title">Section Heading</h2>
        <p>Content...</p>
      </section>
    </article>

    <aside aria-label="Related articles">
      <!-- Supplementary content -->
    </aside>
  </main>

  <footer>
    <!-- Site-wide footer links, copyright -->
  </footer>
</body>
```

## 3. The Heading Hierarchy

Headings (`<h1>` through `<h6>`) create the table of contents for your page. Screen readers allow users to navigate by heading level. A broken heading hierarchy is one of the most common accessibility failures.

**Rules:**
- One `<h1>` per page (the main page title).
- Never skip heading levels (no jumping from `<h2>` to `<h4>`).
- Choose heading levels based on document structure, not visual size. Use CSS to control size.

```html
<!-- ❌ Broken hierarchy -->
<h1>Dashboard</h1>
<h3>Recent Orders</h3>  <!-- Skipped h2! -->
<h5>Order #123</h5>     <!-- Skipped h4! -->

<!-- ✅ Correct hierarchy -->
<h1>Dashboard</h1>
<h2>Recent Orders</h2>
<h3>Order #123</h3>
```

## 4. Semantic Elements in Vue Components

In Nuxt/Vue components, use native semantic elements as the root element of your components wherever possible.

```vue
<!-- ❌ Bad: Wrapping everything in divs -->
<template>
  <div class="product-card">
    <div class="product-title">{{ product.name }}</div>
    <div class="product-price">{{ product.price }}</div>
    <div class="product-cta" @click="addToCart">Add to Cart</div>
  </div>
</template>

<!-- ✅ Good: Semantic HTML communicates meaning -->
<template>
  <article class="product-card">
    <h3 class="product-title">{{ product.name }}</h3>
    <p class="product-price">
      <span class="sr-only">Price:</span> {{ product.price }}
    </p>
    <button type="button" class="product-cta" @click="addToCart">
      Add to Cart
    </button>
  </article>
</template>
```

## 5. Forms

Forms require the most careful semantic attention. Every input must be associated with a label.

```vue
<template>
  <form @submit.prevent="handleSubmit" novalidate>
    <div class="form-group">
      <!-- Associate label with input via 'for'/'id' -->
      <label for="email">Email address</label>
      <input 
        id="email"
        type="email"
        name="email"
        v-model="form.email"
        autocomplete="email"
        required
        :aria-describedby="emailError ? 'email-error' : undefined"
        :aria-invalid="emailError ? 'true' : undefined"
      />
      <p v-if="emailError" id="email-error" role="alert" class="error">
        {{ emailError }}
      </p>
    </div>
    
    <button type="submit">Subscribe</button>
  </form>
</template>
```

## 6. Common Semantic Elements Reference

| Element | Use For |
|---|---|
| `<button>` | Any clickable action that does NOT navigate |
| `<a href>` | Navigation to another URL or page |
| `<nav>` | Navigation landmark regions |
| `<main>` | The page's primary content (one per page) |
| `<aside>` | Supplementary content tangentially related to main |
| `<article>` | Self-contained, independently distributable content |
| `<section>` | Thematic group of content, should have a heading |
| `<header>` | Introductory content for a page or section |
| `<footer>` | Closing content for a page or section |
| `<figure>` + `<figcaption>` | Images, diagrams, or code with captions |
| `<time datetime>` | Dates and times (machine-readable) |
| `<abbr title>` | Abbreviations and acronyms |
