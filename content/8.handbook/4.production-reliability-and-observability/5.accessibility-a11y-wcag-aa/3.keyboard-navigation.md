# Keyboard Navigation

Approximately 7 million Americans with motor disabilities rely on keyboard navigation instead of a mouse or trackpad. Additionally, power users and developers frequently navigate the web using only a keyboard. WCAG 2.1 Success Criterion 2.1.1 (Level A) requires that **all functionality must be operable via keyboard**.

## 1. Core Keyboard Principles

### The Tab Order
When a user presses `Tab`, focus moves to the next focusable element. `Shift+Tab` moves backwards. The tab order must be:
- **Logical:** Follow the visual reading order (top-to-bottom, left-to-right).
- **Complete:** Every interactive element (links, buttons, inputs, custom widgets) must be reachable.
- **Predictable:** Users should never be surprised by where focus lands.

**Natively focusable elements (require no special handling):**
- `<a href="...">` links
- `<button>` buttons
- `<input>`, `<select>`, `<textarea>` form controls
- Elements with `tabindex="0"`

### The Activation Keys
| Element | Keyboard Trigger |
|---|---|
| `<button>` | `Enter` or `Space` |
| `<a href>` | `Enter` |
| `<input type="checkbox">` | `Space` |
| `<select>` | `Enter`, `Arrow` keys |
| Custom widget | Must be implemented manually |

## 2. `tabindex` Usage

```html
<!-- tabindex="0": Adds element to the natural tab order -->
<div role="button" tabindex="0" @keydown.enter="activate" @keydown.space.prevent="activate">
  Custom Button
</div>

<!-- tabindex="-1": Focusable programmatically but NOT via Tab key -->
<!-- Use for elements that receive focus via JS (modals, menu items) -->
<div tabindex="-1" ref="modalContent">...</div>

<!-- tabindex > 0: NEVER use this — it breaks the natural tab order -->
<!-- ❌ BAD -->
<button tabindex="5">Submit</button>
```

## 3. Common Keyboard Patterns for Custom Widgets

ARIA Authoring Practices Guide (APG) defines the expected keyboard behavior for common widget patterns. Always follow these conventions.

### Dropdown Menu / Disclosure
```
Enter or Space on trigger → Opens menu
Arrow Down / Arrow Up → Navigate items
Escape → Closes menu, returns focus to trigger
Tab → Closes menu, moves to next focusable element
```

### Modal Dialog
```
Opening trigger → Focus moves to first focusable element inside modal
Tab → Cycles through focusable elements WITHIN the modal (focus trap!)
Shift+Tab → Cycles backwards
Escape → Closes modal, focus returns to the element that opened it
```

### Tab Panel
```
Arrow Left / Arrow Right → Switch between tabs
Home → First tab
End → Last tab
(Focus follows the active tab; Tab moves into the tab panel content)
```

## 4. Implementing Keyboard Navigation in Vue

### Focus Trap for Modals

```typescript
// composables/useFocusTrap.ts
export function useFocusTrap(containerRef: Ref<HTMLElement | null>) {
  const FOCUSABLE_SELECTORS = [
    'a[href]', 'button:not([disabled])', 'input:not([disabled])',
    'select:not([disabled])', 'textarea:not([disabled])', '[tabindex]:not([tabindex="-1"])'
  ].join(', ')

  const trapFocus = (event: KeyboardEvent) => {
    if (!containerRef.value || event.key !== 'Tab') return
    
    const focusable = Array.from(containerRef.value.querySelectorAll<HTMLElement>(FOCUSABLE_SELECTORS))
    const first = focusable[0]
    const last = focusable[focusable.length - 1]
    
    if (event.shiftKey) {
      if (document.activeElement === first) {
        event.preventDefault()
        last.focus()
      }
    } else {
      if (document.activeElement === last) {
        event.preventDefault()
        first.focus()
      }
    }
  }

  onMounted(() => document.addEventListener('keydown', trapFocus))
  onUnmounted(() => document.removeEventListener('keydown', trapFocus))
}
```

### Arrow Key Navigation for Menu Items

```vue
<script setup lang="ts">
const menuItems = ref<HTMLElement[]>([])
let currentIndex = 0

const onKeydown = (event: KeyboardEvent, index: number) => {
  if (event.key === 'ArrowDown') {
    event.preventDefault()
    currentIndex = (index + 1) % menuItems.value.length
    menuItems.value[currentIndex].focus()
  } else if (event.key === 'ArrowUp') {
    event.preventDefault()
    currentIndex = (index - 1 + menuItems.value.length) % menuItems.value.length
    menuItems.value[currentIndex].focus()
  } else if (event.key === 'Escape') {
    closeMenu()
    triggerRef.value?.focus()
  }
}
</script>
```

## 5. Skip Links

Skip links allow keyboard users to bypass repetitive navigation and jump directly to the main content. They must be the very first element in the page body.

```vue
<!-- app.vue or your layout -->
<template>
  <a href="#main-content" class="skip-link">Skip to main content</a>
  <AppHeader />
  <main id="main-content" tabindex="-1">
    <NuxtPage />
  </main>
</template>

<style>
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  
  /* Becomes visible when focused (via Tab key) */
  &:focus {
    top: 0;
    z-index: 9999;
  }
}
</style>
```

## 6. Keyboard Testing Checklist
- [ ] All interactive elements are reachable via `Tab`.
- [ ] Tab order is logical and follows visual layout.
- [ ] No keyboard focus traps outside of intentional modal dialogs.
- [ ] Modals implement a proper focus trap and restore focus on close.
- [ ] Custom widgets follow APG keyboard interaction patterns.
- [ ] Skip link is present and functional.
- [ ] Dropdown menus can be opened, navigated, and closed via keyboard.
