# Focus Management

Focus management is the practice of programmatically controlling which element receives keyboard focus in response to user interactions and UI state changes. When a user clicks a button and a modal appears, where does focus go? When the modal closes, where does focus return? These decisions are invisible to mouse users but critically important to keyboard and screen reader users.

Poor focus management is the most common accessibility failure in Single-Page Applications (SPAs).

## 1. Why Focus Management Matters in SPAs

Traditional multi-page websites naturally manage focus: when you navigate to a new page, the browser resets focus to the top of the document. In SPAs and Nuxt apps with client-side routing, page content is swapped without a full page reload—meaning focus can get "lost" somewhere in the previous page's content that no longer exists in the DOM.

## 2. Core Focus Management Rules

### Rule 1: When a UI element appears, move focus into it
When a modal, dialog, sidebar, or dropdown opens, focus must move into it.

```typescript
// composables/useModal.ts
export function useModal() {
  const isOpen = ref(false)
  const modalRef = ref<HTMLElement | null>(null)
  const openerRef = ref<HTMLElement | null>(null)

  const open = (opener: HTMLElement) => {
    openerRef.value = opener // Remember who opened the modal
    isOpen.value = true
    
    nextTick(() => {
      // Move focus to the first focusable element inside the modal
      const firstFocusable = modalRef.value?.querySelector<HTMLElement>(
        'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
      )
      firstFocusable?.focus()
    })
  }

  const close = () => {
    isOpen.value = false
    // Return focus to the element that triggered the modal
    nextTick(() => openerRef.value?.focus())
  }

  return { isOpen, modalRef, open, close }
}
```

### Rule 2: When a UI element disappears, restore focus
When a modal, dialog, or overlay closes, focus must return to the element that triggered it.

```vue
<script setup lang="ts">
const triggerRef = ref<HTMLButtonElement>()

const onModalClose = () => {
  isModalOpen.value = false
  
  // Restore focus to the button that opened the modal
  nextTick(() => triggerRef.value?.focus())
}
</script>

<template>
  <button ref="triggerRef" @click="isModalOpen = true">Open Settings</button>
  <SettingsModal :open="isModalOpen" @close="onModalClose" />
</template>
```

### Rule 3: Manage focus on route changes (SPA Navigation)
When the Nuxt router navigates to a new page, focus must be explicitly managed.

```typescript
// plugins/focus-management.client.ts
export default defineNuxtPlugin((nuxtApp) => {
  const router = useRouter()
  
  router.afterEach(() => {
    nextTick(() => {
      // Option 1: Move focus to a dedicated "page title" heading
      const h1 = document.querySelector<HTMLElement>('h1')
      if (h1) {
        // tabindex="-1" allows programmatic focus without adding to tab order
        h1.setAttribute('tabindex', '-1')
        h1.focus()
        // Optionally remove tabindex after focus to keep tab order clean
        h1.addEventListener('blur', () => h1.removeAttribute('tabindex'), { once: true })
        return
      }
      
      // Option 2: Move focus to the <main> element
      const main = document.querySelector<HTMLElement>('#main-content')
      main?.focus()
    })
  })
})
```

## 3. Focus Indicators (`:focus-visible`)

Never remove focus indicators without providing a replacement. `outline: none` makes your site impossible to navigate by keyboard.

Use `:focus-visible` instead of `:focus` to show focus rings only when navigating via keyboard (not on mouse clicks):

```css
/* ❌ BAD: Removes all focus indicators */
* { outline: none; }

/* ✅ GOOD: Preserve focus for keyboard users, hide for mouse users */
*:focus-visible {
  outline: 3px solid var(--color-focus-ring);
  outline-offset: 2px;
  border-radius: 4px;
}

/* Suppress focus ring on mouse click (browsers handle this automatically with :focus-visible) */
*:focus:not(:focus-visible) {
  outline: none;
}
```

## 4. Focus Trapping

Inside modal dialogs, focus must be trapped—Tab should cycle only through elements within the modal, never reaching elements in the background.

See the `useFocusTrap` composable in the **Keyboard Navigation** chapter for the implementation.

**Critical detail:** When the modal opens, the background content should have `aria-hidden="true"` applied to prevent screen readers from reading the background content.

```vue
<template>
  <div>
    <!-- Background content: hidden from screen readers when modal is open -->
    <div id="app-content" :aria-hidden="isModalOpen ? 'true' : undefined">
      <AppHeader />
      <main><NuxtPage /></main>
    </div>

    <!-- Modal: only visible and accessible when open -->
    <BaseModal v-if="isModalOpen" @close="isModalOpen = false" />
  </div>
</template>
```

## 5. Focus Management After Async Operations

When an action causes async content to appear (e.g., a search result, a form validation error), move focus to the new content so keyboard users are informed.

```typescript
// After search results load
const performSearch = async () => {
  isLoading.value = true
  results.value = await searchApi(query.value)
  isLoading.value = false
  
  await nextTick()
  
  if (results.value.length > 0) {
    // Move focus to the results heading so screen readers announce the count
    resultsHeadingRef.value?.focus()
  } else {
    // Move focus to the "no results" message
    noResultsRef.value?.focus()
  }
}
```

## 6. Focus Management Checklist

- [ ] Opening a modal moves focus inside it.
- [ ] Closing a modal returns focus to the triggering element.
- [ ] Modals trap focus within their boundaries.
- [ ] Background content is `aria-hidden` when a modal is open.
- [ ] Route changes move focus to the `<h1>` or `<main>` of the new page.
- [ ] Focus indicators are visible for all keyboard-focused elements.
- [ ] Async content updates move focus to the new content when relevant.
