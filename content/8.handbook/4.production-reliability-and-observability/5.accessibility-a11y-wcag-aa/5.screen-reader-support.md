# Screen Reader Support

Screen readers are assistive technology software that reads digital content aloud for users who are blind, have low vision, or have cognitive disabilities. They interpret the accessibility tree (a structured representation of your page built from HTML semantics and ARIA attributes) rather than the visual DOM.

Supporting screen readers is not a separate feature—it is the result of correctly implementing semantic HTML, ARIA, and focus management.

## 1. The Accessibility Tree

When a browser parses your HTML, it builds two trees:
1. **The DOM Tree:** The full HTML structure rendered visually.
2. **The Accessibility Tree:** A filtered, annotated version of the DOM that assistive technologies consume.

Not all DOM nodes appear in the accessibility tree. Elements with `aria-hidden="true"`, `display: none`, or `visibility: hidden` are excluded.

## 2. Screen Reader Announcements

### Text Alternatives for Non-Text Content

Every image, icon, and graphic must have a text alternative.

```vue
<!-- Meaningful image: use descriptive alt text -->
<img src="/team/alice.jpg" alt="Alice Chen, Head of Engineering" />

<!-- Decorative image: empty alt to skip it -->
<img src="/background-pattern.svg" alt="" />

<!-- Functional icon button: label the action, not the icon name -->
<!-- ❌ Bad: "icon close" announced -->
<button aria-label="Close dialog">
  <XMarkIcon aria-hidden="true" />
</button>

<!-- Icon with adjacent text: hide the icon from screen readers -->
<button>
  <HeartIcon aria-hidden="true" />
  <span>Add to Wishlist</span>
</button>
```

### Visually Hidden Text (Screen Reader Only)

The `.sr-only` class provides context to screen reader users without displaying it visually.

```css
/* styles/utilities.css */
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border-width: 0;
}
```

```vue
<button>
  <!-- Visual: "3" badge only -->
  <!-- Screen reader: "3 notifications" -->
  3 <span class="sr-only">notifications</span>
</button>

<ul>
  <li v-for="product in products">
    {{ product.name }}
    <!-- Gives each button a unique accessible name -->
    <button>
      Add to Cart
      <span class="sr-only">: {{ product.name }}</span>
    </button>
  </li>
</ul>
```

## 3. Live Regions for Dynamic Content

When content changes dynamically (without a page reload), screen readers won't announce the change unless you use a live region.

```vue
<script setup lang="ts">
// Status message composable for announcing dynamic changes
export function useAnnounce() {
  const message = ref('')
  const politeness = ref<'polite' | 'assertive'>('polite')

  const announce = (text: string, urgent = false) => {
    politeness.value = urgent ? 'assertive' : 'polite'
    // Reset and re-set to force re-announcement
    message.value = ''
    nextTick(() => { message.value = text })
  }

  return { message, politeness, announce }
}
</script>
```

```vue
<!-- Place this once in your app layout -->
<template>
  <!-- Visually hidden, but announced to screen readers when content changes -->
  <div 
    class="sr-only" 
    :aria-live="politeness" 
    aria-atomic="true"
  >
    {{ message }}
  </div>
</template>
```

**Usage:**
```typescript
const { announce } = useAnnounce()

const addToCart = async () => {
  await cartStore.addItem(product)
  // Screen reader users hear: "Shoes added to cart. Cart now has 3 items."
  announce(`${product.name} added to cart. Cart now has ${cartStore.count} items.`)
}
```

## 4. Table Accessibility

Data tables need proper semantic markup to be navigable by screen readers.

```html
<table>
  <caption>Monthly Sales Report for Q1 2024</caption>
  <thead>
    <tr>
      <th scope="col">Month</th>
      <th scope="col">Revenue</th>
      <th scope="col">Orders</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">January</th>  <!-- Row header -->
      <td>$12,400</td>
      <td>340</td>
    </tr>
  </tbody>
</table>
```

## 5. Nuxt Route Change Announcements

When using client-side routing, screen readers do not automatically announce page changes. Use an ARIA live region to announce route changes.

```typescript
// plugins/route-announcer.client.ts
export default defineNuxtPlugin((nuxtApp) => {
  const router = useRouter()
  
  // Create a visually hidden live region
  const announcer = document.createElement('div')
  announcer.setAttribute('aria-live', 'polite')
  announcer.setAttribute('aria-atomic', 'true')
  announcer.className = 'sr-only'
  announcer.id = 'route-announcer'
  document.body.prepend(announcer)
  
  router.afterEach((to) => {
    nextTick(() => {
      // Announce the new page title
      const title = document.title || to.path
      announcer.textContent = ''
      requestAnimationFrame(() => {
        announcer.textContent = `Navigated to ${title}`
      })
    })
  })
})
```

## 6. Testing with Screen Readers

Automated tools (axe-core, Lighthouse) catch ~30% of accessibility issues. You must also test manually.

**Recommended Free Screen Readers:**
| Tool | Platform | Browser |
|---|---|---|
| **NVDA** | Windows | Firefox, Chrome |
| **VoiceOver** | macOS / iOS | Safari |
| **TalkBack** | Android | Chrome |

**Basic VoiceOver Test Workflow (macOS):**
1. `Cmd+F5` → Enable VoiceOver.
2. `Tab` → Navigate interactive elements.
3. `VO+U` → Open the Rotor (navigate by landmark, heading, link, form control).
4. Interact with your modal, form, and navigation.
5. `Cmd+F5` → Disable VoiceOver.
