# Performance Monitoring (APM)

Application Performance Monitoring (APM) provides deep visibility into the performance characteristics of your running application. While error tracking tells you *that* something broke, APM tells you *why it is slow*—tracing requests from the browser, through the Nuxt SSR server, to the database query that caused the bottleneck.

## 1. What APM Monitors in a Nuxt Application

In a full-stack Nuxt application, APM instruments the complete request lifecycle:

```
Browser         →  Nuxt SSR Server  →  API  →  Database
[Frontend Trace]  [Server Trace]     [API Trace] [DB Query Trace]
```

For each request, APM collects:
- **Transaction Duration:** How long did the entire operation take?
- **Span Breakdown:** Which part was slowest (DB query? Render? External API call?)?
- **Error Rate:** What percentage of transactions are failing?
- **Throughput (RPM):** How many requests per minute is the server handling?

## 2. Sentry Performance Monitoring

Sentry's APM is the simplest entry point for teams already using Sentry for error tracking, as it shares the same SDK.

```typescript
// sentry.client.config.ts
import * as Sentry from '@sentry/nuxt'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  
  // Sample 10% of transactions in production
  tracesSampleRate: 0.1,
  
  integrations: [
    Sentry.browserTracingIntegration({
      // Automatically create spans for page navigations
      tracePropagationTargets: ['localhost', /^https:\/\/api\.yourapp\.com/],
    }),
  ]
})
```

### Creating Custom Performance Spans

For business-critical operations that are not automatically instrumented, create manual spans.

```typescript
import * as Sentry from '@sentry/nuxt'

const processCheckout = async (cart: Cart) => {
  return Sentry.startSpan({ name: 'checkout.process', op: 'business' }, async () => {
    
    // Child spans are automatically nested under the parent
    const order = await Sentry.startSpan({ name: 'checkout.validateInventory', op: 'http.client' }, 
      () => $fetch('/api/inventory/validate', { body: cart })
    )
    
    const payment = await Sentry.startSpan({ name: 'checkout.processPayment', op: 'http.client' }, 
      () => $fetch('/api/payments', { body: { orderId: order.id } })
    )
    
    return payment
  })
}
```

## 3. Nuxt Server Route Tracing

For server-side API routes in Nuxt's Nitro, Sentry's `@sentry/nuxt` module automatically instruments incoming requests as transactions and outgoing `$fetch` calls as child spans.

Ensure your `sentry.server.config.ts` is correctly configured (see Error Tracking chapter) to get server-side transaction data.

## 4. Core Web Vitals as Performance Signals

For client-side performance, Core Web Vitals (LCP, INP, CLS) are the primary metrics to monitor.

```typescript
// plugins/web-vitals.client.ts
import { onCLS, onINP, onLCP } from 'web-vitals'

export default defineNuxtPlugin(() => {
  const reportVital = (metric) => {
    // Send to your APM tool
    Sentry.setMeasurement(metric.name, metric.value, metric.unit)
    
    // Or send to a custom analytics endpoint
    $fetch('/api/vitals', { method: 'POST', body: metric })
  }
  
  onCLS(reportVital)
  onINP(reportVital)
  onLCP(reportVital)
})
```

**Target Thresholds (Google's "Good" range):**
| Metric | Good | Needs Improvement |
|---|---|---|
| LCP (Largest Contentful Paint) | < 2.5s | 2.5s - 4.0s |
| INP (Interaction to Next Paint) | < 200ms | 200ms - 500ms |
| CLS (Cumulative Layout Shift) | < 0.1 | 0.1 - 0.25 |

## 5. APM Tool Comparison

| Tool | Best For |
|---|---|
| **Sentry Performance** | Teams using Sentry for error tracking who want simple APM |
| **Datadog APM** | Enterprise teams needing deep infrastructure + application correlation |
| **New Relic** | Full-stack observability with a generous free tier |
| **Grafana Tempo + Faro** | Open-source teams with existing Grafana infrastructure |
