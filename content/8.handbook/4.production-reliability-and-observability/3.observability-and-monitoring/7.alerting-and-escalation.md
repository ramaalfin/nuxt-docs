# Alerting & Escalation

Observability data has zero value if no one is notified when something goes wrong. A well-designed alerting and escalation strategy ensures that the right person is notified at the right time with enough context to act—without drowning the team in false alarms.

## 1. Alert on Symptoms, Not Causes

The most important principle in alerting: **alert on what users experience, not on what your infrastructure is doing**.

- ❌ **Don't alert on:** CPU usage > 80%, memory usage > 70%.
- ✅ **Do alert on:** Error rate > 1%, P95 LCP > 4s, checkout success rate drops 20%.

A server running at 95% CPU but serving users perfectly is not an emergency. A server at 40% CPU but where 5% of requests are returning 500 errors absolutely is.

## 2. Alert Severity Levels

Establish clear, team-agreed severity levels. Every alert must have an assigned severity.

| Severity | Definition | Response Time | Channel |
|---|---|---|---|
| **P0 (Critical)** | Production down, revenue loss, data breach | Immediate (< 5min) | PagerDuty / phone call |
| **P1 (High)** | Major feature broken for many users | < 30min | PagerDuty / Slack DM |
| **P2 (Medium)** | Feature degraded, some users affected | < 4 hours | Slack `#alerts-prod` channel |
| **P3 (Low)** | Minor issue, workaround exists | Next business day | Slack `#alerts-low` channel |

## 3. Core Alerts to Implement

### Error Rate Alert (P1/P2)
Alert when the percentage of requests returning 5xx errors rises above a threshold.
```
Rule: error_rate (5xx) > 1% of all requests over the past 5 minutes
Severity: P1
Action: Page on-call engineer
```

### Apdex Score / Latency Alert (P2)
Alert when user-perceived performance degrades significantly.
```
Rule: P95 API response time > 2000ms over the past 10 minutes
Severity: P2
Action: Post to Slack #alerts-prod
```

### Zero Deployments / Regression Alert (P1)
Alert when error rate spikes immediately after a new deployment.
```
Rule: Error rate increases by >200% within 10 minutes of a deployment marker
Severity: P1
Action: Page on-call + post deployment link to Slack
```

### Business Metric Alert (P0/P1)
```
Rule: checkout.completed count < X per hour (compared to same time last week)
Severity: P0 if drop > 50%, P1 if drop > 20%
Action: Page engineering + notify business stakeholders
```

## 4. Escalation Policy

An escalation policy defines what happens when an alert is not acknowledged within a defined time window.

```
Alert fires (P1)
  ↓ (If not acknowledged in 5 min)
Notify primary on-call engineer (Slack + phone)
  ↓ (If not acknowledged in 10 min)
Notify secondary on-call engineer (phone)
  ↓ (If not acknowledged in 15 min)
Notify Engineering Manager
  ↓ (If not resolved in 30 min)
Escalate to VP of Engineering + trigger incident war room
```

Use a tool like **PagerDuty** or **OpsGenie** to manage on-call schedules and escalation chains automatically.

## 5. Fighting Alert Fatigue

Alert fatigue occurs when engineers receive so many noisy, low-quality alerts that they begin to ignore them—including real ones.

**Prevention strategies:**
- **Regular Alert Audits:** Every sprint, review which alerts fired. Was each one actionable? If an alert fired 50 times with no action taken, it needs to be tuned or removed.
- **High Signal-to-Noise Ratio:** Every alert that fires must require a human action. Informational alerts belong in dashboards, not in PagerDuty.
- **Alert Grouping:** Use tools like Sentry's issue grouping or Datadog's alert aggregation to prevent 1000 individual notifications for a single systemic error.
- **Suppress During Deployments:** Temporarily suppress non-critical alerts during known deployment windows to prevent noise from expected restarts.

## 6. Runbook Links in Every Alert

Every alert notification must include a link to a **Runbook**—a document describing:
1. What this alert means.
2. Common causes.
3. Step-by-step investigation procedure.
4. Resolution steps for common cases.
5. Escalation contacts if the runbook steps don't resolve the issue.

```
🚨 [P1 ALERT] Checkout Error Rate > 2%
Current: 3.4% (threshold: 1%)
Duration: 8 minutes
Affected users: ~450/min

📖 Runbook: https://wiki.company.com/runbooks/checkout-errors
📊 Dashboard: https://grafana.company.com/d/checkout
🔗 Sentry Issues: https://sentry.io/issues/?project=frontend&query=transaction:/checkout
```
