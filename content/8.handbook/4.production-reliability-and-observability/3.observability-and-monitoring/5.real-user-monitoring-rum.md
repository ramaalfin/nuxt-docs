# Real User Monitoring (RUM)

Real User Monitoring (RUM) captures performance data from actual users in real browsers on real devices across real networks—as opposed to synthetic monitoring, which runs tests from a controlled lab environment. RUM gives you the ground truth of how your application actually performs for your users.

## 1. RUM vs. Synthetic Monitoring

| Aspect | RUM | Synthetic Monitoring |
|---|---|---|
| **Data Source** | Real users in production | Scripted bots in CI/CD |
| **Coverage** | All user segments, devices, geographies | Fixed test locations |
| **Use Case** | "How fast is it *actually* for users?" | "Is the site up? Did we regress?" |
| **Tools** | Sentry, Datadog RUM, Grafana Faro, New Relic | Lighthouse CI, WebPageTest, Playwright |

Both are necessary. Use synthetic monitoring to catch regressions in CI and RUM to understand real-world impact.

## 2. Core Web Vitals as the Foundation of RUM

Google's Core Web Vitals are the primary RUM metrics used by the industry. They measure page load (LCP), responsiveness (INP), and visual stability (CLS).

```typescript
// plugins/rum.client.ts
import { onCLS, onINP, onLCP, onFCP, onTTFB } from 'web-vitals'

export default defineNuxtPlugin(() => {
  const reportVital = (metric: Metric) => {
    // Enrich with additional context
    const data = {
      name: metric.name,
      value: metric.value,
      rating: metric.rating, // 'good', 'needs-improvement', 'poor'
      page: window.location.pathname,
      connectionType: (navigator as any).connection?.effectiveType ?? 'unknown',
    }
    
    // Send to your observability backend
    navigator.sendBeacon('/api/vitals', JSON.stringify(data))
  }

  onLCP(reportVital, { reportAllChanges: false })
  onINP(reportVital, { reportAllChanges: false })
  onCLS(reportVital, { reportAllChanges: false })
  onFCP(reportVital)
  onTTFB(reportVital)
})
```

## 3. Segmenting RUM Data

Raw averages are misleading. A page with an average LCP of 2.0s might still have 30% of users experiencing > 4.0s (P75 and P95 percentiles). Always segment RUM data by:

- **Percentile:** P50 (median), P75 (Google's recommended target), P95 (worst affected users).
- **Device type:** Mobile vs. desktop. Mobile users typically experience 2-3x worse performance.
- **Geography:** Users in Southeast Asia may have significantly higher TTFB than US users if your server is in `us-east-1`.
- **Connection type:** 4G vs. WiFi vs. 3G.
- **Page/Route:** `/checkout` likely has a different LCP profile than `/blog`.

## 4. Grafana Faro (Open-Source RUM)

If you prefer an open-source solution, **Grafana Faro** is a RUM SDK that integrates with the Grafana observability stack (Loki for logs, Tempo for traces, Grafana for dashboards).

```typescript
// plugins/faro.client.ts
import { initializeFaro, getWebInstrumentations } from '@grafana/faro-web-sdk'

export default defineNuxtPlugin(() => {
  initializeFaro({
    url: process.env.NUXT_PUBLIC_FARO_ENDPOINT!,
    app: {
      name: 'my-nuxt-app',
      version: process.env.NUXT_PUBLIC_APP_VERSION,
      environment: process.env.NODE_ENV
    },
    instrumentations: [
      ...getWebInstrumentations({
        captureConsole: true, // Capture console errors
        enablePerformanceInstrumentation: true, // Auto-collect Web Vitals
      }),
    ]
  })
})
```

## 5. RUM-Driven Performance Budgets

Use RUM data to set and enforce performance budgets—thresholds that, if breached in production, trigger alerts and block future deployments.

```typescript
// Nuxt/Nitro endpoint to receive and store vital data
// server/api/vitals.post.ts
export default defineEventHandler(async (event) => {
  const body = await readBody(event)
  
  // Store in time-series DB (e.g., InfluxDB, Prometheus pushgateway)
  // or forward to Datadog/Grafana
  await storeMetric({
    name: body.name,
    value: body.value,
    page: body.page,
    timestamp: Date.now()
  })
  
  return { ok: true }
})
```

Alert when the P75 LCP for `/checkout` exceeds 3.0s over the last 30 minutes, or when INP exceeds 300ms for mobile users.
