# Custom Metrics

Out-of-the-box APM and error tracking tools give you technical metrics (request latency, error rate, CPU). But the most valuable signals in a production system are often **business metrics** that only your application can define: "How many orders were placed in the last 5 minutes?", "What is the checkout abandonment rate?", "How long does the average user spend on the pricing page?"

Custom metrics bridge the gap between technical observability and business intelligence.

## 1. Types of Custom Metrics

| Type | Description | Example |
|---|---|---|
| **Counter** | A monotonically increasing count. | Number of sign-ups, total API errors. |
| **Gauge** | A value that can go up or down. | Active WebSocket connections, items in cart. |
| **Histogram / Distribution** | Measures the distribution of a value. | Checkout duration, search result load time. |
| **Set** | Count of unique occurrences. | Number of unique users who triggered a feature. |

## 2. Implementing Custom Metrics with Sentry

Sentry's Metrics API allows you to send custom metrics directly from your Nuxt application.

```typescript
import * as Sentry from '@sentry/nuxt'

// COUNTER: Track feature usage
const useFeatureTracking = () => {
  const trackEvent = (eventName: string, tags?: Record<string, string>) => {
    Sentry.metrics.increment(`app.feature.${eventName}`, 1, { tags })
  }
  return { trackEvent }
}

// Example: Track successful checkouts
const onCheckoutSuccess = (orderId: string, totalAmount: number) => {
  Sentry.metrics.increment('checkout.completed', 1, {
    tags: { currency: 'USD', plan: userPlan.value }
  })
  
  // DISTRIBUTION: Track the value distribution of orders
  Sentry.metrics.distribution('checkout.order_value', totalAmount, {
    unit: 'dollar',
    tags: { currency: 'USD' }
  })
}

// GAUGE: Track active items
const onCartUpdated = (itemCount: number) => {
  Sentry.metrics.gauge('cart.item_count', itemCount)
}
```

## 3. Custom Performance Measurements

For measuring durations of specific user-facing operations, use Sentry's measurement API alongside your performance spans.

```typescript
// Measure how long the search result rendering takes
const measureSearchRender = async (query: string) => {
  const startTime = performance.now()
  
  const results = await $fetch('/api/search', { query: { q: query } })
  searchResults.value = results
  
  // Wait for Vue to render the results
  await nextTick()
  
  const duration = performance.now() - startTime
  
  Sentry.setMeasurement('search.render_duration', duration, 'millisecond')
}
```

## 4. Server-Side Custom Metrics with Nitro

For server-side metrics (e.g., tracking database query performance, background job duration), instrument your Nitro server routes.

```typescript
// server/api/products.get.ts
export default defineEventHandler(async (event) => {
  const startTime = Date.now()
  
  const products = await db.query('SELECT * FROM products WHERE active = true')
  
  const dbDuration = Date.now() - startTime
  
  // Track the DB query duration as a custom metric
  Sentry.metrics.distribution('db.query.products_list', dbDuration, {
    unit: 'millisecond',
    tags: { database: 'postgres', query: 'products_list' }
  })
  
  return products
})
```

## 5. A Custom Metrics Strategy

Avoid measuring everything. Define your **Golden Signals** first—the 4-6 metrics that, if they deviate, indicate something is wrong.

### Recommended Golden Signals for an E-commerce Nuxt App:

| Signal | Metric | Alert Threshold |
|---|---|---|
| Revenue | `checkout.completed` (count) | < X orders/hour |
| Order Value | `checkout.order_value` (distribution P50/P95) | P95 drops by >30% |
| Search Success | `search.results.count` gauge | P50 of results = 0 |
| Error Rate | Sentry's built-in error rate | > 1% of sessions |
| Page Performance | LCP (from Web Vitals) | P75 > 2.5s |

By building your alerting around these golden signals (instead of raw CPU or memory), you ensure your on-call team is paged about things that actually matter to users.
