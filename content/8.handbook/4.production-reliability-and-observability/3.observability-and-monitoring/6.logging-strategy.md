# Logging Strategy

Structured, consistent logging is the backbone of operational visibility. Without it, diagnosing production incidents requires guesswork instead of evidence. A good logging strategy ensures that the right information is captured at the right level, in a machine-parseable format, without overwhelming log storage or exposing sensitive data.

## 1. Log Levels

Use standardized log levels to categorize the severity and purpose of each log entry.

| Level | When to Use | Example |
|---|---|---|
| `DEBUG` | Detailed dev-only information. Never in production. | Variable states inside a loop |
| `INFO` | Normal operational events. High-level flow tracking. | "User logged in", "Order processed" |
| `WARN` | Potential issues that don't require immediate action. | "Retry attempt 2/3", "Cache miss" |
| `ERROR` | A failure that requires investigation. | API call failed, database unreachable |
| `FATAL` | System is in an unrecoverable state. | Server startup failed |

**Rule:** Production logs should contain only `INFO`, `WARN`, `ERROR`, and `FATAL` entries. `DEBUG` logs must be disabled in production.

## 2. Structured Logging (JSON Format)

Always log in **JSON format** in production. Plain text logs are human-readable but cannot be efficiently parsed, filtered, or aggregated by log management tools (Datadog, Loki, CloudWatch).

```typescript
// utils/logger.ts (Server-side Nitro logger)
export const logger = {
  info: (message: string, context?: Record<string, unknown>) => {
    console.log(JSON.stringify({
      level: 'info',
      message,
      timestamp: new Date().toISOString(),
      service: 'nuxt-app',
      ...context
    }))
  },
  error: (message: string, error?: Error, context?: Record<string, unknown>) => {
    console.error(JSON.stringify({
      level: 'error',
      message,
      timestamp: new Date().toISOString(),
      service: 'nuxt-app',
      error: {
        name: error?.name,
        message: error?.message,
        stack: error?.stack,
      },
      ...context
    }))
  }
}
```

## 3. What to Log (And What Not To)

### ✅ Always Log
- User authentication events (login, logout, failed attempts).
- Critical business events (order placed, payment received, account deleted).
- All `ERROR` and `WARN` level events.
- Request/response summary for server API routes (method, route, status code, duration—NOT body).

### ❌ Never Log (PII & Secrets)
- Passwords, tokens, API keys, secrets.
- Credit card numbers, CVVs, bank account numbers.
- Full request/response bodies (they may contain sensitive data).
- Health data (for HIPAA compliance).
- Personally Identifiable Information (PII) unless explicitly required and anonymized.

```typescript
// ❌ BAD: Logging sensitive data
logger.info('User login', { email: user.email, password: req.body.password })

// ✅ GOOD: Log only non-sensitive identifiers
logger.info('User login', { userId: user.id, method: 'email' })
```

## 4. Correlation IDs for Request Tracing

When a user action triggers multiple server calls, use a **Correlation ID** (or Request ID) to link all related log entries together. Without this, tracing a complex flow across multiple log entries is extremely difficult.

```typescript
// server/middleware/correlation-id.ts
import { randomUUID } from 'crypto'

export default defineEventHandler((event) => {
  const correlationId = getRequestHeader(event, 'X-Correlation-ID') ?? randomUUID()
  
  // Attach to the event so all handlers in this request can use it
  event.context.correlationId = correlationId
  
  // Return in the response header so clients can include it in support tickets
  setResponseHeader(event, 'X-Correlation-ID', correlationId)
})
```

```typescript
// server/api/orders.post.ts
export default defineEventHandler(async (event) => {
  const { correlationId } = event.context
  
  logger.info('Order submission started', { correlationId, userId: event.context.userId })
  
  try {
    const order = await processOrder(await readBody(event))
    logger.info('Order created successfully', { correlationId, orderId: order.id })
    return order
  } catch (error) {
    logger.error('Order creation failed', error, { correlationId })
    throw error
  }
})
```

## 5. Log Management Infrastructure

| Tool | Best For |
|---|---|
| **Datadog Logs** | Enterprise teams needing unified APM + Logs + Metrics |
| **Grafana Loki** | Open-source teams with existing Grafana infrastructure |
| **AWS CloudWatch** | Teams deployed on AWS |
| **Logtail / BetterStack** | Cost-effective, modern logging for indie/startup teams |
