# Error Tracking with Sentry

Error tracking is the foundation of production observability. Without it, you are effectively flying blind—bugs that affect your users go undetected until someone reports them. Sentry is the industry-standard tool for capturing, grouping, and alerting on runtime errors in Nuxt applications.

## 1. Installing Sentry in Nuxt 3

The official Sentry Nuxt SDK (`@sentry/nuxt`) provides seamless integration with Nuxt's SSR pipeline, capturing errors from both client-side Vue rendering and server-side Nitro routes.

```bash
pnpm add @sentry/nuxt
```

Run the Sentry wizard for guided setup:
```bash
npx @sentry/wizard@latest -i nuxt
```

The wizard will:
- Create `sentry.client.config.ts` and `sentry.server.config.ts`.
- Add `@sentry/nuxt` as a module to `nuxt.config.ts`.
- Create source map upload configuration.

## 2. Client-Side Configuration

```typescript
// sentry.client.config.ts
import * as Sentry from '@sentry/nuxt'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  
  // Capture 10% of all transactions for performance monitoring
  tracesSampleRate: 0.1,
  
  // Only enable in production
  enabled: process.env.NODE_ENV === 'production',
  
  // Track the release version (helps pinpoint when a bug was introduced)
  release: process.env.NUXT_PUBLIC_APP_VERSION,
  
  // Define environments
  environment: process.env.NODE_ENV,

  integrations: [
    // Automatically capture Vue component errors with context
    Sentry.vueIntegration({ tracingOptions: { trackComponents: true } }),
    
    // Capture route changes as transactions
    Sentry.browserTracingIntegration(),
  ],

  // Scrub sensitive data before sending to Sentry
  beforeSend(event) {
    if (event.request?.cookies) {
      delete event.request.cookies
    }
    return event
  }
})
```

## 3. Server-Side Configuration

```typescript
// sentry.server.config.ts
import * as Sentry from '@sentry/nuxt'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  tracesSampleRate: 0.1,
  enabled: process.env.NODE_ENV === 'production',

  integrations: [
    // Automatically trace Nitro server requests
    Sentry.nitroIntegration(),
  ]
})
```

## 4. Capturing Errors Manually

For errors that are caught and handled (not automatically propagated to Sentry), capture them manually.

```typescript
// In a Pinia store action
import * as Sentry from '@sentry/nuxt'

const processPayment = async (payload: PaymentPayload) => {
  try {
    await $fetch('/api/payments', { method: 'POST', body: payload })
  } catch (error) {
    // Attach useful context before sending to Sentry
    Sentry.captureException(error, {
      tags: { feature: 'checkout' },
      extra: { userId: authStore.userId, orderId: payload.orderId }
    })
    
    // Show user-friendly message
    errorMessage.value = 'Payment processing failed.'
  }
}
```

## 5. User Context for Error Triage

Attaching user information to Sentry errors dramatically improves triage speed. You can tell immediately "how many users are affected" and look up affected users.

```typescript
// plugins/sentry-user-context.ts
export default defineNuxtPlugin(() => {
  const authStore = useAuthStore()
  
  watchEffect(() => {
    if (authStore.user) {
      Sentry.setUser({
        id: authStore.user.id,
        email: authStore.user.email,
        // Never include passwords or payment info!
      })
    } else {
      Sentry.setUser(null) // Clear on logout
    }
  })
})
```

## 6. Source Maps for Readable Stack Traces

Without source maps, Sentry shows minified stack traces like `e.u(n,t)` instead of `useCheckout.ts line 42`. Configure source map uploads in your CI pipeline.

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  modules: ['@sentry/nuxt/module'],
  sentry: {
    sourceMapsUploadOptions: {
      org: 'your-org',
      project: 'your-project',
      authToken: process.env.SENTRY_AUTH_TOKEN
    }
  }
})
```

## 7. Alert Configuration

In the Sentry dashboard, configure **Issue Alerts** to notify your team:
- **New Issue Alert:** Notify on Slack/email when a new, previously-unseen error type occurs.
- **Regression Alert:** Notify when a previously resolved issue reappears (typically after a bad deployment).
- **Error Spike Alert:** Notify when error frequency increases by >200% over the past hour.
