# Session Replay

Session replay is a feature that records users' browser interactions—mouse movements, clicks, scrolls, and DOM mutations—and allows you to play them back as a video-like recording. It provides the critical missing context that error logs and stack traces cannot: *what the user was actually doing before the error occurred*.

Session replay is **optional** and should be adopted deliberately, as it has privacy implications and a performance cost.

## 1. When to Use Session Replay

Session replay is most valuable when:
- Users report confusing bugs that are impossible to reproduce from a description alone.
- You want to understand why users are rage-clicking on a non-interactive element.
- You need to identify UX confusion points (e.g., why users repeatedly abandon a specific form step).
- A critical error is intermittent and its reproduction steps are unclear.

## 2. Privacy Considerations (Non-Negotiable)

Session replay records everything the user sees. This creates significant privacy obligations.

**Mandatory Protections:**
- **Input Masking:** All password, credit card, and sensitive form fields must be masked by default.
- **Network Request Scrubbing:** Ensure API response bodies are not recorded.
- **Consent:** Inform users via your Privacy Policy that session recordings may be collected. For GDPR compliance in the EU, consider requiring explicit opt-in consent before activating replay.
- **Data Residency:** Be aware of where session data is stored (EU vs. US servers).

## 3. Sentry Session Replay Integration

Sentry offers Session Replay as part of its SDK, making it easy to activate alongside error tracking.

```typescript
// sentry.client.config.ts
import * as Sentry from '@sentry/nuxt'
import { Replay } from '@sentry/nuxt'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  
  integrations: [
    new Replay({
      // Mask ALL text content (maximum privacy)
      maskAllText: true,
      
      // Block all images from being captured
      blockAllMedia: true,
      
      // Mask all form inputs
      maskAllInputs: true,
    }),
  ],
  
  // Record 1% of all sessions (background sampling)
  replaysSessionSampleRate: 0.01,
  
  // Record 100% of sessions where an ERROR occurred
  // This is the most valuable use case
  replaysOnErrorSampleRate: 1.0,
})
```

### Sampling Strategy
The recommended sampling strategy is:
- **Low background rate (1-5%):** For understanding general user behavior and UX flows.
- **100% on error:** Always capture a replay when a Sentry error is triggered. This is invaluable for bug reproduction.

## 4. Alternative Tools

| Tool | Best For |
|---|---|
| **Sentry Replay** | Developers who already use Sentry for error tracking |
| **LogRocket** | Product teams needing advanced UX analytics + replay |
| **FullStory** | Enterprise product analytics with powerful DX data querying |
| **PostHog** | Open-source teams wanting self-hosted replay + analytics |

## 5. Performance Impact

Session replay works by serializing the DOM and network activity. It has a non-trivial client-side overhead. Mitigate this by:
- Using low background sampling rates (< 5%).
- Enabling replay only in production (not in development).
- Using `replaysOnErrorSampleRate` as the primary activation mechanism rather than `replaysSessionSampleRate`.
- Verifying your Core Web Vitals (LCP, INP) are not degraded after enabling replay in production.

## Summary

Session replay is a powerful debugging tool, not a mass surveillance system. Implement it with the minimum sampling rate that serves your debugging needs, apply all available privacy masking features, and document its usage clearly in your team's privacy policy.
