# Dashboard Creation

Dashboards transform raw observability data (metrics, logs, traces) into a shared visual understanding of your system's health. A well-designed dashboard lets an engineer, an on-call responder, or an engineering manager answer the question "Is everything okay?" in under 10 seconds.

## 1. Dashboard Design Principles

### Principle 1: One Dashboard Per Audience
Do not create one giant dashboard that tries to serve every role. Build targeted dashboards for specific audiences.

| Dashboard | Primary Audience | Focus |
|---|---|---|
| **Executive Overview** | VP Eng, CTO | Uptime, Revenue metrics, SLO compliance |
| **Service Health** | On-call engineers | Error rate, Latency P50/P95, Throughput |
| **Frontend Performance** | Frontend team | Core Web Vitals, JS error rate, INP by page |
| **Checkout Funnel** | Product + Engineering | Conversion, cart abandonment, payment errors |

### Principle 2: Show the Golden Signals First
Place the four Golden Signals (Latency, Traffic, Errors, Saturation) at the top of every operational dashboard. These are the first things an on-call engineer checks.

### Principle 3: Use Time Comparisons
Every metric panel should include a comparison to the same time period last week (or last day). A raw number is meaningless without context. "Error rate: 0.8%" is fine. "Error rate: 0.8% (up from 0.2% last week)" demands immediate attention.

## 3. Building Dashboards in Grafana

Grafana is the most widely adopted open-source dashboarding tool and integrates with virtually every data source (Prometheus, Loki, Datadog, Sentry, InfluxDB).

### Connecting to Prometheus / Datadog
Configure your data sources in Grafana's `provisioning/datasources` directory for infrastructure-as-code dashboards.

### Dashboard as Code (JSON)
Store your Grafana dashboard definitions as JSON files in your repository. This enables version control, code review, and automated provisioning.

```bash
# Export an existing dashboard as JSON
curl -H "Authorization: Bearer $GRAFANA_TOKEN" \
  https://grafana.yourcompany.com/api/dashboards/uid/service-health \
  | jq '.dashboard' > dashboards/service-health.json
```

Commit this file to your repository under `infra/dashboards/`. Use Grafana's provisioning system or Terraform to automatically import it into new environments.

## 4. Sentry Dashboards

For teams using Sentry, the built-in dashboards are the fastest way to get started without setting up a separate visualization tool.

**Recommended Sentry Widgets:**
- **Issues by Frequency:** Top errors sorted by occurrence count.
- **Error Rate over Time:** A time-series of new errors vs. resolved issues.
- **Performance by Transaction:** Identify the slowest Nuxt pages/routes.
- **Apdex Score:** A combined performance score based on your latency target.

## 5. Dashboard Maintenance

- **Review dashboards quarterly:** Remove panels nobody reads. Add panels for new features.
- **Add annotations for deployments:** Automatically add a vertical marker on time-series graphs when a new version is deployed. This makes performance regressions immediately visually obvious.
- **Link to runbooks:** Add a description or link on every dashboard panel pointing to the relevant runbook for that metric.
