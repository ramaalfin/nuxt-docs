# XSS Prevention & Input Sanitization

Cross-Site Scripting (XSS) is one of the most prevalent web security vulnerabilities. It occurs when an attacker injects malicious scripts into content that is then rendered and executed in another user's browser—allowing them to steal session tokens, redirect users, or deface the UI.

## 1. How Vue/Nuxt Prevents XSS by Default

The best defense against XSS in a Vue/Nuxt application is the framework itself. Vue automatically HTML-escapes all dynamic bindings by default.

```vue
<template>
  <!-- SAFE: Vue escapes the content automatically -->
  <!-- If userInput = '<script>alert("xss")</script>' -->
  <!-- It renders as the literal text, not as a script tag -->
  <p>{{ userInput }}</p>
  <div :title="userInput" />
</template>
```

**The golden rule: Never use `v-html` with user-controlled content.**

`v-html` bypasses Vue's built-in escaping and renders raw HTML directly into the DOM. It is the most common source of XSS vulnerabilities in Vue applications.

```vue
<!-- ❌ NEVER DO THIS with user content -->
<div v-html="userProvidedBio" />

<!-- ✅ SAFE: Use text binding instead -->
<div>{{ userProvidedBio }}</div>
```

## 2. When You Must Use `v-html`

If you have a legitimate use case for `v-html` (e.g., rendering a Markdown editor's output, or a rich-text CMS field), you **must** sanitize the HTML before rendering it.

### Using DOMPurify

`DOMPurify` is the gold standard HTML sanitization library. It strips all malicious HTML while preserving safe, whitelisted tags.

```bash
pnpm add dompurify
pnpm add -D @types/dompurify
```

```typescript
// utils/sanitize.ts
import DOMPurify from 'dompurify'

export function sanitizeHtml(dirty: string): string {
  // Only allow a safe subset of HTML tags and attributes
  return DOMPurify.sanitize(dirty, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'ul', 'ol', 'li', 'br'],
    ALLOWED_ATTR: ['href', 'target', 'rel'],
    // Force all links to be safe
    FORCE_BODY: false,
  })
}
```

```vue
<script setup lang="ts">
import { sanitizeHtml } from '~/utils/sanitize'

const props = defineProps<{ rawHtml: string }>()
const safeHtml = computed(() => sanitizeHtml(props.rawHtml))
</script>

<template>
  <!-- Now safe to use v-html because it's been sanitized -->
  <div v-html="safeHtml" />
</template>
```

*Note: DOMPurify runs in the browser. For server-side sanitization in Nuxt's Nitro layer, use `isomorphic-dompurify` or `sanitize-html`.*

## 3. Input Validation at the API Layer

Input sanitization is a defense-in-depth strategy. Validate and sanitize all user input on the server side as well, never trust the client.

```typescript
// server/api/post.post.ts
import { z } from 'zod'
import sanitizeHtml from 'sanitize-html' // Server-safe library

const CreatePostSchema = z.object({
  title: z.string().min(1).max(200).trim(),
  content: z.string().max(50000),
})

export default defineEventHandler(async (event) => {
  const body = await readValidatedBody(event, CreatePostSchema.parse)
  
  // Sanitize rich text content on the server before storing
  const sanitizedContent = sanitizeHtml(body.content, {
    allowedTags: sanitizeHtml.defaults.allowedTags,
    allowedAttributes: { 'a': ['href', 'rel'] }
  })
  
  await db.posts.create({ title: body.title, content: sanitizedContent })
  return { success: true }
})
```

## 4. URL Sanitization

Attackers can use `javascript:` URLs in links to execute XSS through user-controlled `href` attributes.

```typescript
// utils/sanitize.ts
export function sanitizeUrl(url: string): string {
  const allowedProtocols = ['http:', 'https:', 'mailto:']
  
  try {
    const parsed = new URL(url)
    if (!allowedProtocols.includes(parsed.protocol)) {
      return '#' // Replace dangerous URLs with a safe fallback
    }
    return url
  } catch {
    return '#' // Invalid URL
  }
}
```

```vue
<a :href="sanitizeUrl(userProvidedLink)">Click here</a>
```

## 5. Content Security Policy (See CSP Chapter)

A Content Security Policy (CSP) HTTP header acts as a last line of XSS defense by instructing browsers to refuse to execute inline scripts or load scripts from unauthorized origins, even if an XSS attack successfully injects a `<script>` tag into the DOM.

Always implement CSP alongside input sanitization.
