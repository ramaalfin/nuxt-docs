# Security Headers

HTTP security headers are response headers that instruct the browser to enable or enforce specific security mechanisms. They are one of the easiest, highest-value security improvements you can make—often just a few lines of configuration that protect against entire classes of attacks.

Think of security headers as a security hardening layer on top of your application code. They do not replace secure coding practices, but they provide defense-in-depth.

## 1. Essential Security Headers

### `Strict-Transport-Security` (HSTS)
Forces browsers to use HTTPS exclusively. See the [HTTPS Enforcement chapter](./5.https-enforcement.md) for full details.
```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

### `Content-Security-Policy` (CSP)
Defines allowed content sources to prevent XSS. See the [CSP chapter](./3.content-security-policy.md) for full details.
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{nonce}'
```

### `X-Frame-Options`
Prevents your page from being embedded in an `<iframe>`, which is the core mechanism of **Clickjacking attacks**.
```
X-Frame-Options: DENY
```
*Note: CSP's `frame-ancestors 'none'` directive is the modern replacement for this header. Use both for maximum compatibility.*

### `X-Content-Type-Options`
Prevents browsers from "MIME-type sniffing"—interpreting a response as a different content type than declared. This prevents attacks where a malicious file uploaded as an image is executed as JavaScript.
```
X-Content-Type-Options: nosniff
```

### `Referrer-Policy`
Controls how much information about the origin URL is included in the `Referer` request header when navigating to external links. Prevents leaking internal URL paths (which may contain sensitive query parameters) to third-party sites.
```
Referrer-Policy: strict-origin-when-cross-origin
```

### `Permissions-Policy` (formerly Feature-Policy)
Allows you to selectively enable or disable browser features (camera, microphone, geolocation) for your site and its embedded iframes.
```
Permissions-Policy: camera=(), microphone=(), geolocation=(self), payment=(self)
```
The `()` means the feature is disabled entirely. `(self)` means only your own origin can use it.

### `Cross-Origin-Opener-Policy` (COOP)
Prevents other origins from gaining a reference to your `window` object, which is required for certain Spectre-class CPU attacks.
```
Cross-Origin-Opener-Policy: same-origin
```

### `Cross-Origin-Resource-Policy` (CORP)
Prevents other sites from embedding your resources (images, scripts) in their pages.
```
Cross-Origin-Resource-Policy: same-origin
```

## 2. Implementing Headers in Nuxt (`nuxt-security` module)

The `nuxt-security` module provides sensible defaults for all security headers with a single installation.

```bash
pnpm add nuxt-security
```

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  modules: ['nuxt-security'],
  security: {
    headers: {
      // Automatically sets all recommended security headers
      crossOriginEmbedderPolicy: 'require-corp',
      crossOriginOpenerPolicy: 'same-origin',
      crossOriginResourcePolicy: 'same-origin',
      referrerPolicy: 'strict-origin-when-cross-origin',
      xContentTypeOptions: 'nosniff',
      xFrameOptions: 'DENY',
      xXSSProtection: '0', // Disabled (modern browsers don't need it and it can cause issues)
      permissionsPolicy: {
        camera: [],
        microphone: [],
        geolocation: [],
        payment: ['self']
      }
    }
  }
})
```

## 3. Verifying Your Security Headers

Use the following tools to audit your headers in staging/production:

| Tool | URL |
|---|---|
| **Security Headers** | `https://securityheaders.com` |
| **Mozilla Observatory** | `https://observatory.mozilla.org` |
| **SSL Labs** | `https://ssllabs.com/ssltest` (for TLS configuration) |

Target a **Grade A or A+** on all of these tools before going to production.

## 4. Summary: Minimum Required Headers

Every production Nuxt application must have at minimum:

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: [appropriate policy for your app]
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

These six headers protect against HTTPS downgrade, XSS, MIME sniffing, clickjacking, referrer leakage, and unauthorized feature usage—with essentially zero performance cost.
