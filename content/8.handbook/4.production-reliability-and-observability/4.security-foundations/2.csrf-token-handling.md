# CSRF Token Handling

Cross-Site Request Forgery (CSRF) is an attack that tricks an authenticated user into unknowingly submitting a malicious request to a web application they are already logged into. Because the user's browser automatically sends cookies (including session cookies) with every request, the server cannot distinguish a legitimate request from a forged one without additional protection.

## 1. Understanding the CSRF Attack

1. A user logs into `bank.com`. Their browser stores a session cookie.
2. The user visits `evil.com` in another tab.
3. `evil.com` contains `<img src="https://bank.com/transfer?to=attacker&amount=10000">`.
4. The browser makes the request to `bank.com` and **automatically includes the session cookie**.
5. `bank.com` sees a valid session and executes the transfer.

## 2. The Modern Defense: SameSite Cookies

The most effective and modern defense against CSRF is the `SameSite` cookie attribute. Setting `SameSite=Strict` or `SameSite=Lax` prevents the browser from including the session cookie in cross-site requests.

```typescript
// server/api/auth/login.post.ts
export default defineEventHandler(async (event) => {
  // ... authenticate user ...
  
  setCookie(event, 'session_token', sessionToken, {
    httpOnly: true,       // Not accessible via JavaScript
    secure: true,         // HTTPS only
    sameSite: 'strict',   // NEVER sent in cross-site requests (strongest protection)
    path: '/',
    maxAge: 60 * 60 * 24  // 24 hours
  })
})
```

**`SameSite` values:**
| Value | Behavior | Use Case |
|---|---|---|
| `strict` | Cookie never sent on cross-site requests | Maximum security, may break OAuth flows |
| `lax` | Cookie sent on top-level navigation (GET), not on embedded requests | Balanced security (default in modern browsers) |
| `none` | Always sent (must also set `Secure: true`) | Required for cross-site embeds/iframes |

## 3. CSRF Tokens (Double Submit Cookie Pattern)

For APIs that cannot rely solely on `SameSite` cookies (e.g., legacy browser support, or when using `SameSite=None` for specific cross-origin flows), implement the **Synchronizer Token Pattern** or the **Double Submit Cookie Pattern**.

### Implementation in Nuxt with H3 (Nitro)

```typescript
// server/utils/csrf.ts
import { randomBytes } from 'crypto'

export const generateCsrfToken = (): string => {
  return randomBytes(32).toString('hex')
}

export const validateCsrfToken = (event: H3Event): void => {
  const cookieToken = getCookie(event, 'csrf_token')
  const headerToken = getRequestHeader(event, 'X-CSRF-Token')
  
  if (!cookieToken || !headerToken || cookieToken !== headerToken) {
    throw createError({ statusCode: 403, statusMessage: 'Invalid CSRF token' })
  }
}
```

```typescript
// server/api/auth/session.get.ts
// Endpoint to get a CSRF token for the client to use
export default defineEventHandler((event) => {
  const csrfToken = generateCsrfToken()
  
  // Set as a cookie (JS-readable, so NOT httpOnly)
  setCookie(event, 'csrf_token', csrfToken, { 
    sameSite: 'strict', 
    secure: true,
    httpOnly: false // Must be readable by JS to send as header
  })
  
  return { csrfToken }
})
```

```typescript
// composables/useApiFetch.ts
// Automatically include the CSRF token in all mutating requests
export function useApiFetch(url: string, options: FetchOptions = {}) {
  const csrfToken = useCookie('csrf_token')
  
  return $fetch(url, {
    ...options,
    headers: {
      ...options.headers,
      // Include in header for all POST/PUT/PATCH/DELETE requests
      'X-CSRF-Token': csrfToken.value ?? '',
    }
  })
}
```

## 4. CSRF Protection for Nuxt API Routes

Add CSRF validation as Nitro middleware for all state-mutating routes.

```typescript
// server/middleware/csrf-protection.ts
export default defineEventHandler((event) => {
  const method = getMethod(event)
  const mutatingMethods = ['POST', 'PUT', 'PATCH', 'DELETE']
  
  if (mutatingMethods.includes(method)) {
    validateCsrfToken(event) // Throws 403 if invalid
  }
})
```

## 5. When CSRF Protection is Not Needed

CSRF is only a concern for cookie-based authentication. If your API uses **Authorization header tokens** (e.g., `Bearer <JWT>` sent explicitly by JavaScript), CSRF is not a threat because browsers never automatically include `Authorization` headers in cross-site requests.

Most modern SPAs using JWT in `localStorage` are inherently CSRF-safe—though they have different vulnerabilities (XSS can steal localStorage tokens, hence why `httpOnly` cookies are still preferable for session storage).
