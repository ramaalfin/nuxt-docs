# Secure Cookie Attributes

Cookies are the primary mechanism for maintaining authenticated sessions in web applications. Misconfigured cookies are a critical security risk—exposing them to XSS, CSRF, or man-in-the-middle attacks. Setting the correct attributes on every cookie is non-negotiable in production.

## 1. The Complete Security Attribute Checklist

For every session or authentication cookie, verify all of the following attributes are set:

| Attribute | Purpose | Required Value |
|---|---|---|
| `HttpOnly` | Prevents JavaScript access via `document.cookie` | ✅ Always `true` for session cookies |
| `Secure` | Cookie only sent over HTTPS, never plain HTTP | ✅ Always `true` in production |
| `SameSite` | Controls cross-site sending behavior (CSRF protection) | ✅ `Strict` or `Lax` |
| `Path` | Restrict cookie scope to specific paths | `/` or narrower path |
| `Domain` | Restrict cookie scope to specific domains | Set explicitly |
| `Max-Age` / `Expires` | Define cookie lifetime | Shortest acceptable duration |
| `__Host-` prefix | Forces `Secure`, no `Domain`, `Path=/` | Recommended for session cookies |

## 2. Implementation in Nuxt/Nitro

```typescript
// server/api/auth/login.post.ts
export default defineEventHandler(async (event) => {
  const { email, password } = await readBody(event)
  
  const user = await authenticateUser(email, password)
  if (!user) {
    throw createError({ statusCode: 401, statusMessage: 'Invalid credentials' })
  }
  
  const sessionToken = await createSession(user.id)
  
  // ✅ SECURE: All critical attributes set
  setCookie(event, '__Host-session', sessionToken, {
    httpOnly: true,                      // No JS access
    secure: true,                        // HTTPS only
    sameSite: 'strict',                  // CSRF protection
    path: '/',                           // Required with __Host- prefix
    maxAge: 60 * 60 * 24 * 7,           // 7 days (in seconds)
    // Note: Do NOT set 'domain' with __Host- prefix
  })
  
  return { success: true }
})
```

## 3. The `__Host-` and `__Secure-` Cookie Prefixes

These special prefixes provide additional browser-enforced security guarantees:

### `__Host-` Prefix (Strongest)
A cookie named `__Host-session` must:
- Have `Secure` attribute set.
- Not have a `Domain` attribute (cannot be sent to subdomains).
- Have `Path=/` (broadest path, no path restriction exploitation).

This is the most secure option for session cookies in a single-domain application.

### `__Secure-` Prefix (Moderate)
A cookie named `__Secure-token` must:
- Have `Secure` attribute set.

Weaker than `__Host-` as it still allows `Domain` and `Path` attributes.

## 4. What to Never Store in Cookies

Cookies are sent with every request. Keep them small and store only the minimum necessary data.

```
❌ NEVER store in cookies:
- Passwords (even hashed)
- Credit card numbers
- Sensitive PII (date of birth, SSN)
- Large data objects (put them in the server session store, store only the session ID in the cookie)

✅ SAFE to store in cookies:
- Opaque session identifiers (random, server-side session ID)
- JWT access tokens (controversial — see below)
- Preferences (theme, language)
```

## 5. JWT in Cookies vs. localStorage

This is a common debate:

| Storage | XSS Risk | CSRF Risk | Recommendation |
|---|---|---|---|
| `localStorage` | ❌ High (accessible by any JS) | ✅ None (not sent automatically) | Avoid for session tokens |
| `httpOnly` Cookie | ✅ None (not accessible by JS) | ❌ Requires mitigation (SameSite / CSRF tokens) | ✅ **Preferred for session tokens** |

**Recommendation:** Store session JWTs in `httpOnly` cookies with `SameSite=Strict`. Mitigate CSRF with the `SameSite` attribute (modern browsers) and a CSRF token (legacy browsers).

## 6. Cookie Expiry Strategy

Short-lived cookies reduce the window of opportunity if a cookie is stolen:

| Cookie Type | Recommended `maxAge` |
|---|---|
| Session (auth) | 1-7 days (with silent refresh) |
| Remember Me | 30 days |
| CSRF Token | Same as session |
| Preferences | 1 year |

Always clear all auth-related cookies on explicit logout:

```typescript
// server/api/auth/logout.post.ts
export default defineEventHandler((event) => {
  // Overwrite with an expired cookie to force deletion
  deleteCookie(event, '__Host-session')
  deleteCookie(event, 'csrf_token')
  return { success: true }
})
```
