# HTTPS Enforcement

HTTPS (HTTP over TLS) encrypts all communication between the browser and your server, preventing man-in-the-middle attacks, eavesdropping, and content tampering. In 2024, HTTPS is a baseline requirement—not a nice-to-have. Every Nuxt application in production must enforce HTTPS at every layer.

## 1. Why Plain HTTP is Unacceptable

Without HTTPS:
- Session cookies sent over HTTP can be intercepted and hijacked (**session hijacking**).
- Attackers on the same network (coffee shop Wi-Fi) can read or modify every request and response (**packet sniffing**).
- ISPs and middleboxes can inject ads or malicious scripts into your pages.
- Modern browser features (Service Workers, WebCrypto, Geolocation) are disabled without HTTPS.
- Google penalizes non-HTTPS sites in search rankings.

## 2. HTTPS Enforcement Layers

HTTPS enforcement should happen at **every layer** of your infrastructure. A weakness at any one layer can undermine the others.

### Layer 1: DNS / CDN Level (Cloudflare, AWS CloudFront)
Configure your CDN to:
- Automatically redirect all HTTP traffic to HTTPS (301 redirect).
- Enable "Full (Strict)" SSL mode so the CDN also enforces HTTPS to your origin server.

### Layer 2: Reverse Proxy / Load Balancer (Nginx, Caddy)
```nginx
# nginx.conf — Force redirect HTTP to HTTPS
server {
  listen 80;
  server_name yourapp.com;
  return 301 https://$host$request_uri;
}

server {
  listen 443 ssl;
  # ... SSL certificate configuration
}
```

**Caddy** automatically provisions TLS certificates via Let's Encrypt and enforces HTTPS by default with zero configuration.

### Layer 3: Application Level (Nuxt)
```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  // In production, redirect HTTP to HTTPS via the Nitro server
  nitro: {
    routeRules: {
      '/**': {
        headers: {
          // Enforce HTTPS for all routes
          'Strict-Transport-Security': 'max-age=31536000; includeSubDomains; preload'
        }
      }
    }
  }
})
```

## 3. HTTP Strict Transport Security (HSTS)

HSTS is an HTTP header that instructs browsers to **only** connect to your domain over HTTPS for a specified duration—even if the user types `http://` in the address bar.

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

| Directive | Meaning |
|---|---|
| `max-age=31536000` | Remember this rule for 1 year (in seconds) |
| `includeSubDomains` | Apply to all subdomains too (e.g., `api.yourapp.com`) |
| `preload` | Opt into browser's HSTS preload list (hardcoded into Chrome/Firefox) |

**⚠️ Warning:** Once HSTS is deployed with `preload`, it is **very difficult to undo** (requires waiting for `max-age` to expire for all users, and HSTS preload list removal takes months). Start with a short `max-age` (e.g., `max-age=86400` for 1 day) and gradually increase it.

### HSTS Preload

Submit your domain to [https://hstspreload.org](https://hstspreload.org) to be hardcoded into browsers, providing HTTPS enforcement for first-time visitors (before they ever receive the HSTS header).

**Requirements:** `max-age >= 31536000`, `includeSubDomains`, and `preload` must all be set.

## 4. TLS Configuration Best Practices

### Certificate Management
- Use **Let's Encrypt** for free, auto-renewing certificates on self-managed servers.
- Use **AWS ACM**, **GCP-managed SSL**, or **Cloudflare** for cloud-managed certificates.
- Set up certificate expiry alerts at 30 days before expiry. An expired certificate causes complete site unavailability.

### TLS Version
- **Disable TLS 1.0 and 1.1** — they are deprecated and contain known vulnerabilities.
- **Require TLS 1.2 or higher** — TLS 1.3 is preferred for its improved security and reduced handshake latency.

### Cipher Suites
Use a modern, forward-secrecy-enabled cipher suite configuration. The Mozilla SSL Configuration Generator at `ssl-config.mozilla.org` provides ready-to-use, security-vetted configurations for Nginx, Apache, and Caddy.

## 5. Mixed Content

Mixed content occurs when an HTTPS page loads resources (images, scripts, styles) over HTTP. Browsers block or warn on mixed content.

```typescript
// Always use protocol-relative or absolute HTTPS URLs
// ❌ BAD
const imageUrl = 'http://static.example.com/logo.png'

// ✅ GOOD
const imageUrl = 'https://static.example.com/logo.png'

// ✅ ALSO GOOD (protocol-relative)
const imageUrl = '//static.example.com/logo.png'
```

Use CSP's `upgrade-insecure-requests` directive to automatically upgrade HTTP sub-resource requests to HTTPS:
```
Content-Security-Policy: upgrade-insecure-requests;
```
