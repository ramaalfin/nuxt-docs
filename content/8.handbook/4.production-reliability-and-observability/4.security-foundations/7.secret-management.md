# Secret Management

Secrets are credentials that grant access to protected resources: API keys, database passwords, JWT signing keys, OAuth client secrets, and encryption keys. A leaked secret can result in a complete data breach, financial fraud, or service disruption.

The fundamental rule: **secrets must never exist in your codebase, version control history, or client-side JavaScript bundle.**

## 1. The Most Common Mistakes

### ❌ Hardcoded Secrets (Never Do This)
```typescript
// Catastrophically bad — GitHub bots scan for these patterns within seconds of a push
const stripe = new Stripe('your_stripe_api_key_here')
const dbPassword = 'MyProductionPassword123!'
```

### ❌ Committing `.env` Files
```bash
# .gitignore — ENSURE these are listed
.env
.env.local
.env.production
.env.*.local
```

A `.env` file committed even once—then later deleted—still exists in the Git history. Use `git-secrets` or `truffleHog` to scan your entire Git history for exposed secrets.

### ❌ Exposing Server Secrets to the Client
In Nuxt, any variable prefixed with `NUXT_PUBLIC_` is bundled into the client-side JavaScript and visible to all users.

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  runtimeConfig: {
    // ✅ Server-only (not exposed to the client)
    stripeSecretKey: '',  // Read from NUXT_STRIPE_SECRET_KEY env var
    dbPassword: '',

    public: {
      // ⚠️ Exposed to the client — ONLY put truly public values here
      apiBaseUrl: '',     // Read from NUXT_PUBLIC_API_BASE_URL env var
      stripePublishableKey: '', // This is safe — Stripe publishable keys are meant to be public
    }
  }
})
```

## 2. The Correct Pattern: Environment Variables

All secrets must be provided to the application via **environment variables**, injected at runtime by the deployment environment (not baked into the build artifact).

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  runtimeConfig: {
    stripeSecretKey: '',  // Populated from process.env.NUXT_STRIPE_SECRET_KEY at startup
    jwtSigningKey: '',
    databaseUrl: '',
  }
})
```

```typescript
// server/api/payments.post.ts
export default defineEventHandler(async (event) => {
  const config = useRuntimeConfig()
  
  // ✅ Accessed securely on the server side only
  const stripe = new Stripe(config.stripeSecretKey)
})
```

## 3. Secret Managers (Production Standard)

For production systems, environment variables alone are insufficient. A secret manager provides:
- **Centralized storage** with fine-grained access control.
- **Audit logs** of who accessed which secret and when.
- **Secret rotation** without redeployment.
- **Versioning** for rollbacks.

### Popular Secret Managers

| Service | Best For |
|---|---|
| **HashiCorp Vault** | Self-hosted, maximum control, complex enterprise needs |
| **AWS Secrets Manager** | Teams on AWS infrastructure |
| **Google Secret Manager** | Teams on GCP |
| **Azure Key Vault** | Teams on Azure |
| **Doppler** | SaaS, simple multi-environment management, excellent DX |
| **Infisical** | Open-source Doppler alternative |

### Integrating with Doppler (Example)
```bash
# Install Doppler CLI
brew install dopplerhq/cli/doppler

# Inject secrets at runtime — no .env files needed
doppler run -- node .output/server/index.mjs
```

In CI/CD, use the secret manager's native GitHub Actions integration instead of storing secrets in GitHub Secrets directly.

## 4. Secret Rotation

All secrets should have a defined rotation schedule:

| Secret Type | Rotation Frequency |
|---|---|
| Database passwords | Every 90 days |
| API keys (third-party) | Every 90 days or on team member offboarding |
| JWT signing keys | Every 30 days (with a transition window) |
| Service account keys | Every 6 months |
| SSL/TLS certificates | Auto-renewed by Let's Encrypt (every 90 days) |

Use your secret manager's built-in rotation automation where possible. Manual rotation is error-prone and will be skipped.

## 5. Scanning for Leaked Secrets

Integrate secret scanning into your CI pipeline to catch accidental commits before they reach remote:

```bash
# Pre-commit hook using gitleaks
brew install gitleaks
gitleaks protect --staged  # Scans staged files before commit
```

GitHub Advanced Security also provides native secret scanning that alerts you if a known secret pattern (AWS key, Stripe key, etc.) is pushed to your repository.
