# Eventual Consistency Patterns

Eventual consistency is a consistency model where distributed nodes are guaranteed to converge to the same state *eventually*, given that no new updates are made and all messages are delivered. It sacrifices immediate consistency for availability and partition tolerance (the CAP theorem's "AP" choice).

For frontend applications, eventual consistency is the practical reality of any system that supports offline-first behavior, real-time collaboration, or multi-device sync.

## 1. The CAP Theorem in Practice

| System Type | Choice | Behavior |
|---|---|---|
| Traditional RDBMS | CP | Consistent & Partition-tolerant. Rejects writes during partitions. |
| Collaborative App (Yjs) | AP | Available & Partition-tolerant. Accepts writes during partitions, merges later. |
| Real-time Chat (Redis) | AP + eventual | Available during partition, catches up on reconnect. |

For most collaborative frontend applications, **AP (high availability + eventual consistency)** is the correct choice because users expect to keep working when offline.

## 2. Read-Your-Writes Consistency

One of the most jarring eventual consistency failures: a user submits a form, is redirected to a list page, and their new record doesn't appear.

**Problem:** The write went to Server A, but the subsequent read was served by Server B which hasn't replicated yet.

```typescript
// Pattern 1: Include the written record in the redirect
const createPost = async () => {
  const newPost = await $fetch('/api/posts', { method: 'POST', body: form })
  
  // Don't redirect to /posts (may not show the new post yet)
  // Instead, add to local state immediately + redirect
  postsStore.addPost(newPost) // Optimistic local add
  await router.push(`/posts/${newPost.id}`)
}

// Pattern 2: Write token — tell the read endpoint to wait for replication
const createPost = async () => {
  const { post, writeToken } = await $fetch('/api/posts', { method: 'POST', body: form })
  
  // Pass the write token in subsequent reads
  await router.push({ path: `/posts`, query: { afterWrite: writeToken } })
}

// Server: Wait for replication before responding to reads with writeToken
// server/api/posts.get.ts
const afterWrite = getQuery(event).afterWrite
if (afterWrite) {
  await waitForReplication(afterWrite) // Block until replica is caught up
}
```

## 3. Monotonic Reads

Ensure users never see an older version of data after seeing a newer one (going "back in time").

```typescript
// Track the last seen version of each entity
const versionCache = useLocalStorage<Record<string, number>>('version-cache', {})

const fetchPost = async (id: string) => {
  const post = await $fetch<Post>(`/api/posts/${id}`)
  
  const lastSeenVersion = versionCache.value[id] ?? 0
  
  if (post.version < lastSeenVersion) {
    // Received an older version than we've seen — discard to prevent going back in time
    console.warn('Received stale version, ignoring')
    return
  }
  
  versionCache.value[id] = post.version
  currentPost.value = post
}
```

## 4. Conflict-Free UI Patterns

Design your UI to minimize the likelihood of conflicts in the first place.

### Presence Indicators
Show which users are editing a section to naturally reduce conflicts through social coordination.

```vue
<template>
  <div class="editable-section" :class="{ 'being-edited': otherEditors.length > 0 }">
    <div v-if="otherEditors.length > 0" class="editing-indicator">
      <AvatarGroup :users="otherEditors" />
      <span>{{ otherEditors.map(u => u.name).join(', ') }} {{ otherEditors.length === 1 ? 'is' : 'are' }} editing</span>
    </div>
    <textarea v-model="content" :disabled="otherEditors.length > 0 && lockOnConflict" />
  </div>
</template>
```

### Field Locking
For critical fields (e.g., invoice amount), implement soft locks to prevent simultaneous editing.

```typescript
const acquireLock = async (fieldId: string) => {
  const lock = await $fetch(`/api/locks/${fieldId}`, {
    method: 'POST',
    body: { userId: currentUser.id, expiresIn: 30_000 } // 30s TTL
  })
  return lock.acquired // true if lock was granted
}
```

## 5. Convergence Verification

Periodically verify that all clients have converged to the same state.

```typescript
// Server: Compute a hash of the canonical state
// server/api/rooms/[id]/checksum.get.ts
export default defineEventHandler(async (event) => {
  const state = await getRoomState(getRouterParams(event).id)
  const checksum = createHash('sha256').update(JSON.stringify(state)).digest('hex')
  return { checksum, version: state.version }
})

// Client: Periodically compare checksums
const verifyConvergence = async () => {
  const { checksum: serverChecksum } = await $fetch(`/api/rooms/${roomId}/checksum`)
  const localChecksum = computeLocalChecksum(localState.value)

  if (serverChecksum !== localChecksum) {
    console.warn('State divergence detected — requesting full sync')
    await requestFullSync()
  }
}

// Run every 5 minutes
const interval = setInterval(verifyConvergence, 5 * 60 * 1000)
onUnmounted(() => clearInterval(interval))
```
