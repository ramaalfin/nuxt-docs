# Last-Write-Wins vs Semantic Merging

When two concurrent edits conflict, the system must decide which value wins. The choice of merge strategy fundamentally impacts data integrity and user experience. The two primary strategies are **Last-Write-Wins (LWW)** and **Semantic Merging**.

## 1. Last-Write-Wins (LWW)

LWW is the simplest conflict resolution strategy: the operation with the most recent timestamp overwrites all others. It is used by Cassandra, DynamoDB (by default), Redis, and many other distributed systems.

### Implementation

```typescript
// LWW Register CRDT
class LWWValue<T> {
  private value: T
  private timestamp: number

  constructor(initialValue: T) {
    this.value = initialValue
    this.timestamp = Date.now()
  }

  set(newValue: T, timestamp = Date.now()) {
    if (timestamp >= this.timestamp) {
      this.value = newValue
      this.timestamp = timestamp
    }
    // If timestamp is older, silently discard (the existing value wins)
  }

  get(): T { return this.value }

  merge(other: LWWValue<T>): void {
    if (other.timestamp > this.timestamp) {
      this.value = other.value
      this.timestamp = other.timestamp
    }
  }
}
```

### When LWW is Appropriate

✅ **Good for:**
- User preference settings (theme, language — only one correct value exists).
- Profile picture uploads (the latest upload is what the user wants).
- Status fields (online/offline — the latest state is always correct).
- Cache invalidation (latest fetched data replaces stale data).

❌ **Bad for:**
- Shopping cart contents (LWW would discard items added by concurrent devices).
- Collaborative documents (overwrites the other person's work).
- Counters (LWW causes lost increments).
- Financial data (loses transactions).

### The Clock Problem

LWW relies on timestamps, but distributed clocks are **not reliable**:
- NTP drift between servers can be 100ms+.
- Client clocks can be manipulated.
- Two events can have the same millisecond timestamp.

For production systems, use **Hybrid Logical Clocks (HLC)** or **vector clocks** instead of wall-clock time.

## 2. Semantic Merging

Semantic merging understands the *intent* of an operation and merges it meaningfully instead of blindly overwriting.

### Set Union (Additive Merging)
For collections where both sides' additions should be preserved:

```typescript
// Shopping cart: merge both carts, don't overwrite one with the other
const mergeCartItems = (cartA: CartItem[], cartB: CartItem[]): CartItem[] => {
  const itemMap = new Map<string, CartItem>()

  // Add all items from cart A
  cartA.forEach(item => itemMap.set(item.productId, item))

  // Merge items from cart B
  cartB.forEach(item => {
    const existing = itemMap.get(item.productId)
    if (existing) {
      // Semantic merge: take the maximum quantity (user intent was to add, not replace)
      itemMap.set(item.productId, {
        ...item,
        quantity: Math.max(existing.quantity, item.quantity)
      })
    } else {
      itemMap.set(item.productId, item)
    }
  })

  return [...itemMap.values()]
}
```

### Three-Way Merge
The git-style approach: compare two conflicting versions against their common ancestor.

```typescript
interface DocumentState {
  title: string
  content: string
  tags: string[]
}

const threeWayMerge = (
  base: DocumentState,    // Last known common ancestor
  version1: DocumentState, // One concurrent version
  version2: DocumentState  // Another concurrent version
): DocumentState => {
  const merged: DocumentState = { ...base }

  // For each field: if only one version changed it, accept that change
  // If both changed it, we have a true conflict
  for (const key of Object.keys(base) as (keyof DocumentState)[]) {
    const v1Changed = JSON.stringify(version1[key]) !== JSON.stringify(base[key])
    const v2Changed = JSON.stringify(version2[key]) !== JSON.stringify(base[key])

    if (v1Changed && !v2Changed) {
      (merged as any)[key] = version1[key] // Accept v1's change
    } else if (!v1Changed && v2Changed) {
      (merged as any)[key] = version2[key] // Accept v2's change
    } else if (v1Changed && v2Changed) {
      // TRUE CONFLICT: both versions changed the same field
      // Must escalate to user or apply domain-specific logic
      if (key === 'tags') {
        // Semantic merge: union of both tag sets
        (merged as any)[key] = [...new Set([...version1.tags, ...version2.tags])]
      } else {
        // For text: prefer latest by timestamp, or show conflict UI
        (merged as any)[key] = version2[key] // Fallback to LWW
      }
    }
  }

  return merged
}
```

## 3. Decision Matrix

| Data Type | Strategy | Reason |
|---|---|---|
| User preferences | LWW | Only one "correct" value |
| Document text | CRDT (Yjs) | Character-level semantic merge |
| Set of items (cart, tags) | Set Union CRDT | Both additions are valid |
| Counter (likes, views) | G-Counter CRDT | Additions should never be lost |
| File upload | LWW | Latest upload wins |
| Structured JSON | Three-way merge | Field-level granularity |
