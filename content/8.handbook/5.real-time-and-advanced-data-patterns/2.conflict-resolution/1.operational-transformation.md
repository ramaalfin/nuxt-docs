# Operational Transformation (OT)

Operational Transformation is an algorithm for enabling real-time collaborative editing of shared documents. It transforms operations made by concurrent users so that all clients converge to the same state, even when edits were made simultaneously on different versions of the document.

OT is the foundational technology behind Google Docs (before they moved to CRDTs) and many early collaborative editing systems.

## 1. The Problem OT Solves

Imagine two users editing the string `"Hello"`:
- User A (at position 5) types `" World"` → wants `"Hello World"`
- User B (at position 0) types `"Say "` → wants `"Say Hello"`

If both operations are applied naively:
- Server applies A's insert: `"Hello World"`
- Server then applies B's insert at position 0: `"Say Hello World"` ✅

But on User B's client:
- B's insert was applied first: `"Say Hello"`
- Then A's insert comes in at position 5: `"Say WHelllo"` ❌ (wrong position!)

OT **transforms** A's operation against B's operation to produce the correct result.

## 2. Core OT Concepts

### Operations
All document mutations are expressed as atomic operations:
```typescript
type InsertOp = { type: 'insert'; position: number; text: string }
type DeleteOp = { type: 'delete'; position: number; length: number }
type RetainOp = { type: 'retain'; count: number }
type Operation = InsertOp | DeleteOp | RetainOp
```

### The Transform Function
The heart of OT: given two concurrent operations A and B, compute A' (how A should be applied after B) and B' (how B should be applied after A).

```typescript
function transformInsertInsert(opA: InsertOp, opB: InsertOp): [InsertOp, InsertOp] {
  if (opA.position <= opB.position) {
    // A comes before B — B's position shifts right by A's length
    return [
      opA,
      { ...opB, position: opB.position + opA.text.length }
    ]
  } else {
    // B comes before A — A's position shifts right by B's length
    return [
      { ...opA, position: opA.position + opB.text.length },
      opB
    ]
  }
}
```

## 3. OT with ShareDB

**ShareDB** is the most mature OT library for JavaScript, providing both server and client SDK.

```bash
pnpm add sharedb @teamwork/websocket-json-stream
```

### Server Setup
```typescript
// server/sharedb.ts
import ShareDB from 'sharedb'
import { WebSocketJSONStream } from '@teamwork/websocket-json-stream'
import http from 'http'
import WebSocket from 'ws'

const backend = new ShareDB()

const server = http.createServer()
const wss = new WebSocket.Server({ server })

wss.on('connection', (ws) => {
  const stream = new WebSocketJSONStream(ws)
  backend.listen(stream) // ShareDB handles all OT protocol
})

server.listen(3001)
```

### Client Setup
```typescript
// composables/useShareDBDoc.ts
import ShareDB from 'sharedb/lib/client'
import ReconnectingWebSocket from 'reconnecting-websocket'

export function useShareDBDoc(collection: string, docId: string) {
  const content = ref<any>(null)

  const socket = new ReconnectingWebSocket('ws://localhost:3001')
  const connection = new ShareDB.Connection(socket as any)

  const doc = connection.get(collection, docId)

  doc.subscribe((error) => {
    if (error) throw error
    content.value = doc.data

    // Listen for remote operations
    doc.on('op', () => {
      content.value = { ...doc.data } // Trigger Vue reactivity
    })
  })

  const submitOp = (op: any) => {
    doc.submitOp(op)
  }

  onUnmounted(() => {
    doc.destroy()
    connection.close()
  })

  return { content, submitOp }
}
```

## 4. OT vs CRDT: When to Choose OT

| Aspect | OT | CRDT |
|---|---|---|
| **Maturity** | Proven (20+ years, Google Docs) | Newer, rapidly maturing |
| **Server requirement** | Requires central server for ordering | Works peer-to-peer |
| **Complexity** | High (correctness hard to prove) | Lower (mathematically guaranteed) |
| **Data types** | Text-optimized | Any data structure |
| **Best for** | Rich text editors (character-level) | JSON, counters, sets, presence |

**Rule of thumb:** For rich text collaborative editing with fine-grained character operations, OT with ShareDB is battle-tested. For everything else (JSON state, counters, sets), prefer CRDTs with Yjs.
