# Scaling to 100k Concurrent Connections

A single Node.js server can handle approximately 10,000-50,000 concurrent WebSocket connections before performance degrades. Reaching 100,000+ concurrent connections requires a fundamentally different architecture: horizontal scaling, a shared publish/subscribe layer, and stateless connection handlers.

## 1. The Core Scaling Problem: Server Affinity

WebSockets are stateful — a client is connected to a specific server instance. When you scale horizontally (multiple server instances), a message broadcast by Server A won't reach clients connected to Server B or Server C.

```
User A (connected to Server 1) sends a message
Server 1 broadcasts to: User B (Server 1) ✅
                         User C (Server 2) ❌ — Can't reach!
                         User D (Server 3) ❌ — Can't reach!
```

The solution is a **shared Pub/Sub layer** between servers.

## 2. Architecture: Pub/Sub with Redis

Redis is the standard solution for WebSocket fan-out at scale. When any server receives a message to broadcast, it publishes to a Redis channel. All server instances are subscribed to that channel and forward the message to their locally connected clients.

```
User A → Server 1 → Redis PubSub → Server 1 → User B
                                  → Server 2 → User C
                                  → Server 3 → User D
```

### Implementation with Nitro + Redis

```typescript
// server/plugins/redis.ts
import { createClient } from 'redis'

const publisher = createClient({ url: process.env.REDIS_URL })
const subscriber = publisher.duplicate()

await publisher.connect()
await subscriber.connect()

export { publisher, subscriber }
```

```typescript
// server/routes/ws.ts
import { publisher, subscriber } from '../plugins/redis'

// Subscribe to Redis on startup
await subscriber.subscribe('room:broadcast', (messageStr) => {
  const { roomId, message, excludePeerId } = JSON.parse(messageStr)
  
  // Forward to locally connected peers only
  localPeers.get(roomId)?.forEach(peer => {
    if (peer.id !== excludePeerId) {
      peer.send(JSON.stringify(message))
    }
  })
})

export default defineWebSocketHandler({
  message(peer, rawMessage) {
    const data = JSON.parse(rawMessage.text())
    
    if (data.type === 'BROADCAST') {
      // Publish to Redis — all server instances will receive this
      publisher.publish('room:broadcast', JSON.stringify({
        roomId: data.roomId,
        message: data.payload,
        excludePeerId: peer.id
      }))
    }
  }
})
```

## 4. Load Balancing WebSockets

WebSocket connections require **sticky sessions** (session affinity) at the load balancer level if your Pub/Sub layer is incomplete. With a proper Redis Pub/Sub layer, you don't need sticky sessions.

**Nginx Load Balancer Configuration:**
```nginx
upstream websocket_servers {
  # Without sticky sessions — requires Redis Pub/Sub
  least_conn; # Route to server with fewest connections
  server ws-server-1:3000;
  server ws-server-2:3000;
  server ws-server-3:3000;
}

server {
  listen 80;
  location /ws {
    proxy_pass http://websocket_servers;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 86400; # 24 hours — don't timeout idle WebSockets
  }
}
```

## 5. Edge-Native Scaling with Cloudflare Durable Objects

For ultimate scale without managing infrastructure, use **Cloudflare Durable Objects**. Each "Room" or "Document" is a Durable Object that handles all WebSocket connections for that entity from a globally-distributed edge network.

```typescript
// Cloudflare Durable Object (deployed at edge)
export class Room {
  private connections = new Set<WebSocket>()

  async webSocketMessage(ws: WebSocket, message: string) {
    // Broadcast to all connections in this Room instance
    this.connections.forEach(conn => {
      if (conn !== ws && conn.readyState === WebSocket.OPEN) {
        conn.send(message)
      }
    })
  }
}
```

## 6. Connection Lifecycle Management at Scale

At 100k connections, even small per-connection overheads multiply:

| Optimization | Savings |
|---|---|
| **Reduce message size** (MessagePack over JSON) | 30-50% bandwidth |
| **Heartbeat interval** (60s vs 15s) | 4x fewer ping messages |
| **Disconnect stale clients** (no pong in 90s) | Frees resources from ghost connections |
| **Compression** (`permessage-deflate`) | 60-80% bandwidth reduction for text |
| **Connection pooling** in Redis | Reduces Redis connection overhead |
