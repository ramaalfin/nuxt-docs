# Polling Patterns & Optimization

Polling is the simplest real-time technique: the client periodically sends HTTP requests to ask the server "is there anything new?". While WebSockets and SSE are more efficient for high-frequency updates, polling remains the right tool for many scenarios—and when implemented correctly, its overhead can be minimized significantly.

## 1. Short Polling (Naive Polling)

The client sends a request every N seconds, regardless of whether new data exists.

```typescript
// composables/usePolling.ts
export function usePolling<T>(
  fetcher: () => Promise<T>,
  options: {
    interval?: number
    enabled?: Ref<boolean>
    onData?: (data: T) => void
  } = {}
) {
  const { interval = 5000, enabled = ref(true) } = options
  const data = ref<T | null>(null)
  const isLoading = ref(false)
  const error = ref<Error | null>(null)
  let timerId: ReturnType<typeof setTimeout> | null = null

  const poll = async () => {
    if (!enabled.value) return

    isLoading.value = true
    try {
      data.value = await fetcher()
      options.onData?.(data.value as T)
      error.value = null
    } catch (err) {
      error.value = err as Error
    } finally {
      isLoading.value = false
    }

    // Schedule next poll AFTER the current one completes (no overlapping requests)
    timerId = setTimeout(poll, interval)
  }

  onMounted(poll)
  onUnmounted(() => timerId && clearTimeout(timerId))

  // Pause polling when tab is not visible
  useEventListener(document, 'visibilitychange', () => {
    if (document.visibilityState === 'hidden') {
      timerId && clearTimeout(timerId)
    } else {
      poll() // Resume immediately when tab becomes visible again
    }
  })

  return { data, isLoading, error }
}
```

**Key optimization:** Scheduling the next poll *after* the current response arrives (not at a fixed interval) prevents overlapping requests when the network is slow.

## 2. Long Polling (HTTP Hanging)

Long polling is a clever optimization: the server holds the request open until new data is available (or a timeout is reached), then responds immediately. The client then immediately opens a new request.

This approach closely approximates real-time push without WebSockets.

```typescript
// server/api/notifications/poll.get.ts
// The server holds the connection open until there's data or timeout
export default defineEventHandler(async (event) => {
  const userId = getQuery(event).userId as string
  const timeout = 30_000 // 30 seconds max hold

  return new Promise((resolve) => {
    const timer = setTimeout(() => {
      resolve([]) // No new data — return empty after timeout
    }, timeout)

    // Subscribe to a notification channel for this user
    notificationBus.once(`user:${userId}`, (notifications) => {
      clearTimeout(timer)
      resolve(notifications)
    })
  })
})
```

```typescript
// composables/useLongPolling.ts
export function useLongPolling<T>(url: string) {
  const data = ref<T[]>([])
  let isActive = true

  const poll = async () => {
    while (isActive) {
      try {
        const result = await $fetch<T[]>(url, {
          signal: AbortSignal.timeout(35_000) // Client-side timeout slightly > server's
        })

        if (result.length > 0) {
          data.value.push(...result)
        }
        // Immediately open the next long-poll request
      } catch (error: any) {
        if (error.name !== 'TimeoutError') {
          await new Promise(r => setTimeout(r, 2000)) // Backoff on real errors
        }
      }
    }
  }

  onMounted(() => poll())
  onUnmounted(() => { isActive = false })

  return { data }
}
```

## 3. Smart Polling Optimizations

### Exponential Backoff on Error
Never poll at a fixed rate when requests are failing. Apply exponential backoff to avoid hammering a struggling server.

```typescript
const backoff = (attempt: number, base = 1000) => 
  Math.min(base * Math.pow(2, attempt), 30_000) // Max 30 seconds

// In your poll function:
let attempt = 0
try {
  await fetcher()
  attempt = 0 // Reset on success
} catch {
  await sleep(backoff(attempt++))
}
```

### Pause When Tab is Hidden
Users with many open tabs create unnecessary server load. Pause polling when the tab is hidden.

```typescript
const poll = () => {
  if (document.visibilityState === 'hidden') return // Skip if not visible
  // ... proceed with fetch
}
```

### Conditional Requests (ETag / If-Modified-Since)
Use HTTP caching headers to allow the server to return a `304 Not Modified` response with no body when data hasn't changed, saving bandwidth.

```typescript
const { data, headers: lastHeaders } = await $fetch.raw('/api/orders')

// On the next poll, send the ETag from the previous response
const response = await $fetch('/api/orders', {
  headers: {
    'If-None-Match': lastHeaders.get('etag') ?? ''
  }
})
// Server responds with 304 if nothing changed (no body transferred)
```

## 4. Polling vs SSE vs WebSocket: Choosing

| Factor | Short Polling | Long Polling | SSE | WebSocket |
|---|---|---|---|---|
| **Update frequency** | Minutes | Seconds | Sub-second | Sub-second |
| **Server load** | High | Medium | Low | Low |
| **Complexity** | ⭐ Lowest | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| **Infrastructure** | Standard HTTP | Standard HTTP | Standard HTTP | Requires WS support |
| **Use case** | Status checks | Notifications | Live feeds | Chat, games |

**Recommendation:** Use polling for infrequent updates (> 30 second intervals). Use SSE for server-push at sub-30-second intervals. Reserve WebSockets for bidirectional real-time communication.
