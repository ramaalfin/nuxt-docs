# WebSocket Implementation & Lifecycle

WebSockets provide a persistent, full-duplex communication channel over a single TCP connection. Unlike HTTP's request-response model, WebSockets allow both the server and client to send messages at any time, making them the foundation of real-time applications: live chat, collaborative tools, financial dashboards, and multiplayer games.

## 1. The WebSocket Lifecycle

Understanding the complete lifecycle is essential for building robust real-time features.

```
Client                                Server
  |                                     |
  |--- HTTP Upgrade Request ----------->|
  |<-- 101 Switching Protocols ---------|
  |===== WebSocket Connection Open =====|
  |                                     |
  |<-- Server Message -----------------|  (Server Push)
  |--- Client Message ----------------->|  (Client Send)
  |                                     |
  |--- Ping ---------------------------->|  (Heartbeat)
  |<-- Pong -----------------------------|
  |                                     |
  |--- Close Frame --------------------->|  (Graceful Close)
  |<-- Close Frame ----------------------|
  |===== Connection Closed ==============|
```

**Lifecycle Events:**
- `onopen`: Connection established and ready to use.
- `onmessage`: A message received from the server.
- `onerror`: An error occurred (usually followed by `onclose`).
- `onclose`: Connection closed (may be intentional or unexpected).

## 2. Native WebSocket Client

```typescript
// composables/useWebSocket.ts
export function useWebSocket(url: string) {
  const socket = ref<WebSocket | null>(null)
  const status = ref<'connecting' | 'open' | 'closing' | 'closed'>('closed')
  const lastMessage = ref<any>(null)
  const error = ref<Event | null>(null)

  const connect = () => {
    if (socket.value?.readyState === WebSocket.OPEN) return

    socket.value = new WebSocket(url)
    status.value = 'connecting'

    socket.value.onopen = () => {
      status.value = 'open'
      console.log('[WS] Connected')
    }

    socket.value.onmessage = (event: MessageEvent) => {
      try {
        lastMessage.value = JSON.parse(event.data)
      } catch {
        lastMessage.value = event.data
      }
    }

    socket.value.onerror = (event: Event) => {
      error.value = event
      console.error('[WS] Error:', event)
    }

    socket.value.onclose = (event: CloseEvent) => {
      status.value = 'closed'
      console.log(`[WS] Closed: code=${event.code}, reason=${event.reason}`)
    }
  }

  const send = (data: object | string) => {
    if (socket.value?.readyState !== WebSocket.OPEN) {
      console.warn('[WS] Cannot send — connection is not open')
      return
    }
    socket.value.send(typeof data === 'string' ? data : JSON.stringify(data))
  }

  const close = (code = 1000, reason = 'Normal closure') => {
    status.value = 'closing'
    socket.value?.close(code, reason)
  }

  // Clean up on component unmount
  onUnmounted(() => close())

  return { status, lastMessage, error, connect, send, close }
}
```

## 3. WebSocket in Nuxt (Client-Only)

WebSockets are a browser API and must only run on the client side.

```vue
<script setup lang="ts">
// useWebSocket is called within onMounted or wrapped in a client-only plugin
const { status, lastMessage, connect, send } = useWebSocket('wss://api.yourapp.com/ws')

onMounted(() => connect())

// React to incoming messages
watch(lastMessage, (message) => {
  if (message?.type === 'NEW_ORDER') {
    ordersStore.addOrder(message.payload)
  }
})
</script>

<template>
  <div>
    <span :class="status === 'open' ? 'text-green' : 'text-red'">
      {{ status }}
    </span>
    <button @click="send({ type: 'PING' })">Send Ping</button>
  </div>
</template>
```

## 4. Server-Side WebSocket with Nuxt Nitro

Nuxt 3's Nitro server supports WebSockets natively via the H3 `defineWebSocketHandler`.

```typescript
// server/routes/ws.ts
import { defineWebSocketHandler } from 'h3'

export default defineWebSocketHandler({
  open(peer) {
    console.log('[Server WS] Client connected:', peer.id)
    // Send a welcome message immediately on connection
    peer.send(JSON.stringify({ type: 'WELCOME', message: 'Connected to server' }))
  },

  message(peer, message) {
    const data = JSON.parse(message.text())
    console.log('[Server WS] Received:', data)

    // Echo back or broadcast to other peers
    if (data.type === 'BROADCAST') {
      peer.publish('global', JSON.stringify({
        type: 'BROADCAST',
        from: peer.id,
        payload: data.payload
      }))
    }
  },

  close(peer, details) {
    console.log('[Server WS] Client disconnected:', peer.id, details)
  },

  error(peer, error) {
    console.error('[Server WS] Error:', peer.id, error)
  }
})
```

## 5. Message Protocol Design

Always use a structured message format. A JSON protocol with a `type` field enables easy routing.

```typescript
// types/ws-messages.ts
export type WsMessage =
  | { type: 'SUBSCRIBE'; channel: string }
  | { type: 'UNSUBSCRIBE'; channel: string }
  | { type: 'MESSAGE'; channel: string; payload: unknown }
  | { type: 'PING' }
  | { type: 'PONG' }
  | { type: 'ERROR'; code: string; message: string }
```

## 6. WebSocket Close Codes

| Code | Meaning |
|---|---|
| `1000` | Normal closure |
| `1001` | Going away (browser tab closing) |
| `1008` | Policy violation |
| `1011` | Unexpected server error |
| `4000`-`4999` | Application-defined custom codes |

Use custom close codes (4000+) to communicate application-specific disconnect reasons (e.g., `4001: Authentication expired`, `4002: Rate limited`).
