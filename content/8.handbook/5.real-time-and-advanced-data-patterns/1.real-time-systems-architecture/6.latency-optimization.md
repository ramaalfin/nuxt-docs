# Latency Optimization for Real-Time Systems

Latency is the enemy of real-time systems. In collaborative applications, users perceive latency above 100ms as "sluggish" and above 300ms as "broken." Optimizing for low latency requires a holistic approach spanning network, protocol, server architecture, and client rendering.

## 1. Measure First

Before optimizing, establish a baseline. You cannot improve what you don't measure.

```typescript
// Measure round-trip latency with a PING/PONG
export function useLatencyMonitor(send: (msg: object) => void) {
  const latency = ref<number | null>(null)
  let pingTimestamp = 0

  const sendPing = () => {
    pingTimestamp = performance.now()
    send({ type: 'PING', timestamp: pingTimestamp })
  }

  const onPong = () => {
    latency.value = Math.round(performance.now() - pingTimestamp)
  }

  // Ping every 10 seconds
  const interval = setInterval(sendPing, 10_000)
  onMounted(sendPing)
  onUnmounted(() => clearInterval(interval))

  return { latency, onPong }
}
```

## 2. Network Layer Optimizations

### Geographic Proximity (Edge Computing)
The single biggest latency reducer: deploy your WebSocket server closest to your users.
- Use **Cloudflare Workers** with Durable Objects for globally-distributed WebSocket connections.
- Use **Fly.io** to deploy Nitro servers on the edge across 30+ regions.
- A server in Singapore vs. one in Virginia can mean 200ms vs 20ms for a Southeast Asian user.

### Use HTTP/2 or HTTP/3
HTTP/3 (QUIC) eliminates head-of-line blocking and significantly reduces connection establishment latency (0-RTT reconnection). Configure your Nginx or Caddy to use HTTP/3.

### Binary Protocol (MessagePack over JSON)
JSON serialization is expensive. For high-frequency messages (cursor positions, game state), use binary formats.

```typescript
import { pack, unpack } from 'msgpackr'

// Sending binary instead of JSON — ~30% smaller, faster to serialize
const sendBinary = (data: object) => {
  socket.value?.send(pack(data)) // Binary ArrayBuffer
}

socket.onmessage = (event: MessageEvent<ArrayBuffer>) => {
  const data = unpack(new Uint8Array(event.data))
  handleMessage(data)
}
```

## 3. Client-Side Latency Tricks

### Predictive State Updates (Optimistic UI)
Don't wait for the server to confirm an action before updating the UI. Apply changes locally immediately — this makes the app feel instantaneous even at 200ms RTT.

See the **Optimistic Updates** chapter in State Management for full implementation.

### Interpolation for Cursor/Position Data
When receiving position updates at 20fps but rendering at 60fps, interpolate between received positions to create smooth motion.

```typescript
// Linear interpolation between last known position and target
const lerp = (start: number, end: number, t: number) => start + (end - start) * t

// In the render loop
const smoothX = ref(0)
const smoothY = ref(0)
const targetX = ref(0)
const targetY = ref(0)

// Called every animation frame
const smoothUpdate = () => {
  smoothX.value = lerp(smoothX.value, targetX.value, 0.2)
  smoothY.value = lerp(smoothY.value, targetY.value, 0.2)
  requestAnimationFrame(smoothUpdate)
}
```

### Message Batching
Instead of sending 20 individual WebSocket messages per second, batch them into a single message.

```typescript
// Batch cursor positions — send once per frame instead of on every event
let pendingUpdates: object[] = []

const queueUpdate = (update: object) => {
  pendingUpdates.push(update)
}

// Flush on animation frame
const flushUpdates = () => {
  if (pendingUpdates.length > 0) {
    send({ type: 'BATCH', updates: pendingUpdates })
    pendingUpdates = []
  }
  requestAnimationFrame(flushUpdates)
}
requestAnimationFrame(flushUpdates)
```

## 4. Server-Side Latency Optimizations

### In-Memory Over Database
For ephemeral real-time data (cursor positions, typing status, presence), **never** write to a database on every update. Use in-memory state (or Redis) exclusively.

### Non-Blocking Message Broadcasting
Broadcasting to 1000 connected peers must not block the event loop.

```typescript
// Nitro WebSocket handler — batch and async broadcast
const broadcastToRoom = async (roomId: string, message: object, excludePeerId?: string) => {
  const peers = roomStore.getPeers(roomId)
  const payload = JSON.stringify(message)
  
  // Use Promise.allSettled to send to all peers concurrently
  await Promise.allSettled(
    peers
      .filter(p => p.id !== excludePeerId)
      .map(peer => peer.send(payload))
  )
}
```

## 5. Latency Budget

Define a latency budget for your critical real-time interactions:

| Interaction | Target Latency | Optimization |
|---|---|---|
| Cursor movement | < 50ms | Throttle to 20fps, interpolate on client |
| Typing indicator | < 200ms | Acceptable for human perception |
| Chat message delivery | < 100ms | Edge server + optimistic local echo |
| Collaborative edit | < 150ms | CRDT + optimistic update |
