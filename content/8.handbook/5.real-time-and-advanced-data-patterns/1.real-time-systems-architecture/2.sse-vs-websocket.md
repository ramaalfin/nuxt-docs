# Server-Sent Events (SSE) vs WebSocket

Choosing between SSE and WebSockets is one of the most important architectural decisions for a real-time feature. They are not interchangeable—each excels in specific scenarios. Using the wrong one leads to unnecessary complexity or performance problems.

## 1. The Fundamental Difference

| Aspect | Server-Sent Events (SSE) | WebSockets |
|---|---|---|
| **Direction** | Server → Client only (unidirectional) | Bidirectional (full-duplex) |
| **Protocol** | HTTP/1.1 or HTTP/2 | Custom WS protocol (upgraded from HTTP) |
| **Connection** | Standard HTTP connection | Persistent TCP connection |
| **Browser API** | `EventSource` (built-in) | `WebSocket` (built-in) |
| **Auto-reconnect** | ✅ Built-in (automatic) | ❌ Manual implementation required |
| **HTTP/2 multiplexing** | ✅ Yes (no connection limit issue) | ❌ No |
| **Load balancer compatibility** | ✅ Excellent (standard HTTP) | ⚠️ Requires sticky sessions or pub/sub |
| **Proxies/Firewalls** | ✅ Standard HTTP passes through | ⚠️ Some proxies block WebSocket upgrades |
| **Complexity** | Low | High |

## 2. When to Use SSE

Choose SSE whenever data flows **one-way from server to client**:
- Live notifications (new comment, new order, system alerts).
- Real-time dashboards (stock prices, metrics, sports scores).
- Progress updates for long-running server operations (file uploads, AI generation streaming).
- Activity feeds and news streams.

**The golden rule:** If the client never needs to send messages over the real-time channel (only via regular HTTP requests), use SSE.

## 3. Implementing SSE in Nuxt

### Server Route (Nitro)

```typescript
// server/routes/events.ts
export default defineEventHandler(async (event) => {
  // Set SSE response headers
  setResponseHeaders(event, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
    'X-Accel-Buffering': 'no', // Disable Nginx buffering for SSE
  })

  // Send an initial comment to establish the connection
  const stream = createEventStream(event)

  // Send events using the H3 event stream helper
  const interval = setInterval(async () => {
    const data = await getDashboardMetrics()
    await stream.push({
      event: 'metrics',
      data: JSON.stringify(data)
    })
  }, 2000)

  // Cleanup when client disconnects
  stream.onClosed(async () => {
    clearInterval(interval)
    await stream.close()
  })

  return stream.send()
})
```

### Client Composable

```typescript
// composables/useSSE.ts
export function useSSE<T>(url: string) {
  const data = ref<T | null>(null)
  const error = ref<Event | null>(null)
  const status = ref<'connecting' | 'open' | 'closed'>('connecting')
  let eventSource: EventSource | null = null

  const connect = () => {
    eventSource = new EventSource(url, { withCredentials: true })
    
    eventSource.onopen = () => { status.value = 'open' }
    
    eventSource.onmessage = (event: MessageEvent) => {
      data.value = JSON.parse(event.data)
    }
    
    // Listen for named events
    eventSource.addEventListener('metrics', (event: MessageEvent) => {
      data.value = JSON.parse(event.data)
    })
    
    eventSource.onerror = (event: Event) => {
      error.value = event
      status.value = 'closed'
      // EventSource automatically reconnects — no manual logic needed
    }
  }

  const close = () => {
    eventSource?.close()
    status.value = 'closed'
  }

  onMounted(connect)
  onUnmounted(close)

  return { data, error, status, close }
}
```

## 4. When to Use WebSockets

Choose WebSockets when communication is **bidirectional in real-time**:
- Live chat and messaging (client sends messages, server delivers them).
- Collaborative editing (multiple users sending cursor positions and edits simultaneously).
- Multiplayer games (constant bidirectional state updates).
- Trading platforms (client places orders, server streams updates).
- IoT device control (send commands, receive sensor data).

## 5. SSE over HTTP/2

A significant advantage of SSE over WebSockets is HTTP/2 multiplexing. HTTP/2 can multiplex multiple SSE streams over a single TCP connection, bypassing browsers' HTTP/1.1 connection limits (typically 6 per domain). This makes SSE dramatically more scalable for large numbers of concurrent users.

WebSockets use a separate, non-HTTP/2 protocol and do not benefit from multiplexing.

## 6. Decision Matrix

```
Does the client need to SEND data in real-time?
  ├── NO  → Use SSE (simpler, better proxy/infra compatibility)
  └── YES → Is the data being sent low-frequency (e.g., button clicks)?
              ├── YES → Use SSE for server push + REST API for client actions
              └── NO  → Use WebSocket (high-frequency bidirectional, e.g., games, cursors)
```

The hybrid approach—SSE for server pushes + regular `$fetch` POST for client actions—is underutilized but often the ideal architecture for notification-heavy apps.
