# Real-Time State Synchronization

Real-time state synchronization is the challenge of keeping the UI state on multiple clients consistent when the underlying data changes. It is architecturally more complex than simply displaying a live data stream, because it must handle conflicts, partial updates, and eventual consistency.

## 1. The Core Challenge: Divergent State

Consider a collaborative document:
1. User A and User B both have the document open.
2. User A changes the title to "Q4 Report."
3. User B changes the title to "Annual Summary."
4. Both changes arrive at the server simultaneously.

Without a synchronization strategy, one change will silently overwrite the other. This is a **write conflict**.

## 2. Pattern 1: Server as Single Source of Truth

The simplest approach: the server holds the canonical state. Every client mutation goes through the server, and the server broadcasts the authoritative result to all clients.

```typescript
// Pinia store with real-time sync
export const useDocumentStore = defineStore('document', () => {
  const document = ref<Document | null>(null)
  const { send, lastMessage } = useWebSocket('wss://api.yourapp.com/ws')

  // Listen for server-pushed updates
  watch(lastMessage, (message) => {
    if (message?.type === 'DOCUMENT_UPDATED') {
      // Server state ALWAYS wins — overwrite local state
      document.value = message.payload
    }
  })

  // Send mutations to the server, DO NOT update local state
  // Wait for the server's broadcast to update the UI
  const updateTitle = (title: string) => {
    send({ type: 'UPDATE_TITLE', payload: { title } })
    // Local state NOT updated here — server will broadcast the confirmed state
  }

  return { document, updateTitle }
})
```

**Pros:** Simple. No conflicts by definition.
**Cons:** High perceived latency (user sees no change until server responds and broadcasts).

## 3. Pattern 2: Optimistic Updates with Reconciliation

Apply changes locally immediately (for instant UI feedback), then reconcile with the server's authoritative response.

```typescript
const updateTitle = async (title: string) => {
  // 1. Save the old value for rollback
  const previousTitle = document.value?.title

  // 2. Apply optimistically
  if (document.value) document.value.title = title

  try {
    // 3. Send to server
    const serverDocument = await $fetch(`/api/documents/${id}`, {
      method: 'PATCH',
      body: { title }
    })

    // 4. Reconcile: use server's exact response (handles timestamps, versioning)
    document.value = serverDocument
  } catch {
    // 5. Rollback on failure
    if (document.value) document.value.title = previousTitle ?? ''
    showError('Failed to update title. Your changes were reverted.')
  }
}
```

## 4. Pattern 3: Operational Transformation (OT)

For character-level collaborative text editing (like Google Docs), use **Operational Transformation**. OT defines operations (INSERT char at position, DELETE char at position) and applies transformation rules to resolve concurrent operations without conflicts.

**Libraries for Vue/Nuxt:**
- **Yjs** (CRDT-based, recommended): `yjs.dev`
- **ShareDB** (OT-based): `share.github.io/sharedb`

### Yjs Integration Example

```typescript
// composables/useCollaborativeDoc.ts
import * as Y from 'yjs'
import { WebsocketProvider } from 'y-websocket'
import { QuillBinding } from 'y-quill'

export function useCollaborativeDoc(docId: string) {
  const ydoc = new Y.Doc()
  
  // Connect to a y-websocket server for sync
  const provider = new WebsocketProvider(
    'wss://sync.yourapp.com', 
    docId,
    ydoc,
    { connect: true }
  )

  // The shared text type automatically synchronizes across all connected clients
  const ytext = ydoc.getText('content')
  
  // Bind to a Quill editor (or tiptap, prosemirror, etc.)
  // const binding = new QuillBinding(ytext, quillEditor)

  const awareness = provider.awareness // For presence (cursor positions)

  onUnmounted(() => {
    provider.destroy()
    ydoc.destroy()
  })

  return { ydoc, ytext, awareness }
}
```

## 5. Pattern 4: Delta Synchronization

Instead of sending the full state on every change, send only the **delta** (the diff). This is critical for large state objects.

```typescript
// Server sends only what changed
const broadcastDelta = (previousState: State, newState: State) => {
  const delta: Partial<State> = {}
  
  for (const key of Object.keys(newState) as (keyof State)[]) {
    if (newState[key] !== previousState[key]) {
      delta[key] = newState[key]
    }
  }
  
  if (Object.keys(delta).length > 0) {
    broadcast({ type: 'STATE_DELTA', payload: delta })
  }
}

// Client merges the delta into its local state
watch(lastMessage, (message) => {
  if (message?.type === 'STATE_DELTA') {
    // Merge the delta — only changed fields are updated
    Object.assign(localState.value, message.payload)
  }
})
```

## 6. Handling Late Joiners

When a new user joins a real-time session, they need the full current state, not just the stream of changes from that point forward.

```typescript
// Server pattern: State + Event Stream
// 1. Send the full current state snapshot when the client connects
peer.send(JSON.stringify({ type: 'SNAPSHOT', state: getCurrentState() }))

// 2. Subsequently send only deltas
peer.subscribe('state:changes') // Subscribe to the delta stream
```

The client initializes from the snapshot, then applies all subsequent deltas.
