# Server State vs UI State Separation

One of the most impactful architectural improvements in modern Vue applications is the strict separation of **server state** (data that lives on a server and needs to be fetched, cached, and synchronized) from **UI state** (ephemeral state that belongs only to the client).

Mixing these two categories into a single Pinia store leads to bloated, complex stores that are difficult to maintain and nearly impossible to keep synchronized with the server.

## 1. The Two Categories of State

### Server State
- **Origin:** Lives on the server/database.
- **Characteristics:** Asynchronous, potentially stale, shared with other users, requires caching, needs background refetching.
- **Examples:** Product listings, user profiles, order history, comments.
- **Tool:** TanStack Query (`useQuery`, `useMutation`).

### UI State
- **Origin:** Lives only in the browser.
- **Characteristics:** Synchronous, always fresh, owned by this user's session, no caching needed.
- **Examples:** Modal open/close state, active tab, form field values, sidebar collapsed state, selected items in a multi-select.
- **Tool:** `ref`, `reactive`, Pinia (for global UI state).

## 2. Anti-Pattern: Mixing State in Pinia

```typescript
// ❌ BAD: A Pinia store trying to do everything
export const useProductStore = defineStore('products', () => {
  // Server state (should be in TanStack Query)
  const products = ref<Product[]>([])
  const isLoading = ref(false)
  const error = ref<Error | null>(null)

  // UI state (correctly belongs in a store or local ref)
  const selectedProductId = ref<string | null>(null)
  const isFilterPanelOpen = ref(false)
  const sortOrder = ref<'asc' | 'desc'>('asc')

  // Complex async action with manual loading/error tracking
  const fetchProducts = async () => {
    isLoading.value = true
    try {
      products.value = await $fetch('/api/products')
    } catch (e) {
      error.value = e as Error
    } finally {
      isLoading.value = false
    }
  }

  // No caching, no background refetching, no stale-while-revalidate
  return { products, isLoading, error, selectedProductId, isFilterPanelOpen, fetchProducts }
})
```

**Problems:** 30+ lines of boilerplate. No caching. No deduplication. No background refetching.

## 3. Clean Architecture: Separated State

```typescript
// ✅ GOOD: Server state handled by TanStack Query
// composables/useProducts.ts
export function useProducts(filters: Ref<ProductFilters>) {
  return useQuery({
    queryKey: ['products', filters],
    queryFn: () => $fetch('/api/products', { query: filters.value }),
    staleTime: 1000 * 60 * 5, // Automatically handles caching, deduplication
  })
}

// ✅ GOOD: UI state in a lean Pinia store (or simple refs)
// stores/productUI.ts
export const useProductUIStore = defineStore('product-ui', () => {
  const selectedProductId = ref<string | null>(null)
  const isFilterPanelOpen = ref(false)
  const sortOrder = ref<'asc' | 'desc'>('asc')
  
  return { selectedProductId, isFilterPanelOpen, sortOrder }
})
```

```vue
<script setup lang="ts">
// Server state: TanStack Query handles all the complexity
const { data: products, isPending, isError } = useProducts(activeFilters)

// UI state: Simple, local, synchronous
const uiStore = useProductUIStore()
</script>
```

## 4. The Rule of Thumb

> **"If it involves an `await $fetch()`—it's server state. Use TanStack Query."**
> **"If it only lives in the browser—it's UI state. Use `ref` or Pinia."**

### Gray Areas

| State | Classification | Tool |
|---|---|---|
| Currently authenticated user | Server state (fetched from `/api/me`) | TanStack Query |
| User preferences (theme/language) | Hybrid — fetched from server, cached locally | TanStack Query + `localStorage` |
| Shopping cart (guest user) | UI state | Pinia + `localStorage` |
| Shopping cart (logged-in user) | Server state | TanStack Query |
| Search query string | UI state | `ref` in composable |
| Search results | Server state | TanStack Query |

## 5. Integration: Using Server State in UI Store Actions

Sometimes UI actions must trigger server state mutations. The mutation callback is the bridge.

```typescript
// Using TanStack Query mutation in response to UI actions
const { mutate: deleteProduct } = useMutation({
  mutationFn: (id: string) => $fetch(`/api/products/${id}`, { method: 'DELETE' }),
  onSuccess: () => {
    uiStore.selectedProductId = null // Clear UI selection
    queryClient.invalidateQueries({ queryKey: ['products'] }) // Refresh server state
  }
})

// UI action triggers server mutation
const onDeleteClick = () => deleteProduct(uiStore.selectedProductId!)
```
