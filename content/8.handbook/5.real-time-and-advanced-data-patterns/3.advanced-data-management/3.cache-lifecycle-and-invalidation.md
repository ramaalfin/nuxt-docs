# Cache Lifecycle & Invalidation

Understanding TanStack Query's cache lifecycle is essential for building apps that feel instant while always showing fresh data. The cache is not a simple key-value store—it manages the full lifecycle of each piece of server data, including freshness, garbage collection, and background updates.

## 1. The Cache Lifecycle States

```
                     queryFn called
              ┌──────────────────────────────┐
              │                              ▼
           FRESH  ──(staleTime expires)──► STALE
              │                              │
              │                              ├── Background refetch on window focus
              │                              ├── Background refetch on reconnect
              │                              └── Refetch on next mount
              │
           (all observers unmount)
              │
              ▼
           INACTIVE ──(gcTime expires)──► DELETED from cache
```

**Key timings:**
- `staleTime`: How long data is considered "fresh" (no refetch needed). Default: `0`.
- `gcTime`: How long inactive (unmounted) query data stays in cache before deletion. Default: `5 minutes`.

## 2. Configuring Stale Times

```typescript
// Global defaults for your app
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 5,  // 5 mins — most API data
      gcTime: 1000 * 60 * 30,    // 30 mins — keep in cache even when not displayed
    }
  }
})

// Per-query overrides
useQuery({
  queryKey: ['user-profile'],
  queryFn: fetchUserProfile,
  staleTime: Infinity, // User profile rarely changes — never re-fetch automatically
})

useQuery({
  queryKey: ['stock-prices'],
  queryFn: fetchStockPrices,
  staleTime: 1000 * 10, // 10 seconds — financial data must be fresh
  refetchInterval: 1000 * 30, // Actively poll every 30 seconds
})
```

## 3. Manual Invalidation (The Most Important Pattern)

After a mutation (create, update, delete), invalidate related queries so the UI reflects the new server state.

```typescript
const queryClient = useQueryClient()

// After creating a new product:
const { mutate: createProduct } = useMutation({
  mutationFn: (data: CreateProductInput) => $fetch('/api/products', {
    method: 'POST', body: data
  }),
  onSuccess: () => {
    // Invalidate all product list queries — they'll refetch on next access
    queryClient.invalidateQueries({ queryKey: ['products'] })
    
    // For immediate refetch (don't wait for next mount):
    // queryClient.refetchQueries({ queryKey: ['products'] })
  }
})
```

### Surgical Invalidation

```typescript
// Invalidate only the specific product that was updated
const { mutate: updateProduct } = useMutation({
  mutationFn: ({ id, data }: { id: string; data: UpdateProductInput }) =>
    $fetch(`/api/products/${id}`, { method: 'PATCH', body: data }),
  
  onSuccess: (updatedProduct) => {
    // Update the cache directly — no network request needed
    queryClient.setQueryData(['product', updatedProduct.id], updatedProduct)
    
    // Invalidate the list (the list has a summary that may have changed)
    queryClient.invalidateQueries({
      queryKey: ['products'],
      exact: false // Matches all queries starting with 'products'
    })
  }
})
```

## 4. Programmatic Cache Updates (`setQueryData`)

Instead of invalidating and refetching, inject the server response directly into the cache for instant updates.

```typescript
const { mutate: toggleLike } = useMutation({
  mutationFn: (postId: string) => $fetch(`/api/posts/${postId}/like`, { method: 'POST' }),
  
  onMutate: async (postId) => {
    // Cancel any in-flight refetch (would overwrite our optimistic update)
    await queryClient.cancelQueries({ queryKey: ['post', postId] })
    
    // Snapshot the current value for rollback
    const previous = queryClient.getQueryData<Post>(['post', postId])
    
    // Optimistically update the cache
    queryClient.setQueryData<Post>(['post', postId], old => old ? {
      ...old,
      isLiked: !old.isLiked,
      likeCount: old.isLiked ? old.likeCount - 1 : old.likeCount + 1
    } : old)
    
    return { previous } // Return context for onError
  },
  
  onError: (_err, postId, context) => {
    // Rollback on failure
    queryClient.setQueryData(['post', postId], context?.previous)
  },
  
  onSettled: (_, __, postId) => {
    // Always sync with server truth after mutation settles
    queryClient.invalidateQueries({ queryKey: ['post', postId] })
  }
})
```

## 5. Cache Persistence Across Sessions

Persist the query cache to `localStorage` or `IndexedDB` so users see cached data instantly on their next visit.

```typescript
import { persistQueryClient } from '@tanstack/query-persist-client-core'
import { createSyncStoragePersister } from '@tanstack/query-sync-storage-persister'

const persister = createSyncStoragePersister({
  storage: window.localStorage
})

persistQueryClient({
  queryClient,
  persister,
  maxAge: 1000 * 60 * 60 * 24, // 24 hours
  buster: APP_VERSION, // Bust cache when app version changes
})
```
