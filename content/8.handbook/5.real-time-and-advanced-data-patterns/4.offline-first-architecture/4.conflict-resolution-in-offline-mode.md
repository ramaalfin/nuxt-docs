# Conflict Resolution in Offline Mode

When a user makes changes while offline and those changes are later synced with the server, conflicts can arise if the same data was modified by another user or device in the meantime. Handling these conflicts correctly is the hardest part of offline-first architecture.

## 1. Types of Offline Conflicts

### Write-Write Conflict
The most common: the same field was changed by the offline client and by the server while the client was offline.
```
Client (offline): task.status = "done"
Server (updated by another user): task.status = "cancelled"
→ When client reconnects: true conflict
```

### Delete-Update Conflict
A record deleted on the server was modified offline, or vice versa.
```
Client (offline): updated task.title
Server: deleted the task
→ Resurrect or discard the offline update?
```

## 2. Conflict Detection During Sync

```typescript
// When syncing, attach the entity's version number
const syncOperation = async (op: SyncQueueItem) => {
  try {
    const result = await $fetch(op.endpoint, {
      method: op.method,
      body: op.body,
      headers: {
        // Tell the server which version we were editing
        'If-Match': op.entityVersion  // ETag-style version
      }
    })
    return { success: true, result }
  } catch (error: any) {
    if (error.status === 409) {
      // Conflict! Server has a newer version
      const serverVersion = error.data?.serverVersion
      return { success: false, conflict: true, serverVersion }
    }
    return { success: false, conflict: false, error }
  }
}
```

```typescript
// server/api/tasks/[id].patch.ts — Server-side conflict detection
export default defineEventHandler(async (event) => {
  const clientVersion = getRequestHeader(event, 'If-Match')
  const current = await db.tasks.findById(getRouterParams(event).id)

  if (clientVersion && current.version !== parseInt(clientVersion)) {
    throw createError({
      statusCode: 409,
      data: {
        serverVersion: current,
        conflict: true
      }
    })
  }

  const updated = await db.tasks.update(/* ... */)
  setResponseHeader(event, 'ETag', updated.version.toString())
  return updated
})
```

## 3. Automatic Resolution Strategies

Before surfacing conflicts to users, attempt automatic resolution:

```typescript
// composables/useConflictResolver.ts
interface ConflictContext {
  localVersion: any
  serverVersion: any
  localUpdatedAt: number
  serverUpdatedAt: number
}

const tryAutoResolve = (conflict: ConflictContext): any | null => {
  // Strategy 1: Same value — no real conflict
  if (JSON.stringify(conflict.localVersion) === JSON.stringify(conflict.serverVersion)) {
    return conflict.serverVersion
  }
  
  // Strategy 2: Array fields — merge (union)
  if (Array.isArray(conflict.localVersion) && Array.isArray(conflict.serverVersion)) {
    return [...new Set([...conflict.localVersion, ...conflict.serverVersion])]
  }
  
  // Strategy 3: Non-overlapping field changes — merge objects
  if (typeof conflict.localVersion === 'object' && typeof conflict.serverVersion === 'object') {
    const changedLocalFields = getChangedFields(conflict.localVersion, conflict.serverVersion)
    const changedServerFields = getChangedFields(conflict.serverVersion, conflict.localVersion)
    const overlap = changedLocalFields.filter(f => changedServerFields.includes(f))
    
    if (overlap.length === 0) {
      // No overlapping changes — safe to merge
      return { ...conflict.serverVersion, ...pick(conflict.localVersion, changedLocalFields) }
    }
  }
  
  // Cannot auto-resolve
  return null
}
```

## 4. Conflict Resolution UI

```vue
<!-- components/OfflineConflictBanner.vue -->
<script setup lang="ts">
const { getConflicts, resolveConflict } = useSyncQueue()
const conflicts = ref<SyncQueueItem[]>([])

onMounted(async () => {
  conflicts.value = await getConflicts()
})

const keepLocal = async (conflict: SyncQueueItem) => {
  // Force-push local version to server (ignore server version)
  await $fetch(conflict.endpoint, {
    method: conflict.method,
    body: conflict.body,
    headers: { 'X-Force-Write': 'true' }
  })
  await resolveConflict(conflict.id!)
  conflicts.value = conflicts.value.filter(c => c.id !== conflict.id)
}

const keepServer = async (conflict: SyncQueueItem) => {
  // Discard local changes — pull server version into local DB
  const serverData = await $fetch(conflict.endpoint)
  await db.table(conflict.entityType).put(serverData)
  await resolveConflict(conflict.id!)
  conflicts.value = conflicts.value.filter(c => c.id !== conflict.id)
}
</script>

<template>
  <div v-if="conflicts.length > 0" class="conflict-banner" role="alert">
    <h3>⚠️ {{ conflicts.length }} sync conflict{{ conflicts.length > 1 ? 's' : '' }}</h3>
    <p>Changes you made offline conflict with server updates.</p>
    
    <div v-for="conflict in conflicts" :key="conflict.id" class="conflict-item">
      <span>{{ conflict.entityType }}: {{ conflict.entityId }}</span>
      <button @click="keepLocal(conflict)">Keep my changes</button>
      <button @click="keepServer(conflict)">Keep server version</button>
    </div>
  </div>
</template>
```

## 5. Prevention: Field-Level Locking

The best conflict resolution is prevention. Use last-edited-field tracking to reduce the scope of conflicts.

```typescript
// Instead of versioning the whole document, version each field independently
interface Task {
  id: string
  // Each field carries its own version
  title: string
  titleVersion: number
  status: string
  statusVersion: number
}

// Merge by taking the latest version of each field
const fieldMerge = (local: Task, server: Task): Task => ({
  ...local,
  title: local.titleVersion >= server.titleVersion ? local.title : server.title,
  titleVersion: Math.max(local.titleVersion, server.titleVersion),
  status: local.statusVersion >= server.statusVersion ? local.status : server.status,
  statusVersion: Math.max(local.statusVersion, server.statusVersion),
})
```
