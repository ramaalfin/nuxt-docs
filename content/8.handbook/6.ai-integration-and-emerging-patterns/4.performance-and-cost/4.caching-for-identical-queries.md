# Caching for Identical Queries

Caching LLM responses is one of the easiest ways to cut costs and reduce latency to zero for repeated queries. If users frequently ask the same questions or trigger the same automated analysis, there is no reason to pay an LLM provider to compute the exact same answer twice.

## 1. The Semantic Caching Strategy

Caching LLM responses is harder than caching normal API responses because inputs are rarely *exactly* identical. 

- **Exact Match Caching:** Fast, cheap, but low hit rate. ("What is the refund policy?" != "What's your refund policy?")
- **Semantic Caching:** Uses embeddings to cache based on meaning. High hit rate, slightly more complex.

## 2. Exact Match Caching (Redis/KV)

The simplest approach. Hash the input and system prompt, and use it as a Redis key.

```typescript
// server/utils/llm-cache.ts
import { createHash } from 'crypto'
import { useStorage } from '#imports' // Nuxt Nitro storage (Redis, Cloudflare KV, etc.)

export const generateCacheKey = (systemPrompt: string, userPrompt: string) => {
  return createHash('sha256')
    .update(`${systemPrompt}:::${userPrompt.trim().toLowerCase()}`)
    .digest('hex')
}

export const getCachedLLMResponse = async (key: string) => {
  const cache = useStorage('cache:llm')
  return await cache.getItem<string>(key)
}

export const setCachedLLMResponse = async (key: string, response: string, ttlSeconds = 86400) => {
  const cache = useStorage('cache:llm')
  // Store with expiration (e.g., 24 hours)
  await cache.setItem(key, response)
}
```

```typescript
// server/api/chat.post.ts
export default defineEventHandler(async (event) => {
  const { prompt } = await readBody(event)
  const system = 'You are a helpful assistant.'
  
  const cacheKey = generateCacheKey(system, prompt)
  const cachedResponse = await getCachedLLMResponse(cacheKey)
  
  if (cachedResponse) {
    console.log('LLM Cache HIT ⚡')
    // Fast return, $0 cost
    return { text: cachedResponse, cached: true }
  }
  
  console.log('LLM Cache MISS 🐌')
  const { text } = await generateText({ model: openai('gpt-4o-mini'), prompt, system })
  
  // Save in background
  event.waitUntil(setCachedLLMResponse(cacheKey, text))
  
  return { text, cached: false }
})
```

## 3. Semantic Caching (Vector Database)

Semantic caching finds *similar* previous queries. If a new query has a cosine similarity of >0.95 to a cached query, return the cached answer.

```typescript
// Semantic Cache Flow
export const semanticCacheLookup = async (userQuery: string) => {
  // 1. Embed the incoming query
  const queryEmbedding = await embedText(userQuery)
  
  // 2. Search the vector DB for previous identical/similar queries
  const results = await vectorDB.query({
    namespace: 'llm-query-cache',
    vector: queryEmbedding,
    topK: 1
  })
  
  const topMatch = results.matches[0]
  
  // 3. If similarity is very high (e.g., 95% identical meaning), use it
  if (topMatch && topMatch.score > 0.95) {
    return topMatch.metadata.cachedAnswer
  }
  
  return null // Cache miss
}
```

## 4. Cache Invalidation Strategies

LLM answers can become stale if your underlying data changes.

### 1. Time-to-Live (TTL)
The simplest approach: automatically expire cached answers after 24 hours.

### 2. RAG Dependency Tagging
If an LLM answer was generated using RAG, tag the cache entry with the IDs of the documents used. If those documents are updated, invalidate the cache.

```typescript
// Store this when caching the response
const cacheMetadata = {
  response: generatedText,
  sourceDocumentIds: retrievedDocs.map(d => d.id), // e.g., ['doc-123', 'doc-456']
  timestamp: Date.now()
}

// When a document is updated:
const onDocumentUpdated = async (docId: string) => {
  // Find and delete all cache entries that relied on this document
  await cacheDB.deleteWhere({ sourceDocumentIds: { contains: docId } })
}
```

## 5. Streaming Cached Responses

When serving a cached response, simulate the streaming experience so the UX remains consistent (otherwise cached answers feel instantly jarring, while uncached answers stream slowly).

```typescript
// server/api/chat.post.ts
if (cachedResponse) {
  // Convert string to a stream that yields chunks over time
  const stream = new ReadableStream({
    async start(controller) {
      const words = cachedResponse.split(' ')
      for (const word of words) {
        controller.enqueue(new TextEncoder().encode(`data: {"text":"${word} "}\n\n`))
        // Simulate streaming delay (~10ms per word)
        await new Promise(r => setTimeout(r, 10))
      }
      controller.enqueue(new TextEncoder().encode('data: [DONE]\n\n'))
      controller.close()
    }
  })
  
  return new Response(stream, { headers: { 'Content-Type': 'text/event-stream' } })
}
```

## 6. Pre-computation (Warm Caching)

For predictable queries (e.g., "Weekly Summary" generated every Monday), don't wait for a user to request it. Run a cron job to generate the response via the cheap Batch API, and store it in the cache for instant retrieval when the user logs in.
