# Edge Inference (For Fast Models)

Edge inference moves the execution of AI models (or the API calls to them) away from centralized servers and out to the "edge" of the network, as close to the user as possible. For latency-sensitive AI features like auto-complete, suggestions, and real-time chat, edge execution shaves 50–200ms off the Time-to-First-Token (TTFT).

## 1. Why Edge?

**Traditional Node.js Server (e.g., US-East):**
User in Tokyo → Request travels to US-East (150ms) → Server calls OpenAI (US-East) (50ms) → Stream starts traveling back to Tokyo (150ms) = **350ms TTFT.**

**Edge Function (Cloudflare/Vercel):**
User in Tokyo → Request hits Tokyo Edge Node (10ms) → Edge Node calls OpenAI API (closest region) (50ms) → Stream starts traveling back (10ms) = **70ms TTFT.**

## 2. Deploying Nuxt API Routes to the Edge

Nuxt's Nitro engine supports deploying API routes to edge runtimes (Cloudflare Workers, Vercel Edge, Deno) out of the box.

```typescript
// server/api/chat.post.ts
// Nitro will compile this specific route for the Edge runtime
export const config = {
  runtime: 'edge'
}

import { openai } from '@ai-sdk/openai'
import { streamText } from 'ai'

export default defineEventHandler(async (event) => {
  const { messages } = await readBody(event)

  // This API call originates from the Edge node closest to the user
  const result = await streamText({
    model: openai('gpt-4o-mini'),
    messages
  })

  return result.toDataStreamResponse()
})
```

## 3. WebAssembly (Wasm) Local Inference

True "edge" inference means running the model directly inside the user's browser using WebAssembly and WebGPU. This results in **zero server latency**, **zero API costs**, and **total privacy**.

The `@xenova/transformers` library brings Hugging Face models to the browser.

```bash
pnpm add @xenova/transformers
```

```typescript
// composables/useLocalAI.ts
import { pipeline } from '@xenova/transformers'

export function useLocalAI() {
  const isReady = ref(false)
  const isGenerating = ref(false)
  let generator: any = null

  const initModel = async () => {
    // Downloads the model weights to browser cache on first load (e.g., 50MB-200MB)
    // Uses WebGPU if available, falls back to Wasm CPU
    generator = await pipeline('text-generation', 'Xenova/TinyLlama-1.1B-Chat-v1.0')
    isReady.value = true
  }

  const generate = async (prompt: string) => {
    if (!generator) await initModel()
    isGenerating.value = true
    
    try {
      // Runs entirely on the user's device! Zero cloud cost.
      const result = await generator(prompt, { max_new_tokens: 100 })
      return result[0].generated_text
    } finally {
      isGenerating.value = false
    }
  }

  return { isReady, isGenerating, initModel, generate }
}
```

### When to Use Browser Inference
- **Text Classification / Sentiment:** Small models (10-50MB) run instantly in browser.
- **Embeddings:** Generate search embeddings locally to save API costs.
- **Offline Mode:** The only way to provide AI features when offline.
- **Privacy-Critical:** When user data cannot legally leave the device (HIPAA, etc).

*Note: Large language models (like LLaMA 8B) require too much RAM to reliably run in web browsers today. Stick to small, specialized models (<1B parameters).*

## 4. Cloudflare Workers AI

Cloudflare Workers AI allows you to run open-weight models (like LLaMA 3) directly on Cloudflare's global edge network GPUs.

```typescript
// server/api/edge-chat.post.ts
// Assuming deployment to Cloudflare Pages/Workers
export const config = { runtime: 'edge' }

export default defineEventHandler(async (event) => {
  const { prompt } = await readBody(event)
  
  // Access the Cloudflare AI binding (provided by Nitro/Cloudflare environment)
  const ai = event.context.cloudflare.env.AI
  
  // Run LLaMA 3 directly on Cloudflare's edge GPU
  const response = await ai.run('@cf/meta/llama-3-8b-instruct', {
    prompt,
    stream: true
  })
  
  // Return standard SSE stream
  return new Response(response, {
    headers: { 'content-type': 'text/event-stream' }
  })
})
```

## 5. Architectural Trade-offs

| Approach | Latency | Cloud Cost | Privacy | Model Quality |
|---|---|---|---|---|
| Node.js Server → OpenAI | High (200-500ms) | $$ | Low | Highest (GPT-4o) |
| Edge Function → OpenAI | Medium (100-200ms) | $$ | Low | Highest (GPT-4o) |
| Cloudflare Workers AI | Fast (50-100ms) | $ | Medium | High (LLaMA 3 8B) |
| Browser Wasm (Local) | Instant (0ms) | Free | Highest | Low (Tiny models) |

**Recommendation:** 
- Use **Edge Functions → OpenAI/Anthropic** for 90% of Nuxt applications.
- Use **Browser Wasm** for generating vector embeddings or simple classifications to cut costs.
