# Index Optimization

A vector index is the data structure that enables fast approximate nearest-neighbor (ANN) search. Without an index, every vector search requires comparing the query against every stored vector — O(n) complexity that becomes impossibly slow at scale. Index optimization dramatically improves search performance.

## 1. ANN Index Algorithms

| Algorithm | Type | Query Speed | Index Build | Memory | Best For |
|---|---|---|---|---|---|
| **IVFFlat** | Cluster-based | ⚡ Fast | Medium | Medium | pgvector, general use |
| **HNSW** | Graph-based | ⚡⚡ Very Fast | Slow | High | Qdrant, Weaviate |
| **IVF-PQ** | Quantized | ⚡ Fast | Slow | Low | Billion-scale |
| **Flat (brute force)** | None | 🐌 Slow | Instant | Low | < 10k vectors |

## 2. pgvector Index Configuration

```sql
-- IVFFlat Index (good balance of speed and accuracy)
-- Rule: lists ≈ sqrt(number_of_rows)
CREATE INDEX ON documents 
USING ivfflat (embedding vector_cosine_ops) 
WITH (lists = 100);  -- For ~10,000 vectors: sqrt(10000) = 100

-- At query time, set probes (accuracy vs speed tradeoff)
-- More probes = more accurate but slower
SET ivfflat.probes = 10;  -- Check 10 clusters (10% of lists)

-- HNSW Index (faster queries, more memory)
-- Better for production if you have the RAM
CREATE INDEX ON documents 
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- At query time
SET hnsw.ef_search = 40;  -- Higher = more accurate but slower
```

### Choosing Between IVFFlat and HNSW for pgvector

```
Dataset < 100k rows → HNSW (fastest queries, memory not a concern)
Dataset 100k-1M rows → HNSW with higher m value or IVFFlat
Dataset > 1M rows → IVFFlat (more memory efficient)
Frequent updates → IVFFlat (HNSW updates are expensive)
```

## 3. Index Rebuilding After Bulk Inserts

After bulk loading data, rebuild indexes for optimal performance:

```sql
-- After bulk inserting many vectors, rebuild the index
REINDEX INDEX CONCURRENTLY documents_embedding_idx;

-- Or with pgvector's recommended approach: drop and recreate
DROP INDEX CONCURRENTLY documents_embedding_idx;
CREATE INDEX CONCURRENTLY ON documents
  USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

## 4. Pinecone Index Configuration

```typescript
// Pinecone: Pod-based vs Serverless
// Serverless: Pay per query, scales automatically (recommended for most)
await pc.createIndex({
  name: 'my-docs',
  dimension: 1536,
  metric: 'cosine',
  spec: {
    serverless: {
      cloud: 'aws',
      region: 'us-east-1'
    }
  }
})

// Pod-based: Fixed cost, predictable performance (for high-volume)
await pc.createIndex({
  name: 'my-docs-high-volume',
  dimension: 1536,
  metric: 'cosine',
  spec: {
    pod: {
      environment: 'us-east1-gcp',
      podType: 'p1.x1',  // p1 = performance, p2 = storage efficiency
      pods: 1
    }
  }
})
```

## 5. Qdrant HNSW Configuration

```typescript
await client.createCollection('documents', {
  vectors: {
    size: 1536,
    distance: 'Cosine',
    hnsw_config: {
      m: 16,              // Number of edges per node (higher = more accurate, more memory)
      ef_construct: 100,  // Beam size during index construction (higher = better, slower build)
      on_disk: false      // Set true for datasets > RAM
    }
  },
  optimizers_config: {
    memmap_threshold: 100_000  // Use memory-mapped files above this vector count
  }
})
```

## 6. Dimensionality Reduction

Reduce embedding dimensions for faster indexing and lower storage cost, with minor quality tradeoff.

```typescript
// OpenAI allows dimension reduction in the embedding call
const { embedding } = await embed({
  model: openai.embedding('text-embedding-3-large'),
  value: text,
  // Reduce from 3072 → 512 dimensions (Matryoshka representation)
  // 83% storage reduction with ~5% quality loss
  experimental_dimensions: 512
})
```

## 7. Performance Benchmarking

Always benchmark before and after index changes:

```typescript
// Measure query performance
const measureSearch = async (queries: string[], iterations = 100) => {
  const times: number[] = []
  
  for (let i = 0; i < iterations; i++) {
    const query = queries[i % queries.length]
    const start = performance.now()
    await semanticSearch(query)
    times.push(performance.now() - start)
  }
  
  times.sort((a, b) => a - b)
  return {
    p50: times[Math.floor(iterations * 0.5)],
    p95: times[Math.floor(iterations * 0.95)],
    p99: times[Math.floor(iterations * 0.99)],
    avg: times.reduce((a, b) => a + b) / times.length
  }
}
```
