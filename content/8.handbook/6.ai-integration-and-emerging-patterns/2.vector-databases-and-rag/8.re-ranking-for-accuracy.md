# Re-ranking for Accuracy

Vector search retrieves candidates by approximate semantic similarity, but this first-pass retrieval is inherently imprecise. **Re-ranking** applies a more powerful (but slower) scoring model to the retrieved candidates to sort them by true relevance before injecting them into the LLM context. This two-stage approach significantly improves RAG answer quality.

## 1. The Two-Stage Retrieval Pipeline

```
Stage 1: Fast Retrieval (Recall-oriented)
  Query → Vector search → Top-50 candidates (fast, approximate)

Stage 2: Re-ranking (Precision-oriented)
  Top-50 candidates → Cross-encoder model → Top-5 most relevant (slower, precise)

Stage 3: Generation
  Top-5 re-ranked documents → LLM context → Answer
```

Retrieval (bi-encoder) is fast but less accurate. Re-ranking (cross-encoder) is slower but far more accurate. The combination captures the best of both.

## 2. Cohere Re-ranking API

The easiest way to add re-ranking — Cohere provides a dedicated re-ranking API.

```typescript
// server/utils/reranker.ts
import { CohereClient } from 'cohere-ai'

const cohere = new CohereClient({ token: process.env.COHERE_API_KEY! })

export const rerankDocuments = async (
  query: string,
  documents: Array<{ id: string; content: string }>,
  topN = 5
) => {
  const response = await cohere.rerank({
    model: 'rerank-english-v3.0',
    query,
    documents: documents.map(d => d.content),
    topN
  })

  // Map re-ranked results back to original documents
  return response.results.map(result => ({
    ...documents[result.index],
    relevanceScore: result.relevanceScore
  }))
}
```

```typescript
// server/api/rag-chat.post.ts — Integrated re-ranking
export default defineEventHandler(async (event) => {
  const { question } = await readBody(event)

  // Stage 1: Broad retrieval (fetch 20 candidates)
  const candidates = await semanticSearch(question, { topK: 20 })

  // Stage 2: Re-rank to top 5
  const reranked = await rerankDocuments(question, candidates, 5)

  // Stage 3: Build context from re-ranked results
  const context = reranked
    .map((doc, i) => `[Source ${i + 1}] (relevance: ${doc.relevanceScore.toFixed(2)})\n${doc.content}`)
    .join('\n\n')

  // Stage 4: Generate with high-quality context
  const result = await streamText({
    model: getModel(),
    system: `Answer based only on the provided context:\n\n${context}`,
    messages: [{ role: 'user', content: question }]
  })

  return result.toDataStreamResponse()
})
```

## 3. LLM-as-Reranker (No External API)

Use your LLM to score relevance — more flexible but slower and more expensive.

```typescript
import { generateObject } from 'ai'
import { z } from 'zod'

const RerankSchema = z.object({
  rankedIds: z.array(z.string()).describe('Document IDs sorted by relevance, most relevant first'),
  reasoning: z.string()
})

export const llmRerank = async (query: string, documents: Array<{ id: string; content: string }>) => {
  const { object } = await generateObject({
    model: openai('gpt-4o-mini'), // Fast model for re-ranking
    schema: RerankSchema,
    prompt: `
Query: "${query}"

Score these documents by how relevant they are to the query. 
Return their IDs sorted from most to least relevant.

Documents:
${documents.map(d => `ID: ${d.id}\nContent: ${d.content.slice(0, 300)}...`).join('\n\n')}
`
  })

  return object.rankedIds
    .map(id => documents.find(d => d.id === id))
    .filter(Boolean)
}
```

## 4. Local Re-ranking (Self-Hosted, Free)

Use a local cross-encoder model via Hugging Face Transformers for zero API cost.

```bash
# Install transformers.js for server-side model inference
pnpm add @xenova/transformers
```

```typescript
// server/utils/local-reranker.ts
import { pipeline } from '@xenova/transformers'

let reranker: any = null

const getReranker = async () => {
  if (!reranker) {
    reranker = await pipeline(
      'text-classification',
      'cross-encoder/ms-marco-MiniLM-L-6-v2' // Lightweight cross-encoder
    )
  }
  return reranker
}

export const localRerank = async (
  query: string,
  documents: Array<{ id: string; content: string }>,
  topN = 5
) => {
  const model = await getReranker()
  const pairs = documents.map(d => [query, d.content])
  const scores = await model(pairs)

  return documents
    .map((doc, i) => ({ ...doc, score: scores[i].score }))
    .sort((a, b) => b.score - a.score)
    .slice(0, topN)
}
```

## 5. Re-ranking ROI

Re-ranking adds ~100-300ms latency but delivers significant quality improvements:

| Metric | Without Re-rank | With Re-rank | Improvement |
|---|---|---|---|
| Answer faithfulness | 72% | 89% | +24% |
| Relevant context in top-5 | 61% | 84% | +38% |
| User satisfaction | 3.4/5 | 4.2/5 | +24% |

The latency cost is almost always worth the quality improvement for user-facing RAG applications.
