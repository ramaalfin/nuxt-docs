# Rate Limiting & Retry Logic

LLM APIs enforce strict rate limits on requests per minute (RPM) and tokens per minute (TPM). Exceeding them results in `429 Too Many Requests` errors. A production-grade integration must handle rate limits gracefully with intelligent retry logic, request queuing, and exponential backoff.

## 1. Understanding Rate Limits

| Provider | Default Limits (Tier 1) |
|---|---|
| **OpenAI GPT-4o** | 500 RPM / 30,000 TPM |
| **OpenAI GPT-4o-mini** | 2,000 RPM / 200,000 TPM |
| **Anthropic Claude 3.5 Sonnet** | 1,000 RPM / 80,000 TPM |

Rate limits increase as you spend more money with the provider. For high-volume applications, request limit increases proactively.

## 2. Retry with Exponential Backoff

When you receive a `429`, wait before retrying. Use exponential backoff with jitter to spread retries across time.

```typescript
// server/utils/llm-retry.ts
import { generateText, streamText } from 'ai'

interface RetryOptions {
  maxRetries?: number
  baseDelay?: number
  maxDelay?: number
}

export async function withRetry<T>(
  fn: () => Promise<T>,
  options: RetryOptions = {}
): Promise<T> {
  const { maxRetries = 3, baseDelay = 1000, maxDelay = 30_000 } = options
  let lastError: Error

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn()
    } catch (error: any) {
      lastError = error

      // Don't retry on non-retryable errors
      const isRateLimit = error.status === 429
      const isServerError = error.status >= 500
      if (!isRateLimit && !isServerError) throw error

      if (attempt === maxRetries) break

      // Exponential backoff with jitter
      const exponential = baseDelay * Math.pow(2, attempt)
      const jitter = Math.random() * 1000
      const delay = Math.min(exponential + jitter, maxDelay)

      // Respect Retry-After header if present
      const retryAfter = error.headers?.get?.('retry-after')
      const waitMs = retryAfter ? parseInt(retryAfter) * 1000 : delay

      console.log(`[LLM] Rate limited. Retrying in ${Math.round(waitMs)}ms (attempt ${attempt + 1}/${maxRetries})`)
      await new Promise(resolve => setTimeout(resolve, waitMs))
    }
  }

  throw lastError!
}
```

```typescript
// server/api/chat.post.ts
export default defineEventHandler(async (event) => {
  const { messages } = await readBody(event)

  const result = await withRetry(() =>
    streamText({ model: getModel(), messages, maxTokens: 1000 })
  , { maxRetries: 3 })

  return result.toDataStreamResponse()
})
```

## 3. Server-Side Request Queue

When concurrent requests exceed your rate limit, queue them server-side rather than failing.

```typescript
// server/utils/llm-queue.ts
import PQueue from 'p-queue'

// Limit concurrent LLM requests to respect RPM limits
// GPT-4o: 500 RPM ≈ ~8 per second — allow 5 concurrent with some buffer
const llmQueue = new PQueue({
  concurrency: 5,
  interval: 1000,      // Per second
  intervalCap: 8,      // Max 8 requests per second
})

export const queuedLLMCall = <T>(fn: () => Promise<T>): Promise<T> => {
  return llmQueue.add(fn) as Promise<T>
}

// Usage:
export default defineEventHandler(async (event) => {
  const { messages } = await readBody(event)

  const result = await queuedLLMCall(() =>
    withRetry(() => streamText({ model: getModel(), messages }))
  )

  return result.toDataStreamResponse()
})
```

## 4. Application-Level Rate Limiting (Per User)

Protect your API from individual users consuming all your rate limit budget.

```typescript
// server/middleware/user-rate-limit.ts
import { LRUCache } from 'lru-cache'

const userRequestCounts = new LRUCache<string, number[]>({ max: 10_000 })

const MAX_REQUESTS_PER_MINUTE = 10 // Per user

export default defineEventHandler(async (event) => {
  if (!event.path.startsWith('/api/ai/')) return

  const userId = event.context.userId ?? getRequestIP(event) ?? 'anonymous'
  const now = Date.now()
  const windowStart = now - 60_000 // Last 60 seconds

  // Get user's request timestamps, filter to current window
  const timestamps = (userRequestCounts.get(userId) ?? [])
    .filter(ts => ts > windowStart)

  if (timestamps.length >= MAX_REQUESTS_PER_MINUTE) {
    const oldestTimestamp = timestamps[0]
    const retryAfter = Math.ceil((oldestTimestamp + 60_000 - now) / 1000)

    throw createError({
      statusCode: 429,
      statusMessage: 'Too many requests',
      data: { retryAfter, message: `Rate limit: ${MAX_REQUESTS_PER_MINUTE} requests per minute` }
    })
  }

  // Record this request
  timestamps.push(now)
  userRequestCounts.set(userId, timestamps)
})
```

## 5. Circuit Breaker Pattern

If an LLM provider is down, fail fast instead of queuing requests that will all timeout.

```typescript
// server/utils/circuit-breaker.ts
type State = 'CLOSED' | 'OPEN' | 'HALF_OPEN'

class CircuitBreaker {
  private state: State = 'CLOSED'
  private failures = 0
  private lastFailureTime = 0

  constructor(
    private readonly threshold = 5,    // Open after 5 consecutive failures
    private readonly resetTimeout = 30_000  // Try again after 30 seconds
  ) {}

  async call<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state === 'OPEN') {
      if (Date.now() - this.lastFailureTime > this.resetTimeout) {
        this.state = 'HALF_OPEN'
      } else {
        throw new Error('LLM service temporarily unavailable. Please try again shortly.')
      }
    }

    try {
      const result = await fn()
      this.onSuccess()
      return result
    } catch (error) {
      this.onFailure()
      throw error
    }
  }

  private onSuccess() { this.failures = 0; this.state = 'CLOSED' }
  private onFailure() {
    this.failures++
    this.lastFailureTime = Date.now()
    if (this.failures >= this.threshold) this.state = 'OPEN'
  }
}

export const llmCircuitBreaker = new CircuitBreaker()
```
