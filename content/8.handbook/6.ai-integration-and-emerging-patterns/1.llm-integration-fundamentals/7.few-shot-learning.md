# Few-Shot Learning

Few-shot learning is the technique of providing the LLM with examples of the desired input-output pattern directly in the prompt. Instead of describing what you want, you *show* the model what you want with 2–5 examples. This dramatically improves consistency, format adherence, and output quality for specialized tasks.

## 1. Zero-Shot vs Few-Shot vs Fine-Tuning

| Approach | How | Best For | Cost |
|---|---|---|---|
| **Zero-shot** | Instructions only, no examples | General tasks, simple requests | Cheapest |
| **Few-shot** | 2–5 examples in the prompt | Specific formats, domain tasks | Moderate (extra input tokens) |
| **Fine-tuning** | Training the model weights | High-volume consistent tasks | Most expensive (training cost) |

**Start with few-shot before considering fine-tuning.** Few-shot can often achieve 80–90% of fine-tuning quality at zero infrastructure cost.

## 2. Basic Few-Shot Pattern

```typescript
// server/api/classify-sentiment.post.ts
const FEW_SHOT_EXAMPLES = `
Examples:

User: "This product is absolutely fantastic! Best purchase I've made."
Response: {"sentiment": "positive", "confidence": 0.97}

User: "Arrived on time, packaging was fine. Does what it says."
Response: {"sentiment": "neutral", "confidence": 0.82}

User: "Completely broken out of the box. Waste of money. Avoid."
Response: {"sentiment": "negative", "confidence": 0.95}

User: "It's okay I guess, nothing special but no major issues either."
Response: {"sentiment": "neutral", "confidence": 0.71}
`

export default defineEventHandler(async (event) => {
  const { review } = await readBody(event)

  const { text } = await generateText({
    model: openai('gpt-4o-mini'),
    messages: [
      { role: 'system', content: `You classify product reviews into sentiment categories. Always respond with valid JSON only.\n\n${FEW_SHOT_EXAMPLES}` },
      { role: 'user', content: review }
    ],
    maxTokens: 100
  })

  return JSON.parse(text)
})
```

## 3. Few-Shot for Complex Format Adherence

When you need a very specific output structure, examples are more reliable than instructions alone.

```typescript
// Extract structured data from unstructured text
const EXTRACTION_EXAMPLES = `
Examples:

Input: "The meeting is scheduled for next Tuesday at 3pm with John Smith from Engineering to discuss the Q4 roadmap."
Output: {
  "date": "next Tuesday",
  "time": "15:00",
  "attendees": ["John Smith"],
  "department": "Engineering",
  "topic": "Q4 roadmap",
  "duration": null
}

Input: "Quick 30-min sync tomorrow morning at 9 with the entire design team re: new dashboard mockups."
Output: {
  "date": "tomorrow",
  "time": "09:00",
  "attendees": [],
  "department": "Design",
  "topic": "new dashboard mockups",
  "duration": 30
}
`

export const extractMeetingDetails = async (text: string) => {
  const { object } = await generateObject({
    model: openai('gpt-4o-mini'),
    schema: MeetingSchema,
    messages: [
      { role: 'system', content: `Extract meeting details from natural language text.\n\n${EXTRACTION_EXAMPLES}` },
      { role: 'user', content: text }
    ]
  })
  return object
}
```

## 4. Dynamic Few-Shot Selection

Instead of hardcoding examples, retrieve the most relevant examples from a database based on the current input (semantic few-shot selection).

```typescript
// Select examples most similar to the current input using embeddings
export const getDynamicExamples = async (input: string, k = 3): Promise<Example[]> => {
  const inputEmbedding = await embedText(input)
  
  // Query your vector DB for the most similar examples
  const similar = await vectorDB.query({
    vector: inputEmbedding,
    topK: k,
    namespace: 'few-shot-examples'
  })
  
  return similar.matches.map(m => m.metadata as Example)
}

// Use in the prompt
const examples = await getDynamicExamples(userInput)
const fewShotBlock = examples
  .map(ex => `Input: ${ex.input}\nOutput: ${ex.output}`)
  .join('\n\n')
```

## 5. Few-Shot Best Practices

### Example Quality Over Quantity
- 3 excellent, diverse examples > 10 mediocre ones.
- Cover edge cases in your examples (not just happy paths).
- Examples should represent the actual distribution of real inputs.

### Example Ordering
- Put the most representative example last (closest to the actual request).
- Include both easy and challenging examples.

### Label Diversity
For classification tasks, ensure your examples cover all possible output classes to prevent bias.

```typescript
// ❌ BAD: Only positive examples biases the model
const examples = [
  { input: 'Great product!', output: 'positive' },
  { input: 'Love it!', output: 'positive' },
  { input: 'Excellent quality', output: 'positive' }
]

// ✅ GOOD: Balanced examples
const examples = [
  { input: 'Great product!', output: 'positive' },
  { input: 'It is okay', output: 'neutral' },
  { input: 'Terrible experience', output: 'negative' }
]
```
