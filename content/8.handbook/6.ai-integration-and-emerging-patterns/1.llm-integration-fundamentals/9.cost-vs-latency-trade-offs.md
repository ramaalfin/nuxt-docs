# Cost vs Latency Trade-offs

Every LLM integration requires balancing three competing forces: **quality**, **cost**, and **latency**. The right balance depends on your specific feature — a real-time autocomplete has different requirements than a nightly report generation job.

## 1. The Trade-off Triangle

```
        QUALITY
          ▲
         /|\
        / | \
       /  |  \
      /   |   \
COST ◄────────► LATENCY
```

You can optimize for at most two of these simultaneously:
- **High quality + Low cost** → High latency (use batching/async processing)
- **High quality + Low latency** → High cost (use best models, minimal optimizations)
- **Low cost + Low latency** → Lower quality (use smaller/faster models)

## 2. Model Selection by Use Case

```typescript
// server/utils/model-selector.ts
type UseCase =
  | 'realtime-autocomplete'   // Speed critical
  | 'chat-assistant'          // Balanced
  | 'document-analysis'       // Quality critical
  | 'classification'          // Cost critical
  | 'code-generation'         // Quality + speed balanced
  | 'batch-processing'        // Cost critical, latency irrelevant

export const selectModel = (useCase: UseCase) => {
  switch (useCase) {
    // Speed-critical: ~200ms response
    case 'realtime-autocomplete':
      return openai('gpt-4o-mini')     // Fast + cheap: ~200ms p50

    // Balanced: ~500-800ms first token
    case 'chat-assistant':
    case 'code-generation':
      return openai('gpt-4o')          // Best quality/speed balance

    // Quality-critical (latency acceptable): ~1-3s
    case 'document-analysis':
      return anthropic('claude-3-5-sonnet-20241022')  // Best instruction-following

    // Cost-critical: Cheapest possible
    case 'classification':
      return openai('gpt-4o-mini')     // 100x cheaper than GPT-4o

    // Batch: Can wait, use best quality
    case 'batch-processing':
      return openai('gpt-4o', {
        // Use Batch API for 50% cost discount on async processing
      })
  }
}
```

## 3. OpenAI Batch API (50% Cost Discount)

For non-real-time tasks (nightly reports, bulk classification, content generation), the OpenAI Batch API processes requests asynchronously with a 24-hour window at half the cost.

```typescript
// server/api/batch-classify.post.ts
import OpenAI from 'openai'

const openai = new OpenAI()

export const submitBatchJob = async (items: Array<{ id: string; text: string }>) => {
  // Create a JSONL batch file
  const requests = items.map(item => ({
    custom_id: item.id,
    method: 'POST',
    url: '/v1/chat/completions',
    body: {
      model: 'gpt-4o',
      messages: [
        { role: 'system', content: 'Classify the sentiment.' },
        { role: 'user', content: item.text }
      ],
      max_tokens: 50
    }
  }))

  // Upload batch file
  const file = await openai.files.create({
    file: new Blob([requests.map(r => JSON.stringify(r)).join('\n')]),
    purpose: 'batch'
  })

  // Submit batch job
  const batch = await openai.batches.create({
    input_file_id: file.id,
    endpoint: '/v1/chat/completions',
    completion_window: '24h' // Results within 24 hours, 50% cheaper
  })

  return batch.id // Poll later for results
}
```

## 4. Time-to-First-Token (TTFT) Optimization

For streaming chat, **Time to First Token** matters more than total response time. Users feel the app is fast if the first word appears quickly.

```typescript
// Optimize for TTFT: shorter prompts → faster first token
const optimizeForTTFT = (systemPrompt: string) => ({
  // Compress system prompt to reduce processing time
  system: systemPrompt.replace(/\s+/g, ' ').trim(),
  
  // Use streaming (user sees output as it generates)
  streamText,
  
  // Lower max_tokens bounds the response and speeds up initial generation
  maxTokens: 500 // For chat, 500 tokens ≈ 375 words — usually sufficient
})
```

## 5. Cost Optimization Patterns Summary

| Pattern | Cost Saving | Quality Impact | Latency Impact |
|---|---|---|---|
| Use `gpt-4o-mini` instead of `gpt-4o` | -97% | Medium (-) | Better (+) |
| Semantic caching | -30% to -70% | None | Better (+) |
| Trim conversation history | -20% to -40% | Slight (-) | Better (+) |
| Batch API (async tasks) | -50% | None | +24h delay |
| Compress system prompts | -5% to -15% | Slight (-) | Better (+) |
| Output length control | -10% to -30% | Slight (-) | Better (+) |

## 6. Decision Framework

```
Is this user-facing and real-time?
  ├── YES → Is quality critical?
  │         ├── YES → GPT-4o / Claude 3.5 Sonnet ($)
  │         └── NO  → GPT-4o-mini ($$)
  └── NO  → Is it high volume?
             ├── YES → Batch API + GPT-4o (50% discount)
             └── NO  → Async job with GPT-4o-mini
```
